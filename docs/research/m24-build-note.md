## Builder / platform appendix

### Spike decisions needed before content (the "M24 MCP packaging" gate)
1. **Neutral client.** (A, recommended) **house Quarkus LangChain4j CLI** (`apps/mcp-agent-cli`) — controlled, deterministic, reuses M23; (B) **mcphost** pinned (frozen upstream) or its successor **Kit** (needs its own verification); (C) a che-code MCP-consuming extension (licensing/openness risk). 2. **MCP server + transport.** (A, recommended) **in-cluster Deployment of upstream `containers/kubernetes-mcp-server` over SSE/HTTP** with an in-cluster `mcp-agent` SA (cleanest "no kubeconfig" story, GitOps-friendly); (B) stdio-in-workspace + `oc create token` bound-token kubeconfig (the RH-article shape, simplest single-user). Product path = `openshift-mcp-server` TP + MCP Lifecycle Operator, **documented not defaulted**.
3. **Pre-stage vs attendee-deploy the read-only SA.** Recommend the entry state ships the SA + read-only `view` binding + MCP manifests, and the **attendee performs the deploy + connect** (the lab's first beat); the **scoped-write grant is always an attendee action** (the lesson).

### Least-privilege SA + RBAC shape (the spine — all namespaced, ties M14)
- **SA:** `mcp-agent` in `{user}-dev` (dedicated; not `default`; manage `automountServiceAccountToken`). The MCP-server pod runs as this SA → its **projected bound token is the only cluster credential in the system**; the CLI agent holds none.
- **Phase 1 (read-only):** `RoleBinding mcp-agent → ClusterRole view` in `{user}-dev` (namespaced) + server `--read-only --toolsets core,config` (+ optional `denied_resources` on Secrets, product path). Agent can `pods_list`, `events_list`, get the Deployment, read logs — **diagnose** — every write 403 by RBAC even if the flag flipped.
- **Phase 2 (scoped write):** attendee adds `RoleBinding mcp-agent → edit` (or a custom `deployer` Role: `patch/update deployments`, **no secrets**) in `{user}-dev` ONLY, and restarts the server without `--read-only`. Agent can now patch the probe.
- **Forbidden test:** ask the agent to touch `{user}-stage` (or a neighbor ns) → **no binding there → RBAC 403 → the agent reports the denial.** Deterministic; reinforced by the CVE story.

### Seeded broken-deployment design (bad readiness probe — confirmed clean)
- Base `parasol-claims:1.0` (Quarkus, in `parasol-images`) in `{user}-dev`, with `readinessProbe.httpGet.path` set **wrong** (e.g. `/q/health/reddy` or `/ready`) on the correct container port. Result: pod **Running, `0/1` Ready**, Service has **no ready endpoints**, `oc get events` shows a single crisp `Readiness probe failed: HTTP probe failed with statuscode: 404`. **Keep liveness correct/absent** so there is **no restart loop** (determinism). Fix = patch the path to **`/q/health/ready`** — a **write**, exercising Phase 2. Low-temperature-friendly (one unambiguous signal); cleaner than wrong-image (ImagePullBackOff noise) or wrong-port (connection-refused, less legible). Document under an **"Intentional flaws — do not fix"** note on the entry state.

### Lab spine (grounded step outline — dual-path where genuine)
1. **Meet the world / deploy the MCP server (read-only SA)** — `oc apply` the MCP server + SA + `view` binding (Console:: Topology / CLI:: `oc`), confirm it's Ready; the agent has read-only cluster tools and no kubeconfig. *(dual-path)*
2. **Connect the assistant + diagnose** — run the CLI agent in the Dev Spaces terminal; imperative prompt "diagnose why `parasol-claims` is not Ready in `{user}-dev`"; **watch the tool calls** (`pods_list`/`events_list`/`resources_get`/`pods_log`); **verify its claims yourself** with `oc`. *(single-path CLI — POSTing to the agent; Console:: to view pods/events)*
