== A pod is `Pending` after I added a `nodeSelector` for a pool

Symptom:: You set a `nodeSelector` to target a dedicated pool and the pod won't schedule — it sits `Pending`.
Cause:: The pool node is *tainted*, and a `nodeSelector` only *attracts* — it doesn't grant *permission* to land on a tainted node. The selector sends the pod at the pool node, the taint repels it, and no other node matches the selector.
Fix:: Add the matching *toleration* so the pod is permitted on the tainted node (keep the selector — you need both):
+
[source,sh,role=execute]
----
oc get events -n $NS --field-selector reason=FailedScheduling -o jsonpath='{range .items[*]}{.message}{"\n"}{end}' | tail -1
# "... 1 node(s) had untolerated taint(s), 5 node(s) didn't match Pod's node affinity/selector"
oc patch deploy statement-batch -n $NS --type=merge -p '{"spec":{"template":{"spec":{"tolerations":[{"key":"workshop.redhat.com/pool","operator":"Equal","value":"batch","effect":"NoSchedule"}]}}}}'
oc rollout status deploy/statement-batch -n $NS --timeout=60s
----
+
Selector attracts, toleration permits. A selector alone is exactly this `Pending`.

== The batch pod schedules *somewhere else*, not on the pool

Symptom:: You added a *toleration* for the batch taint, but the pod landed on a general node, not the dedicated pool.
Cause:: The inverse of the previous entry. A toleration only *permits* the tainted node; it doesn't *send* the pod there. Without a `nodeSelector`, the scheduler is free to place the (now taint-tolerant) pod anywhere.
Fix:: Add the `nodeSelector` so the pod is actually *directed* at the pool:
+
[source,sh,role=execute]
----
oc patch deploy statement-batch -n $NS --type=merge -p '{"spec":{"template":{"spec":{"nodeSelector":{"workshop.redhat.com/pool":"batch"}}}}}'
oc get pods -n $NS -l app=statement-batch -o wide     # now on the pool node
----
+
Pinning a workload to a pool always needs the *pair*: toleration (permit) + nodeSelector (attract).

== A replica is `Pending` after I required anti-affinity

Symptom:: With required `podAntiAffinity` on `kubernetes.io/hostname`, one or more replicas stay `Pending` — usually after scaling up.
Cause:: Required anti-affinity means *no two replicas share a node*, so you can never have more running replicas than eligible nodes. If replicas exceed the eligible-node count (remember the tainted pool node is excluded), the extra pods have nowhere legal to go.
Fix:: Read the event to confirm, then scale within the node budget (or relax to `preferred`):
+
[source,sh,role=execute]
----
oc get events -n $NS --field-selector reason=FailedScheduling -o jsonpath='{range .items[*]}{.message}{"\n"}{end}' | tail -1
# "0/6 nodes are available: 1 node(s) had untolerated taint(s), 5 node(s) didn't match pod anti-affinity rules"
oc scale deploy/parasol-claims -n $NS --replicas=3
----
+
"Didn't match pod anti-affinity rules" on every non-tainted node means "they all already hold one of me." Scale to at most the eligible-node count.

== The rolling update is crawling — or seems stuck

Symptom:: A `parasol-claims` roll takes far longer than a normal Deployment — one replica at a time, with long pauses.
Cause:: This is *expected* with required anti-affinity. With `maxSurge: 1`, the surge pod must find a node that has *no* other claims pod, so it waits for an old pod to vacate before each step — the roll goes serial, not parallel.
Fix:: Nothing to fix if it's progressing — watch it advance:
+
[source,sh,role=execute]
----
oc rollout status deploy/parasol-claims -n $NS --timeout=180s
----
+
It genuinely *wedges* only if replicas equal the eligible-node count (the surge pod can't place at all). Keep replicas below the number of schedulable non-pool nodes so there's always a free node to surge onto.

== `oc patch` fails with `spec.containers[0].image: Required value`

Symptom:: Patching a container field (raising a CPU limit, setting an env var) with `--type=merge` errors: `The Deployment "parasol-claims" is invalid: spec.template.spec.containers[0].image: Required value`.
Cause:: A JSON *merge* patch (`--type=merge`) treats `containers` as a whole-list *replace* — your patch, which named only the container and the new field, wiped the image and everything else.
Fix:: Use a *strategic* merge patch, which understands the container `name` as a merge key and edits *into* the existing container (or an `oc set` helper, which does it for you):
+
[source,sh,role=execute]
----
oc patch deploy parasol-claims -n $NS --type=strategic -p '{"spec":{"template":{"spec":{"containers":[{"name":"parasol-claims","resources":{"requests":{"cpu":"200m","memory":"256Mi"},"limits":{"cpu":"2","memory":"512Mi"}}}]}}}}'
# or the helper that sidesteps the footgun entirely:
oc set resources deploy/parasol-claims -n $NS --limits=cpu=2,memory=512Mi --requests=cpu=200m,memory=256Mi
----
+
Rule of thumb: patching a single field inside a *named list element* (a container, a volume) → `--type=strategic`, not `merge`.

== A node drain hangs, or an eviction is refused

Symptom:: `oc adm drain` won't finish, or an eviction returns `Cannot evict pod as it would violate the pod's disruption budget`.
Cause:: A `PodDisruptionBudget` is doing its job — evicting the pod would drop below `minAvailable`. Either the budget is legitimately exhausted (the drain is *waiting*, which is correct), or `minAvailable` was set *equal to* the replica count, which leaves zero room and *deadlocks* the drain forever.
Fix:: Check the budget math; if `ALLOWED DISRUPTIONS` is `0` because `minAvailable == replicas`, lower it:
+
[source,sh,role=execute]
----
oc get pdb parasol-claims -n $NS
oc get pdb parasol-claims -n $NS -o jsonpath='minAvailable={.spec.minAvailable} healthy={.status.currentHealthy} allowed={.status.disruptionsAllowed}{"\n"}'
oc patch pdb parasol-claims -n $NS --type=merge -p '{"spec":{"minAvailable":1}}'
----
+
`minAvailable` must be *strictly less* than the replica count (`1` of `3`) so a disruption can proceed while the service stays up.

== A rollout drops requests even with `maxUnavailable: 0`

Symptom:: You set `maxUnavailable: 0`, capacity holds flat at the desired replica count through the roll, and the API *still* returns 5xx/000 during a rollout — often a burst of them, not a stray one or two.
Cause:: The disruption is in the *data plane, at pod startup*, not the connection plane at shutdown. The `parasol-claims` image boots with Hibernate `schema-management.strategy=drop-and-create`, so *every* pod, on *every* boot, drops and reseeds the *shared* `claims-db`. During a rolling update each new pod's boot wipes the tables out from under the old pods still serving — which is why the shutdown-side knobs (`maxUnavailable`, `preStop`, `terminationGracePeriodSeconds`) can never reach zero. The *deterministic* tell: a value written to the database *reverts to its seed* after a single pod restart.
Fix:: Confirm the signature, then stop pods rebuilding shared state on boot (`schema-management=none`; seed the DB once at deploy from a migration Job instead of from every app pod):
+
[source,sh,role=execute]
----
# signature: write a non-seed status, restart ONE pod, watch it revert (CLM-1001 seeds as UnderReview)
oc exec deploy/claims-load -n $NS -- curl -s -X PUT http://parasol-claims:8080/api/claims/CLM-1001/status \
  -H 'Content-Type: application/json' -d '{"status":"Approved"}' -o /dev/null -w 'wrote %{http_code}\n'
oc delete pod -n $NS "$(oc get pods -n $NS -l app=parasol-claims -o jsonpath='{.items[0].metadata.name}')"
sleep 3; oc wait --for=condition=Ready pod -l app=parasol-claims -n $NS --timeout=180s
oc exec deploy/claims-load -n $NS -- curl -s http://parasol-claims:8080/api/claims/CLM-1001 | grep -o '"status":"[^"]*"'
# reverted to "UnderReview" ⇒ the boot-time reseed is the cause. The primary fix:
oc set env deployment/parasol-claims QUARKUS_HIBERNATE_ORM_SCHEMA_MANAGEMENT_STRATEGY=none -n $NS
----
+
After `schema-management=none` the same write *survives* a restart and a re-run of the drop probe reads `dropped=0`. Graceful `SIGTERM` draining (`quarkus.shutdown.timeout`) is a separate concern this image already handles — it was not what dropped requests here.

== The zone-spread step shows empty zones

Symptom:: The `topologySpreadConstraints` zone-knob step runs, but every pod prints `zone=` (empty).
Cause:: The synthetic `workshop.redhat.com/zone` labels aren't on this cluster's nodes. Those are lab topology, added at bootstrap; the *hostname* spread is the always-true story and needs no node labels.
Fix:: Confirm whether the zone labels exist; if not, use the hostname spread (the real lesson) and treat zones as the "on a cloud, point at the provider's zone" narrative:
+
[source,sh,role=execute]
----
oc get nodes -L workshop.redhat.com/zone
oc patch deploy parasol-claims -n $NS --type=merge -p '{"spec":{"template":{"spec":{"topologySpreadConstraints":[{"maxSkew":1,"topologyKey":"kubernetes.io/hostname","whenUnsatisfiable":"ScheduleAnyway","labelSelector":{"matchLabels":{"app":"parasol-claims"}}}]}}}}'
----
+
On a cloud cluster the same constraint on `topology.kubernetes.io/zone` spreads across *real* failure zones.
