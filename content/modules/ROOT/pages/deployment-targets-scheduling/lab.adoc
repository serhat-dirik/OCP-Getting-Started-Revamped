You are the platform engineer for Parasol's *claims* application.
It runs in `{user}-dev`: `parasol-web` (front end), `parasol-claims` (the API, 2 replicas), `claims-db` (Postgres), a `statement-batch` worker that churns monthly statements, and a `claims-load` generator driving steady traffic at the API.
Right now *nothing* about where these pods run is under your control — the scheduler dropped them wherever they fit.
In this lab you take that control: spread the API across nodes so one failure can't sink it, pin the batch worker onto a *dedicated pool*, make a rollout *zero-downtime*, and protect the service with a disruption budget.

== Demo arc — Dedicated nodes and a zero-downtime roll in ten minutes (10 min)

[NOTE]
.Before you present
====
Run `ws start deployment-targets-scheduling --user {user}` a few minutes ahead, and set `NS={user}-dev` in your terminal.
This demo runs from the *entry* state (default scheduling, batch worker unpinned), *not* from `ws solve`.
The money moment is beat 1 — the batch pod stuck *Pending* on an untolerated taint, then *snapping* onto the batch node the instant you add the toleration.
Keep the terminal font large and `oc get pods -o wide` handy; node placement is the whole visual.
====

[demo-block]
[TIME 3m]
Say:: "Parasol's monthly-statement *batch* job is running on the same nodes as the live claims API — fighting it for CPU. We have a *dedicated* pool of nodes for batch work. Watch me move it there — and watch it refuse, first, for a reason worth understanding."
Show:: `statement-batch` on a general node; the batch pool node (labelled + tainted); then the pod going *Pending* on `nodeSelector` alone, and *scheduling* once the toleration is added.
Do:: Pin the batch worker onto its pool:
+
[source,sh,role=execute]
----
NS=user2-dev
oc get pods -n $NS -l app=statement-batch -o wide
oc get nodes -l workshop.redhat.com/pool=batch
# nodeSelector ALONE — the pod is attracted but not permitted (the taint blocks it)
oc patch deploy statement-batch -n $NS --type=merge -p '{"spec":{"template":{"spec":{"nodeSelector":{"workshop.redhat.com/pool":"batch"}}}}}'
sleep 10
oc get pods -n $NS -l app=statement-batch
oc get events -n $NS --field-selector reason=FailedScheduling | tail -1
----
+
"'Untolerated taint' — the selector *sent* it to the batch node, but the node *repels* it. A selector attracts; it doesn't permit."

[demo-block]
[TIME 2m]
Say:: "One toleration — 'this pod is allowed on the batch node' — and it snaps into place. Selector to *send* it, toleration to *permit* it. You need both."
Show:: The toleration added; the pod now *Running* on the batch pool node.
Do:: Add the toleration and confirm the pin:
+
[source,sh,role=execute]
----
oc patch deploy statement-batch -n $NS --type=merge -p '{"spec":{"template":{"spec":{"tolerations":[{"key":"workshop.redhat.com/pool","operator":"Equal","value":"batch","effect":"NoSchedule"}]}}}}'
oc rollout status deploy/statement-batch -n $NS --timeout=60s
oc get pods -n $NS -l app=statement-batch -o wide
----
+
"On its own nodes now — the API never has to share a CPU with a statement run again."

[demo-block]
[TIME 3m]
Say:: "Now the claims API. Every time a new pod boots it *reseeds the shared database* — silently discarding every claim filed since the last deploy. That's the real reason a naive rollout loses data, and no shutdown tuning fixes it. I'll stop the reseed, hold capacity flat, and redeploy under live traffic — the claim I plant first will still be there afterward."
Show:: The schema strategy flipped to `none`, `maxUnavailable: 0` and a raised CPU limit applied; a value written before the roll still present after it; `availableReplicas` holding flat at 3 through the rollout restart.
Do:: Stop the boot-time reseed, harden the roll, then redeploy under load:
+
[source,sh,role=execute]
----
oc set env deployment/parasol-claims QUARKUS_HIBERNATE_ORM_SCHEMA_MANAGEMENT_STRATEGY=none -n $NS
oc patch deploy parasol-claims -n $NS --type=strategic -p '{"spec":{"strategy":{"type":"RollingUpdate","rollingUpdate":{"maxSurge":1,"maxUnavailable":0}},"template":{"spec":{"containers":[{"name":"parasol-claims","resources":{"requests":{"cpu":"200m","memory":"256Mi"},"limits":{"cpu":"2","memory":"512Mi"}}}]}}}}'
oc rollout status deploy/parasol-claims -n $NS --timeout=180s
# plant a claim state the seed doesn't carry, then redeploy under live traffic
oc exec deploy/claims-load -n $NS -- curl -s -X PUT http://parasol-claims:8080/api/claims/CLM-1001/status -H 'Content-Type: application/json' -d '{"status":"Approved"}' -o /dev/null
oc rollout restart deploy/parasol-claims -n $NS
for i in $(seq 1 10); do oc get deploy parasol-claims -n $NS -o jsonpath='available={.status.availableReplicas}/{.spec.replicas}{"\n"}'; sleep 4; done
oc exec deploy/claims-load -n $NS -- curl -s http://parasol-claims:8080/api/claims/CLM-1001 | grep -o '"status":"[^"]*"'
----
+
"`available=3/3` the whole way through — and `CLM-1001` is *still* `Approved` after the roll, because the pods no longer reseed the database on boot. A mid-day redeploy the customers never felt."

[demo-block]
[TIME 2m]
Say:: "Last piece: protect it during a *drain*. A disruption budget says 'never take me below one healthy pod.' An admin draining nodes has to *wait its turn* instead of evicting my last replica."
Show:: The PDB created; an eviction *allowed*, then — with the budget exhausted — *denied* with the exact message a node drain hits.
Do:: Show the budget in force:
+
[source,sh,role=execute]
----
oc apply -n $NS -f - <<'EOF'
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata: {name: parasol-claims}
spec: {minAvailable: 1, selector: {matchLabels: {app: parasol-claims}}}
EOF
oc get pdb parasol-claims -n $NS
POD=$(oc get pods -n $NS -l app=parasol-claims -o jsonpath='{.items[0].metadata.name}')
oc patch pdb parasol-claims -n $NS --type=merge -p '{"spec":{"minAvailable":3}}'
sleep 3
echo '{"apiVersion":"policy/v1","kind":"Eviction","metadata":{"name":"'"$POD"'"}}' | oc create --raw /api/v1/namespaces/$NS/pods/$POD/eviction -f - || true
oc patch pdb parasol-claims -n $NS --type=merge -p '{"spec":{"minAvailable":1}}'
----
+
"'Cannot evict pod as it would violate the pod's disruption budget' — that's a node patch politely waiting instead of taking the service down."

[NOTE]
.Reset between runs
====
`ws prep deployment-targets-scheduling --yes` returns the world to the entry state — default scheduling, the batch worker unpinned, no PDB.
Run it before a second demo so beat 1 lands clean.
====
endif::demo[]

⏱ *~8 min* · read placement, requests, and the scheduler's own decision

Start by seeing what the scheduler did *for* you, with no input from you.
`-o wide` adds the *node* each pod landed on:

[tabs]
====

+
--
In the {ocp_console_url}[web console^] (project `{user}-dev`), open *Workloads* → *Pods*.
The list shows each pod's *Node* — note that the two `parasol-claims` pods, `parasol-web`, and `statement-batch` are scattered across whatever nodes had room.
--
CLI::
+
--
[source,sh,role=execute]
----
oc get pods -n $NS -o wide
----
--
====

.Expected output (your node names differ; the scatter is the point)
[source,texinfo]
----
NAME                              READY   STATUS    NODE              NOMINATED NODE
claims-db-8976f6bdb-wkh4r         1/1     Running   worker-3          <none>
claims-load-6dd4499cbd-qj2px      1/1     Running   worker-2          <none>
parasol-claims-74b5d4fb86-4cl9x   1/1     Running   worker-2          <none>
parasol-claims-74b5d4fb86-q7gnw   1/1     Running   control-plane-1   <none>
parasol-web-bf7d5f795-f7r4z       1/1     Running   control-plane-2   <none>
statement-batch-db8794f7c-88cvm   1/1     Running   worker-2          <none>
----

The scheduler placed each pod by its *resource requests* — the CPU and memory it reserves.
Look at what `parasol-claims` asks for, and the *Quality of Service* class that follows from it:

[source,sh,role=execute]
----
oc get deploy parasol-claims -n $NS -o jsonpath='requests={.spec.template.spec.containers[0].resources.requests} limits={.spec.template.spec.containers[0].resources.limits}{"\n"}'
oc get pods -n $NS -l app=parasol-claims -o jsonpath='{range .items[*]}{.metadata.name}{"  qos="}{.status.qosClass}{"\n"}{end}'
----

.Expected output
[source,texinfo]
----
requests={"cpu":"200m","memory":"256Mi"} limits={"cpu":"500m","memory":"512Mi"}
parasol-claims-74b5d4fb86-4cl9x  qos=Burstable
parasol-claims-74b5d4fb86-q7gnw  qos=Burstable
----

Requests below limits makes this pod *Burstable* — it's guaranteed its 200m/256Mi and may burst toward the limit if a node has spare.
The scheduler placed it against that *200m*, not against what it actually uses.

Now read the scheduler's *own account* of one placement — every pod carries a `Scheduled` event that names the node it chose:

[source,sh,role=execute]
----
POD=$(oc get pods -n $NS -l app=parasol-claims -o jsonpath='{.items[0].metadata.name}')
oc get events -n $NS --field-selector involvedObject.name=$POD,reason=Scheduled -o custom-columns='REASON:.reason,MESSAGE:.message'
----

.Expected output
[source,texinfo]
----
REASON      MESSAGE
Scheduled   Successfully assigned user2-dev/parasol-claims-74b5d4fb86-4cl9x to worker-2
----

One node in the cluster is deliberately *held back* from ordinary pods — a *taint*.
This is the dedicated *batch pool* you'll use later; peek at it now so you know it's there:

[source,sh,role=execute]
----
oc get nodes -l workshop.redhat.com/pool=batch -o name
oc get nodes -l workshop.redhat.com/pool=batch -o jsonpath='{.items[0].spec.taints}{"\n"}'
----

.Expected output
[source,texinfo]
----
node/worker-1
[{"effect":"NoSchedule","key":"workshop.redhat.com/pool","value":"batch"}]
----

`NoSchedule` means "no pod lands here unless it explicitly tolerates `workshop.redhat.com/pool=batch`."
That's *why* none of your app's pods are on `worker-1` — none of them tolerate it.

== 2. Direct the API with pod anti-affinity
⏱ *~14 min* · force the replicas apart, read a scheduler refusal, watch a reschedule

Your two `parasol-claims` replicas *happen* to be on different nodes — but that's the scheduler's *soft preference*, not a guarantee.
On a busy cluster both could land on one node, and one node failure would take the whole API down.
Turn "prefers apart" into *required* apart with `podAntiAffinity` on `kubernetes.io/hostname` (no two claims pods on the same node):

// [CAPTURE-VERIFY] Workloads → Deployments → parasol-claims → Actions → Edit Deployment (YAML) exposes spec.template.spec.affinity (OCP 4.21) — CLI authoritative.
[tabs]
====
Console::
+
--
Open *Workloads* → *Deployments* → `parasol-claims` → *Actions* → *Edit Deployment* (YAML).
Under `spec.template.spec`, add an `affinity.podAntiAffinity` with a `requiredDuringSchedulingIgnoredDuringExecution` rule matching `app: parasol-claims` on `topologyKey: kubernetes.io/hostname`, then *Save* — the same object the CLI tab patches.
--
CLI::
+
--
[source,sh,role=execute]
----
oc patch deploy parasol-claims -n $NS --type=merge -p '{"spec":{"template":{"spec":{"affinity":{"podAntiAffinity":{"requiredDuringSchedulingIgnoredDuringExecution":[{"labelSelector":{"matchLabels":{"app":"parasol-claims"}},"topologyKey":"kubernetes.io/hostname"}]}}}}}}'
oc rollout status deploy/parasol-claims -n $NS --timeout=90s
----
--
====

.Expected output (the roll takes ~40 s — anti-affinity makes it deliberate)
[source,texinfo]
----
deployment.apps/parasol-claims patched
Waiting for deployment "parasol-claims" rollout to finish: 1 out of 2 new replicas have been updated...
deployment "parasol-claims" successfully rolled out
----

Confirm the replicas are now on *distinct* nodes — guaranteed, not lucky:

[source,sh,role=execute]
----
oc get pods -n $NS -l app=parasol-claims -o wide
----

.Expected output
[source,texinfo]
----
NAME                              READY   STATUS    NODE
parasol-claims-585785cf56-464cj   1/1     Running   control-plane-3
parasol-claims-585785cf56-5jwzr   1/1     Running   control-plane-2
----

Now *kill* one pod and watch the scheduler honour the rule again — the replacement lands on a node that has *no* other claims pod:

[source,sh,role=execute]
----
VICTIM=$(oc get pods -n $NS -l app=parasol-claims -o jsonpath='{.items[0].metadata.name}')
oc delete pod $VICTIM -n $NS
sleep 12
oc get pods -n $NS -l app=parasol-claims -o wide
----

.Expected output (a fresh pod, still on a distinct node)
[source,texinfo]
----
pod "parasol-claims-585785cf56-464cj" deleted
NAME                              READY   STATUS    NODE
parasol-claims-585785cf56-5jwzr   1/1     Running   control-plane-2
parasol-claims-789ff476b-jkb6t    1/1     Running   worker-3
----

Finally, see what happens when the scheduler *can't* satisfy the rule.
Ask for *more* replicas than there are eligible nodes (the batch node is tainted off, so only the rest are available), and one pod has nowhere to go:

[source,sh,role=execute]
----
oc scale deploy/parasol-claims -n $NS --replicas=6
sleep 12
oc get pods -n $NS -l app=parasol-claims -o wide | grep -E 'NAME|Pending|Running' | head -8
----

.Expected output (one pod stuck `Pending`)
[source,texinfo]
----
NAME                              READY   STATUS    NODE
parasol-claims-585785cf56-hhcs9   0/1     Pending   <none>
parasol-claims-585785cf56-prjvp   1/1     Running   control-plane-1
parasol-claims-585785cf56-5jwzr   1/1     Running   control-plane-2
parasol-claims-585785cf56-464cj   1/1     Running   control-plane-3
parasol-claims-585785cf56-cj784   1/1     Running   worker-2
parasol-claims-585785cf56-bhjcd   1/1     Running   worker-3
----

The scheduler tells you *exactly* why — read its refusal:

[source,sh,role=execute]
----
oc get events -n $NS --field-selector reason=FailedScheduling -o jsonpath='{range .items[*]}{.message}{"\n"}{end}' | tail -1
----

.Expected output
[source,texinfo]
----
0/6 nodes are available: 1 node(s) had untolerated taint(s), 5 node(s) didn't match pod anti-affinity rules. ...
----

Six nodes, none available: *one* is the tainted batch node, and the other *five* already hold a claims pod, so the anti-affinity rule rejects them.
That single line is the answer to almost every "why is my pod Pending?" — always read it first.
Scale back to three (a good spread, room for the disruption budget later):

[source,sh,role=execute]
----
oc scale deploy/parasol-claims -n $NS --replicas=3
oc rollout status deploy/parasol-claims -n $NS --timeout=90s
----

[.checkpoint]
****
✔ *Verify:* Required `podAntiAffinity` put the replicas on *distinct* nodes; deleting one rescheduled it onto another distinct node; scaling past the available-node count left a pod `Pending` with a `FailedScheduling` event reading *"untolerated taint … didn't match pod anti-affinity rules"*; and you scaled back to three.
****

== 3. Spread evenly with topology spread constraints
⏱ *~8 min* · even distribution across a domain — nodes, and (the knob) zones

Anti-affinity says "never two on a node." `topologySpreadConstraints` says the stronger thing — "keep them *evenly* spread across a domain."
Add one on `kubernetes.io/hostname`, `maxSkew: 1`, `ScheduleAnyway` (a strong preference that never blocks a deploy):

[source,sh,role=execute]
----
oc patch deploy parasol-claims -n $NS --type=merge -p '{"spec":{"template":{"spec":{"topologySpreadConstraints":[{"maxSkew":1,"topologyKey":"kubernetes.io/hostname","whenUnsatisfiable":"ScheduleAnyway","labelSelector":{"matchLabels":{"app":"parasol-claims"}}}]}}}}'
oc rollout status deploy/parasol-claims -n $NS --timeout=120s
oc get pods -n $NS -l app=parasol-claims -o wide
----

.Expected output (three replicas, three distinct nodes)
[source,texinfo]
----
deployment "parasol-claims" successfully rolled out
NAME                             READY   STATUS    NODE
parasol-claims-...-drrr4         1/1     Running   control-plane-2
parasol-claims-...-pfnj6         1/1     Running   control-plane-1
parasol-claims-...-xrcc7         1/1     Running   control-plane-3
----

The topology key is a *knob*.
Point it at a *zone* label instead of the hostname, and the same constraint spreads your replicas across *failure zones* — one per zone:

[source,sh,role=execute]
----
oc patch deploy parasol-claims -n $NS --type=merge -p '{"spec":{"template":{"spec":{"topologySpreadConstraints":[{"maxSkew":1,"topologyKey":"workshop.redhat.com/zone","whenUnsatisfiable":"ScheduleAnyway","labelSelector":{"matchLabels":{"app":"parasol-claims"}}}]}}}}'
oc rollout status deploy/parasol-claims -n $NS --timeout=120s
oc get pods -n $NS -l app=parasol-claims --field-selector=status.phase=Running -o jsonpath='{range .items[*]}{.spec.nodeName}{"\n"}{end}' | while read n; do echo "$n zone=$(oc get node $n -o jsonpath='{.metadata.labels.workshop\.redhat\.com/zone}')"; done
----

.Expected output (one replica per zone)
[source,texinfo]
----
control-plane-2 zone=b
control-plane-1 zone=a
control-plane-3 zone=c
----

[NOTE]
.These zones are synthetic — real ones ride the provider
====
This bare-metal cluster has no real failure zones, so the workshop *synthesized* `workshop.redhat.com/zone={a,b,c}` on the nodes to make the spread visible.
On a cloud cluster you'd point `topologyKey` at the provider's real `topology.kubernetes.io/zone` and the same one-per-zone spread would protect you from a whole *zone* outage.
The `kubernetes.io/hostname` spread, by contrast, is *always* true — every node carries that label on every cluster — so put the topology key back to hostname for the rest of the lab:
====

[source,sh,role=execute]
----
oc patch deploy parasol-claims -n $NS --type=merge -p '{"spec":{"template":{"spec":{"topologySpreadConstraints":[{"maxSkew":1,"topologyKey":"kubernetes.io/hostname","whenUnsatisfiable":"ScheduleAnyway","labelSelector":{"matchLabels":{"app":"parasol-claims"}}}]}}}}'
----

[.checkpoint]
****
✔ *Verify:* A `topologySpreadConstraints` on `kubernetes.io/hostname` spread the three replicas evenly across nodes; flipping the topology key to `workshop.redhat.com/zone` spread them one-per-zone (synthetic here, real on a cloud); and you set the key back to hostname.
****

== 4. Reserve a dedicated pool for the batch worker
⏱ *~12 min* · the break-and-fix: a selector attracts, but a toleration is what permits

The `statement-batch` worker churns monthly statements and shouldn't compete with the live API for CPU.
You have a *dedicated pool* — the tainted `worker-1` from exercise 1 — and now you'll pin the batch worker onto it.
It's on a general node today:

[source,sh,role=execute]
----
oc get pods -n $NS -l app=statement-batch -o wide
----

.Expected output
[source,texinfo]
----
NAME                              READY   STATUS    NODE
statement-batch-db8794f7c-88cvm   1/1     Running   worker-2
----

The instinct is "add a `nodeSelector` for the pool label and I'm done."
Do exactly that — *only* the selector — and watch it *fail*, on purpose:

[source,sh,role=execute]
----
oc patch deploy statement-batch -n $NS --type=merge -p '{"spec":{"template":{"spec":{"nodeSelector":{"workshop.redhat.com/pool":"batch"}}}}}'
sleep 10
oc get pods -n $NS -l app=statement-batch
----

.Expected output (the new pod is stuck `Pending`)
[source,texinfo]
----
deployment.apps/statement-batch patched
NAME                               READY   STATUS    NODE
statement-batch-64b4659bdf-2l2tp   0/1     Pending   <none>
statement-batch-db8794f7c-88cvm    1/1     Running   worker-2
----

Read the refusal — it names *both* halves of the problem:

[source,sh,role=execute]
----
BATCHPOD=$(oc get pods -n $NS -l app=statement-batch --field-selector=status.phase=Pending -o jsonpath='{.items[0].metadata.name}')
oc get events -n $NS --field-selector involvedObject.name=$BATCHPOD,reason=FailedScheduling -o jsonpath='{range .items[*]}{.message}{"\n"}{end}' | tail -1
----

.Expected output
[source,texinfo]
----
0/6 nodes are available: 1 node(s) had untolerated taint(s), 5 node(s) didn't match Pod's node affinity/selector. ...
----

Read it carefully — this is the whole lesson.
The *one* node your selector points at (the batch pool) has an *untolerated taint* — the selector *sent* the pod there, but the node *repels* it.
The other *five* don't match the selector at all.
So the pod has nowhere to go.

The fix is the *toleration* — the pod's permission slip for the tainted node.
Add it, and the selector and toleration together do the job:

// [CAPTURE-VERIFY] Workloads → Deployments → statement-batch → Actions → Edit Deployment (YAML) exposes spec.template.spec.tolerations (OCP 4.21) — CLI authoritative.
[tabs]
====
Console::
+
--
Open *Workloads* → *Deployments* → `statement-batch` → *Actions* → *Edit Deployment* (YAML).
Under `spec.template.spec`, add a `tolerations` entry for key `workshop.redhat.com/pool`, value `batch`, effect `NoSchedule` (the `nodeSelector` is already there), then *Save*.
--
CLI::
+
--
[source,sh,role=execute]
----
oc patch deploy statement-batch -n $NS --type=merge -p '{"spec":{"template":{"spec":{"tolerations":[{"key":"workshop.redhat.com/pool","operator":"Equal","value":"batch","effect":"NoSchedule"}]}}}}'
oc rollout status deploy/statement-batch -n $NS --timeout=60s
----
--
====

Confirm it landed on the batch pool node — and that the node it landed on really is the tainted one:

[source,sh,role=execute]
----
oc get pods -n $NS -l app=statement-batch -o wide
BN=$(oc get pods -n $NS -l app=statement-batch --field-selector=status.phase=Running -o jsonpath='{.items[0].spec.nodeName}')
echo "landed on: $BN  pool-label=$(oc get node $BN -o jsonpath='{.metadata.labels.workshop\.redhat\.com/pool}')"
----

.Expected output
[source,texinfo]
----
NAME                               READY   STATUS    NODE
statement-batch-667dc64bc7-zcjxm   1/1     Running   worker-1
landed on: worker-1  pool-label=batch
----

There it is: *selector to send it, toleration to permit it — you need both.*
A toleration without a selector would *allow* the batch node but not *choose* it; a selector without a toleration is exactly the `Pending` you just fixed.

[.checkpoint]
****
✔ *Verify:* The `nodeSelector` alone left `statement-batch` `Pending` with an *"untolerated taint … didn't match node selector"* event; adding the matching *toleration* let it schedule *onto the dedicated batch pool node* (`worker-1`, `pool=batch`). Selector attracts, toleration permits — both required.
****

== 5. Make the rollout zero-downtime
⏱ *~18 min* · measure the drops, diagnose the real cause, fix it at the source, and protect with a PDB

A rollout replaces every API pod. Done carelessly it drops requests; done right a customer never notices.
First, *measure* a default roll. This one-shot block drives a fast probe at the API from the load generator while a rollout runs, and tallies any request that isn't a clean `2xx`:

[source,sh,role=execute]
----
( oc exec deploy/claims-load -n $NS -- bash -c '
ok=0; bad=0; SECONDS=0
while [ $SECONDS -lt 55 ]; do
  c=$(curl -s -o /dev/null -w "%{http_code}" --max-time 1 http://parasol-claims:8080/api/claims 2>/dev/null || echo 000)
  [ "${c:0:1}" = "2" ] && ok=$((ok+1)) || bad=$((bad+1))
  sleep 0.05
done
echo "default roll: ok=$ok dropped=$bad"' ) &
sleep 6
oc rollout restart deploy/parasol-claims -n $NS >/dev/null
wait
----

// [CAPTURE-VERIFY] exact counts come from your run; dropped is well above zero across the reseed windows of the roll.
.Expected output (your exact counts vary — the point is `dropped` is *not zero*, and not a stray one or two)
[source,texinfo]
----
default roll: ok=241 dropped=15
----

Requests dropped — and more than a stray one or two. Before reaching for a rollout knob, *find out why*.

=== Diagnose: what does a new pod do on boot?

The `parasol-claims` image ships a Hibernate setting that decides what a pod does to its database at startup. Read it:

[source,sh,role=execute]
----
oc set env deploy/parasol-claims -n $NS --list | grep SCHEMA_MANAGEMENT
----

.Expected output
[source,texinfo]
----
QUARKUS_HIBERNATE_ORM_SCHEMA_MANAGEMENT_STRATEGY=drop-and-create
----

`drop-and-create` means *every pod, on every boot, drops the schema, recreates it, and reloads the seed* — against the *shared* `claims-db` that all the replicas use.
Prove that's what's biting you. Plant a value the seed doesn't carry (`CLM-1001` seeds as `UnderReview`; move it to `Approved`) and confirm it stuck:

[source,sh,role=execute]
----
oc exec deploy/claims-load -n $NS -- curl -s -X PUT http://parasol-claims:8080/api/claims/CLM-1001/status \
  -H 'Content-Type: application/json' -d '{"status":"Approved"}' -o /dev/null -w 'wrote HTTP %{http_code}\n'
oc exec deploy/claims-load -n $NS -- curl -s http://parasol-claims:8080/api/claims/CLM-1001 | grep -o '"status":"[^"]*"'
----

// [CAPTURE-VERIFY] confirm the HTTP code and JSON rendering live; the `Approved` write landing in the shared DB is the grounded behaviour.
.Expected output (your write is live in the shared database)
[source,texinfo]
----
wrote HTTP 200
"status":"Approved"
----

Now restart *one* pod and read `CLM-1001` again:

[source,sh,role=execute]
----
ONEPOD=$(oc get pods -n $NS -l app=parasol-claims -o jsonpath='{.items[0].metadata.name}')
oc delete pod $ONEPOD -n $NS
sleep 3
oc wait --for=condition=Ready pod -l app=parasol-claims -n $NS --timeout=180s
oc exec deploy/claims-load -n $NS -- curl -s http://parasol-claims:8080/api/claims/CLM-1001 | grep -o '"status":"[^"]*"'
----

// [CAPTURE-VERIFY] the revert to the seed value `UnderReview` is the grounded, deterministic signature (proven live: the planted value is gone after one restart).
.Expected output (your marker is *gone* — one pod boot reseeded the shared database)
[source,texinfo]
----
pod "parasol-claims-...-..." deleted
pod/parasol-claims-...-... condition met
"status":"UnderReview"
----

There it is. A *single* pod restart wiped your write and reverted `CLM-1001` to the seed `UnderReview` — because that one boot ran `drop-and-create` and rebuilt the *shared* database out from under the other, still-serving pods.
That is exactly what a rolling update does, over and over: each new pod's boot drops the tables the old pods are mid-query on.
The disruption is in the *data plane, at startup* — which is why no `maxUnavailable` or shutdown-hook tuning could ever reach zero.
One more thing to notice: that replacement pod took a while to reach `Ready` — about *27 s* on the build cluster, because the container's *500m CPU limit* throttles the JVM cold-start. Hold that thought; it matters for the capacity dip.

[.checkpoint]
****
✔ *Verify (diagnosed):* The schema strategy is `drop-and-create`; a write to `CLM-1001` *vanished* after a single pod restart — reverting to the seed `UnderReview` — proving every pod boot reseeds the *shared* `claims-db`. Under the 500m CPU limit the replacement took ~27 s to reach `Ready`.
****

=== Fix it at the source

*Step 1 — stop the boot-time reseed (the primary fix).*
Flip the schema strategy to `none` so a pod boot never touches the database:

// [CAPTURE-VERIFY] Workloads → Deployments → parasol-claims → Environment tab exposes the schema-management env var (OCP 4.21) — CLI authoritative.
[tabs]
====
Console::
+
--
Open *Workloads* → *Deployments* → `parasol-claims` → *Environment*.
Change `QUARKUS_HIBERNATE_ORM_SCHEMA_MANAGEMENT_STRATEGY` from `drop-and-create` to `none`, then *Save* — the same field the CLI tab sets.
--
CLI::
+
--
[source,sh,role=execute]
----
oc set env deployment/parasol-claims QUARKUS_HIBERNATE_ORM_SCHEMA_MANAGEMENT_STRATEGY=none -n $NS
oc rollout status deploy/parasol-claims -n $NS --timeout=180s
----
--
====

Prove the fix the same way you proved the fault — plant the value again, restart one pod, and this time watch it *survive*:

[source,sh,role=execute]
----
oc exec deploy/claims-load -n $NS -- curl -s -X PUT http://parasol-claims:8080/api/claims/CLM-1001/status \
  -H 'Content-Type: application/json' -d '{"status":"Approved"}' -o /dev/null -w 'wrote HTTP %{http_code}\n'
ONEPOD=$(oc get pods -n $NS -l app=parasol-claims -o jsonpath='{.items[0].metadata.name}')
oc delete pod $ONEPOD -n $NS
sleep 3
oc wait --for=condition=Ready pod -l app=parasol-claims -n $NS --timeout=180s
oc exec deploy/claims-load -n $NS -- curl -s http://parasol-claims:8080/api/claims/CLM-1001 | grep -o '"status":"[^"]*"'
----

// [CAPTURE-VERIFY] the planted value SURVIVING the restart is the grounded proof the reseed is gone.
.Expected output (this time your write *survives* the restart)
[source,texinfo]
----
wrote HTTP 200
pod "parasol-claims-...-..." deleted
pod/parasol-claims-...-... condition met
"status":"Approved"
----

[NOTE]
.Seed once at deploy, not on every boot
====
With `schema-management=none` a pod never rebuilds the database — so the schema and seed data have to come from *somewhere else, once*.
The real-world pattern is a decoupled *migration Job* (Flyway or Liquibase) that runs at deploy time, before the app pods start.
Your `claims-db` is already populated (the app seeded it while it was still `drop-and-create`), so there's nothing to run here — but the principle *is* the fix: *setup work belongs to the deploy, not to every pod's startup path.*
====

*Step 2 — harden the roll (hold capacity, and speed the cold-start).*
`maxSurge: 1`/`maxUnavailable: 0` brings a new pod up *before* retiring an old one, and raising the container's CPU *limit* lets the JVM cold-start without throttling — so a fresh pod reaches `Ready` in about *17 s* instead of ~27 s, shrinking the brief capacity dip while the roll waits on it:

// [CAPTURE-VERIFY] Workloads → Deployments → parasol-claims → Actions → Edit Deployment (YAML) exposes spec.strategy + spec.template.spec.containers[].resources (OCP 4.21) — CLI authoritative.
[tabs]
====
Console::
+
--
Open *Workloads* → *Deployments* → `parasol-claims` → *Actions* → *Edit Deployment* (YAML).
Set `spec.strategy.rollingUpdate` to `maxSurge: 1`, `maxUnavailable: 0`; and on the `parasol-claims` container raise `resources.limits.cpu` to `2` (leave `requests.cpu` at `200m`), then *Save*.
--
CLI::
+
--
[source,sh,role=execute]
----
oc patch deploy parasol-claims -n $NS --type=strategic -p '{"spec":{"strategy":{"type":"RollingUpdate","rollingUpdate":{"maxSurge":1,"maxUnavailable":0}},"template":{"spec":{"containers":[{"name":"parasol-claims","resources":{"requests":{"cpu":"200m","memory":"256Mi"},"limits":{"cpu":"2","memory":"512Mi"}}}]}}}}'
oc rollout status deploy/parasol-claims -n $NS --timeout=180s
----
--
====

[IMPORTANT]
.Use a *strategic* patch for the container change
====
The *resources* edit reaches inside a named container, and `--type=merge` (JSON merge) *replaces* the whole `containers` list — dropping the image and failing with `spec.containers[0].image: Required value`.
`--type=strategic` understands the container `name` as a merge key, so it edits *into* the existing container.
Reach for strategic merge (or an `oc set resources`/`oc set env` helper, which do it for you) whenever you patch a single field inside a named list element.
====

Now prove the load-bearing win: during a rollout, the number of *available* (ready) replicas never dips below your desired count.
Trigger a roll and sample it:

[source,sh,role=execute]
----
oc rollout restart deploy/parasol-claims -n $NS
for i in $(seq 1 18); do
  oc get deploy parasol-claims -n $NS -o jsonpath='available={.status.availableReplicas}/{.spec.replicas}{"\n"}'
  sleep 4
done | sort | uniq -c
----

.Expected output (every sample shows full capacity — `maxUnavailable: 0` holds the floor)
[source,texinfo]
----
  18 available=3/3
----

Capacity never dropped below 3 — the Deployment surged a replacement to `Ready` *before* retiring each old pod.
Now re-run the *same drop probe* from the top of this exercise — with the reseed gone, it comes back clean:

[source,sh,role=execute]
----
( oc exec deploy/claims-load -n $NS -- bash -c '
ok=0; bad=0; SECONDS=0
while [ $SECONDS -lt 55 ]; do
  c=$(curl -s -o /dev/null -w "%{http_code}" --max-time 1 http://parasol-claims:8080/api/claims 2>/dev/null || echo 000)
  [ "${c:0:1}" = "2" ] && ok=$((ok+1)) || bad=$((bad+1))
  sleep 0.05
done
echo "hardened roll: ok=$ok dropped=$bad"' ) &
sleep 6
oc rollout restart deploy/parasol-claims -n $NS >/dev/null
wait
----

// [CAPTURE-VERIFY] the ok count varies; `dropped=0` is the deterministic expectation once the boot-time reseed is removed.
.Expected output (zero dropped — deterministically, because the reseed is gone)
[source,texinfo]
----
hardened roll: ok=274 dropped=0
----

[NOTE]
.Why this reaches zero, not almost-zero
====
`maxUnavailable: 0` holds *capacity*; stopping the boot-time reseed removes the *data-plane* disruption that was actually failing requests; and the raised CPU limit shrinks the cold-start so the capacity dip is brief.
Because the reseed was *deterministic* — one boot wiped the shared database *every* time — removing it is a deterministic fix, not a "usually fine."
Graceful connection-draining on `SIGTERM` is a separate, real concern, but this Quarkus image already handles it (`quarkus.shutdown.timeout`), and it was never what dropped requests here.
====

Finally, protect the service during *voluntary* disruptions (node drains, upgrades) with a *PodDisruptionBudget* — "never below one healthy pod":

// [CAPTURE-VERIFY] Workloads → PodDisruptionBudgets → Create shows ALLOWED DISRUPTIONS once created (OCP 4.21) — CLI authoritative.
[tabs]
====
Console::
+
--
Open *Workloads* → *PodDisruptionBudgets* → *Create PodDisruptionBudget* → *YAML view*, paste the manifest from the *CLI* tab, and *Create*.
The list shows *ALLOWED DISRUPTIONS* — with 3 healthy pods and `minAvailable: 1`, that's *2*.
--
CLI::
+
--
[source,sh,role=execute]
----
oc apply -n $NS -f - <<'EOF'
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: parasol-claims
spec:
  minAvailable: 1
  selector:
    matchLabels:
      app: parasol-claims
EOF
sleep 3
oc get pdb parasol-claims -n $NS
----
--
====

.Expected output
[source,texinfo]
----
NAME             MIN AVAILABLE   MAX UNAVAILABLE   ALLOWED DISRUPTIONS   AGE
parasol-claims   1               N/A               2                     3s
----

`ALLOWED DISRUPTIONS: 2` — 3 healthy minus the 1 you must keep.
The eviction API that `oc adm drain` uses checks this budget. Prove it: *exhaust* the budget (tighten `minAvailable` to 3, so *zero* disruptions are allowed) and try to evict a pod — the API *refuses*, with the exact message a node drain hits:

[source,sh,role=execute]
----
oc patch pdb parasol-claims -n $NS --type=merge -p '{"spec":{"minAvailable":3}}'
sleep 3
POD=$(oc get pods -n $NS -l app=parasol-claims -o jsonpath='{.items[0].metadata.name}')
echo '{"apiVersion":"policy/v1","kind":"Eviction","metadata":{"name":"'"$POD"'"}}' | oc create --raw /api/v1/namespaces/$NS/pods/$POD/eviction -f - || true
# restore the sensible budget
oc patch pdb parasol-claims -n $NS --type=merge -p '{"spec":{"minAvailable":1}}'
----

.Expected output
[source,texinfo]
----
poddisruptionbudget.policy/parasol-claims patched
Error from server (TooManyRequests): Cannot evict pod as it would violate the pod's disruption budget.
poddisruptionbudget.policy/parasol-claims patched
----

That refusal is exactly what makes a node drain *wait its turn* instead of taking your last replica.
Back at `minAvailable: 1`, a drain can evict one claims pod (the reschedule respects your anti-affinity), then the next — one at a time, service up throughout.
