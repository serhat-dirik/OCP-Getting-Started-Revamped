[.lead]
A new engineer joins Parasol's claims team, and on day one they need a service that already builds into a trusted image, ships through a pipeline, deploys by GitOps, and reports health, metrics, and traces.
Handed nine separate tools and a wiki, they lose a week to plumbing before writing a line of business logic.
This module is the payoff of everything you have built so far: the platform team packages *modules 2 through 10 into a single button*, and the new engineer gets a running, standards-compliant service from a web form in minutes.

== What you'll learn

* What an *Internal Developer Platform* (IDP) is, and why platform teams stand up a *developer portal* — *Red Hat Developer Hub* — instead of a wiki of tribal knowledge.
* How the *software catalog* becomes the org's live map: *Components*, *APIs*, *Systems*, and the *Groups* that own them — so "who owns claims, and what does it talk to?" has one answer, not five.
* What a *golden path* is — a *paved road, not a cage* — and how a *Software Template* encodes the build, container, GitOps, and observability decisions from the rest of Block B into one self-service action.
* Why *self-service with standards-by-default* changes an organization: new services start compliant instead of being audited into compliance later.
* Where an IDP is *worth its maintenance cost* and where it is not — so you reach for it deliberately, not reflexively.

== The problem: cognitive load, and standards nobody follows

By this point in the workshop you have assembled a real platform.
You can build a container image the trusted way, run it through a pipeline, sign and verify it, hand deployment to GitOps, and scale it safely.
Every one of those is a capability — and every one is *another tool, another repository layout, another set of conventions* a developer has to learn before they are productive.

Multiply that by a growing team and the cost is not the tools; it is the *cognitive load*.
The tenth engineer to create a service either reverse-engineers how the last nine did it, copies a repo that was already subtly wrong, or asks the platform team to do it by hand.
Standards that live in a wiki are standards that drift: the probes get forgotten, the base image is stale, the pipeline is copied from a service that did it differently.
The platform is powerful and *nobody can find the on-ramp*.

An *Internal Developer Platform* fixes the on-ramp.
The piece you meet in this module is its front door: a *developer portal*, *Red Hat Developer Hub* (RHDH), Red Hat's build of the open-source Backstage project.
It gives an organization two things — a *catalog* that answers "what do we run and who owns it," and *golden paths* that answer "how do I start something new, correctly, without asking anyone."

== The catalog: the org's map, not a spreadsheet

The first thing a portal gives you is a *software catalog* — a live inventory of what the organization runs, modelled as *entities* with typed relationships.

Four entity kinds carry most of the value:

Component:: A thing you build and run — a service, a website, a library.
`parasol-claims` is a Component of type `service`.
API:: A contract a Component *provides* or *consumes* — `parasol-claims-api` is an OpenAPI definition the claims service provides, browsable in the portal.
System:: A group of Components and APIs that work together — `parasol-insurance` is the System the claims platform belongs to.
Group:: The team that *owns* entities — the `parasol` Group owns the claims service and its API.

Because the relationships are typed, the catalog is a *map*, not a spreadsheet: it renders who owns `parasol-claims`, which System it is part of, which API it provides, and which other Components depend on that API.

Each entity's source of truth is a small `catalog-info.yaml` file that lives *in the service's own Git repository*, next to the code.
The portal reads those files and keeps the catalog current — so ownership and structure are described *as code, reviewed in pull requests*, not maintained by hand in a separate system that rots.
When you onboard a new engineer, you do not hand them a diagram that is three reorgs out of date; you point them at the catalog, which is as current as the last commit.

== The golden path: a paved road, not a cage

A catalog tells you what exists.
A *golden path* is how you *create* something new — correctly — without learning the whole platform first.

Concretely, a golden path is a *Software Template*: a form plus a set of steps.
The developer fills in a couple of fields ("name your service"), and the template *scaffolds a complete, standards-compliant service*: source code with health, metrics, and tracing already wired; a trusted UBI base image; a build recipe; a catalog entry; a workspace definition — every decision the rest of Block B taught, made correctly, once, by the platform team, and applied on every click.

The phrase that matters is *paved road, not cage*.
A cage forces one way and punishes deviation; a paved road makes the *right* thing the *easy* thing while leaving you free to drive off it when you have a real reason.
The scaffolded service is plain, standard, editable source in *your* Git repository — not a locked black box.
You own it the moment it is created: change the code, add dependencies, adjust the pipeline.
The template gave you a correct starting line and got out of the way.

This is why the golden path is the payoff of the whole delivery-and-trust block.
Everything you did by hand in the earlier modules — choosing a trusted base image, wiring probes and metrics, describing the app for GitOps, laying out the repo — is a *decision the platform team makes once* and *encodes in the template*.
The new engineer does not re-derive those decisions; they inherit them.
One form submission stands up what used to take a week of copying and asking.

[NOTE]
.Standards by construction, not by audit
====
The quiet shift a golden path makes is *when* compliance happens.
Without one, a service is created some ad-hoc way and then audited *later* — someone finds the missing probe, the stale base image, the undocumented owner, and files the ticket to fix it.
With one, the service starts *already* carrying probes, a trusted image, an owner, and a catalog entry — compliance is the *default state at creation*, not a correction applied after the fact.
Auditing shifts from "find and fix the drift" to "confirm the road was used."
====

== Red Hat Developer Hub in one paragraph

*Red Hat Developer Hub* is Red Hat's supported build of *Backstage*, the open-source developer-portal framework.
Backstage is a framework, not a finished product: you assemble the portal your organization needs from *plugins*.
RHDH packages that assembly the Red Hat way — a supported core, a curated set of *dynamic plugins* you enable through configuration rather than a custom build, and integration with the rest of the OpenShift platform.
In this workshop the portal is configured with a *guest sign-in* (one click to enter — real deployments wire single sign-on instead), a catalog fed from the Parasol Git repositories, and the *golden-path template* that publishes new services into your own Git organization.
It is a shared, read-mostly portal for the whole room: everyone browses the same catalog and runs the same template, and each attendee's scaffolded services land in their *own* Git organization.

== How this shows up in real orgs

* *"How do I start a new service?" gets one answer, not ten.*
Before a portal, the tenth service is created by copying the ninth — including whatever was wrong with it.
After, it is created from a template the platform team owns, so it starts with the current base image, the current probes, and an owner already recorded.
The on-ramp is the same for everyone, and it is correct.
* *The catalog answers ownership and blast-radius questions in seconds.*
"Who owns claims? What depends on its API? What System is it part of?" is a portal query, not an archaeology project across wikis and Slack.
When an incident hits, the map is current because it is generated from files that live with the code.
* *Compliance moves from audit to default.*
Standards encoded in a template are standards that new services *have* by construction.
The platform team's job shifts from chasing drift to maintaining the paved road — a much smaller, more leveraged surface.
* *The platform team's leverage compounds.*
Improve the template once — a better base image, a new required label, an extra pipeline step — and every service created after inherits it.
The portal turns "we should standardize on X" from a migration project into a one-line change to a golden path.
