== What you built

You handed Parasol's claims environments over to a controller.
You toured the `claims-config` repository — one Kustomize *base* and three *overlays* — then created an Argo CD *Application* in the student instance, signed in as yourself, and watched it reconcile `overlays/dev` into `{user}-dev`: seven objects, deployed from Git and stamped as GitOps-managed.
You made the cluster *drift* — scaled the app by hand — and watched Argo CD flag it `OutOfSync` and revert it, first when you pressed *Sync* and then, with *self-heal* on, entirely on its own.
Finally you promoted to `{user}-stage` and changed it the GitOps way: by editing a file in Git and letting the change flow in.

The idea under all of it: *the cluster is a projection of Git.*
Desired state lives in a repository, a controller reconciles the cluster to match it continuously, and anything that diverges is either surfaced for you to reconcile or corrected automatically — so "what is running, and why" has one honest answer.

== Map to your org

. *Where does the true state of your environments live today — Git, or the cluster?*
You now have environments that converge to a folder in a repo.
Pick a real service in your org: if someone asks "what exactly is running in staging and how did it get there," is the answer a Git commit, or is it "let me check the cluster and ask around"?
What would it take to make the repo the answer?
. *What was your last "someone changed something and forgot" incident?*
You watched self-heal erase an un-tracked console edit.
Think of an outage that started with an undocumented change to a running environment: would drift detection have surfaced it, and would you want self-heal *enforcing* Git there — or is that an environment where you would keep sync manual and review every change?
. *How do changes reach production now — a pull request, or a privileged deploy?*
You promoted by editing Git.
In your org, does moving a change to prod mean merging a reviewed commit, or a person (or CI job) with cluster credentials running `apply`?
What would it take to make production a *pull*, so no pipeline needs standing write access to it?

== When *not* to reach for GitOps

GitOps is the right tool for *declarative state you want continuously reconciled* — which means being honest about its edges:

* *Don't put fast, ephemeral, imperative work in GitOps.*
A one-off data migration, a `debug` pod, a quick `oc adm` operation — these are actions, not desired state.
GitOps reconciles *what should exist*; it is the wrong tool for *do this once, now*.
Reach for a Job, a script, or the CLI.
* *Don't turn on self-heal where humans legitimately hold the wheel.*
Self-heal reverting your change is a *feature* in production and a *nuisance* in a break-glass or actively-debugged environment — it will undo the very edit you are making.
Keep sync *manual* (or self-heal off) where interactive change is the point; turn it on where Git must be the only author.
* *Don't commit secrets to Git to make GitOps "complete."*
The one thing that must *not* live in your GitOps repo is a plaintext credential.
Reconciling a Secret from Git means the secret is *in* Git.
Source secrets from a manager (the External Secrets pattern the config module previews) and let GitOps reconcile the *reference*, not the value.
* *Don't reach for GitOps to paper over missing environments or config discipline.*
GitOps reconciles the structure you give it; it does not invent base/overlays or a clean config boundary for you.
If the config is a tangle of hand-edited manifests, fix that first (the config module's job) — GitOps then keeps the clean structure clean.
