== 1. Git is the source of truth
⏱ *~10 min* · tour the repository that describes your environments

Before a controller can reconcile your cluster to Git, you should know what is *in* Git.
The workshop forked a small config repository into your Git account — `{user}/claims-config` — the *same* repo the config module used.
Clone it in the terminal and look:

[source,sh,role=execute,subs="attributes"]
----
cd ${HOME:-/tmp}
git clone {gitea_url}/{user}/claims-config
cd claims-config
find . -type f -not -path './.git/*' | sort
----

.Expected output
[%collapsible]
====
[source,texinfo]
----
./README.md
./base/claims-app.yaml
./base/claims-configmap.yaml
./base/claims-creds-secret.yaml
./base/claims-db.yaml
./base/kustomization.yaml
./overlays/dev/kustomization.yaml
./overlays/prod/kustomization.yaml
./overlays/stage/kustomization.yaml
----
====

This is plain Kustomize: one `base/` that describes a claims environment once (a ConfigMap, a Secret, the database, and the app's Deployment + Service + Route), and three `overlays/` that change only what differs per environment.
The `dev` overlay is *yours* — when your fork was created, its namespace placeholder was rewritten to your username.
Read it:

[source,sh,role=execute,subs="attributes"]
----
cat overlays/dev/kustomization.yaml
----

.Expected output (the YAML below the comment header)
[%collapsible]
====
[source,texinfo]
----
namespace: user1-dev

resources:
  - ../../base

replicas:
  - name: parasol-claims
    count: 1

patches:
  - target:
      kind: ConfigMap
      name: claims-config
    patch: |-
      - op: replace
        path: /data/APP_ENV
        value: dev
      - op: replace
        path: /data/QUARKUS_LOG_LEVEL
        value: DEBUG
----
====

The overlay is tiny on purpose: it pins the *namespace* to `{user}-dev`, sets *1 replica*, and patches two ConfigMap values (`APP_ENV=dev`, log level `DEBUG`).
Everything else — the image, the probes, the database — comes from `base/`.
Now render what Argo CD will actually reconcile.
`oc kustomize` renders an overlay to the exact YAML the cluster would receive, *without applying anything*:

[source,sh,role=execute]
----
oc kustomize overlays/dev | grep -E '^kind:|^  name:|^  namespace:|^  replicas:'
----

.Expected output
[%collapsible]
====
[source,texinfo]
----
kind: ConfigMap
  name: claims-config
  namespace: user1-dev
kind: Secret
  name: claims-creds
  namespace: user1-dev
kind: Service
  name: claims-db
  namespace: user1-dev
kind: Service
  name: parasol-claims
  namespace: user1-dev
kind: Deployment
  name: claims-db
  namespace: user1-dev
  replicas: 1
kind: Deployment
  name: parasol-claims
  namespace: user1-dev
  replicas: 1
kind: Route
  name: parasol-claims
  namespace: user1-dev
----
====

That rendered YAML — seven objects (ConfigMap, Secret, two Services, two Deployments, and a Route), all in `{user}-dev`, the claims app at 1 replica — is the *desired state*.
In the config module you applied it by hand with `oc apply -k`.
This module hands that job to a controller: you point an Argo CD Application at `overlays/dev`, and it renders and applies exactly this, then keeps the cluster matching it.

== 3. Drift and self-heal: the platform argues back
⏱ *~25 min* · break the cluster by hand, and watch Git win — first on demand, then on its own

This is the beat that makes GitOps matter.
Your app is deployed from Git and `Synced`.
Now do what happens in every real cluster: reach in and change the running app by hand, *behind Git's back*.
Scale the Deployment from its Git-defined 1 replica up to 3:

[tabs]
====
Console::
+
--
In *Topology* (Project `{user}-dev`), click the `parasol-claims` node to open its side panel, then click the *up arrow* on the Pod donut twice to raise the count to *3* (or *Actions* → *Edit Pod count*).
The ring fills to three Pods — a hand edit, exactly the kind of out-of-band change Git is about to notice.
--
CLI::
+
--
[source,sh,role=execute]
----
oc scale deployment/parasol-claims --replicas=3 -n $DEV
oc get deployment/parasol-claims -n $DEV -o jsonpath='live replicas now: {.spec.replicas}{"\n"}'
----

.Expected output
[source,texinfo]
----
deployment.apps/parasol-claims scaled
live replicas now: 3
----
--
====

Switch to the Argo CD tab and press *Refresh* on the `claims-dev-{user}` app.
It flips to *OutOfSync*.
Open the `parasol-claims` Deployment in the tree and click *DIFF*: Argo CD shows you the exact divergence — live `replicas: 3`, desired `replicas: 1`.

// media-pass: screenshot gitops-fundamentals-05-drift-diff.png — the Argo CD app OutOfSync with the
// Deployment DIFF panel open, showing `replicas: 3` (live) vs `replicas: 1` (desired/Git), red/green.
// HIGH priority — this is the module's signature visual. See media-manifest.md.

Here is the important part, and the reason self-heal exists.
With *Manual* sync, Argo CD *reports* the drift but does *not* fix it.
Wait, then check — the drift persists; Argo CD is watching, not enforcing:

[source,sh,role=execute]
----
sleep 30
oc get deployment/parasol-claims -n $DEV -o jsonpath='still live: {.spec.replicas} replicas (Argo sees OutOfSync but has not reverted){"\n"}'
----

.Expected output
[source,texinfo]
----
still live: 3 replicas (Argo sees OutOfSync but has not reverted)
----

Now make Git win.
In Argo CD, click *SYNC → SYNCHRONIZE*.
Argo CD reapplies the desired state and the drift is erased — back in the terminal, within a few seconds:

[source,sh,role=execute]
----
sleep 8
oc get deployment/parasol-claims -n $DEV -o jsonpath='after Sync: {.spec.replicas} replica (reverted to Git){"\n"}'
----

.Expected output
[source,texinfo]
----
after Sync: 1 replica (reverted to Git)
----

That is the manual model: Argo CD shows drift, *you* decide, Sync reconciles.
Powerful — but it still needs a human to notice.

*Now turn on the automatic version.*
In the Argo CD app, open *APP DETAILS* (or *DETAILS*), find *SYNC POLICY*, and click *ENABLE AUTO-SYNC*, then *ENABLE* *SELF HEAL*.

// media-pass: screenshot gitops-fundamentals-06-enable-selfheal.png — the App Details → Sync Policy
// panel with AUTO-SYNC enabled and SELF HEAL enabled. Embed here. See media-manifest.md.

With self-heal on, Argo CD stops waiting.
Drift the cluster one more time — and this time *do not touch Argo CD at all*.
Watch it snap back on its own.

[tabs]
====
Console::
+
--
In *Topology* (Project `{user}-dev`), scale `parasol-claims` up to *3* again with the Pod-donut up arrow, then *keep Topology open and do nothing else*.
Within a few seconds the ring shrinks back to a single Pod on its own — Argo CD caught the drift and reverted it, with no Sync click.
--
CLI::
+
--
This command polls the live replica count once a second and stops the moment Argo CD has put it back to 1:

[source,sh,role=execute]
----
oc scale deployment/parasol-claims --replicas=3 -n $DEV >/dev/null
echo "drifted to 3 — now watch, hands off (Argo CD snaps it back):"
for i in $(seq 1 30); do
  R=$(oc get deployment/parasol-claims -n $DEV -o jsonpath='{.spec.replicas}')
  echo "  $(date +%T)  replicas=$R"
  [ "$R" = "1" ] && { echo "  ✔ self-heal reverted the drift — no human touched Argo CD"; break; }
  sleep 1
done
----

.Expected output — the revert lands in a few seconds
[source,texinfo]
----
drifted to 3 — now watch, hands off (Argo CD snaps it back):
  02:44:30  replicas=3
  02:44:32  replicas=3
  02:44:34  replicas=1
  ✔ self-heal reverted the drift — no human touched Argo CD
----
--
====

The console edit vanished — in a few seconds, with *nobody* pressing Sync.
That is self-heal: Argo CD *watches* the resources it manages, so it catches drift the moment it happens and corrects it — not on a slow timer.

[NOTE]
.How fast, and the one number that bounds it
====
Self-heal is *watch-triggered*: Argo CD reacts to the live change almost immediately (measured ~4 seconds here), not on a slow poll.
The controller's periodic reconcile interval — `90s` on this cluster — is only the *ceiling*, the longest drift could linger if a watch event were ever missed under heavy load.
Either way you *cannot out-edit Git*: the manual *Sync* button reverts drift instantly, and self-heal reverts it within seconds, unasked.
====

[.checkpoint]
****
✔ *Verify:* after scaling to 3 with *Manual* sync, the Deployment *stayed* at 3 until you pressed *Sync*, which reverted it to 1; after enabling *Self Heal*, a fresh scale-to-3 reverted to 1 *on its own within seconds* — with no Sync click.
Git won both times.
****

[NOTE]
.If the count does not move for a few minutes
====
The student Argo CD instance is shared; on a busy cluster a sync can lag while the controller works through everyone's apps.
The change is queued (the app is `OutOfSync` at your new commit, as the diff showed); pressing *SYNC* again, or waiting a reconcile cycle, applies it.
If it stays stuck, tell your instructor — the shared controller may need attention (see the troubleshooting page).
====

Stop and notice what you did *not* do: you never ran `oc scale`, never touched `{user}-stage` directly.
You edited a file in Git, Argo CD *computed the exact change* (the diff you saw: desired 3 vs live 2), and Sync *applied* it — reviewed as a commit, diffed before it landed, and revertible by reverting the commit.
That is promotion the GitOps way, and it is exactly how you would move a change from stage to prod: another edit, another environment's overlay, another Application.
