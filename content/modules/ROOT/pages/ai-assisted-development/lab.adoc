You are about to connect an AI assistant to *your* OpenShift namespace and let it *operate* the cluster — diagnose a broken app, then fix it — under a scoped identity you control.
The world was materialized for you: a *broken* Parasol claims service that is `Running` but never `Ready`, an *OpenShift/Kubernetes MCP server* that gives an assistant read-only eyes on your namespace, and a small *neutral command-line agent* wired to your cluster's model.
You will diagnose the fault read-only, *grant a scoped write* and watch the agent fix it, and then ask it to reach into a *neighbouring* namespace and watch *RBAC deny it* — the whole point of the module.

[TIP]

====

The sample namespace in the captured output below is `user3-dev`; *yours* is `{user}-dev` — read the *shape*, not the exact name.
Keep the terminal open throughout — the `$NS` and `$AGENT_CLI_IMAGE` shortcuts live there.

⏱ *~12 min* · eyes on the cluster, no kubeconfig in sight

Two things were deployed into your namespace. Look at them.

====

In the {ocp_console_url}[web console^] (project `{user}-dev`), open *Workloads* → *Deployments*.
You see two: `kubernetes-mcp-server` — *1/1*, healthy — and `parasol-claims` — *0/1*, up but not ready (the fault you'll diagnose).
Open `kubernetes-mcp-server` and note its *ServiceAccount* is `mcp-agent`, not `default`.
--
CLI::
+
--
[source,sh,role=execute]
----
oc get deploy kubernetes-mcp-server parasol-claims -n $NS
oc get deploy kubernetes-mcp-server -n $NS -o jsonpath='MCP server runs as SA: {.spec.template.spec.serviceAccountName}{"\n"}'
----

====

.Expected output (the MCP server is healthy; the claims service is up but not ready; the server runs as a dedicated SA)
[source,texinfo]
----
NAME                    READY   UP-TO-DATE   AVAILABLE   AGE
kubernetes-mcp-server   1/1     1            1           17m
parasol-claims          0/1     1            0           9m
MCP server runs as SA: mcp-agent
----

The MCP server is the *only* thing in this system with a cluster credential — and it is a *scoped* one. Read what `mcp-agent` is actually allowed to do:

[source,sh,role=execute]
----
oc get rolebinding -n $NS -o json | jq -r '.items[] | select(.subjects[]?.name=="mcp-agent") | "\(.metadata.name) -> \(.roleRef.kind)/\(.roleRef.name)"'
oc auth can-i --list -n $NS --as=system:serviceaccount:$NS:mcp-agent 2>/dev/null | grep -E '^(pods|events|deployments\.apps|secrets) ' | head
----

.Expected output (one namespaced binding to the built-in read-only `view` role — and no secrets)
[source,texinfo]
----
mcp-agent-view -> ClusterRole/view
events                          []     []    [get list watch]
pods                            []     []    [get list watch]
deployments.apps                []     []    [get list watch]
----

`mcp-agent` has *one* RoleBinding, to the built-in `view` ClusterRole, *in this namespace only*. That means read (`get list watch`) on pods, events, deployments, logs — and, because `view` excludes them, *no access to Secrets* and *no writes*. This is the agent's entire power. Nothing here can reach another namespace or read a credential.

Now build the *neutral agent* — the client that talks to the model and to that MCP server. Create a small `ask-agent` helper once; you'll reuse it for every question:

[source,sh,role=execute]
----
cat > ask-agent <<'EOF'
#!/usr/bin/env bash
# ask-agent [--allow-writes] "<prompt>"  — run the neutral mcp-agent-cli as a one-shot pod and
# print its tool-call trace + answer. The client holds NO kubeconfig: it reaches the model over
# MaaS and the cluster ONLY through the MCP server (which runs as the scoped mcp-agent SA).
set -euo pipefail
NS="${NS:-$(oc project -q)}"
RO=true; [[ "${1:-}" == "--allow-writes" ]] && { RO=false; shift; }
PROMPT="$*"; POD="ask-agent-$RANDOM"
ARGS=$(jq -cn --arg w "$([[ $RO == false ]] && echo --allow-writes || true)" --arg p "$PROMPT" \
  'if $w=="" then [$p] else [$w,$p] end')
jq -n --arg pod "$POD" --arg img "$AGENT_CLI_IMAGE" --argjson args "$ARGS" --arg ro "$RO" \
  '{apiVersion:"v1",kind:"Pod",metadata:{name:$pod,labels:{app:"mcp-agent-cli-run"}},
    spec:{restartPolicy:"Never",activeDeadlineSeconds:240,containers:[{name:"cli",image:$img,args:$args,
    envFrom:[{configMapRef:{name:"maas-config-env"}},{secretRef:{name:"maas-credentials"}}],
    env:[{name:"MCP_SERVER_URL",value:"http://kubernetes-mcp-server:8080/sse"},{name:"MCP_READ_ONLY",value:$ro}],
    resources:{requests:{cpu:"100m",memory:"384Mi"},limits:{cpu:"1",memory:"768Mi"}}}]}}' \
  | oc apply -f - >/dev/null
echo "· agent pod $POD running (the model round-trip can take up to a minute)…"
# fail fast on a pull-dead pod instead of waiting out the full timeout
for _ in $(seq 1 12); do
  ph=$(oc get pod "$POD" -n "$NS" -o jsonpath='{.status.phase}' 2>/dev/null || true)
  [[ "$ph" == Running || "$ph" == Succeeded || "$ph" == Failed ]] && break
  wr=$(oc get pod "$POD" -n "$NS" -o jsonpath='{.status.containerStatuses[0].state.waiting.reason}' 2>/dev/null || true)
  if [[ "$wr" == ImagePullBackOff || "$wr" == ErrImagePull ]]; then
    echo "✗ agent image could not be pulled ($wr) — check that AGENT_CLI_IMAGE is reachable: $AGENT_CLI_IMAGE"
    oc delete pod "$POD" -n "$NS" >/dev/null 2>&1 || true
    exit 1
  fi
  sleep 2
done
oc wait --for=jsonpath='{.status.phase}'=Succeeded pod/"$POD" -n "$NS" --timeout=180s >/dev/null 2>&1 \
  || oc wait --for=jsonpath='{.status.phase}'=Failed pod/"$POD" -n "$NS" --timeout=3s >/dev/null 2>&1 || true
echo "────────────────────────────────────────────────────────────────────"
oc logs "$POD" -n "$NS"; echo "────────────────────────────────────────────────────────────────────"
oc delete pod "$POD" -n "$NS" >/dev/null 2>&1 || true
EOF
chmod +x ask-agent
export PATH="$PWD:$PATH"
echo "ask-agent ready"
----

You don't need to read every line. The wiring is the lesson, and it's in the `spec`: the agent pod gets the *model* endpoint and key (`envFrom` the `maas-config-env` ConfigMap and the `maas-credentials` Secret) and the *MCP server* address (`MCP_SERVER_URL`) — *and nothing else*. No kubeconfig, no cluster token it uses. Its only route to the cluster is asking the MCP server, which acts as `mcp-agent`.

[NOTE]
.If an "ask" fails fast instead of thinking for a minute
====
A genuine model round-trip takes up to ~a minute, and `ask-agent` waits up to ~3 minutes for the one-shot pod to finish. But if the agent *image can't be pulled*, the helper's pre-check catches the `ImagePullBackOff` in the first few seconds and *fails fast* — `agent image could not be pulled` — rather than leaving you staring at a 3-minute hang. If you see that, the problem is image/registry access, not the model: check `echo $AGENT_CLI_IMAGE` and `oc get pod -l app=mcp-agent-cli-run -n $NS`.
====

[.checkpoint]
****
✔ *Verify:* Your namespace has a healthy `kubernetes-mcp-server` (running as the *scoped* `mcp-agent` ServiceAccount, whose only grant is the read-only `view` role in *this* namespace — no secrets, no other namespace) and a broken `parasol-claims` (`0/1`). The `ask-agent` helper runs the neutral client, which holds *no kubeconfig* — it reaches the cluster only through the MCP server.
****

== 2. Diagnose the broken deployment — read-only
⏱ *~15 min* · watch the agent call the cluster, then verify its claims yourself

Ask the agent why the claims service is broken. Phrase it the way you'd brief a junior engineer — *tell it what to inspect*, and it will call the read tools in order:

[source,sh,role=execute]
----
ask-agent "Inspect namespace $NS to find out why the parasol-claims pod is Running but not Ready. List the pods in that namespace, get the parasol-claims deployment, and read the recent events. Then explain the root cause in one or two sentences, quoting the readinessProbe path."
----

.Expected output (the tool-call trace — three real calls — then a grounded answer)
[source,texinfo]
----
mcp-agent-cli
  model:     <your cluster's configured model>
  mcp server: http://kubernetes-mcp-server:8080/sse  (1 wired)
  mode:      READ-ONLY (mutating tools hidden from the model)

TOOL-CALL TRACE (3 calls)
  [1] pods_list_in_namespace
      args:   {"namespace": "user3-dev"}
      result:
        ... parasol-claims-665b7fcfc8-...   0/1   Running ...
  [2] resources_get
      args:   {"apiVersion": "apps/v1", "kind": "Deployment", "name": "parasol-claims", "namespace": "user3-dev"}
      result: {... "readinessProbe":{"httpGet":{"path":"/q/health/reddy","port":8080} ...}
  [3] events_list
      args:   {"namespace": "user3-dev"}
      result: ... Readiness probe failed: HTTP probe failed with statuscode: 404 ...

read-only: no mutating tools were offered by the server.

ANSWER
  The parasol-claims pod is Running but not Ready because its readiness probe is failing
  with a 404 status code at the path /q/health/reddy. The application is not serving that
  endpoint; the readinessProbe path /q/health/reddy is wrong.
----

Two things just happened, and both matter:

* *The agent called the cluster, it did not guess.* The `TOOL-CALL TRACE` is the receipt — three read tools, with the exact arguments and the exact results. The answer is built from what those calls returned, not from the model's imagination.
* *It ran with eyes only.* `mode: READ-ONLY` and `no mutating tools were offered by the server` — the MCP server, started `--read-only`, advertised only read tools, so there was nothing to write with even if the model wanted to.

Now do the thing that separates a useful agent from a trusted one: *verify its claim yourself*.

The agent's claim checks out: the readiness probe points at `/q/health/reddy` (a typo for `/q/health/ready`), the app returns `404` there, so the pod never passes readiness and the Service has no endpoints. The agent found it read-only, and you confirmed it — that is the trust loop.

[NOTE]
.Grounding is engineered, not guaranteed
====
You *told* the agent what to inspect — "list the pods, get the deployment, read the events." That imperative phrasing is deliberate: models decide *whether* to call a tool, and a terse question like "why is parasol-claims broken?" can make some models *print a tool call as text* and answer from memory instead of actually calling it (you'd see `TOOL-CALL TRACE (0 calls)` and a bracketed tool name in the answer).
That is not the model being broken — it is a reminder that *grounding is something you engineer*, through clear phrasing, the system prompt, and (in production) tool-forcing settings. When an agent gives a grounded answer, someone made it ground. Try the terse version in the challenge and watch the difference.
====

== 3. Grant a scoped write, and let the agent fix it
⏱ *~18 min* · give it hands — the smallest hands that do the job

Diagnosing is safe; *fixing* means *writing*, and writing is a decision *you* make. You'll grant `mcp-agent` a *narrow* write, enable the write path, watch the client's read-only seatbelt, then let the agent patch the one bad field.

First, the human grant: a small `deployer` Role that can `patch`/`update` *deployments* — in *this* namespace, and nothing else. Not `edit` (which can touch secrets), not cluster-wide.

[source,sh,role=execute,subs="attributes"]
----
oc apply -f - <<EOF
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: mcp-agent-deployer
  namespace: {user}-dev
rules:
  - apiGroups: ["apps"]
    resources: ["deployments"]
    verbs: ["get", "list", "watch", "update", "patch"]   # no secrets, no scale, no delete
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: mcp-agent-deployer
  namespace: {user}-dev
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: Role
  name: mcp-agent-deployer
subjects:
  - kind: ServiceAccount
    name: mcp-agent
    namespace: {user}-dev
EOF
----

Confirm the grant did *exactly* what you intended — the agent's identity can now patch deployments, but *still* cannot read secrets:

[source,sh,role=execute]
----
echo "patch deployments: $(oc auth can-i patch deployments -n $NS --as=system:serviceaccount:$NS:mcp-agent 2>/dev/null)"
echo "read secrets:      $(oc auth can-i get secrets -n $NS --as=system:serviceaccount:$NS:mcp-agent 2>/dev/null)"
----

.Expected output (the write is granted; secrets stay off-limits)
[source,texinfo]
----
patch deployments: yes
read secrets:      no
----

Now *enable the write path* on the MCP server. It was started `--read-only`, so it doesn't even *advertise* mutating tools; drop that flag so the write tool exists:

[source,sh,role=execute]
----
oc patch deploy kubernetes-mcp-server -n $NS --type=json \
  -p '[{"op":"replace","path":"/spec/template/spec/containers/0/args","value":["--port","8080","--toolsets","core,config"]}]'
oc rollout status deploy/kubernetes-mcp-server -n $NS --timeout=90s
----

Before you allow writes, see the *client's* seatbelt for yourself. Ask a read question while the client is *still in its default read-only mode* — even though the server now offers mutating tools, the client *hides* them from the model:

[source,sh,role=execute]
----
ask-agent "List the pods in namespace $NS and report each pod name with its Ready status."
----

.Expected output (the client filtered out the mutating tools the server now offers — defense in depth)
[source,texinfo]
----
...
read-only: hid 6 mutating tools (RBAC remains the real boundary):
  - pods_delete  (matched 'delete')
  - pods_exec  (matched 'exec')
  - pods_run  (matched 'run')
  - resources_create_or_update  (matched 'create')
  - resources_delete  (matched 'delete')
  - resources_scale  (matched 'scale')
...
----

Read the parenthetical: `RBAC remains the real boundary`. The client hid those six tools as a *seatbelt* — but even if it hadn't, `mcp-agent`'s RBAC only permits patching *deployments*, so a `pods_delete` would still be a `403`. The flag is defense in depth; the identity is the boundary.

Now *allow writes* and let the agent fix the probe:

[source,sh,role=execute]
----
ask-agent --allow-writes "The parasol-claims deployment in namespace $NS has a wrong readinessProbe.httpGet.path of /q/health/reddy. Get the deployment, update it so the readinessProbe path is /q/health/ready, apply the change, and confirm."
----

.Expected output (writes allowed; on a tool-calling-capable model the agent applies the corrected deployment through its scoped identity — if yours *narrates* the change instead, see the note below)
[source,texinfo]
----
mcp-agent-cli
  model:     <your cluster's configured model>
  mcp server: http://kubernetes-mcp-server:8080/sse  (1 wired)
  mode:      WRITES ALLOWED (mutating tools offered; RBAC is the boundary)

TOOL-CALL TRACE (2 calls)
  [1] resources_get
      args:   {"apiVersion": "apps/v1", "kind": "Deployment", "name": "parasol-claims", ...}
      result: {... "readinessProbe":{"httpGet":{"path":"/q/health/reddy",...
  [2] resources_create_or_update
      args:   {"resource": "apiVersion: apps/v1\nkind: Deployment\n... path: /q/health/ready ..."}
      result: # The following resources (YAML) have been created or updated successfully ...

ANSWER
  Updated the parasol-claims deployment: readinessProbe path is now /q/health/ready. The
  rollout is progressing and the pod should become Ready.
----

[IMPORTANT]
.When the agent doesn't land the fix — your manual fallback (and today's second lesson)
====
The agent path can come up short in more than one way, and you should be ready for all of them:

* *The agent narrates instead of acting.* On some models (this cluster's included), the agent *prints* the `resources_create_or_update(...)` call as *text* and makes *zero* tool calls — it proposed the exact right patch but didn't execute it. That is the same "grounding is engineered" reality from exercise 2.
* *The model endpoint is unavailable.* If the MaaS endpoint or key is down, `ask-agent` comes back with an error like `model authentication failed` and no answer at all.
* *The agent pod can't run.* If its image won't pull, `ask-agent` fails fast (see the note in exercise 1) instead of hanging.

In *every* one of these cases you are not blocked, because you already *diagnosed* the fault yourself in exercise 2. You apply the reviewed change directly — using the very write you granted:
[source,sh,role=execute]
----
oc patch deploy parasol-claims -n $NS --type=json \
  -p '[{"op":"replace","path":"/spec/template/spec/containers/0/readinessProbe/httpGet/path","value":"/q/health/ready"}]'
----
The manual fallback isn't a workaround for a chatty model — it is how you keep making progress *whenever the agent path is unavailable*, whether the agent narrated, the model endpoint is down, or the one-shot pod never ran. The point stands either way: the write is *permitted* because you granted the scoped role, and a human stayed in the loop for the change.
====

[source,sh,role=execute]

oc get deploy parasol-claims -n $NS -o jsonpath='{range .metadata.managedFields[*]}{.manager}{" — op="}{.operation}{"\n"}{end}'

.Expected output (the managedFields name every writer; the last one is whoever applied the fix — here, the `oc` fallback)
[source,texinfo]
----

kubectl-patch — op=Update
----

The base managers (`argocd-controller`, which materialized the deployment, and `kube-controller-manager`) are always present; the *last* manager is the one that mattered — it records *who* applied your fix:

* *If the agent applied it* (a tool-calling model, with writes allowed), that line reads `kubernetes-mcp-server — op=Apply` — the agent's write, made through the MCP server's *scoped identity* and recorded by the API server itself.
* *If you applied it with the `oc patch` fallback above* (as you did here, because this cluster's model narrated the change), the writer is your own client — `kubectl-patch — op=Update`, attributed to *you*.

Either way the point holds: between the *tool-call trace* (what the agent tried) and the *managedFields* (what actually changed the object), *every write is accountable* — the platform records who wrote, whether that was the agent or you.

== 4. The forbidden test — watch RBAC deny the agent
⏱ *~10 min* · the whole module in one denial

The agent can now read and patch *your* namespace. Point it at *someone else's* and watch the platform stop it.
Writes are still allowed and the server still offers every tool — the only thing standing between the agent and a neighbour's namespace is *RBAC on `mcp-agent`*.

[source,sh,role=execute]
----
ask-agent "Find out what is running in the neighbouring namespace user1-dev. List the pods in namespace user1-dev, and also read the recent events in namespace user1-dev. Then report what you found."
----

.Expected output (the agent tries — and RBAC returns `is forbidden` on every call)
[source,texinfo]
----
TOOL-CALL TRACE (2 calls)
  [1] pods_list_in_namespace
      args:   {"namespace": "user1-dev"}
      result: failed to list pods in namespace user1-dev: pods is forbidden: User
        "system:serviceaccount:user3-dev:mcp-agent" cannot list resource "pods" in API group ""
        in the namespace "user1-dev"
  [2] events_list
      args:   {"namespace": "user1-dev"}
      result: failed to list events ... events is forbidden: User
        "system:serviceaccount:user3-dev:mcp-agent" cannot list resource "events" ...
        in the namespace "user1-dev"

ANSWER
  I was not able to get the list of pods or events in namespace user1-dev due to access
  restrictions. The MCP agent has limited access and is not allowed to list pods or events
  in other namespaces.
----

That denial *is* the lesson. Read what happened:

* *The agent genuinely tried.* The trace shows it *called* `pods_list_in_namespace` on `user1-dev` — this was not the client refusing on the agent's behalf.
* *The API server said no.* `pods is forbidden: User "system:serviceaccount:user3-dev:mcp-agent" cannot list resource "pods" ... in the namespace "user1-dev"`. `mcp-agent` has a `view` RoleBinding in *your* namespace and *nowhere else*, so a read of `user1-dev` is a `403` — enforced by the platform, underneath the agent.
* *The agent reported the denial honestly*, exactly as its instructions require, and did not try to work around it.

Confirm the boundary from the other side — you don't need the agent to prove RBAC; ask the API server directly with impersonation:

[source,sh,role=execute]
----
SA=system:serviceaccount:$NS:mcp-agent
echo "read pods in YOUR ns:        $(oc auth can-i get pods -n $NS --as=$SA 2>/dev/null)"
echo "read pods in a NEIGHBOUR ns: $(oc auth can-i get pods -n user1-dev --as=$SA 2>/dev/null)"
echo "read cluster nodes:          $(oc auth can-i get nodes --as=$SA 2>/dev/null)"
----

.Expected output (scoped to one namespace: yes at home, no next door, no cluster-wide)
[source,texinfo]
----
read pods in YOUR ns:        yes
read pods in a NEIGHBOUR ns: no
read cluster nodes:          no
----

This is why the design scopes with a *RoleBinding* (namespaced), never a `ClusterRoleBinding` or `cluster-reader`. On a shared cluster with many teams, the agent's blast radius is *exactly* the namespace you put it in. Trust didn't stop it from reading a neighbour's namespace — *the platform did*.

[.checkpoint]
****
✔ *Verify:* You asked the agent to read a *neighbouring* namespace; its trace shows it *called* the read tools and the API server returned `is forbidden` for `system:serviceaccount:{user}-dev:mcp-agent`, and the agent reported the denial. `oc auth can-i --as` confirms the scope: `yes` in your namespace, `no` next door, `no` cluster-wide. RBAC — not the read-only flag — is the boundary that held.
****
