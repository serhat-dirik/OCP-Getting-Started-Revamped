== 1. Scaffold a chart with `helm create`
⏱ *~8 min* · read what a chart actually is

`helm create` scaffolds a working chart you then shape to your app.
Generate one for the notifications service and read the tree:

[source,sh,role=execute]
----
helm create parasol-notifications
find parasol-notifications -type f | sort
----

.Expected output (a complete, installable chart skeleton)
[source,texinfo]
----
parasol-notifications/.helmignore
parasol-notifications/Chart.yaml
parasol-notifications/templates/NOTES.txt
parasol-notifications/templates/_helpers.tpl
parasol-notifications/templates/deployment.yaml
parasol-notifications/templates/hpa.yaml
parasol-notifications/templates/ingress.yaml
parasol-notifications/templates/service.yaml
parasol-notifications/templates/serviceaccount.yaml
parasol-notifications/templates/tests/test-connection.yaml
parasol-notifications/values.yaml
----

Four pieces carry the whole idea:

* *`Chart.yaml`* — the chart's identity: its `name`, its `version` (the chart's own semver), and `appVersion` (the app it ships).
* *`values.yaml`* — the *inputs*. Everything a consumer might change — image, replica count, ports, probes — has a default here, overridable at install with `--set` or `-f`.
* *`templates/`* — Go-templated Kubernetes manifests. `{{ .Values.image.repository }}` in a template pulls from `values.yaml`. `_helpers.tpl` holds reusable naming/label snippets.
* *`templates/tests/`* — a `helm test` hook (exercise 5) that proves a release actually works.

The skeleton ships an `nginx` Deployment on port 80 with an Ingress — a generic starting point.
Your job in the next exercise is to make it *your* app.

[.checkpoint]
****
✔ *Verify:* `helm create` produced a `parasol-notifications/` chart with `Chart.yaml`, `values.yaml`, and a `templates/` directory (Deployment, Service, Ingress, NOTES, `_helpers.tpl`, a test) — a complete, installable skeleton you will now shape to the notifications service.
****

== 2. Template the notifications service
⏱ *~12 min* · values drive the manifests — image, port, probes, env, an OpenShift Route

Shape the chart to the real app.
The prebuilt notifications image sits in *your* registry namespace as an ImageStream tag — the entry state records it in a marker.
First set the chart's *values* — image, the service's real port (`8080`), health probes on `/health`, a small resource footprint, the `SITE` env marker, and a flag to publish an OpenShift Route:

[source,sh,role=execute]
----
cd parasol-notifications
cat > values.yaml <<EOF
replicaCount: 1

image:
  # The prebuilt notifications image seeded into your namespace as an ImageStream tag.
  repository: image-registry.openshift-image-registry.svc:5000/$NS/parasol-notifications
  pullPolicy: IfNotPresent
  tag: "1.0"

imagePullSecrets: []
nameOverride: ""
fullnameOverride: ""
serviceAccount: {create: true, automount: true, annotations: {}, name: ""}
podAnnotations: {}
podLabels: {}
podSecurityContext: {}
securityContext: {}

service:
  type: ClusterIP
  port: 8080          # the service listens on 8080; containerPort follows this value

route:
  enabled: true       # publish an OpenShift edge Route (replaces the stock Ingress)

site: "$NS"           # origin-site marker the service echoes on GET /

resources:
  requests: {cpu: 50m, memory: 64Mi}
  limits: {cpu: 200m, memory: 128Mi}

livenessProbe:
  httpGet: {path: /health, port: http}
readinessProbe:
  httpGet: {path: /health, port: http}

autoscaling: {enabled: false, minReplicas: 1, maxReplicas: 100, targetCPUUtilizationPercentage: 80}
nodeSelector: {}
tolerations: []
affinity: {}
EOF
echo "values.yaml written"
----

The image and probe defaults now describe the notifications service, not nginx.
Two template edits finish the job.
First, the app reads a `SITE` env var — add it to the container by editing `templates/deployment.yaml` so the container block carries an `env`:

[source,sh,role=execute]
----
cat > templates/deployment.yaml <<'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "parasol-notifications.fullname" . }}
  labels:
    {{- include "parasol-notifications.labels" . | nindent 4 }}
spec:
  {{- if not .Values.autoscaling.enabled }}
  replicas: {{ .Values.replicaCount }}
  {{- end }}
  selector:
    matchLabels:
      {{- include "parasol-notifications.selectorLabels" . | nindent 6 }}
  template:
    metadata:
      labels:
        {{- include "parasol-notifications.labels" . | nindent 8 }}
    spec:
      serviceAccountName: {{ include "parasol-notifications.serviceAccountName" . }}
      containers:
        - name: {{ .Chart.Name }}
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag | default .Chart.AppVersion }}"
          imagePullPolicy: {{ .Values.image.pullPolicy }}
          env:
            - name: SITE
              value: {{ .Values.site | quote }}
          ports:
            - name: http
              containerPort: {{ .Values.service.port }}
              protocol: TCP
          {{- with .Values.livenessProbe }}
          livenessProbe:
            {{- toYaml . | nindent 12 }}
          {{- end }}
          {{- with .Values.readinessProbe }}
          readinessProbe:
            {{- toYaml . | nindent 12 }}
          {{- end }}
          {{- with .Values.resources }}
          resources:
            {{- toYaml . | nindent 12 }}
          {{- end }}
EOF
echo "deployment.yaml updated (SITE env + values-driven probes)"
----

Second — and this is the OpenShift-specific move — *replace the stock Ingress with an edge Route*.
An Ingress is generic Kubernetes; on OpenShift a browser-facing app wants a *Route*, and an *edge* Route with `insecureEdgeTerminationPolicy: Allow` serves the landing page cleanly over both HTTP and HTTPS:

[source,sh,role=execute]
----
rm -f templates/ingress.yaml
cat > templates/route.yaml <<'EOF'
{{- if .Values.route.enabled }}
apiVersion: route.openshift.io/v1
kind: Route
metadata:
  name: {{ include "parasol-notifications.fullname" . }}
  labels:
    {{- include "parasol-notifications.labels" . | nindent 4 }}
spec:
  to:
    kind: Service
    name: {{ include "parasol-notifications.fullname" . }}
  port:
    targetPort: http
  tls:
    termination: edge
    insecureEdgeTerminationPolicy: Allow
{{- end }}
EOF
echo "ingress.yaml removed, route.yaml added"
----

[NOTE]
.Why a Route, not an Ingress
====
A plain HTTP Route (or `oc expose`) gets HSTS-upgraded by the router and can dead-end in the browser; an *edge* Route with `insecureEdgeTerminationPolicy: Allow` terminates TLS at the router and serves both schemes.
This is the workshop-wide rule for any browser-facing app — the chart bakes it in so every install gets it right.
====

Now *lint* the chart (catches structural mistakes) and *render* it locally without installing (so you see exactly what will be applied):

[source,sh,role=execute]
----
cd ..
helm lint parasol-notifications
----

.Expected output (a clean chart; the icon note is advisory)
[source,texinfo]
----
==> Linting parasol-notifications

1 chart(s) linted, 0 chart(s) failed
----

[source,sh,role=execute]
----
helm template parasol-notifications ./parasol-notifications | grep -E 'kind:|image:|containerPort|path: /health|name: SITE|termination:'
----

.Expected output (your values, rendered into real manifests — image, port 8080, /health probes, SITE, an edge Route)
[source,texinfo]
----
kind: ServiceAccount
kind: Service
kind: Deployment
          image: "image-registry.openshift-image-registry.svc:5000/user2-dev/parasol-notifications:1.0"
            - name: SITE
              containerPort: 8080
              path: /health
              path: /health
kind: Route
    termination: edge
----

`helm template` is the render-without-installing button: what you see is what `helm install` will apply.
The image points at *your* registry, the port is `8080`, the probes hit `/health`, the `SITE` env is wired, and the Ingress is gone — replaced by an edge Route.

[.checkpoint]
****
✔ *Verify:* Your `values.yaml` drives the image (your namespace's `parasol-notifications:1.0`), `service.port: 8080`, `/health` probes, and the `SITE` marker; the Deployment template carries the `SITE` env; the Ingress is replaced by an edge `Route`. `helm lint` passes and `helm template` renders all of it — the chart now describes *your* app, not nginx.

== 4. Upgrade, break it on purpose, and roll back
⏱ *~15 min* · a release has history — and an undo

A packaged app has a *lifecycle*.
Change an input and `helm upgrade` — here, scale to two replicas:

[source,sh,role=execute]
----
helm upgrade parasol-notifications ./parasol-notifications -n $NS --set replicaCount=2
oc rollout status deploy/parasol-notifications -n $NS --timeout=60s
helm history parasol-notifications -n $NS
----

.Expected output (revision 2, and a history you can audit)
[source,texinfo]
----
deployment "parasol-notifications" successfully rolled out
REVISION  UPDATED                   STATUS      CHART                        APP VERSION  DESCRIPTION
1         Thu Jul 16 00:21:03 2026  superseded  parasol-notifications-0.1.0  1.0          Install complete
2         Thu Jul 16 00:21:40 2026  deployed    parasol-notifications-0.1.0  1.0          Upgrade complete
----

Two revisions, each recorded — *which* config is live is now a fact, not a memory.

*Now the deliberate break.* Ship a bad config the way a real mistake arrives: point the probes at a health path the app doesn't serve (`/healthz` instead of `/health`).
`--wait` tells Helm to *hold the release open until the new pods are healthy* — so a bad probe makes the upgrade *fail loudly* instead of silently:

[source,sh,role=execute]
----
helm upgrade parasol-notifications ./parasol-notifications -n $NS \
  --set replicaCount=2 \
  --set 'livenessProbe.httpGet.path=/healthz' \
  --set 'readinessProbe.httpGet.path=/healthz' \
  --wait --timeout=90s
----

.Expected output (Helm waited, the probe never passed, the upgrade is refused)
[source,texinfo]
----
Error: UPGRADE FAILED: context deadline exceeded
----

That is the safety net working.
Look at what actually happened — the new pod never became ready, but the *old* pods kept serving:

[source,sh,role=execute]
----
oc get pods -n $NS -l app.kubernetes.io/instance=parasol-notifications
oc get events -n $NS --field-selector reason=Unhealthy | grep 404 | tail -2
----

.Expected output (the new pod fails its probe with 404; the old pods stay Ready)
[source,texinfo]
----
NAME                                     READY   STATUS    RESTARTS      AGE
parasol-notifications-79cbd49cc6-q727r   0/1     Running   3 (12s ago)   103s
parasol-notifications-7f69567464-h5hf5   1/1     Running   0             2m49s
parasol-notifications-7f69567464-l5cwz   1/1     Running   0             4m31s
...  Liveness probe failed: HTTP probe failed with statuscode: 404
...  Readiness probe failed: HTTP probe failed with statuscode: 404
----

The `/healthz` path returns 404, so the new pod (`79cbd49cc6-…`) never passes its probe and the rolling update *stalls* — it will not retire the healthy old pods (`7f69567464-…`) for a pod that isn't ready.
Prove the app never went down:

[source,sh,role=execute]
----
curl -sk "https://$ROUTE/health"; echo
----

.Expected output (still serving — no outage during the failed upgrade)
[source,texinfo]
----
{"status":"UP"}
----

`helm history` now shows revision 3 as `failed`.
*Roll back* to the last good revision — one command, and you're recovered.

====

CLI::

====

.Expected output (rolled back; a new revision 4 that is a copy of the good revision 2)
[source,texinfo]

deployment "parasol-notifications" successfully rolled out
2  Thu Jul 16 00:21:40 2026  superseded  parasol-notifications-0.1.0  1.0  Upgrade complete
3  Thu Jul 16 00:22:11 2026  failed      parasol-notifications-0.1.0  1.0  Upgrade "parasol-notifications" failed: context deadline exceeded

Revision 4 is the *good* config again — Helm rolls *forward to a copy of* revision 2 rather than deleting history, so the failed revision 3 stays on the record.
That is the whole value of a packaged release: the bad change was caught by probes, never took traffic, and the undo was one command.

== 5. Test the release and understand hooks
⏱ *~7 min* · a chart that proves itself, and lifecycle actions

A chart can run actions at defined points in a release's life — *lifecycle hooks* — and prove itself with a *test*.
First, add a *post-install / post-upgrade hook* that seeds a welcome notification once the service is up.
Two annotations turn an ordinary Job into a hook:

[source,sh,role=execute]
----
cat > parasol-notifications/templates/hooks-seed.yaml <<'EOF'
apiVersion: batch/v1
kind: Job
metadata:
  name: "{{ include "parasol-notifications.fullname" . }}-seed"
  labels:
    {{- include "parasol-notifications.labels" . | nindent 4 }}
  annotations:
    "helm.sh/hook": post-install,post-upgrade
    "helm.sh/hook-delete-policy": before-hook-creation,hook-succeeded
spec:
  backoffLimit: 3
  activeDeadlineSeconds: 120
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: seed
          image: registry.access.redhat.com/ubi9/ubi-minimal:latest
          command: ["/bin/sh", "-c"]
          args:
            - |
              set -eu
              BASE="http://{{ include "parasol-notifications.fullname" . }}:{{ .Values.service.port }}"
              for i in $(seq 1 30); do curl -sf "$BASE/health" >/dev/null 2>&1 && break; sleep 2; done
              curl -sf -X POST "$BASE/api/notify" -H 'content-type: application/json' \
                -d '{"claimNumber":"CLM-0000","message":"notifications service online"}'
              echo "seeded welcome notification"
EOF
echo "seed hook added"
----

`helm.sh/hook: post-install,post-upgrade` runs the Job *after* every install and upgrade (not as part of the release's normal objects); `hook-delete-policy` cleans the Job up when it succeeds.
Now *add a test* — a Pod under `templates/tests/` that `helm test` runs on demand, here a real `notify` → `notifications` round-trip:

[source,sh,role=execute]
----
cat > parasol-notifications/templates/tests/test-connection.yaml <<'EOF'
apiVersion: v1
kind: Pod
metadata:
  name: "{{ include "parasol-notifications.fullname" . }}-test"
  labels:
    {{- include "parasol-notifications.labels" . | nindent 4 }}
  annotations:
    "helm.sh/hook": test
spec:
  restartPolicy: Never
  containers:
    - name: api-test
      image: registry.access.redhat.com/ubi9/ubi-minimal:latest
      command: ["/bin/sh", "-c"]
      args:
        - |
          set -eu
          BASE="http://{{ include "parasol-notifications.fullname" . }}:{{ .Values.service.port }}"
          echo "1) health check"
          curl -sf "$BASE/health" | grep -q '"status":"UP"'
          echo "2) POST a notification"
          curl -sf -X POST "$BASE/api/notify" -H 'content-type: application/json' \
            -d '{"claimNumber":"CLM-TEST","message":"helm test round-trip"}' | grep -q 'CLM-TEST'
          echo "3) read it back"
          curl -sf "$BASE/api/notifications" | grep -q 'CLM-TEST'
          echo "PASS: health + notify round-trip"
EOF
echo "test added"
----

Upgrade to apply both — the *post-upgrade hook fires* and seeds the welcome notification into the store that was empty in exercise 3:

[source,sh,role=execute]
----
helm upgrade parasol-notifications ./parasol-notifications -n $NS >/dev/null && echo "upgraded"
curl -sk "https://$ROUTE/api/notifications" | grep -o 'CLM-0000'
----

.Expected output (the hook ran — the store now carries the seeded welcome)
[source,texinfo]
----
upgraded
CLM-0000
----

Now run the *test* — Helm launches the test Pod and reports pass/fail:

[source,sh,role=execute]
----
helm test parasol-notifications -n $NS --logs
----

.Expected output (the test Pod runs the round-trip and passes)
[source,texinfo]

...

`Phase: Succeeded` means the release is not just *running* but *working* — health plus a real write-then-read a consumer can run after *their* install, on *their* cluster.
`helm.sh/hook: test` (the test Pod) and `helm.sh/hook: post-install,post-upgrade` (the seed Job) are the *same mechanism* at different moments.

== 6. Package and push the chart as an OCI artifact
⏱ *~12 min* · a chart is a versioned artifact you can store in the registry

A chart you can `install` from a *directory* is useful; a chart you can *distribute* is a product.
`helm package` bundles the chart into a versioned `.tgz`:

[source,sh,role=execute]
----
helm package parasol-notifications
ls parasol-notifications-*.tgz
----

.Expected output (a versioned chart archive, named `chart-version.tgz`)
[source,texinfo]
----
Successfully packaged chart and saved it to: .../parasol-notifications-0.1.0.tgz
parasol-notifications-0.1.0.tgz
----

Modern Helm speaks *OCI* — the same artifact format as container images — so a chart can live in any OCI registry, including the cluster's *internal registry*.
The registry is an in-cluster Service (`image-registry.openshift-image-registry.svc:5000`) with a certificate signed by the cluster's *service CA*, so you log in with your OpenShift token and trust that CA.
Grab the CA bundle, log in, and push:

[source,sh,role=execute]
----
oc get configmap openshift-service-ca.crt -n $NS -o jsonpath='{.data.service-ca\.crt}' > /tmp/service-ca.crt
REG=image-registry.openshift-image-registry.svc:5000
oc whoami -t | helm registry login $REG -u $(oc whoami) --password-stdin --ca-file /tmp/service-ca.crt
----

.Expected output
[source,texinfo]
----
Login Succeeded
----

Now push the packaged chart.
The internal registry maps every OCI repository to *`<namespace>/<name>`*, so you push into *your* namespace and the chart lands as a tag there:

[source,sh,role=execute]
----
helm push parasol-notifications-0.1.0.tgz oci://$REG/$NS --ca-file /tmp/service-ca.crt
----

.Expected output (pushed as an OCI artifact; note the sha256 digest)
[source,texinfo]
----
Pushed: image-registry.openshift-image-registry.svc:5000/user2-dev/parasol-notifications:0.1.0
Digest: sha256:ab9cc8033b1bf5be306912262264ea43626b49c37ba2f6e48b5e75b8c637204a
----

The chart is now a *`0.1.0` tag* on a `parasol-notifications` repository in your namespace — an OCI artifact sitting alongside the app image (which is tag `1.0` on the same path), distinguished by its version tag and artifact type.

[NOTE]
.The integrated registry uses `<namespace>/<name>` — no deeper paths
====
On a general-purpose OCI registry (Quay, Harbor) you would push charts under a tidy prefix like `oci://registry/{user}-dev/charts`.
The OpenShift *integrated* registry maps repositories to ImageStreams — exactly `<namespace>/<name>` — so a deeper `.../charts/parasol-notifications` path is *rejected* (a `401` from the registry's token endpoint).
Push to `oci://<registry>/<your-namespace>` and let the chart's own name and version tag keep it distinct from the app image.
====

*Prove the round-trip* — pull the chart back and confirm the digest matches what you pushed:

[source,sh,role=execute]
----
mkdir -p /tmp/pulled && cd /tmp/pulled
helm pull oci://$REG/$NS/parasol-notifications --version 0.1.0 --ca-file /tmp/service-ca.crt
cd - >/dev/null
----

.Expected output (the same sha256 digest — a verified round-trip)
[source,texinfo]
----
Pulled: image-registry.openshift-image-registry.svc:5000/user2-dev/parasol-notifications:0.1.0
Digest: sha256:ab9cc8033b1bf5be306912262264ea43626b49c37ba2f6e48b5e75b8c637204a
----

Push digest and pull digest match — the artifact in the registry is exactly the chart you packaged.
Anyone with pull access to your namespace can now `helm install oci://$REG/$NS/parasol-notifications --version 0.1.0` — install-from-registry, no chart directory needed.

[NOTE]
.Your digest will differ — that's the point
====
A `sha256` digest is a *content hash*: any change to the chart produces a different digest.
Yours will not be the `ab9cc8…` shown above, and that's correct — what matters is that *your* push digest and *your* pull digest are *identical*, proving the bytes made the round-trip intact.
====

[.checkpoint]
****
✔ *Verify:* `helm package` produced `parasol-notifications-0.1.0.tgz`; you logged in to the internal registry with your OpenShift token and the service-CA bundle, `helm push`ed the chart as an OCI artifact to `<your-namespace>/parasol-notifications:0.1.0`, and `helm pull` returned the *same sha256 digest* — a verified distribution round-trip through the cluster registry.
****

== 7. Dissect a live operator's OLM bundle
⏱ *~15 min* · what actually happens when a customer clicks "Install"

Helm is one end of the packaging spectrum; an *operator* is the other.
You don't have to imagine one — the *Pipelines operator* that ran your builds all week is installed on this cluster, and you can read its bundle *read-only* (the `platform-observer` role grants cluster-wide *read*; you never mutate it).

First, a *trap* worth knowing: several APIs define a `Subscription` kind, so a bare `oc get subscription` resolves to the *wrong* one.

[source,sh,role=execute]
----
oc api-resources | awk 'NR==1 || tolower($0) ~ /subscription/'
----

.Expected output (more than one `subscriptions` kind — the bare name is ambiguous)
[source,texinfo]
----
NAME            SHORTNAMES   APIVERSION                      NAMESPACED   KIND
subscriptions   sub          messaging.knative.dev/v1        true         Subscription
subscriptions   sub,subs     operators.coreos.com/v1alpha1   true         Subscription
...
----

Always *fully-qualify* `subscriptions.operators.coreos.com` for OLM.
Read the Pipelines operator's Subscription — the "I want this operator, on this channel, from this catalog" declaration:

[tabs]
====
Console::
+
--
// [CAPTURE-VERIFY] Operators → Installed Operators → Red Hat OpenShift Pipelines → Subscription tab (project openshift-operators, OCP 4.21) shows Channel / Update approval / Catalog. CLI authoritative.
In the {ocp_console_url}[web console^], open *Operators* → *Installed Operators*, select *Red Hat OpenShift Pipelines*, and open its *Subscription* tab.
It shows the *Update channel* (`latest`), the *Update approval* (Automatic), and the *Catalog source* (Red Hat Operators) — the same fields the CLI prints.
--
CLI::
+
--
[source,sh,role=execute]
----
oc get subscriptions.operators.coreos.com -n openshift-operators openshift-pipelines-operator-rh
----
--
====

.Expected output (the Subscription: package, catalog source, chosen channel)
[source,texinfo]
----
NAME                              PACKAGE                           SOURCE             CHANNEL
openshift-pipelines-operator-rh   openshift-pipelines-operator-rh   redhat-operators   latest
----

The *channels* are the operator's upgrade streams — you subscribe to one.
List them from the PackageManifest (using `--field-selector` to avoid another bare-name trap):

[source,sh,role=execute]
----
oc get packagemanifests --field-selector=metadata.name=openshift-pipelines-operator-rh -n openshift-marketplace -o json \
  | jq -r '.items[0].status | "default channel: \(.defaultChannel)", "channels: \([.channels[].name] | join(", "))"'
----

.Expected output (a default channel plus version-pinned streams)
[source,texinfo]
----
default channel: latest
channels: latest, pipelines-1.15, pipelines-1.16, pipelines-1.17, pipelines-1.18, pipelines-1.19, pipelines-1.20, pipelines-1.21, pipelines-1.22
----

Now the *heart of the bundle* — the *ClusterServiceVersion (CSV)*.
The Subscription resolved to an installed CSV; read what it declares:

[source,sh,role=execute]
----
CSV=$(oc get subscriptions.operators.coreos.com -n openshift-operators openshift-pipelines-operator-rh -o jsonpath='{.status.installedCSV}')
echo "installed CSV: $CSV"
oc get csv "$CSV" -n openshift-operators -o json | jq -r '
  "owned CRDs: \(.spec.customresourcedefinitions.owned | length)",
  "install strategy: \(.spec.install.strategy)",
  "install modes: \([.spec.installModes[] | select(.supported) | .type] | join(", "))"'
----

.Expected output (the CSV: the CRDs it owns, the controller it installs, its scope)
[source,texinfo]
----
installed CSV: openshift-pipelines-operator-rh.v1.22.4
owned CRDs: 14
install strategy: deployment
install modes: AllNamespaces
----

A CSV declares the *custom resource definitions the operator owns*, the *controller Deployment* to run (`install strategy: deployment`), and its *install mode*.
See the CRDs it owns — these are the API types the operator adds to the cluster:

[source,sh,role=execute]
----
oc get csv "$CSV" -n openshift-operators -o jsonpath='{range .spec.customresourcedefinitions.owned[*]}{.kind}{"\n"}{end}' | head -6
echo "--- and now live on the cluster: ---"
oc get crd | grep -c tekton
----

.Expected output (owned API types; and the CRDs the operator installed cluster-wide)
[source,texinfo]
----
ManualApprovalGate
OpenShiftPipelinesAsCode
TektonChain
TektonConfig
TektonPipeline
TektonTrigger
--- and now live on the cluster: ---
30
----

Those `Tekton…` kinds are what let you write a `Pipeline` or a `TektonConfig` — *the operator extended the Kubernetes API*.
Finally, trace *where the bundle came from* and *how it was approved*:

[source,sh,role=execute]
----
oc get catalogsource redhat-operators -n openshift-marketplace -o jsonpath='catalog: {.spec.displayName}  image: {.spec.image}{"\n"}'
oc get installplan -n openshift-operators -o custom-columns='NAME:.metadata.name,APPROVAL:.spec.approval,APPROVED:.spec.approved,CSV:.spec.clusterServiceVersionNames[0]' | grep pipelines
----

.Expected output (the catalog index image, and the InstallPlan that approved the install)
[source,texinfo]
----
catalog: Red Hat Operators  image: registry.redhat.io/redhat/redhat-operator-index:v4.21
install-hrcwp   Automatic   true   openshift-pipelines-operator-rh.v1.22.4
----

That is the whole chain, read backwards from a running operator: a *CatalogSource* (an index image) offered the package; a *Subscription* picked the `latest` channel; an *InstallPlan* approved it *automatically*; a *CSV* installed the controller and 14 owned CRDs.
*That is what happened when someone clicked "Install."*

== 8. Where OLM is going, and map your product onto the spectrum
⏱ *~8 min* · the direction of travel, then a decision with a straight face

Everything in exercise 7 was *classic OLM* — the model that installed every operator on this cluster.
A newer generation, *OLM v1*, is generally available and *running here*, reshaped into two cluster-scoped objects.
Read them:

[source,sh,role=execute]
----
oc get clustercatalog
echo "--- installed the new way: ---"
oc get clusterextension
----

.Expected output (the new catalogs are serving; nothing is installed through them yet)
[source,texinfo]
----
NAME                            LASTUNPACKED   SERVING   AGE
openshift-certified-operators   ...            True      ...
openshift-community-operators   ...            True      ...
openshift-redhat-marketplace    ...            True      ...
openshift-redhat-operators      ...            True      ...
--- installed the new way: ---
No resources found
----

Four *ClusterCatalogs* are *serving*, but there are *zero ClusterExtensions* — OLM v1 is present and ready, yet nothing uses it here and the current web console does not surface it.
Classic OLM (a `Subscription` → `CSV`) is what drives OperatorHub today and what you dissected; OLM v1 (a `ClusterCatalog` + a `ClusterExtension` with a ServiceAccount and RBAC you supply) is the *direction*.
Reading `oc get clustercatalog` returning four `Serving` entries is enough to know it is real and coming.

*Now the decision, made honestly.*
You have now seen both ends of the spectrum — you *shipped a Helm chart* and *dissected an operator*.
For a service you own, place it:

[cols="1,3,3",options="header"]
|===
| Package | Day-2 it covers | Reach for it when…

| Raw manifests | none — you own every change | it's a throwaway or a one-off
| Kustomize | environment overlays, GitOps diffs | Git is your control plane and there's no release lifecycle
| *Helm* | install / upgrade / rollback / test / hooks | you want *versioned releases* with an undo — *most apps*
| Operator | *continuous reconcile* — repair, stateful upgrade, failover | the *reconcile itself is the value*, not just install/upgrade
|===

[NOTE]
.The honest line: when *not* to write an operator
====
If your day-2 is "occasionally `helm upgrade` with new values," a *chart already does that* — an operator would be a reconcile loop with nothing to reconcile, plus a controller to build, certify, and keep alive for the product's life.
Write the operator when it must *continuously* act — repair drift the instant it happens, orchestrate a stateful upgrade, fail over across sites, or manage many instances from one spec.
The notifications service you packaged today wants *Helm*, and saying so out loud is the senior move.
====

****

You can now choose template vs Kustomize vs Helm vs operator with a straight face.
