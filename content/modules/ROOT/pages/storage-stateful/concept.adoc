[.lead]
Parasol's claims database has been running in `{user}-dev` all week, and nobody noticed it was a time bomb.
Its data lives in an *emptyDir* volume — memory-and-disk that belongs to the Pod — so the first time that Pod is rescheduled, upgraded, or simply dies, every claim vanishes.
This module is about the difference between storage that dies with a Pod and storage that outlives it, and about running databases on the platform honestly.

== What you'll learn

* The *storage abstraction chain* — PersistentVolumeClaim, PersistentVolume, StorageClass — and how *dynamic provisioning* turns a one-line request into real disk.
* What *access modes* (RWO, RWX, ROX) mean, and which ones your cluster's storage actually offers — measured, not assumed.
* Why a *StatefulSet* is not just "a Deployment that keeps its data": stable network identity, a headless Service, per-Pod volumes (`volumeClaimTemplates`), and controlled *partitioned* updates.
* How an *init container* runs a schema migration to completion before your app is allowed to start.
* How real databases run *on* the platform through Operators — and the honest Day-2 questions (who operates it, backup/restore, storage performance) you answer before production.

== The storage abstraction chain

A container's own filesystem is *ephemeral*: it is part of the Pod, and when the Pod goes away, so does everything written to it.
That is correct behavior for stateless apps (the whole point of "cattle, not pets"), but a database needs storage that survives the Pod.

OpenShift gives you that through three objects, and the separation between them is the entire idea:

PersistentVolumeClaim (PVC)::
What your app *asks for* — "I need 2Gi of ReadWriteOnce storage."
It lives in your namespace, next to your app.
You write PVCs; you never write the disk directly.
PersistentVolume (PV)::
What the platform *provisioned* — the actual piece of network storage bound to your claim.
It is cluster-scoped and you rarely create one by hand.
StorageClass (SC)::
The *rules* the platform followed to create the PV — which storage backend, which access modes, whether volumes can grow, what happens when the claim is deleted.

You ask (PVC); the platform provisions (PV) according to a recipe (SC); the PV is backed by real storage.

*Dynamic provisioning* is what makes this a one-liner.
You never pre-create disks or hand-craft PVs.
You submit a PVC, and the StorageClass's provisioner creates a matching PV on demand and binds them.

One behavior of the default StorageClass will surprise you, and the lab makes you meet it head-on: *binding mode*.
A class with `volumeBindingMode: WaitForFirstConsumer` deliberately leaves a new PVC in `Pending` until a Pod that uses it is scheduled — so the volume is created in the same place the Pod lands.
A `Pending` PVC with no Pod is not broken; it is waiting, on purpose.

== Access modes: what your app may assume

A PVC requests an *access mode*, and this is where mental models from one cloud mislead you on another:

* *ReadWriteOnce (RWO)* — mountable read-write by Pods on *one node* at a time.
Correct for a single-writer database.
This is what the default class here provides.
* *ReadWriteMany (RWX)* — mountable read-write by many Pods across many nodes at once.
Needed for shared-file workloads (uploaded media, a shared content directory).
Not every backend offers it.
* *ReadOnlyMany (ROX)* — mounted read-only by many Pods.
Niche.

The rule that matters: *do not assume RWX exists.*
Many block-storage backends only offer RWO, and an app that quietly relies on many Pods sharing one volume will fail to schedule on them.
On this cluster you can see both — the default class is RWO, and a separate file class offers RWX — and in the lab you confirm which is which by reading the StorageClasses directly instead of guessing.

== StatefulSet vs Deployment

A Deployment treats its Pods as interchangeable: same spec, random names, one shared Service load-balancing across them, and any Pod can use any volume.
That is exactly wrong for stateful software where *identity* matters — where "replica 0" owns a specific slice of data and clients must reach *that* member, not a random one.

A *StatefulSet* changes four things:

Stable network identity::
Pods are named by ordinal — `pg-sts-0`, `pg-sts-1` — and each keeps its name across restarts.
Paired with a *headless Service* (one with `clusterIP: None`), each Pod gets its own stable DNS name, `pg-sts-0.pg-sts.<namespace>.svc`, so clients can address a specific member.
The Pod's IP changes on restart; the *name* does not.
Per-Pod storage::
A `volumeClaimTemplate` gives *each* Pod its own PVC — `data-pg-sts-0`, `data-pg-sts-1` — not one shared volume.
When `pg-sts-0` is rescheduled, it re-attaches to *its* PVC, so its data follows its identity.
Ordered lifecycle::
Pods are created and scaled in order (0, then 1, …), which matters when members must join a cluster in sequence.
Controlled updates::
Change the Pod template — a new image, a new setting — and a StatefulSet does *not* replace every Pod at once.
By default (the `RollingUpdate` strategy) it updates them one at a time, highest ordinal first, and waits for each Pod to go `Ready` before it touches the next.
+
A *partition* turns that automatic rollout into a staged canary.
The partition is a stop line: only Pods whose ordinal is *at or above* it take the new version; Pods below it stay on the old one.
Say you have three replicas `db-0`, `db-1`, `db-2`.
Set `rollingUpdate.partition: 2` and change the image — only `db-2` rolls to the new version; `db-0` and `db-1` are left untouched.
Verify `db-2` is healthy on the new version; if the image is bad you have lost exactly one member, and reverting the template rolls it back.
Once you trust it, drop the partition to `0` and the rest follow, `db-1` then `db-0`.
(You do exactly this in the lab with two replicas: `partition: 1` rolls `pg-sts-1` first, then `partition: 0` finishes `pg-sts-0`.)
+
The other strategy, `OnDelete`, updates *nothing* on its own: a Pod picks up the new template only when *you* delete it — a fully manual rollout you drive one Pod at a time.

== Stateful workloads on Kubernetes, honestly

Running databases on Kubernetes is not a compromise — it is how modern data software is built to run.
The datastores you would reach for run *on* the platform through *Operators*: the EDB and CloudNativePG Operators for PostgreSQL, or the CockroachDB, MongoDB, and Couchbase Operators — and for big data, Streams for Apache Kafka.
An Operator is what makes this safe in production: it wraps the same StatefulSets, Services, and PersistentVolumeClaims you build by hand in this lab into a single custom resource, then keeps running them — provisioning replicas, failing over, taking and restoring backups, and rolling minor-version upgrades.
The hand-built StatefulSet here teaches you the primitives; an Operator is what you actually run.

So the honest question is not "in the cluster or out of it" — it is the set of Day-2 things you decide *before* you go to production:

Who operates it::
Someone owns the Operator and the database's lifecycle — usually a platform or data team.
Choosing a database Operator is choosing who has encoded that operational knowledge.
(If you would rather not run it at all, a managed cloud database is a legitimate alternative — you trade control and portability for someone else carrying the pager.)
Backup and restore::
Persistence survives a Pod restart; it does *not* survive a deleted PVC, a corrupt volume, or a bad migration.
You need backups you have actually *restored from* — through the Operator's own backup resources or OpenShift API for Data Protection (its own module).
Storage performance::
Databases are latency-sensitive.
Give them fast block storage (RWO) close to the Pod through the right StorageClass — not a generic shared file volume — and size IOPS for the workload.
Day-2 operations::
Failover, minor-version upgrades, connection pooling, monitoring.
A bare StatefulSet gives you none of these and a mature Operator gives you all of them; running `replicas: 2` by hand is two independent instances, not a replicated database.

One more decision that costs nothing to get right: give the database its *own* project, not the application's.
A datastore has a different lifecycle, backup policy, quota, and access list than the app that reads it, and a dedicated namespace keeps those boundaries — and its blast radius — clean.
The wrap-up returns to all of this with a concrete decision guide.

== How this shows up in real orgs

* *"Storage is a platform service."*
App teams should write a PVC and get storage, without filing a ticket or knowing which array is behind it.
The StorageClass is the contract between the platform team (who picks and tunes the backend) and app teams (who just ask).
When that contract is clean, storage stops being a project.
* *Match the access mode to the workload.*
ReadWriteMany is the right — and sometimes required — answer when Pods genuinely share files: the internal image registry scaled across nodes needs RWX, as do a shared media or content directory and a primary/standby pair that coordinates through lock files on shared storage.
It becomes a crutch only when it stands in for a different tool — many writers hammering one file as if it were a queue or a database, or blobs dropped on a filesystem instead of an object store.
So ask what the data really is — shared files (RWX is correct), a single writer (RWO), or objects and records (a bucket or a datastore) — and confirm your cluster even offers RWX before you design around it.
* *A StatefulSet is plumbing, not a database strategy by itself.*
It gives you identity and volumes — the primitives.
Running PostgreSQL well in-cluster means pairing it with an Operator that owns failover, backup, and upgrades; a managed service is the alternative when you would rather not operate it at all.
Presenting "we run a StatefulSet" as "we run a resilient database" is how teams get surprised at 3 a.m.
