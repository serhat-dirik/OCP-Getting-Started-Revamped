// and the trust story (cosign verify → attestation → keyless/Rekor → admission) is the spine.

This module has three facts to plan around.
First, *the environment is pre-warmed*: at every prep, `warm-clean-image.yaml` builds the clean `main` branch, so a *scanned-green, Chains-signed* `parasol-claims:latest` is ready before the attendee starts — that adds *~8 minutes to prep* (budget it into provisioning), and the whole trust spine runs against it with no waiting.
Second, *there is exactly one attendee build* — the seeded branch, to a throwaway `:candidate` tag — and it is the single memorable beat: the build *succeeds*, then the scan turns the run *red* on a CVSS-10 vulnerability.
Third, *the scan gate depends on the RHACS vulnerability store being warm*: on a freshly installed cluster the scanner needs *~1 hour* to populate before the Log4Shell policy will fire — pre-warm before any session (see pre-flight).

== Timing

Total hands-on: *~65 min* (spec target 90 min; add ~12–15 min of concept up front).
This is *shorter* than the pre-rebalance module (~80 min) — the second full build (fix-and-re-run to green) is gone; its place is taken by reading the pre-scanned clean verdict and by the deeper trust beats.

[cols="1,4,1,2",options="header"]
|===
| # | Exercise group | Attendee time | Measured core (as user3, {ocp_version}, warm cluster)

| 1 | The scan gate — one refused, one cleared | ~18 min | seeded `:candidate` run `Failed` at `acs-scan` in *7m11s* (build *Succeeded*, scan blocked on Log4Shell CVSS 10); reading the pre-scanned `scan-passed=true` is instant; `mvn ...:makeBom` *~45s* warm, 239 components — done *during* the build
| 2 | Chains signed it — verify with cosign | ~12 min | `cosign` (v3.0.4) + `rekor-cli` fetched from the RHTAS cli-server in *seconds*; `cosign verify --key` RC=0; the negative (stock image) fails "no matching signatures"
| 3 | Provenance and SBOM | ~10 min | read-only — `verify-attestation` + `download attestation` seconds; SLSA provenance v0.2, `builder tekton.dev/chains/v2`
| 4 | Keyless signing and the transparency log | ~15 min | `cosign initialize` + `sign-blob` + `rekor-cli search` + `verify-blob` *~2 min*; Rekor tree grows by one, `verify-blob` → `Verified OK`
| 5 | Only signed images run (admission) | ~10 min | instructor-demo; the `ImagePolicy` applies via a CRI-O *reload* (not a reboot) — seconds
|===

[NOTE]
.How this timing was measured
====
Every exercise was performed as `user3` on the build cluster (OCP {ocp_version}, RHACS {acs_version}, RHTAS {rhtas_version}, Tekton Chains) on 2026-07-18 and timed.

* *Warm clean build (prep)* — the `main` branch built, scanned green (`scan-passed=true`), and was Chains-signed in *~8 min* (cold node longer); this is the `parasol-claims:latest` the trust spine reads.
* *Seeded run (exercise 1)* — `Failed` at `acs-scan` in *7m11s*; the build *Succeeded* and was even signed before the scan refused the `:candidate` image.
* *These are warm-cache numbers.* A cold node downloads the Maven dependency tree, adding several minutes to the first build; the warm prep run pre-heats it.
* *Chains signing latency* — the `chains.tekton.dev/signed=true` annotation appears a *few seconds after* the build TaskRun completes, not instantly.
====

== Pre-flight checklist

Run these *before the session* (as cluster admin unless noted).

. *A sample user's entry state materializes and verifies (including the warm signed image).*
+
[source,sh,role=execute]
----
ws start trusted-supply-chain --user user1     # blocks ~8 min: it builds the warm clean image
ws verify trusted-supply-chain --user user1 --entry-only
----
+
Expect all checks green: the `user1-cicd` namespace + entry marker, the `parasol-claims-supply-chain` Pipeline, the copied `rox-api-token` Secret and `chains-cosign-pub` ConfigMap, the Gitea fork + its `seed-vulnerable` branch carrying the seeded `log4j-core`, the curated `acs-image-check` task, *and the pre-scanned trust artifact* — a Chains-signed `parasol-claims:latest` with its `.sig`/`.att` tags and a run that passed the scan gate.

. *The shared trust stack is up — RHACS and RHTAS.*
+
[source,sh,role=execute]
----
oc get central -n stackrox -o jsonpath='RHACS: {.items[0].status.conditions[?(@.type=="Deployed")].status}{"\n"}'
oc get securesign -n trusted-artifact-signer -o jsonpath='RHTAS: {.items[0].status.phase}{"\n"}'
oc get route -n trusted-artifact-signer | grep -E 'cli-server|rekor-server|tuf'
oc get task acs-image-check -n ogsr-parasol-tasks -o name
----
+
Expect RHACS `Deployed=True`, RHTAS `Ready`, the `cli-server`/`rekor-server`/`tuf` routes serving, and the `acs-image-check` task.
If RHTAS is not `Ready`, exercises 2–4 cannot run (no cli-server to fetch cosign, no Rekor to record to) — re-sync the trust stack (`pp-trust` + `trust-signing`; these are workshop-layer, not per-user).

. *The cli-server actually serves the clients.*
+
[source,sh,role=execute]
----
CLI=https://$(oc get route -n trusted-artifact-signer -l app.kubernetes.io/component=client-server -o jsonpath='{.items[0].spec.host}' 2>/dev/null || oc get route cli-server -n trusted-artifact-signer -o jsonpath='{.spec.host}')
curl -ksI $CLI/clients/linux/cosign-amd64.gz  | head -1
curl -ksI $CLI/clients/linux/rekor-cli-amd64.gz | head -1
----
+
Expect `HTTP/1.1 200` for both — the attendee downloads `cosign` and `rekor-cli` from here (the Showroom terminal ships neither).

. *Pre-warm the scanner AND prove the gate fires (do this ~1 h before).*
Run the seeded pipeline once on a sample user to a `:candidate` tag — it both warms the Maven cache and confirms the RHACS vulnerability store is populated enough for the Log4Shell policy to fire:
+
[source,sh,role=execute]
----
NS=user1-cicd
oc create -n $NS -f - <<'EOF'
apiVersion: tekton.dev/v1
kind: PipelineRun
metadata: {generateName: preflight-vuln-}
spec:
  pipelineRef: {name: parasol-claims-supply-chain}
  params:
    - {name: git-revision, value: seed-vulnerable}
    - {name: image, value: "image-registry.openshift-image-registry.svc:5000/$(context.pipelineRun.namespace)/parasol-claims:candidate"}
  taskRunTemplate: {serviceAccountName: pipeline}
  workspaces:
    - name: shared-workspace
      volumeClaimTemplate: {spec: {accessModes: [ReadWriteOnce], resources: {requests: {storage: 1Gi}}}}
  taskRunSpecs:
    - {pipelineTaskName: sbom-report, computeResources: {requests: {memory: 1Gi}, limits: {memory: 1536Mi}}}
    - {pipelineTaskName: build-image, computeResources: {requests: {memory: 1536Mi}, limits: {memory: 2Gi}}}
EOF
tkn pipelinerun logs --last -f -n $NS | grep -iE 'Block Log4Shell|image check'
----
+
[IMPORTANT]
.The scanner needs ~1 hour of warm-up on a fresh cluster
=====
RHACS Scanner V4 downloads its vulnerability data after install; until it is populated, `roxctl image check` finds *no* CVEs and the gate *passes even on the seeded branch* — the lab's first beat silently fails.
On a cluster up for an hour this is a non-issue.
On a *freshly provisioned* cluster, run the pre-flight above and confirm it prints `Block Log4Shell (CVE-2021-44228) at build` before you trust the gate.
=====

== Demo-mode notes

Beyond the Say/Show/Do arc in the demo flavor:

* *Pre-solve before the room.*
`ws solve trusted-supply-chain --user {user}` lands on the same warm end state as prep (a scanned, signed `parasol-claims:latest`) and *blocks by design* (~8 min warm, ~10–12 cold).
It leaves your "good outcome" ready and warms the Maven cache for the live seeded build.

* *cosign is now attendee-hands-on — no verification pod needed.*
Since RHTAS recovered, `cosign` and `rekor-cli` are fetched from the cli-server (exercise 2), so verification is *hands-on for everyone* — the old "instructor runs cosign in a pod" workaround is retired.
If you present on a machine without them, run the same two `curl … | gunzip` lines from the lab.

* *Talk through the one build, never at it.*
Even warm, the seeded run is ~7 minutes.
Start it, then narrate the threat-model / three-artifacts story while it builds — the payoff is the *red scan* and then the trust beats, not a progress bar.

* *The admission demo (exercise 5) is the only instructor-run beat.*
Apply the `ImagePolicy` as cluster-admin (attendees cannot — see watchouts):
+
[source,sh,role=execute]
----
NS=user1-cicd
KEY=$(oc get configmap chains-cosign-pub -n $NS -o jsonpath='{.data.cosign\.pub}' | base64 -w0)
oc apply -f - <<EOF
apiVersion: config.openshift.io/v1
kind: ImagePolicy
metadata: {name: only-signed-claims, namespace: ${NS}}
spec:
  scopes: ["image-registry.openshift-image-registry.svc:5000/${NS}/parasol-claims"]
  policy:
    rootOfTrust: {policyType: PublicKey, publicKey: {keyData: ${KEY}}}
    signedIdentity: {matchPolicy: MatchRepoDigestOrExact}
EOF
----
+
Then show the block by deploying an *unsigned* image into the scoped repo and reading the pod events: `SignatureValidationFailed: ... A signature was required, but no signature exists`.
Deploying the *signed* `parasol-claims:latest` works.
Clean up with `oc delete imagepolicy only-signed-claims -n $NS` (the entry-state engine also purges ImagePolicies on `ws reset`).
+
[IMPORTANT]
.The ImagePolicy applies via a CRI-O reload, not a node reboot — but keep it instructor-only
=====
On this cluster the node-disruption policy maps the sigstore files (`/etc/containers/policy.json`, `/etc/crio/policies`, `/etc/containers/registries.d`) to *Reload crio.service*, so applying an `ImagePolicy` does *not* reboot nodes.
It does briefly cycle the worker MCP through an "updating" (reload) state that touches *every* node, so do *not* have 30 attendees each apply one — demo it once, cluster-wide, from the instructor seat.
=====

== Top questions attendees ask

*1. "The build succeeded but the run failed — is that a bug?"*::
No, it is the design.
The scan runs *after* the build (`acs-scan` has `runAfter: build-image`) because you cannot scan an image you have not built.
So the image _is_ built (and even signed), and then the scan refuses the `:candidate` — the run is `Failed` with `build-image: Succeeded, acs-scan: Failed`.

*2. "Where did the clean `latest` image come from — I only built `:candidate`?"*::
The entry state pre-built it.
At every prep, `warm-clean-image.yaml` runs the pipeline on the clean `main` branch (no log4j) and leaves a scanned-green, Chains-signed `parasol-claims:latest`.
That is deliberate — the owner's rebalance makes the attendee *read* the clean verdict and *verify* the clean image, instead of waiting through a second build. Their own seeded build goes to `:candidate` so it never clobbers `:latest`.

*3. "What's the difference between the signature, the attestation, and the SBOM?"*::
Three different facts.
The *signature* proves the image is intact and came from us (a signature over the digest).
The *attestation* is *signed provenance* — how the image was built (builder, source, steps, timestamps), as a SLSA statement.
The *SBOM* is what is *inside* — every component and version.
Chains produces the signature and attestation automatically; the build produces the SBOM.

*4. "Key-based versus keyless — and why did we sign the SBOM, not the image, keylessly?"*::
The lab verifies the Chains *key-based* signature (deterministic, no per-build identity) in exercise 2, then demonstrates *keyless* (Fulcio short-lived cert + a Rekor transparency-log entry) in exercise 4.
We sign the *SBOM file* (`sign-blob`) rather than push a keyless *image* signature because the OpenShift internal registry rejects cosign's keyless image-signature manifest (`UNKNOWN` on the manifest PUT) — the Rekor entry is written first regardless, but the image push fails.
In production, keyless *image* signatures go to an external registry (Quay). The SBOM `sign-blob` keeps the transparency-log beat self-contained and clean.

*5. "How does the transparency log actually help?"*::
It makes signing *auditable*.
A key-based signature proves "signed with a key we hold." A keyless signature in Rekor proves "signed by _this identity_ at _this time_, on a public, append-only log nobody — not even us — can quietly alter."
When an auditor asks "prove the image in production came from your pipeline," the Rekor entry is a receipt they can read without contacting you.

*6. "How does this module relate to the Application Security Testing module?"*::
Complementary halves of DevSecOps.
*This* module (trust) proves the image is *provably yours* — signed, attested, transparency-logged, and admission-controlled.
*Application Security Testing* (scanning) proves the image is *free of known flaws* — SAST, SCA, DAST, config, and image-risk gates.
They overlap on one mechanic: both drive an RHACS build-time image check that blocks Log4Shell. Here that scan is *one beat of context*; there it is the subject.

== Watchouts

* *The scanner vulnerability store must be warm.*
On a freshly installed cluster, RHACS Scanner V4 needs *~1 hour* to populate before the Log4Shell policy fires; until then the gate *passes on the seeded branch* and the lab's first beat silently no-ops.
Always run the pre-flight seeded pipeline and confirm it prints `Block Log4Shell` before a session.
* *Prep now blocks ~8 minutes (the warm build).*
`ws start`/`ws prep`/`ws solve` build the clean image before returning (ws-meta `waitSeconds: 1500` covers it). Provision with that in mind; a re-prep on an already-warm namespace skips the rebuild (the hook's idempotency guard).
* *RHTAS must be Ready for exercises 2–4.*
`cosign`/`rekor-cli` come from the RHTAS cli-server, and the keyless beat records to Rekor. If Securesign is not `Ready`, those exercises cannot run — confirm the routes serve in pre-flight.
* *Attendees cannot create an `ImagePolicy`.*
`ImagePolicy` is a `config.openshift.io` resource; the namespace `admin` role does *not* grant it (verified: `oc auth can-i create imagepolicies --as=user3` = `no`), so admission (exercise 5) is an *instructor-demo* — the instructor applies one cluster-wide and the effect lands in each user's namespace.
(To make it hands-on in a future revision, a per-user Role granting `imagepolicies.config.openshift.io` is needed — flagged to the platform team.)
* *cosign v3 sources Fulcio/Rekor from the TUF signing config.*
On cosign v3.0.4, passing `--fulcio-url`/`--rekor-url` to `sign` after `cosign initialize` errors with "cannot specify service URLs and use signing config" — let the signing config (from `cosign initialize`) supply them. The lab commands already do.
* *Chains signs every build — including the seeded one.*
`signed=true` appears on the build TaskRun of the _failed_ `:candidate` run too; that is correct (signing answers "who built it," a different question from "is it clean"). Do not let attendees read "signed" as "passed the scan."
* *The `scan-passed` result is empty on a failed run.*
A failed `acs-scan` task does not propagate the `scan-passed` result, so it is blank (not `false`) on the seeded run; it reads `true` on the pre-warmed clean run.
* *Benign roxctl warning.*
The `acs-scan` logs a `warning: unsuccessful cred copy: ".docker" ... permission denied` line — harmless; `roxctl` authenticates with the token from `rox-api-token`.
* *`ws reset`/`ws prep` restores the vulnerable seed and re-warms the image.*
The fork job re-injects `log4j-core` on `seed-vulnerable` idempotently, and the warm hook rebuilds the signed `:latest` — a reset returns the attendee to a fresh, warm start.
