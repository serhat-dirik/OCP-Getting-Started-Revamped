In this lab you take Parasol's `parasol-claims` pipeline from "it builds" to "it is *trustworthy*."
Most of your time is spent on the *trust* controls — the *signature* and *SLSA provenance* that travel with the image, a *transparency-log* record anyone can audit, and the cluster *refusing to run anything unsigned*.
Scanning is *one compact beat*: your environment already holds a clean, *pre-scanned* image, so you read a verdict rather than wait on a build.

[NOTE]
.Your environment came pre-warmed with a signed image
====
When your project was prepared, the pipeline already ran once on the *clean* `main` branch and left a *scanned-green, signed* `parasol-claims:latest` in your registry.
That is the image the trust exercises below verify.
You will *also* build the *seeded* branch yourself to watch the scan gate catch a real CVE — that build goes to a separate `:candidate` tag, so it never disturbs the trusted `:latest`.
====

[NOTE]
.The scan here is one beat — scanning is its own module
====
You will watch the pipeline's *scan* step fail on a seeded Log4Shell dependency, and read the *clean* image's passing verdict beside it.
That build-time scanning — vulnerabilities, secrets, insecure config — is the subject of the *Application Security Testing* module, which runs SAST, SCA, DAST, and image-risk gates in depth.
*This* module's subject is *trust*: signing, provenance, attestation, transparency, and admission.

====

[NOTE]

The cryptographic steps — `cosign`, `rekor-cli` — are terminal-only by nature and marked where they occur.
One exercise edits a file in the *Gitea web UI*: open your fork at {gitea_url}/{user}/parasol-claims[{user}/parasol-claims^] in a second browser tab now.

Now *start a build of the seeded branch* — to a throwaway `:candidate` tag, so the trusted `:latest` already in your registry is untouched.
This build takes a few minutes, so start it and read on:

[NOTE]
.Import YAML, not the *Start* button
=====
Create the run from *Import YAML* (which takes the whole manifest), *not* from the Pipeline's *Actions → Start* form.
This run raises per-Task memory for the build steps (the `taskRunSpecs`), and the *Start* form cannot set that — a run started without it is killed mid-build.
=====
--
CLI::
+
--
[source,sh,role=execute]
----
oc create -n $NS -f - <<'EOF'
apiVersion: tekton.dev/v1
kind: PipelineRun
metadata:
  generateName: seed-scan-
spec:
  pipelineRef:
    name: parasol-claims-supply-chain
  params:
    - name: git-revision
      value: seed-vulnerable              # the seeded branch — build it to a throwaway tag
    - name: image
      value: image-registry.openshift-image-registry.svc:5000/$(context.pipelineRun.namespace)/parasol-claims:candidate
  taskRunTemplate:
    serviceAccountName: pipeline          # operator-provided SA: can build + push images here
  workspaces:
    - name: shared-workspace
      volumeClaimTemplate:                # one PVC per run, auto-pruned with the run
        spec:
          accessModes: [ReadWriteOnce]
          resources:
            requests:
              storage: 1Gi
  taskRunSpecs:                           # more memory than the 1Gi namespace default for the builds
    - pipelineTaskName: sbom-report
      computeResources:
        requests: {memory: 1Gi}
        limits: {memory: 1536Mi}
    - pipelineTaskName: build-image
      computeResources:
        requests: {memory: 1536Mi}
        limits: {memory: 2Gi}
EOF
----
--

== 2. Chains signed it — verify the signature with cosign
⏱ *~12 min* · fetch cosign from RHTAS, then prove the clean image is intact and yours

The scan proved the clean image is _clean_.
This is the heart of the trust module: *signing* proves the image is _yours_ — byte-for-byte the one your pipeline produced.
*Tekton Chains* — a component of OpenShift Pipelines — watched the build TaskRun and *signed the image it produced*, with no per-build wiring, when your environment was warmed.

`cosign` is the tool that _checks_ what Chains _produced_.
The Showroom terminal does not ship it — but *Red Hat Trusted Artifact Signer* runs a *client-download server*, so fetch `cosign` (and `rekor-cli`, for later) straight from the cluster:

[source,sh,role=execute]
----
mkdir -p ~/bin
curl -ksL $CLI/clients/linux/cosign-amd64.gz  | gunzip > ~/bin/cosign
curl -ksL $CLI/clients/linux/rekor-cli-amd64.gz | gunzip > ~/bin/rekor-cli
chmod +x ~/bin/cosign ~/bin/rekor-cli
export PATH="$HOME/bin:$PATH"
cosign version | grep GitVersion
----

.Expected output
[source,texinfo]
----
GitVersion:    v3.0.4
----

`cosign` verifies signatures by pulling them from the registry, so give it a credential for your internal registry and point it at your signed image:

[source,sh,role=execute]
----
oc registry login --to="$HOME/.docker/config.json" --skip-check
export DOCKER_CONFIG="$HOME/.docker"
# the public key Chains signs with — copied into your namespace by the entry state
oc get cm chains-cosign-pub -n $NS -o jsonpath='{.data.cosign\.pub}' > ~/cosign.pub
# the exact clean image, by digest
IMG=$(oc get istag parasol-claims:latest -n $NS -o jsonpath='{.image.dockerImageReference}')
echo "verifying: $IMG"
----

Now *verify* it against the public key:

[source,sh,role=execute]
----
cosign verify --key ~/cosign.pub --allow-insecure-registry --insecure-ignore-tlog "$IMG"
----

.Expected output (abridged)
[source,texinfo]
----
Verification for image-registry.openshift-image-registry.svc:5000/<you>-cicd/parasol-claims@sha256:... --
The following checks were performed on each of these signatures:
  - The cosign claims were validated
  - The signatures were verified against the specified public key
----

The signature is *valid*: this image is intact and was signed with the key your pipeline signs with.
(`--insecure-ignore-tlog` tells cosign not to look for a transparency-log entry — this key-based signature does not use one; you will meet the transparency log with *keyless* signing in exercise 4. `--allow-insecure-registry` skips the internal registry's service-CA TLS.)

Now watch it *reject* an image that was not signed with your key — a stock public image:

[source,sh,role=execute]
----
cosign verify --key ~/cosign.pub --insecure-ignore-tlog \
  docker.io/library/hello-world:latest 2>&1 | tail -2
----

.Expected output
[source,texinfo]
----
Error: no signatures found
error during command execution: no signatures found
----

That is the whole point of verification: an image your pipeline did not sign does not pass.
Exercise 5 turns that check into a cluster rule.

== 3. Provenance and SBOM: prove *how* it was built
⏱ *~10 min* · read the SLSA attestation Chains attached beside the image

A signature says the image is intact and yours.
An *attestation* says _how it was built_ — the builder, the source, the steps, the timestamps — as a signed *SLSA provenance* statement.
Chains attached one to your image, stored in the registry beside it as a `.att` tag.

First see the artifacts Chains stored — the image plus two tags derived from its digest:

== 4. Keyless signing and the transparency log
⏱ *~15 min* · sign the SBOM with a short-lived identity and find it in Rekor

The signature you verified uses a *cosign key pair* (the private key lives in the cluster).
The enterprise alternative is *keyless* signing: *Red Hat Trusted Artifact Signer* issues a *short-lived certificate* tied to your build identity (via *Fulcio*), and records every signature in a public, append-only *transparency log* (*Rekor*) — so there is no long-lived key to steal, and no signature can be quietly backdated or hidden.

You will sign your *SBOM* keylessly and then find its record in the log.
First point cosign at the RHTAS trust root, and mint a short-lived *identity token* (a Kubernetes service-account token — the identity Fulcio issues a certificate for):

[source,sh,role=execute]
----
cosign initialize --mirror=$TUF --root=$TUF/root.json
TOKEN=$(oc create token pipeline --audience trusted-artifact-signer -n $NS)
----

Now *sign the SBOM* keylessly.
cosign gets a certificate from Fulcio for your identity, signs, and uploads the record to Rekor — writing a *bundle* (certificate + signature + log proof) beside the file:

[source,sh,role=execute]
----
cd ~/parasol-claims
cosign sign-blob -y --identity-token="$TOKEN" \
  --bundle ~/sbom.bundle target/parasol-claims-sbom.json > /dev/null
echo "signed; bundle written to ~/sbom.bundle"
----

The signature is now in the transparency log.
*Find it* — search Rekor for your artifact, and read the log's growth:

[source,sh,role=execute]
----
rekor-cli search --rekor_server $REKOR --artifact target/parasol-claims-sbom.json
rekor-cli loginfo --rekor_server $REKOR | grep -E 'Tree Size'
----

.Expected output (your UUID differs; the log is shared and append-only, so the tree size is a running total)
[source,texinfo]
----
Found matching entries (listed by UUID):
<a 64-character entry UUID>
Active Tree Size:       <a running total — your signature is now one of them>
Total Tree Size:        <a running total — your signature is now one of them>
----

That UUID is a permanent, tamper-evident record: *this artifact was signed by this identity at this time.*
Anyone can audit it without contacting you.

Finally, *verify the keyless signature* — cosign checks the certificate identity, the issuer, and the log proof from the bundle:

[source,sh,role=execute,subs="attributes"]
----
cosign verify-blob --bundle ~/sbom.bundle \
  --certificate-identity=https://kubernetes.io/namespaces/{user}-cicd/serviceaccounts/pipeline \
  --certificate-oidc-issuer=https://kubernetes.default.svc \
  ~/parasol-claims/target/parasol-claims-sbom.json
----

.Expected output
[source,texinfo]
----
Verified OK
----

`Verified OK`: the SBOM was signed by the `pipeline` identity in your namespace, and Rekor has the receipt.

[NOTE]
.Why sign the SBOM, and not push the keyless signature onto the image
====
Keyless signing records in Rekor *before* it stores the signature — which is exactly what you want to see here.
Signing the *SBOM file* (`sign-blob`) keeps the beat self-contained and registry-independent; the OpenShift internal registry does not accept cosign's keyless image-signature manifest (you would see an `UNKNOWN` push error _after_ the log entry is already written).
In production you would push keyless *image* signatures to an external registry (Quay), where the same `cosign sign` you saw in exercise 2 lands them.
You can browse every entry in this log in the *Rekor Search UI* at https://rekor-search-ui-trusted-artifact-signer.{cluster_domain}[the Rekor Search UI^].
====

== 5. Only signed images run: admission control
⏱ *~10 min* · the cluster refuses to pull an unsigned image into your namespace

Signing is worthless if nobody checks the signature.
The final control is *admission*: OpenShift's native `ImagePolicy` (a `config.openshift.io/v1` resource, shipped with OpenShift {ocp_image_policy_version}) tells the node to *verify a signature before it will pull an image* into a namespace.
An image that never went through your signing pipeline is *refused at pull time*.

Because an `ImagePolicy` changes cluster-node behavior (the node's container runtime re-reads its signature-verification config), it is applied by the instructor cluster-wide in this workshop — but the effect lands in *your* namespace, and you verify it.

[NOTE,role=instructor-demo]
.🧑‍🏫 Instructor demo — the cluster blocks an unsigned image
====
The instructor applies an `ImagePolicy` scoped to your `parasol-claims` image, pinned to the same cosign public key Chains signs with:

[source,yaml]
----
apiVersion: config.openshift.io/v1
kind: ImagePolicy
metadata:
  name: only-signed-claims
  namespace: <you>-cicd
spec:
  scopes:
    - image-registry.openshift-image-registry.svc:5000/<you>-cicd/parasol-claims
  policy:
    rootOfTrust:
      policyType: PublicKey              # <1>
      publicKey:
        keyData: <base64 cosign.pub>     # <2>
    signedIdentity:
      matchPolicy: MatchRepoDigestOrExact
----
<1> Key-based verification (the lab's path); the keyless alternative is `policyType: FulcioCAWithRekor`.
<2> The base64-encoded cosign public key — the same `chains-cosign-pub` your namespace already holds.

With the policy in place:

* Deploying your *signed* `parasol-claims:latest` image *works* — the node verifies the signature against the key and pulls it.
* Deploying an *unsigned* image (say a stock `docker.io` image) into the same scope is *refused* — the pod stays `ImagePullBackOff`, and its events read `SignatureValidationFailed: ... A signature was required, but no signature exists`.

That is the point of admission: it does not matter whether anyone _remembered_ to run the scan gate — the cluster will not pull an image it cannot verify.
====
