== When *not* to reach for a pipeline

Pipelines are the right tool for repeatable delivery — which means being honest about when they are the wrong one:

* *Don't build a pipeline for a one-off.*
A pipeline earns its keep through *repetition*.
A script you will run once, or a spike you will throw away, does not need a PipelineRun, a workspace, and a webhook — it needs a terminal.
Reach for a pipeline when the work will happen *again and again the same way*.
* *Don't put the whole delivery in one giant pipeline.*
It is tempting to grow one pipeline until it does everything.
The moment a pipeline is hard to read, split it — small, composable Pipelines that hand off (build here, promote there) beat one monolith, and are the reason Tasks are reusable in the first place.
* *Don't hand-write a Task the catalog or library already ships.*
The whole point of the three reuse layers is that you write as little as possible.
Before authoring a Task, check the shipped catalog and your organization's library — the best Task is the one you reused.
* *Don't treat the pipeline's `-cicd` deploy as promotion to production.*
This module deploys straight from the build into `{user}-cicd` to stay self-contained.
That is fine for a CI sandbox and deliberately *not* how you promote to staging or production — the pipeline builds and publishes an *image*, and a later step (GitOps) reconciles that image into each environment.
Pushing an image straight into prod from CI is the anti-pattern the GitOps modules exist to replace.

* *Trusted Software Supply Chain* — your pipeline now produces an image and data; the next module adds the gates that make it *trustworthy*: image signing and attestation, provenance, and an SBOM, so only artifacts you can prove you built are admitted (a separate module). Scanning that image and its code for flaws is its own later module, *Application Security Testing*.
