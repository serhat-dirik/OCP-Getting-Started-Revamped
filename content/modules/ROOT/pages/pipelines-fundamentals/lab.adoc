== 2. Explore the company task library
⏱ *~15 min* · the three reuse layers, and why Parasol curates its own Tasks

You just saw the Pipeline reference five Tasks by cluster resolver.
This exercise looks at the two *curated* ones — and the concrete reason they exist.

[NOTE]
.Why this exercise stays in the terminal
====
The curated Tasks live in shared namespaces (`parasol-tasks` and `openshift-pipelines`) that you can *read* but not *select* as a project in the console — so the console's *Pipelines → Tasks* view cannot scope to them for you.
Reading a Task's inner fields (its image, its parameters) is also exactly what `oc get … -o jsonpath` does best, so this one is CLI-only.
====

Start by browsing Parasol's shared library.
You have read access to it (the platform team granted it), so you can list and read the Tasks any pipeline reuses:

[source,sh,role=execute]
----
oc get tasks -n parasol-tasks
----

.Expected output
[source,texinfo]
----
NAME                AGE
image-size-report   ...
maven-jdk21         ...
----

Two curated Tasks.
Now the *why*.
The Pipeline's `unit-test` step runs Maven — and OpenShift Pipelines *ships* a `maven` Task, so the obvious choice is to use that shipped one.
Look at what it actually runs:

[source,sh,role=execute]
----
echo "shipped maven task, build step image:"
oc get task maven -n openshift-pipelines -o jsonpath='  {.spec.steps[1].image}{"\n"}'
echo "shipped maven task, parameters:"
oc get task maven -n openshift-pipelines -o jsonpath='  {range .spec.params[*]}{.name}{"  "}{end}{"\n"}'
----

.Expected output
[source,texinfo]
----
shipped maven task, build step image:
  registry.redhat.io/ubi9/openjdk-17@sha256:...
shipped maven task, parameters:
  GOALS  MAVEN_MIRROR_URL  SUBDIRECTORY
----

The shipped `maven` Task runs on a *Java 17* image — and there is *no parameter* to change it (`GOALS`, `MAVEN_MIRROR_URL`, `SUBDIRECTORY`, and nothing for the image).
`parasol-claims` is a *Java 21* application, so the shipped Task cannot build it; Maven fails with `release version 21 not supported`.
That is the gap the curated library fills.
Read Parasol's answer:

[source,sh,role=execute]
----
echo "curated maven-jdk21, build step image:"
oc get task maven-jdk21 -n parasol-tasks -o jsonpath='  {.spec.steps[0].image}{"\n"}'
----

.Expected output
[source,texinfo]
----
curated maven-jdk21, build step image:
  registry.access.redhat.com/ubi9/openjdk-21:latest
----

A *Java 21* runner — the one thing the shipped Task could not give.
Rather than paste a patched Maven Task into every Java 21 pipeline, the platform team curated `maven-jdk21` *once*, and every pipeline references it by cluster resolver.
That is the entire argument for an organization library, in one concrete gap.

Now read the other curated Task — the one that produced the `image-size-mb` result you saw in exercise 1:

[source,sh,role=execute]
----
oc get task image-size-report -n parasol-tasks \
  -o jsonpath='params: {range .spec.params[*]}{.name}{" "}{end}{"\n"}results: {range .spec.results[*]}{.name}{" "}{end}{"\n"}'
----

.Expected output
[source,texinfo]
----
params: IMAGE MAX_SIZE_MB TLS_VERIFY
results: image-size-bytes image-size-mb within-budget
----

This is a Task you could have written: it takes an image reference and a budget as *parameters*, inspects the image, and publishes three *results*.
Read its full body if you like (`oc get task image-size-report -n parasol-tasks -o yaml`) — it is a short shell script.
The point is that it lives in the *shared library*, so every Parasol pipeline gets image-size reporting by adding *one `taskRef`*, not by copying a script.

[NOTE]
.`ClusterTask` is gone — the resolver is how you share Tasks now
====
Older Tekton shared Tasks cluster-wide with a `ClusterTask` kind.
It has since been removed from OpenShift Pipelines (this cluster runs {pipelines_version}), so it does not exist here.
The *cluster resolver* you have been reading — `resolver: cluster` with `name` and `namespace` — is its supported replacement, and it is strictly better: Tasks live in a normal namespace with normal RBAC, and you reference the exact one you want.
====

You can explain why the platform team curated the library.

== 3. Break the build: a failing test stops the pipeline
⏱ *~20 min* · edit a test in your fork, watch the pipeline refuse the bad change, then fix it

A pipeline that tests but ships anyway is theater.
The value is that a *failing test stops the build* — a broken change never becomes a deployed image.
You will prove that by breaking a real rule, in *your own fork*.

`parasol-claims` has a unit test that encodes a Parasol business rule: *a claim cannot be Approved while its adjuster is still Unassigned.*
The test is green as shipped.
You will flip a one-line toggle that violates the rule, watch the pipeline go red at the test, then revert.

In your *Gitea tab* ({gitea_url}/{user}/parasol-claims[{user}/parasol-claims^]):

. Open `src/test/java/com/parasol/claims/ClaimResourceTest.java`.
. Click the *pencil* (Edit) icon.
. Find this line (near line 191):
+
[source,java]
----
final boolean assignAdjusterBeforeApproval = true;
----
. Change `true` to `false`.
. Scroll down and *Commit Changes* directly to the `main` branch (commit message e.g. "break: approve without an adjuster").

[NOTE]
.No Dev Spaces required
====
Editing the file in the Gitea web UI *is* a commit and push — that is all the pipeline needs.
If you did the Dev Spaces module you can make the same edit there and `git push` instead; the result is identical.
====

Now run the pipeline against *your fork* (not the shared default).
This is why `git-url` is a parameter — the same Pipeline builds any repository:

// [CAPTURE-VERIFY] Pipelines > parasol-claims-build-test-deploy > Actions > Start: change the git-url param to the fork URL; Start; PipelineRuns graph goes red at unit-test — CLI 'tkn pipeline start -p git-url=…' authoritative.
[tabs]
====
Console::
+
--
Open the `parasol-claims-build-test-deploy` Pipeline → *Actions* → *Start* again.
This time change the *git-url* parameter to your fork's in-cluster address — `http://gitea.gitea.svc:3000/{user}/parasol-claims.git` — set *shared-workspace* to *VolumeClaimTemplate*, and click *Start*.
Open the new run under *Pipelines* → *PipelineRuns*: the graph goes *red at `unit-test`* and never reaches the build.
--
CLI::

[NOTE]
.Why `http://gitea.gitea.svc:3000/...`
====
That is the *in-cluster* address of the very repo you see in the browser at {gitea_url}/{user}/parasol-claims.
The pipeline runs inside the cluster, so it clones over the internal service — no external round-trip, no TLS to configure.
====

Follow the run — this time it *fails fast*, at the test, and never reaches the build:

[source,sh,role=execute]

[ERROR] Tests run: 14, Failures: 1
[INFO] BUILD FAILURE
----

Now confirm the important part — the build and deploy *never ran*.

[tabs]
====

+
--

CLI::
+

`tkn` lists them under a *Skipped Tasks* heading:

.Expected output

That is the whole lesson: `unit-test` failed, so `build-image`, `image-report`, and `deploy` were *skipped* — the overall run is `Failed`, and its message reads `Tasks Completed: 2 (Failed: 1, Cancelled 0), Skipped: 3`.
The broken change could not become a running image.

*Fix it.*

== 4. Pipelines-as-Code: a Git push builds and deploys
⏱ *~20 min* · wire a webhook so the pipeline fires on every push — no `oc create`

So far *you* started every run.
Pipelines-as-Code (PaC) removes that step: the PipelineRun definition already lives in your fork, at `.tekton/pull-request.yaml`, and a *push* fires it.
Confirm the file is there:

[source,sh,role=execute,subs="attributes"]
----
curl -s -o /dev/null -w "%\{http_code}\n" \
  {gitea_url}/{user}/parasol-claims/raw/branch/main/.tekton/pull-request.yaml
----

A `200` means the entry state seeded it into your fork.
Your namespace also already has a `Repository` resource — the anchor PaC matches an incoming webhook to — and it is pre-wired to read your Gitea credentials from a Secret named `gitea-pac-secret`:

// [CAPTURE-VERIFY] Pipelines > Repositories > parasol-claims (git provider = Gitea; linked PipelineRuns) (OCP 4.21) — CLI 'oc get repository' authoritative.
[tabs]
====

+

Its page shows the Git provider (*Gitea*); once pushes start firing later in this exercise, the PipelineRuns PaC created for it are listed here too.

+

Two pieces remain: create that Secret, and create the webhook that feeds it.

*First*, give PaC a Gitea token (to read your repo and report status) and a *webhook secret* (a shared string that lets PaC trust the incoming push).
Create the token in Gitea, then store both in the `gitea-pac-secret` the `Repository` already expects:

. In Gitea, click your avatar (top right) → *Settings* → *Applications*.
. Under *Manage Access Tokens*, name a token `pac`, and under *Select scopes* choose *write:repository*.
Click *Generate Token* and *copy the value* (Gitea shows it once).
. Store the token plus a webhook secret you generate into the Secret:
+
[source,sh,role=execute]
----
read -rsp "paste your Gitea token, then press Enter: " GITEA_TOKEN; echo
WEBHOOK_SECRET="parasol-$(openssl rand -hex 6)"
echo "your webhook secret is: $WEBHOOK_SECRET"
echo "  ^ copy it — you paste it into the Gitea webhook form next"
oc create secret generic gitea-pac-secret -n $NS \
  --from-literal=token="$GITEA_TOKEN" \
  --from-literal=webhook-secret="$WEBHOOK_SECRET"
----

*Second*, create the webhook in Gitea that POSTs to the PaC controller.
The controller is reachable at a predictable address built from your cluster domain — print it:

[source,sh,role=execute,subs="attributes"]
----
echo "https://pipelines-as-code-controller-openshift-pipelines.{cluster_domain}"
----

In your fork's Gitea page:

. *Settings* → *Webhooks* → *Add Webhook* → *Gitea*.
. *Target URL*: the URL printed above.
. *HTTP Method*: `POST`; *POST Content Type*: `application/json`.
. *Secret*: the `$WEBHOOK_SECRET` value printed earlier.
. *Trigger On*: *Push Events* (leave the default branch filter).
. *Active*: checked. Click *Add Webhook*.

// media-pass: screenshot pipelines-fundamentals-02-gitea-webhook.png — the Gitea Add Webhook
// form (Target URL = the pipelines-as-code-controller route, content type application/json, secret
// filled in, Push Events). Embed here. See media-manifest.md.

== Challenge (optional)
⏱ *no hand-holding*

You have the mechanics.
Try these on your own:

* *Tighten the budget.*
Re-run the pipeline with `-p max-size-mb=300` (add a `params` entry, or `tkn pipeline start ... -p max-size-mb=300`) and read the `image-report` step: the image is 391&nbsp;MB, so `within-budget` flips to `false`.
Notice the run still *succeeds* — the report *flags*, it does not *fail* the build.
How would you make it a hard gate?
* *Read a Task's full body.*
`oc get task image-size-report -n parasol-tasks -o yaml` — find where it writes its three results (the `$(results.<name>.path)` files).
That is how any Task publishes data.
* *Trigger on a pull request.*
Open a PR in Gitea against `main` instead of pushing to it, and watch PaC run the pipeline for the PR — the same `.tekton/` file handles both (`on-event: [push, pull_request]`).
* *Find the run in the console.*
Open *Pipelines → PipelineRuns* in the {ocp_console_url}[OpenShift console^], select your latest run, and read the graph — the same five Tasks, visualized.
