In this lab you take the developer inner loop into the cluster.
You open Parasol's claims service in a browser IDE that starts in under two minutes, run it in Quarkus dev mode against the *real* in-cluster database, change a live endpoint and watch it hot-reload, extend the workspace definition with a Valkey cache, push your change to Git, step through a running request in a debugger, and reach past the workspace to port-forward the database and ship your change as a running Deployment.
Everything happens in your own projects â€” `{user}-devspaces` for the workspace, `{user}-dev` for the app and database â€” so nothing you do can affect another attendee.

[IMPORTANT]
.Two terminals â€” know which one you are in
====
This lab uses two different terminals:

* The *workspace terminal* â€” inside the browser IDE (Red Hat OpenShift Dev Spaces, "Dev Spaces" from here on).
*Almost every command runs here.*
These blocks are copy-enabled; paste them into the IDE terminal.
* The *Showroom terminal* â€” the terminal tab beside these instructions, logged in as `{user}`.
Only steps explicitly labelled *Showroom terminal* run here (they use `oc` to look at the cluster from the outside).
Those are click-to-run.
====

== 2. Wire dev mode to the in-cluster database
âŹ± *~15 min* Â· the deliberate break-and-fix â€” a dev instance needs a real database

Quarkus *dev mode* (`./mvnw quarkus:dev`) runs the app with live reload.
Start it the naive way first, in the *workspace terminal*, and watch it fail on purpose:

[source,sh]
----
cd /projects/parasol-claims
./mvnw quarkus:dev
----

After the dependencies download, it stops with an error:

.Expected output (the deliberate failure)
[%collapsible]
=====
[source,texinfo]
----
ERROR [...DockerClientProviderStrategy] Could not find a valid Docker environment. ...
WARN  [...DevServicesDatasourceProcessor] Please configure the datasource URL for
      default datasource or ensure the Docker daemon is up and running.
ERROR [io.quarkus.runtime.Application] Failed to start application:
  Datasource '<default>' was deactivated automatically because its URL is not set.
  To activate the datasource, set configuration property 'quarkus.datasource.jdbc.url'.
----
=====

Read what happened, because it is the whole lesson.
The claims app needs a database.
With no database configured, Quarkus tried to *start one for you* (a throwaway container, called a Dev Service) â€” but a workspace has no container runtime inside it, so that failed, and the app refused to start without a datasource URL.

You do not want a throwaway database anyway.
You want the *real* `claims-db` that is already running in your `{user}-dev` project â€” the same one the deployed claims app uses.
Your workspace can reach it by ordinary in-cluster DNS, across namespaces: `claims-db.{user}-dev.svc`.

Press *Ctrl+C* to stop the failed attempt, then wire the datasource with environment variables and start dev mode again:

[source,sh,subs="attributes"]
----
export QUARKUS_DATASOURCE_JDBC_URL=jdbc:postgresql://claims-db.{user}-dev.svc:5432/parasol
export QUARKUS_DATASOURCE_USERNAME=parasol
export QUARKUS_DATASOURCE_PASSWORD=parasol
export QUARKUS_HIBERNATE_ORM_SCHEMA_MANAGEMENT_STRATEGY=update
./mvnw quarkus:dev
----

This time it starts.
Watch for these lines:

.Expected output
[source,texinfo,subs="attributes"]
----
INFO  [io.quarkus] parasol-claims 1.0.0 on JVM (powered by Quarkus {quarkus_version}) started in 6.6s. Listening on: http://0.0.0.0:8080
INFO  [io.quarkus] Profile dev activated. Live Coding activated.
INFO  [io.quarkus] Installed features: [..., hibernate-orm, jdbc-postgresql, ...]
----

[NOTE]
.Why environment variables, not an edit to the code
====
The claims service reads its database URL, username, and password from environment variables, so the *same image* runs in dev, stage, and prod with only configuration changing.
You just used that: you pointed a dev instance at the cluster database without touching a line of code.
The schema strategy `update` tells your dev instance to *use* the existing tables, not drop and reseed them â€” so you never clobber the shared database.
Externalizing configuration like this is the subject of a later module (Config, Secrets & Multi-Environment â€” a separate module).
====

With dev mode running, confirm your workspace instance is talking to the in-cluster database.
In the workspace terminal, in a *second* IDE terminal tab (leave dev mode running in the first):

[source,sh]
----
curl -s http://localhost:8080/api/claims | head -c 200
----

You should see real claims â€” `CLM-1001`, `CLM-1002`, â€¦ â€” the 30 seeded rows served from `claims-db`.

[.checkpoint]
****
âś” *Verify:* Dev mode logs `Profile dev activated. Live Coding activated`, and `curl http://localhost:8080/api/claims` returns seeded claims.
If you get `Connection refused`, dev mode has not finished starting â€” wait for the `Listening on` line.
If the app fails with a datasource error, re-check the `export` lines (host `claims-db.{user}-dev.svc`).
****

== 3. Change a live endpoint and watch it hot-reload
âŹ± *~12 min* Â· the inner loop at full speed

Now the payoff: change the running service without rebuilding or restarting it.
Leave dev mode running.

In the IDE Explorer, open `src/main/java/com/parasol/claims/ClaimResource.java`.
Add a small new endpoint just before the final closing brace of the class:

[source,java]
----
    @GET
    @Path("/ping")
    @Produces(MediaType.TEXT_PLAIN)
    public String ping() {
        return "parasol-claims dev mode: hot reload works";
    }
----

Save the file (*Ctrl+S*).
You did *not* restart anything.
Now hit the new endpoint from your second terminal tab:

[source,sh]
----
curl http://localhost:8080/api/claims/ping
----

The first request triggers Quarkus to recompile just what changed and reload it in place â€” you will see it in the dev-mode terminal:

.Expected output (in the dev-mode terminal)
[source,texinfo]
----
INFO  [io.quarkus.deployment.dev.RuntimeUpdatesProcessor] Restarting quarkus due to changes in ClaimResource$NewClaim.class, ClaimResource.class, ClaimResource$StatusUpdate.class.
INFO  [io.quarkus.deployment.dev.RuntimeUpdatesProcessor] Live reload total time: 2.2s
----

â€” and `curl` prints `parasol-claims dev mode: hot reload works`.
That endpoint did not exist a moment ago.
No image was built, no Pod was replaced; the loop was *seconds*.

Prove it is the *same shared database* as the deployed app.
Create a claim through your dev instance, then read it back through the *deployed* app's public Route â€” from the *Showroom terminal* this time:

[source,sh]
----
# workspace terminal: create a claim via your dev instance
curl -s -X POST http://localhost:8080/api/claims \
  -H 'Content-Type: application/json' \
  -d '{"claimant":"Dev Mode Demo","type":"auto","amount":500.00}'
----

[source,sh,role=execute,subs="attributes"]
----
# Showroom terminal: read it back through the DEPLOYED app's Route
CLAIMS_HOST=$(oc -n {user}-dev get route parasol-claims -o jsonpath='{.spec.host}')
curl -s "http://$CLAIMS_HOST/api/claims" | tr ',' '\n' | grep -A1 "Dev Mode Demo"
----

Your new claim, written by the workspace dev instance, is visible through the separately-deployed app â€” because both point at the one `claims-db`.

[.checkpoint]
****
âś” *Verify:* `curl .../api/claims/ping` returns your string, the dev-mode terminal shows a `Live reload` line, and the claim you POSTed appears through the deployed Route.

== 4. Extend the workspace: add a Valkey cache to the devfile
âŹ± *~15 min* Â· the devfile is yours to change â€” restart the workspace from it

A workspace is not fixed â€” it is whatever its devfile says.
Add a second container (a Valkey cache the app could use) and a command to talk to it, then restart the workspace *from the devfile* so the new container appears.

[NOTE]
.Why Valkey, not Redis
====
*Valkey* is the Linux Foundation's BSD-licensed continuation of Redis, created after Redis changed its license.
Red Hat ships a supported *Valkey* image and it is a drop-in for the same in-memory cache role â€” the wire protocol and the client commands (`PING`, `SET`, `GET`, â€¦) are unchanged, so what you learn here transfers directly.
====

Stop dev mode (*Ctrl+C*).
In the IDE Explorer, open `devfile.yaml` at the repository root.
Under `components:`, add a `cache` component, and under `commands:`, add a `cache-ping` command:

[source,yaml]
----
  # add under components:
  - name: cache
    container:
      # pinned by digest: tags move, digests don't â€” a devfile exists to be reproducible
      image: registry.redhat.io/rhel9/valkey-8@sha256:21f04dd3590cbb3561af0ced8e0e1e8cc1039023d808adcbb0c1cf10dca5b6fa
      memoryLimit: 256Mi
      endpoints:
        - name: valkey
          targetPort: 6379
          exposure: internal
----

[source,yaml]
----
  # add under commands:
  - id: cache-ping
    exec:
      label: Ping the Valkey cache
      component: cache
      commandLine: valkey-cli ping
----

Save the file.
Now restart the workspace so Dev Spaces rebuilds it from the updated devfile:

. Open the command palette (*F1* or *Ctrl+Shift+P*).
. Run *Dev Spaces: Restart Workspace from Local Devfile*.

When the workspace comes back, the `cache` container is part of it.
Confirm it by running the devfile command you just declared: open the command palette (*F1*) â†’ *Tasks: Run Task* â†’ *devfile: Ping the Valkey cache*.
It executes `valkey-cli ping` *inside the `cache` container* (the `valkey-cli` binary lives there, not in your `tools` terminal).

.Expected output
[source,texinfo]
----
PONG
----

The two containers share the workspace Pod's network, so your app â€” and your terminal â€” can reach Valkey at `localhost:6379`.
Prove it from the *workspace terminal* without any Valkey tooling, using bash's built-in TCP support:

[source,sh]
----
exec 3<>/dev/tcp/localhost/6379 && printf 'PING\r\n' >&3 && head -c 7 <&3
----

.Expected output
[source,texinfo]
----
+PONG
----

No service, no route â€” just a sidecar you declared in Git.

[.checkpoint]
****
âś” *Verify:* After the restart, the *devfile: Ping the Valkey cache* task returns `PONG`, and the workspace now runs a `cache` container (the terminal TCP probe answers `+PONG` too).
If the restart fails, your devfile has a YAML error â€” check the indentation of the two blocks you added (see xref:devspaces-inner-loop/troubleshooting.adoc[troubleshooting]).
****

== 5. Commit and push your change
âŹ± *~10 min* Â· the hand-off from inner loop to outer loop

Your `/ping` change lives only in the workspace.
Commit it and push it to your fork â€” this is the moment the inner loop hands off to the outer loop that the rest of the workshop builds (pipelines, GitOps).

In the workspace terminal, first tell git who you are â€” a fresh workspace container has no git identity yet (set it once per workspace; the values only label your commits):

[source,sh,subs="attributes"]
----

git config --global user.name "{user}"
----

Then commit and push:

[source,sh]
----
cd /projects/parasol-claims
git add -A
git commit -m "Add /api/claims/ping endpoint"

----

The *first* push will likely fail asking for credentials:

.Expected output (the auth wall)
[source,texinfo]
----
fatal: could not read Username for 'https://...': No such device or address
----

That is expected: your workspace can *read* the public repo but has no credentials to *write* to it yet.
Give it your workshop credentials once, then push again:

[source,sh,subs="attributes"]

.Expected output

=== Guaranteed fallback: attach `jdb` from the terminal

[source,text]

== 7. Beyond dev mode: reach the database and ship a container
âŹ± *~15 min* Â· port-forward an internal service, then deploy your change as a real Deployment

Dev mode is the fast heart of the inner loop, but two everyday developer moves live just past it: reaching a cluster service directly with your own tools, and turning a change into a running Deployment the platform manages.
Both are the bridge from the inner loop to the outer loop.

[IMPORTANT]
.These steps run in the *Showroom terminal*, not the workspace
====
The `oc` inside the *workspace* terminal is authenticated as the workspace's *ServiceAccount*, scoped to your `{user}-devspaces` project â€” it cannot read or change anything in `{user}-dev` (run `oc whoami` in the workspace terminal and you will see a `system:serviceaccount:â€¦` identity, not `{user}`).
So the cluster-side steps below run in the *Showroom terminal*, where you are signed in as `{user}` with full rights in your own `{user}-dev` project.
====

=== Port-forward the database to your localhost

The `claims-db` PostgreSQL database has *no Route* â€” it is internal-only, reachable inside the cluster by DNS (that is the `claims-db.{user}-dev.svc` name your dev instance used).
To reach it from a database tool on your own machine â€” `psql`, a GUI client, or your IDE's database panel â€” bridge it to a local port with `oc port-forward`.
In the *Showroom terminal*:

[source,sh,role=execute,subs="attributes"]
----
oc -n {user}-dev port-forward svc/claims-db 5432:5432
----

.Expected output (it keeps running â€” leave it)
[source,texinfo]
----
Forwarding from 127.0.0.1:5432 -> 5432
Forwarding from [::1]:5432 -> 5432
----

`localhost:5432` in your environment is now the in-cluster database: point any PostgreSQL client at it and you are querying the real `claims-db`, through a temporary tunnel that closes when you press *Ctrl+C*.

[TIP]
.Prove the tunnel carries PostgreSQL (optional)
====
The Showroom terminal ships no `psql`, but bash can speak enough of the wire protocol to prove what is on the other end â€” the same TCP trick you used for the cache.
In a second terminal tab, send a PostgreSQL SSL-negotiation request and read the one-byte reply:

[source,sh]
----
exec 3<>/dev/tcp/localhost/5432
printf '\x00\x00\x00\x08\x04\xd2\x16\x2f' >&3
read -t 3 -n1 reply <&3 && echo "claims-db answered on localhost:5432 with: $reply"
----

A reply of `N` (SSL off) or `S` (SSL on) is PostgreSQL's own answer â€” you have reached the database through the tunnel.
Press *Ctrl+C* on the port-forward when you are done.

=== Ship your pushed change as a running Deployment

Dev mode is for iterating; it is not where the app runs for real.
The `/api/claims/ping` endpoint you pushed to your fork in exercise 5 is only source right now â€” turn it into a running container the platform builds, runs, and heals.
`oc new-app` does the whole source-to-Deployment step from your fork in one command (the same on-ramp the *Ways to Build & Deliver Apps* module covers in depth).

In a *fresh Showroom terminal tab* (so the port-forward keeps running):

[source,sh,role=execute,subs="attributes"]
----
oc -n {user}-dev new-app java-21~{gitea_url}/{user}/parasol-claims.git --name=parasol-claims-mine
----

That creates a BuildConfig, an ImageStream, a Deployment, and a Service â€” all named `parasol-claims-mine`, so they sit *beside* the pre-staged `parasol-claims`, not on top of it.
Wire it to the shared database (the same Secret the pre-staged app uses) and wait for the build and rollout:

[source,sh,role=execute,subs="attributes"]
----
oc -n {user}-dev set env deploy/parasol-claims-mine \
  --from=secret/claims-db \
  POSTGRESQL_HOST=claims-db \
  QUARKUS_HIBERNATE_ORM_SCHEMA_MANAGEMENT_STRATEGY=update
oc -n {user}-dev logs -f buildconfig/parasol-claims-mine
oc -n {user}-dev rollout status deploy/parasol-claims-mine
----

The S2I build takes a few minutes the first time â€” Maven resolves dependencies and compiles your fork.
The `update` schema strategy is the same one you used in dev mode: your new instance *uses* the existing tables instead of dropping and reseeding the shared database.
When `rollout status` reports the Deployment available, give it a browser-reachable Route (edge TLS) and call *your* endpoint on the running container:

[source,sh,role=execute,subs="attributes"]
----
oc -n {user}-dev create route edge parasol-claims-mine \
  --service=parasol-claims-mine --insecure-policy=Redirect
MINE=$(oc -n {user}-dev get route parasol-claims-mine -o jsonpath='{.spec.host}')
curl -s https://$MINE/api/claims/ping; echo
----

.Expected output
[source,texinfo]
----
parasol-claims dev mode: hot reload works
----

The exact string you wrote in dev mode is now served by a container the platform runs â€” replicated, health-checked, restarted for you if it dies.
That is the outer loop taking your change: from here a pipeline and GitOps would build and promote it (later modules), instead of you running `oc new-app` by hand.

[.checkpoint]
****
âś” *Verify:* `oc -n {user}-dev get deploy parasol-claims-mine` shows the Deployment available, and `curl https://$MINE/api/claims/ping` returns your `/ping` string from the *deployed* container (not dev mode).
****

=== Going further: debug and hot-reload the running container

You attached a debugger to dev mode in exercise 6; the same attach works against a *deployed* container once you open its debug port.
Turn JDWP on for your Deployment and forward the port from the *Showroom terminal*:

[source,sh,subs="attributes"]
----
oc -n {user}-dev set env deploy/parasol-claims-mine \
  JAVA_TOOL_OPTIONS='-agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=*:5005'
oc -n {user}-dev rollout status deploy/parasol-claims-mine
oc -n {user}-dev port-forward deploy/parasol-claims-mine 5005:5005
----

// [CAPTURE-VERIFY] The final debugger *attach* is driven from a graphical IDE or a JDK shell on
// the host holding the forwarded port; it cannot be exercised headlessly here. The oc scaffolding
// above (JDWP env + rollout + port-forward) is grounded on-cluster; the attach mirrors exercise 6.
With `5005` forwarded, attach a debugger to `localhost:5005` from a client that can reach that port â€” `jdb -attach localhost:5005`, or your IDE's *Attach* configuration from exercise 6 â€” and you are stepping through the *deployed* container instead of dev mode.

For a live *edit-and-see* loop against a running Pod (not just local dev mode), Quarkus offers *remote development mode*: you run a mutable build in the cluster with `QUARKUS_LAUNCH_DEVMODE=true` and connect to it with `./mvnw quarkus:remote-dev`, so a saved file hot-reloads straight into the Pod.
It is an advanced pattern â€” the Quarkus link in the wrap-up has the details.
