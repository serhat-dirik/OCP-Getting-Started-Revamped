== Creating the PrometheusRule is denied: `Forbidden ... cannot create resource "prometheusrules"`

Symptom:: In exercise 2, `oc apply` (or console *Import YAML*) of the alert rule returns `prometheusrules.monitoring.coreos.com is forbidden: User "<user>" cannot create resource "prometheusrules" in ... namespace "<user>-dev"`.
Cause:: The project `admin` role does not include the `monitoring.coreos.com` API group.
Authoring a `PrometheusRule` (or reading a `ServiceMonitor`) needs the `monitoring-edit` ClusterRole bound in your namespace — a *platform* grant, not something you did wrong.
Fix:: This is a workshop-layer RoleBinding your platform team applies per attendee; it is not self-service.
Flag it to your instructor.
For reference, the binding is:
+
[source,yaml]
----
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: monitoring-edit
  namespace: <user>-dev
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: monitoring-edit
subjects:
  - apiGroup: rbac.authorization.k8s.io
    kind: User
    name: <user>
----
Once it is in place, `oc auth can-i create prometheusrules -n <user>-dev` prints `yes` and the rule applies.
The same grant is what lets `ws verify m12` read your ServiceMonitor and rule.

== My alert never fires even though the app is clearly failing

Symptom:: You scaled the database to zero, the claims API is failing, but `ParasolClaimsErrorRateHigh` sits in *Inactive* (or flaps briefly into *Pending* and falls back) — it never reaches *Firing*.
Cause:: You are banking too few 5xx before the failing pod removes itself from the scrape.
While the database is down the app returns 500s, but within ~60–80s it trips its own `/q/health/ready` probe; Kubernetes then pulls the pod out of the Service endpoints (you will see the sampled requests flip from `500` to `503` as it leaves rotation), the ServiceMonitor's scrape *freezes*, and the 5xx counter stops climbing.
A *serial* one-request-at-a-time loop — plus the load generator's steady trickle — banks only a couple of dozen 5xx before that freeze, a 5-minute rate of ~0.08/s that is just *under* the rule's `0.1/s` threshold, so `rate(...[5m])` never crosses the line and the `for: 2m` timer never completes.
Fix:: Drive a *concurrent* burst, not a serial loop, so you bank hundreds of 5xx in the first half-minute — while the pod is still being scraped — and the rate clears `0.1/s` with room to spare:
+
[source,sh,role=execute]
----
end=$(( $(date +%s) + 90 ))
for w in $(seq 1 20); do
  ( while [ "$(date +%s)" -lt "$end" ]; do curl -s -o /dev/null "$CLAIMS/api/claims"; done ) &
done
wait
----
+
With the burst the 5xx rate holds around ~12/s and the rule reaches *Pending* at ~59s and *Firing* at ~178s (about 3 minutes).
The `for: 2m` window over a `rate(...[5m])` is deliberately slow so a brief blip does not page anyone — confirm the rule loaded (`oc get prometheusrule parasol-claims-alerts -n <user>-dev`) and give it the full window.

== I brought the database back, but the claims API still returns 500

Symptom:: You ran `oc scale deploy/claims-db --replicas=1`, the database pod is `Running`, but `curl $CLAIMS/api/claims` keeps returning `HTTP 500`.
Cause:: The claims application is still holding *dead connections* in its Agroal connection pool — they point at the old database pod that no longer exists.
A healthy database does not heal a poisoned pool.
Fix:: Restart the app so it rebuilds the pool against the current database:
+
[source,sh,role=execute]
----
oc rollout restart deploy/parasol-claims -n <user>-dev
oc rollout status deploy/parasol-claims -n <user>-dev
----
Then `curl $CLAIMS/api/claims` returns `HTTP 200`.
This "recovery needs a restart" step is a real operational lesson — a dependency coming back does not always heal the caller.

== `curl`-ing PromQL against thanos-querier returns `Forbidden`

Symptom:: Following a generic monitoring how-to, you try `curl -H "Authorization: Bearer $(oc whoami -t)" https://thanos-querier.../api/v1/query ...` and get `Forbidden (user=<user>, verb=get, resource=prometheuses, subresource=api)`.
Cause:: The `thanos-querier` route enforces a cluster-monitoring view permission attendees do not have, and this cluster exposes only that route (not the per-namespace tenancy endpoint).
Ad-hoc PromQL over that route is not an attendee capability here.
Fix:: Use *Observe → Metrics* in the console for PromQL — it queries with your identity, scoped to your namespace, and needs no extra role.
Use `curl $CLAIMS/q/metrics` when you want the raw counter values at the source.
That two-surface split (console for queries, `curl` for raw truth) is exactly how the lab is written.

== Observe → Traces is empty

Symptom:: You open *Observe → Traces*, run `{ resource.service.name = "parasol-claims" }`, and get no results.
Cause:: Usually one of: no recent request has been traced (Tempo runs an in-memory backend on the shared cluster and keeps only *recent* traces), the time range is too narrow, or the shared collector is down.
Fix:: Drive a request and widen the range.
The load generator produces traffic continuously, so a request within the last few minutes should appear — hit the app yourself to be sure:
+
[source,sh,role=execute]
----
curl -s -o /dev/null "$CLAIMS/api/claims/CLM-1001/history"
----
+
Then refresh with a *Last 15 minutes* range.
If it is still empty, the collector may be down — that is a shared-stack issue for your instructor (`oc get pods -n observability-workshop`), not something in your namespace.

== The trace only shows one span — where are the database queries?

Symptom:: You open a `parasol-claims` trace expecting to see the N+1 database queries, but there is a single HTTP request span and no `SELECT` children.
Cause:: The claims image emits the *HTTP request* span but not per-query *database* spans — database-span instrumentation is a build-time flag on the image, tracked as an enhancement.
The N+1 is real in the code, but invisible in the trace today.
Fix:: There is nothing to fix in your namespace.
Use the trace to identify *which* endpoint is the latency outlier (rank by duration), and read the code to see *why* (`GET /api/claims/\{claimNumber}/history` issues one query per event).
The database-span "staircase" appears once the image ships that flag.

== My HPA shows `<unknown>/60%` and is not scaling

Symptom:: Just after `oc autoscale`, `oc get hpa` shows `TARGETS` as `<unknown>/60%` and the app is not scaling.
Cause:: Two possibilities: the metrics pipeline has not reported CPU for the new HorizontalPodAutoscaler (HPA) yet (it takes a short interval after creation), or the target container has no CPU *request*, so utilization cannot be computed.
The entry-state app *does* set a 200m CPU request, so on this lab it is almost always the former.
Fix:: Wait ~30 seconds and re-check — `TARGETS` resolves to a real percentage (e.g. `18%/60%`) once the first CPU sample lands.
If it stays `<unknown>`, confirm the request is present: `oc get deploy parasol-claims -n <user>-dev -o jsonpath='{.spec.template.spec.containers[0].resources.requests.cpu}{"\n"}'` should print `200m`.

== The HPA will not scale back down after I stop the load

Symptom:: You stopped the burst, `TARGETS` fell below 60%, but `REPLICAS` stays at 4 for minutes.
Cause:: The default *5-minute scale-down stabilization window*.
The HPA holds the higher count so a brief lull does not cause a shrink-then-grow flap.
Fix:: Nothing — this is correct.
Wait ~5 minutes and it steps back toward the floor of 2.
(Scale-*up* has no such delay, which is why the burst grew it in seconds.)
