= Config, Secrets & Multi-Environment: Instructor Guide
:module-id: config-multienv
include::partial$version-attributes.adoc[]

// Internal delivery runbook. Timings measured on the build cluster (OCP {ocp_version},
// Quarkus {quarkus_version}) as user6 on 2026-07-09 by performing every exercise on-cluster;
// see the "how timing was measured" note below.

This module is where configuration stops being an afterthought and becomes a discipline.
The two ideas that must land are *"the same image runs everywhere; config is what changes"* and *"readiness holds traffic away from a pod that is not ready."*
The three deliberate breaks — a bad datasource URL, a broken readiness probe, and a quota-busting request — are the spine; each one is a "read the error, then fix it" beat, not a detour.

The biggest delivery subtlety is the readiness gate, which now drives *both* the exercise-1 and exercise-4 breaks and does *not* behave the naive way: in exercise 1 the bad datasource makes the new pod fail readiness, so it is held at `0/1` while the *old* pod keeps serving (the Route stays `200`, not `503`); in exercise 4 the same gate only forces a `503` once you scale the deployment 0→1 so there is no ready pod left.
Attendees expect a bad config to "take the app down" and are surprised the old pod keeps serving — that surprise *is* the lesson, and it is in the Top-5 below.

== Timing

Total hands-on: *~80 min* (spec target 75 min — within tolerance).
Add ~10–12 min of in-room concept discussion up front.

[cols="1,4,1,2",options="header"]
|===
| # | Exercise group | Attendee time | Measured core

| 1 | Break the app with a bad setting | ~10 min | new pod `0/1` within *~1–2 s* (readiness); old pod keeps serving (Route `200`); `CrashLoopBackOff` at *~90 s*
| 2 | Fix it via a ConfigMap | ~10 min | each `oc set env` rollout *~15–40 s*
| 3 | Secret + mounted-file config | ~15 min | mount takes effect on next rollout; access-log line immediate
| 4 | Add probes; break/fix readiness | ~15 min | new pod NotReady in *~30 s*; stable Route `503` after scale 0→1; recovery *~15 s*
| 5 | Requests/limits; bust the quota | ~12 min | quota rejection is *immediate* (admission-time `ReplicaFailure`)
| 6 | Promote to stage and prod | ~13 min | each env boots + seeds in *~10 s* (Quarkus `started in 10.1 s`); ready with startup probe *~30–45 s*
| 7 | [OPTIONAL] External Secrets | ~5 min | read / instructor-led — no hands-on (External Secrets Operator not installed)
|===

[NOTE]
.How this timing was measured
====
The *Measured core* column is real.
On the build cluster on 2026-07-09 (with the exercise-1 readiness-gate rework re-measured on 2026-07-15), every exercise was performed from the Showroom-terminal commands in the lab:

* *Config break (exercise 1)* — with a readiness probe added first, a bad `QUARKUS_DATASOURCE_JDBC_URL` makes the new pod fail readiness *immediately* (`0/1` within ~1–2 s) instead of flashing a deceptive `1/1 Running`.
The rolling update is gated, so the old pod keeps serving (Route `200`); the new pod boots (~22 s), throws `UnknownHostException`, exits `1`, and reaches `CrashLoopBackOff` after ~90 s.
* *Readiness break* — pointing readiness at a `404` path makes the *new* pod NotReady in ~30 s while the old pod keeps serving (the rolling update protects users).
Only after the last ready pod is removed does the Route `503`.
Fix + recovery is ~15 s.
* *Quota bust* — a pod requesting `requests.cpu=4` against a `3`-core cap is refused at admission time, immediately, with `exceeded quota: workshop-quota`.
* *Promotion* — each of stage/prod boots the app and seeds its ephemeral DB in ~10 s (`parasol-claims 1.0.0 ... started in 10.101s`); with the startup probe it reports Ready in ~30–45 s.

The *Attendee time* column adds realistic reading, checkpoint, and think-time on top of that core.
====

== Pre-flight checklist

Run these *before the session* (as an instructor/admin).

. *The entry state materializes and verifies* for a sample user.
+
--
[source,sh,role=execute]
----
ws start config-multienv --user user1
ws verify config-multienv --user user1 --entry-only
----

Expect `✅ all 13 checks passed` (three namespaces, three markers, quota, Gitea fork, dev claims + DB ready, Route 200, and stage/prod empty).
--

. *The prebuilt image is present* (the app that all three environments run):
+
[source,sh,role=execute]
----
oc get istag parasol-claims:1.0 -n parasol-images \
  -o jsonpath='{.image.dockerImageReference}{"\n"}'
----
+
If it is missing, no environment will start — rebuild it (see the app README) before the session.

. *The per-namespace guardrails are in place* for the sample user (dev, stage, and prod each carry the same quota/limits):
+
[source,sh,role=execute]
----
for ns in user1-dev user1-stage user1-prod; do
  echo "== $ns =="
  oc get resourcequota workshop-quota -n $ns -o jsonpath='requests.cpu cap={.status.hard.requests\.cpu}{"\n"}'
  oc get limitrange workshop-limits -n $ns -o jsonpath='defaultRequest={.spec.limits[0].defaultRequest}{"\n"}'
done
----
+
Expect `requests.cpu cap=3` and `defaultRequest={"cpu":"100m","memory":"256Mi"}` in each.

. *The demo end state stands up* (for the demo flavor): `ws solve config-multienv --user user1`, then confirm all three environments serve:
+
[source,sh,role=execute]
----
for ns in user1-dev user1-stage user1-prod; do
  H=$(oc -n $ns get route parasol-claims -o jsonpath='{.spec.host}')
  curl -s -o /dev/null -w "$ns -> %{http_code}\n" http://$H/api/claims
done
----
+
Expect `200` from all three.

== Demo-mode notes

Beyond the Say/Show/Do arc in the demo flavor:

* *Solve first, always.*
Run `ws solve config-multienv` a few minutes ahead so dev is configured and stage/prod are promoted.
The promotion boots two databases; do not do it live under time pressure.
* *The readiness break needs the "scale 0→1" beat.*
If you only break readiness, the old pod keeps serving and the Route stays `200` — which is a great lesson (the rollout is *gated*), but not the `503` you are selling.
Scale the deployment to zero and back to one so the *only* pod uses the broken probe, and the audience sees the endpoints drain and a stable `503`.
Say out loud *why* it stayed green first.
(Deleting a pod will not do it — the old ReplicaSet recreates a ready one.)
* *Show the digest side by side.*
The single most convincing frame is the three-line `for ns in ...` output with the *identical* `sha256:` in all three environments.
Let it sit on screen.
* *Prod is quieter on purpose.*
If someone notices prod logs almost nothing, that is the `WARN` log level from its config — a live example of "config changes behavior," not a bug.

== Top questions attendees ask

*1. "Exercise 1 shows *two* pods and the app still answers — did the break fail?"*::
No — that is the readiness gate working.
Because the attendee adds a readiness probe *before* breaking the datasource, the new (broken) pod never passes readiness, so the rolling update is *gated*: the new pod is held at `0/1` while the old, healthy pod keeps serving (the Route stays `200`).
Tell the room to watch the `READY` column (`0/1`), not `STATUS` — `STATUS` oscillates `Running`/`Error`/`CrashLoopBackOff` as the container boots and dies, but the pod is never actually ready.
For the root cause, read the *newest* pod's `--previous` logs by name (as the lab does): a plain `oc logs -l app=parasol-claims --previous` *errors* here, because the healthy old pod has no previous container.

*2. "Why did `oc set env --from=configmap/...` not error when those variables already existed inline?"*::
`oc set env --from` defaults to *overwrite*: for each key in the ConfigMap that matches an existing inline variable, it replaces the inline value with a reference to the ConfigMap.
That is exactly what you want here — it is what turns the hardcoded values into sourced ones.
(There is no `oc set env` flag to remove an `envFrom` block, which is why the entry state uses inline vars — they are cleanly removable.)

*3. "Is a Secret encrypted?"*::
No.
A Secret's values are *base64-encoded*, which is trivially reversible — the lab decodes the password in one command.
What a Secret buys you is *separate RBAC* (deploy without read-credentials) and one place to rotate.
Encryption of the source of truth is what a real secrets manager (Vault, cloud KMS, via the External Secrets Operator) adds — the optional section covers it.

*4. "When I broke readiness in exercise 4, why didn't the Route `503` right away?"*::
For the same reason the bad datasource in exercise 1 did not take the app down: a probe change triggers a *rolling update*, and the rollout is itself gated by readiness, so the new (broken) pod never becomes Ready and is held out of the Service while the old, healthy pod keeps serving.
That is the platform protecting users.
You only see the `503` once there is *no* ready pod at all — which is why the lab scales the deployment to zero and back to one, so the *only* pod uses the broken probe.
(Deleting individual pods does not work: the old ReplicaSet keeps recreating a ready pod, so the Route recovers to `200`.)

*5. "Why does each environment get its own database?"*::
So the environments are genuinely independent — promoting to prod must not touch stage's data.
Each overlay stands up its own ephemeral PostgreSQL, seeded with the same 30 claims at boot.
(They are ephemeral on purpose; persistent storage is the next module.)

== Watchouts

* *Probe timing can flake under load* (the spec's named risk).
The lab uses generous values (startup `period=3s failureThreshold=30` ≈ 90 s of grace; readiness/liveness `period=10s`).
On a busy cluster with ~30 users promoting at once, image pulls and scheduling can push a slow boot past a tight probe — the generous startup window is what prevents a healthy-but-slow pod from being killed.
Do not tighten these for the workshop.
* *Quota headroom across three namespaces.*
Each of `{user}-dev/-stage/-prod` has its *own* `requests.cpu=3` / `pods=30` quota, so the promotion (1 + 2 + 3 claims replicas plus a DB each) is comfortable.
The only place a user hits the quota is the *deliberate* bust in exercise 5 — that is the lesson, not a problem.
* *Two pods in exercise 1 confuse people.*
Because a readiness probe is added before the break, the gated rollout leaves *two* pods on screen — the old one `1/1` (still serving) and the broken new one `0/1`.
Tell the room to watch the `READY` column, not `STATUS` (which oscillates), and that the app staying up is the point, not a sign the break failed.
* *Rapid `oc set env`/`oc set probe` cause several quick rollouts.*
Between rollouts the Route can briefly `503`.
The lab waits for the *final* `oc rollout status`; attendees who curl mid-rollout will see a transient `503` — that is normal, have them wait for the rollout to finish.
* *The promotion clones over HTTPS.*
`git clone` of the public fork needs no credentials.
If a proxy or TLS interception is in play, the clone can fail where `oc` succeeds — check the Gitea route is reachable from the Showroom terminal (`curl -sI {gitea_url}`).
