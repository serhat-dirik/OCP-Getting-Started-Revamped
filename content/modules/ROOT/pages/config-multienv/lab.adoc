In this lab you take the claims service from a fragile, hardcoded configuration to a properly externalized one, and then promote it across three environments.
You break the app with a bad setting and read the failure, move its configuration into a ConfigMap, move its credentials into a Secret, add health probes and watch a readiness gate hold traffic, set resource requests and bump into the namespace quota, and finally promote the *same image* to stage and prod with per-environment config.
Everything happens in your own namespaces — `{user}-dev`, `{user}-stage`, `{user}-prod` — so nothing you do can affect another attendee.

[TIP]
.Two ways to do most steps
====
Most steps below are offered two ways.
Use the *Console* tab for the graphical workflow in the {ocp_console_url}[OpenShift web console^] (sign in as `{user}`), or the *CLI* tab for `oc` in the *Terminal* tab beside these instructions — the two paths create the same objects.
Pick a path once: the tabs stay in sync across the page.
A few steps are CLI-only — promoting with Kustomize, delivering config as a mounted file, and reading a crash-looping pod's _previous_ logs — and say so where they appear.
The console click-paths follow the OpenShift web console; if a label reads slightly differently on your cluster, the `oc` command beside it is the source of truth.
====

== 1. Break the app with a bad setting — and read the failure
⏱ *~10 min* · the first deliberate break: a bad setting fails loudly and safely

Your claims service is running in `{user}-dev`, but it is configured the *naive* way: every setting, including the database password, is hardcoded inline in the Deployment.
See for yourself:

[source,sh,role=execute,subs="attributes"]
----
oc project {user}-dev
oc set env deploy/parasol-claims --list
----

.Expected output
[source,texinfo]
----
# deployments/parasol-claims, container parasol-claims
POSTGRESQL_HOST=claims-db
POSTGRESQL_PORT=5432
POSTGRESQL_DATABASE=parasol
POSTGRESQL_USER=parasol
POSTGRESQL_PASSWORD=parasol
----

Everything — including `POSTGRESQL_PASSWORD` — is sitting in plain text in the Deployment.
We will fix that.
But first, feel *why* configuration matters: change one setting to a bad value and watch what the platform does about it.

To *see* a bad setting the moment it lands, first give the platform a way to judge the app's health.
Add a *readiness probe* — the platform's "can this pod serve traffic *right now*?" check — pointed at the health endpoint the claims service already exposes.
Without one, the platform counts a container as ready the instant its process starts, even while it is seconds from crashing; with one, a pod that cannot serve is held out of rotation.
(You add the full set of probes — startup and liveness too — in exercise 4; this one readiness check is what makes the next failure visible.)

// [CAPTURE-VERIFY] Deployment > Actions > "Add Health Checks" > Readiness probe (HTTP GET :8080 /q/health/ready) — CLI 'oc set probe --readiness' authoritative.
[tabs]
====
Console::
+
--
. Go to *Workloads* → *Deployments* → `parasol-claims` → *Actions* → *Add Health Checks*.
. Add a *Readiness* probe — an *HTTP GET* on port `8080`, path `/q/health/ready` — and click *Save*.
--
CLI::
+
--
[source,sh,role=execute]
----
oc set probe deploy/parasol-claims --readiness \
  --get-url=http://:8080/q/health/ready --period-seconds=10 --failure-threshold=3
oc rollout status deploy/parasol-claims
----
--
====

The app is still healthy — one pod, `1/1` ready.
Now break it: point the datasource at a database host that does not exist.

// [CAPTURE-VERIFY] Deployment > Environment tab: the "Single values (env)" add-row label — CLI 'oc set env' beside it is authoritative.
[tabs]
====
Console::
+
--
. Go to *Workloads* → *Deployments* → `parasol-claims`, and open the *Environment* tab.
. Under *Single values (env)*, add a variable: name `QUARKUS_DATASOURCE_JDBC_URL`, value `jdbc:postgresql://wrong-db-host:5432/parasol`.
. Click *Save*.
The Deployment rolls out a new pod that picks up the bad datasource URL.
--
CLI::
+
--
[source,sh,role=execute]
----
oc set env deploy/parasol-claims \
  QUARKUS_DATASOURCE_JDBC_URL=jdbc:postgresql://wrong-db-host:5432/parasol
----
--
====

`QUARKUS_DATASOURCE_JDBC_URL` is the environment-variable form of the app's datasource URL, and it *overrides* the correct value the image ships with.
Because this is a rolling update — the same strategy you met in *Platform Orientation & First App* — the platform brings up a *new* pod alongside the old one and holds off retiring the old one until the new pod passes readiness.
It never will.
Look right away:

[source,sh,role=execute]
----
oc get pods -l app=parasol-claims
----

.Expected output (the broken pod is held at `0/1`; the old one still serves)
[source,texinfo]
----
NAME                              READY   STATUS             RESTARTS      AGE
parasol-claims-6764c459d6-dgmsm   1/1     Running            0             11m
parasol-claims-7874794948-pw8lr   0/1     CrashLoopBackOff   3 (19s ago)   2m20s
----

The new pod is `0/1` — *NotReady* — almost immediately: it cannot reach the database, so its readiness check never passes and the platform refuses to put it into service.
The old, healthy pod keeps running and keeps serving.
That is the readiness gate doing its job: a broken new version is *held back*, not shipped.
(Without the probe you just added, the platform would have swapped straight to the broken pod, which would have reported a reassuring `1/1 Running` for the better part of a minute before it finally started crash-looping — the failure would have been invisible exactly when you needed to see it.)

[NOTE]
.Watch the READY column, not STATUS
====
The `STATUS` column oscillates — `Running`, `Error`, `CrashLoopBackOff` — as the container boots, fails, and is restarted on a back-off timer.
Ignore it.
The honest signal is the `READY` column: it stays `0/1` the whole time, because the pod never once passes its readiness check.
====

The rollout is now stuck on that unready pod — confirm it, without waiting forever:

[source,sh,role=execute]
----
oc rollout status deploy/parasol-claims --timeout=15s
----

.Expected output
[source,texinfo]
----
Waiting for deployment "parasol-claims" rollout to finish: 1 old replicas are pending termination...
error: timed out waiting for the condition
----

And because the old pod is still in the Service, your users are unaffected — the Route keeps answering:

[source,sh,role=execute]
----
curl -s -o /dev/null -w "Route: %{http_code}\n" "http://$(oc get route parasol-claims -o jsonpath='{.spec.host}')/api/claims"
----

.Expected output
[source,texinfo]
----
Route: 200
----

Now do the thing this whole module is about: *read the logs.*
The broken pod is the *newest* one, so grab its name; then ask for its *previous* container's logs, because a crash-looping container has no current ones (targeting the newest pod also avoids the healthy old pod, which has no previous container to read):

[source,sh,role=execute]
----
BAD=$(oc get pods -l app=parasol-claims --sort-by=.metadata.creationTimestamp -o jsonpath='{.items[-1:].metadata.name}')
oc logs "$BAD" --previous --tail=40 | grep -A2 -iE 'Failed to start|UnknownHost'
----

.Expected output
[source,texinfo]
----
ERROR [io.quarkus.runtime.Application] (main) Failed to start application: java.lang.RuntimeException: Failed to start quarkus
Caused by: org.hibernate.exception.JDBCConnectionException: Unable to open JDBC Connection for DDL execution [The connection attempt failed.]
Caused by: org.postgresql.util.PSQLException: The connection attempt failed.
Caused by: java.net.UnknownHostException: wrong-db-host
----

The root cause is right there at the bottom: `UnknownHostException: wrong-db-host`.
The app cannot resolve the database host you gave it.
The lesson is not the fix — it is the *habit*: when a pod will not start, the logs almost always tell you why in plain language, and the last `Caused by` is usually the real culprit.

[.checkpoint]
****
✔ *Verify:* `oc get pods -l app=parasol-claims` shows the new pod held at `0/1` and crash-looping while the old pod still serves (the Route still answers `200`), and the newest pod's previous-container logs end with `UnknownHostException: wrong-db-host`.
Leave it broken — you fix it, properly, in the next exercise.
****

== 2. Fix it the right way: move config into a ConfigMap
⏱ *~10 min* · externalize non-secret configuration

The quick fix is to remove the bad override:

[source,sh,role=execute]
----
oc set env deploy/parasol-claims QUARKUS_DATASOURCE_JDBC_URL-
oc rollout status deploy/parasol-claims
----

(The trailing `-` on a variable name *removes* it.)
Now the new pod can reach the database, passes its readiness check, and the rollout finally completes — the old pod steps aside.
But you have just proved the deeper problem: configuration is scattered inline in the Deployment, where it is easy to fat-finger and hard to manage.
Externalize it.
Create a *ConfigMap* — a named object that holds *non-secret* configuration — for the environment name, host, port, database name, and log level:

// [CAPTURE-VERIFY] Workloads > ConfigMaps > "Create ConfigMap" form/YAML view — CLI 'oc create configmap' authoritative.
[tabs]
====
Console::
+
--
. Go to *Workloads* → *ConfigMaps* and click *Create ConfigMap* (switch to the *YAML view* if you prefer to paste the keys).
. Name it `claims-config` and add five key/value pairs: `APP_ENV`=`dev`, `POSTGRESQL_HOST`=`claims-db`, `POSTGRESQL_PORT`=`5432`, `POSTGRESQL_DATABASE`=`parasol`, `QUARKUS_LOG_LEVEL`=`INFO`.
. Click *Create*.
--
CLI::
+
--
[source,sh,role=execute]
----
oc create configmap claims-config \
  --from-literal=APP_ENV=dev \
  --from-literal=POSTGRESQL_HOST=claims-db \
  --from-literal=POSTGRESQL_PORT=5432 \
  --from-literal=POSTGRESQL_DATABASE=parasol \
  --from-literal=QUARKUS_LOG_LEVEL=INFO
----
--
====

Look at what you made:

[source,sh,role=execute]
----
oc get configmap claims-config -o yaml | grep -A5 '^data:'
----

.Expected output
[source,texinfo]
----
data:
  APP_ENV: dev
  POSTGRESQL_DATABASE: parasol
  POSTGRESQL_HOST: claims-db
  POSTGRESQL_PORT: "5432"
  QUARKUS_LOG_LEVEL: INFO
----

Now point the Deployment at the ConfigMap — this injects every key as an environment variable and, because the names match, *replaces* the matching inline values with references to the ConfigMap:

[tabs]
====
Console::
+
--
. Go to *Workloads* → *Deployments* → `parasol-claims` → the *Environment* tab.
. In the *All values from existing ConfigMaps or Secrets (envFrom)* section, add the `claims-config` ConfigMap, and click *Save*.
--
CLI::
+
--
[source,sh,role=execute]
----
oc set env deploy/parasol-claims --from=configmap/claims-config
----
--
====

Confirm the non-secret config now comes from the ConfigMap (only the credentials remain inline):

[source,sh,role=execute]
----
oc set env deploy/parasol-claims --list
----

.Expected output
[%collapsible]
=====
[source,texinfo]
----

# POSTGRESQL_HOST from configmap claims-config, key POSTGRESQL_HOST
# POSTGRESQL_PORT from configmap claims-config, key POSTGRESQL_PORT
# POSTGRESQL_DATABASE from configmap claims-config, key POSTGRESQL_DATABASE
POSTGRESQL_USER=parasol
POSTGRESQL_PASSWORD=parasol

# QUARKUS_LOG_LEVEL from configmap claims-config, key QUARKUS_LOG_LEVEL

=====

The `#`-commented lines are the values sourced from the ConfigMap.
The app is now configured from an object you can review, version, and change *without touching the Deployment* — and, crucially, without rebuilding the image.

== 3. Move credentials into a Secret — and mount config as a file
⏱ *~15 min* · secret hygiene, and env vs mounted files

Look again at the last output: `POSTGRESQL_PASSWORD=parasol` is still sitting in plain text in the Deployment.
Anyone who can read the Deployment can read the password.
Move the credentials into a *Secret*, which is stored and access-controlled separately:

// [CAPTURE-VERIFY] Workloads > Secrets > "Create" > "Key/value secret" form — CLI 'oc create secret generic' authoritative.
[tabs]
====
Console::
+
--
. Go to *Workloads* → *Secrets*, click *Create* → *Key/value secret*, and name it `claims-creds`.
. Add two keys: `POSTGRESQL_USER`=`parasol` and `POSTGRESQL_PASSWORD`=`parasol`. Click *Create*.
. Wire it into the app: *Workloads* → *Deployments* → `parasol-claims` → *Environment* tab → in the *All values from existing ConfigMaps or Secrets (envFrom)* section, add the `claims-creds` Secret, and *Save*.
--
CLI::
+
--
[source,sh,role=execute]
----
oc create secret generic claims-creds \
  --from-literal=POSTGRESQL_USER=parasol \
  --from-literal=POSTGRESQL_PASSWORD=parasol

oc set env deploy/parasol-claims --from=secret/claims-creds
----
--
====

As with the ConfigMap, this replaces the inline `POSTGRESQL_USER`/`POSTGRESQL_PASSWORD` with references to the Secret.
Now every setting is sourced from an object, and *nothing* sensitive is inline:

[source,sh,role=execute]
----
oc set env deploy/parasol-claims --list
----

.Expected output
[%collapsible]
=====
[source,texinfo]
----
# deployments/parasol-claims, container parasol-claims
# POSTGRESQL_HOST from configmap claims-config, key POSTGRESQL_HOST
# POSTGRESQL_PORT from configmap claims-config, key POSTGRESQL_PORT
# POSTGRESQL_DATABASE from configmap claims-config, key POSTGRESQL_DATABASE
# POSTGRESQL_USER from secret claims-creds, key POSTGRESQL_USER
# POSTGRESQL_PASSWORD from secret claims-creds, key POSTGRESQL_PASSWORD
# QUARKUS_LOG_LEVEL from configmap claims-config, key QUARKUS_LOG_LEVEL
----
=====

Now the honesty a Secret demands.
A Secret is *not* encrypted — its values are merely base64-encoded.
See for yourself:

[source,sh,role=execute]
----
echo -n "stored:  "; oc get secret claims-creds -o jsonpath='{.data.POSTGRESQL_PASSWORD}'; echo
echo -n "decoded: "; oc get secret claims-creds -o jsonpath='{.data.POSTGRESQL_PASSWORD}' | base64 -d; echo
----

.Expected output
[source,texinfo]
----
stored:  cGFyYXNvbA==
decoded: parasol
----

Anyone with permission to *read* the Secret reads the password.
What a Secret gives you is *separate access control* — you can let people deploy without letting them read credentials — and one place to rotate.
Encryption of the source of truth is what a real secrets manager adds (the optional section at the end).

=== Mount config as a file (the other way to deliver it)

Environment variables are perfect for a handful of values, but they have downsides: they are visible in the process environment (and thus in crash dumps and some logs), and changing one means changing the Deployment.
The alternative is to deliver configuration as a *mounted file*.
Try it: put a small properties file in a ConfigMap and mount it into the container, then tell Quarkus to read it.

[source,sh,role=execute]
----
# a ConfigMap whose single key is a whole application.properties file
oc create configmap claims-file-config \
  --from-literal=application.properties=$'# delivered as a MOUNTED FILE\nquarkus.http.access-log.enabled=true\n'

# mount it read-only at /mnt/claims-config
oc set volume deploy/parasol-claims --add --name=file-config --type=configmap \
  --configmap-name=claims-file-config --mount-path=/mnt/claims-config --read-only=true

# point Quarkus at the mounted file (the documented quarkus.config.locations pattern)
oc set env deploy/parasol-claims QUARKUS_CONFIG_LOCATIONS=/mnt/claims-config/application.properties
oc rollout status deploy/parasol-claims
----

The property you put in that file — `quarkus.http.access-log.enabled=true` — turns on HTTP access logging, which nothing else sets.
So if you now see access-log lines, the mounted file was read.
Make a couple of requests and look:

[source,sh,role=execute,subs="attributes"]
----
CLAIMS=$(oc get route parasol-claims -o jsonpath='{.spec.host}')
curl -s -o /dev/null "http://$CLAIMS/api/claims"; curl -s -o /dev/null "http://$CLAIMS/api/claims"
oc logs -l app=parasol-claims --tail=20 | grep access-log
----

.Expected output
[source,texinfo]
----
... INFO  [io.quarkus.http.access-log] (executor-thread-1) 10.233.2.2 - - [09/Jul/2026:09:38:39 +0000] "GET /api/claims HTTP/1.1" 200 4798
----

That proves the mounted file was loaded.
For this app a few environment variables are simpler, so revert the mount and keep the env-based config for the rest of the lab:

[source,sh,role=execute]
----
oc set env deploy/parasol-claims QUARKUS_CONFIG_LOCATIONS-
oc set volume deploy/parasol-claims --remove --name=file-config
oc rollout status deploy/parasol-claims
----

(Leave the `claims-file-config` ConfigMap itself in place — unreferenced, it does no harm, and `ws reset` cleans it up.)

[NOTE]
.Env vs mounted file — when to use which
====
Reach for *environment variables* for a handful of discrete values.
Reach for a *mounted file* when you have a large or structured config (a whole properties/YAML file), when you want to rotate a value *without* editing the Deployment (update the ConfigMap/Secret and the file changes in place), or when you want to keep secrets *out* of the process environment.
Both are first-class; the choice is about size and rotation, not correctness.
====

== 4. Add probes — and watch a readiness gate hold traffic
⏱ *~15 min* · the platform learns your app's health; the second deliberate break

Your app already carries a *readiness* probe — you added it in exercise 1, and it is what caught the bad datasource.
Now complete the set: add a *startup* and a *liveness* probe alongside it, so the platform can also tell a slow boot from a failure and restart a container that has wedged.
All three point at health endpoints the claims service already exposes:

// [CAPTURE-VERIFY] Deployment > Actions > "Add Health Checks" flow (Startup/Readiness/Liveness, HTTP GET, port 8080) — CLI 'oc set probe' authoritative.
[tabs]
====
Console::
+
--
. Go to *Workloads* → *Deployments* → `parasol-claims` → *Actions* → *Add Health Checks*.
. Add a *Startup* and a *Liveness* probe — each an *HTTP GET* on port `8080`, with paths `/q/health/started` and `/q/health/live` (your *Readiness* probe on `/q/health/ready` is already there from exercise 1; leave the periods and failure thresholds at the values shown in the CLI tab).
. *Save*, then wait for the rollout to finish.
--
CLI::
+
--
[source,sh,role=execute]
----
oc set probe deploy/parasol-claims --startup   --get-url=http://:8080/q/health/started --period-seconds=3  --failure-threshold=30
oc set probe deploy/parasol-claims --readiness --get-url=http://:8080/q/health/ready   --period-seconds=10 --failure-threshold=3   # already set in exercise 1; re-applying is a no-op
oc set probe deploy/parasol-claims --liveness  --get-url=http://:8080/q/health/live    --period-seconds=10 --failure-threshold=3
oc rollout status deploy/parasol-claims
----
--
====

Confirm all three are set:

[source,sh,role=execute]
----
oc get deploy parasol-claims -o jsonpath='startup={.spec.template.spec.containers[0].startupProbe.httpGet.path}{"\n"}readiness={.spec.template.spec.containers[0].readinessProbe.httpGet.path}{"\n"}liveness={.spec.template.spec.containers[0].livenessProbe.httpGet.path}{"\n"}'
----

.Expected output
[source,texinfo]
----
startup=/q/health/started
readiness=/q/health/ready
liveness=/q/health/live
----

Now break *readiness* a different way — point it at a path that returns `404`:

[source,sh,role=execute]
----
oc set probe deploy/parasol-claims --readiness --get-url=http://:8080/q/health/typo
----

You saw this gate in exercise 1: the probe change triggers a rolling update, the new pod never passes readiness, so it is held out of rotation while the old, healthy pod keeps serving — the Route stays `200`.
Breaking the probe *path* (rather than the database) proves the gate reacts to the probe's verdict itself, not just to the app's dependencies.
Now see the *other* half — what happens when there is *no* ready pod at all.
Roll the *whole* deployment onto the broken probe by scaling it to zero and back to one, so the only pod uses the failing readiness check:

[source,sh,role=execute]
----
oc scale deploy/parasol-claims --replicas=0
oc scale deploy/parasol-claims --replicas=1
sleep 40
oc get endpoints parasol-claims
----

.Expected output (endpoints drained — no ready address)
[source,texinfo]
----
Warning: v1 Endpoints is deprecated in v1.33+; use discovery.k8s.io/v1 EndpointSlice
NAME             ENDPOINTS   AGE
parasol-claims   <none>      ...
----

(The deprecation warning is expected — the classic Endpoints view is still the quickest way to see the readiness gate; EndpointSlice is its long-form successor.)
With no ready pods, the Service has *no endpoints*, and the Route has nowhere to send traffic:

[source,sh,role=execute]
----
CLAIMS=$(oc get route parasol-claims -o jsonpath='{.spec.host}')
curl -s -o /dev/null -w "Route: %{http_code}\n" "http://$CLAIMS/api/claims"
----

.Expected output
[source,texinfo]
----
Route: 503
----

That `503` is the platform *holding* the request — refusing to send it to a pod that would fail it.
Now fix readiness and watch the pod return to rotation:

[source,sh,role=execute]
----
oc set probe deploy/parasol-claims --readiness --get-url=http://:8080/q/health/ready
oc rollout status deploy/parasol-claims
curl -s -o /dev/null -w "Route: %{http_code}\n" "http://$(oc get route parasol-claims -o jsonpath='{.spec.host}')/api/claims"
----

.Expected output
[source,texinfo]
----
deployment "parasol-claims" successfully rolled out
Route: 200
----

[.checkpoint]
****
✔ *Verify:* while readiness was broken, `oc get endpoints parasol-claims` showed `<none>` and the Route returned `503`; after the fix the Route returns `200`.
****

== 5. Set requests and limits — and bump into the quota
⏱ *~12 min* · resource governance; the third deliberate break

You met resource *requests* and *limits* — and how the namespace *LimitRange* fills in defaults for a container that declares none — when you deployed your first app in *Platform Orientation & First App*.
Your claims app has been running on exactly those inherited defaults; this exercise adds the *namespace* guardrail on top.
First, see the defaults it inherits, then pin *explicit* values on the claims app:

// [CAPTURE-VERIFY] Administration > LimitRanges detail; Deployment > Actions > "Edit resource limits" — CLI 'oc get limitrange'/'oc set resources' authoritative.
[tabs]
====
Console::
+
--
. See the namespace defaults every pod inherits: *Administration* → *LimitRanges* → `workshop-limits` (its *Default* limit and *Default request*).
. Set explicit values on the app: *Workloads* → *Deployments* → `parasol-claims` → *Actions* → *Edit resource limits*, then set CPU request `100m` / limit `500m` and Memory request `256Mi` / limit `512Mi`, and *Save*.
--
CLI::
+
--
[source,sh,role=execute]
----
oc get limitrange workshop-limits -o jsonpath='default(limit)={.spec.limits[0].default}{"\n"}defaultRequest={.spec.limits[0].defaultRequest}{"\n"}'

oc set resources deploy/parasol-claims \
  --requests=cpu=100m,memory=256Mi \
  --limits=cpu=500m,memory=512Mi
oc rollout status deploy/parasol-claims
----

.Expected output
[source,texinfo]
----
default(limit)={"cpu":"500m","memory":"1Gi"}
defaultRequest={"cpu":"100m","memory":"256Mi"}
deployment "parasol-claims" successfully rolled out
----
--
====

Now meet the other guardrail: the namespace *ResourceQuota*.
It caps the *total* your namespace may request:

[source,sh,role=execute]
----
oc describe resourcequota workshop-quota | grep -E 'requests.cpu|requests.memory|pods'
----

.Expected output
[source,texinfo]
----
pods               2       30
requests.cpu       200m    3
requests.memory    512Mi   6Gi
----

You may request at most `3` CPU cores across the whole namespace, and you are using `200m`.
Ask for more than the cap and watch the platform refuse.
Create a deployment whose one pod requests `4` cores:

[source,sh,role=execute]
----
cat <<'EOF' | oc apply -f -
apiVersion: apps/v1
kind: Deployment
metadata:
  name: claims-hog
  labels: { app: claims-hog }
spec:
  replicas: 1
  selector: { matchLabels: { app: claims-hog } }
  template:
    metadata: { labels: { app: claims-hog } }
    spec:
      containers:
        - name: hog
          image: registry.access.redhat.com/ubi9/ubi-minimal:latest
          command: ["sleep", "infinity"]
          resources:
            requests: { cpu: "4", memory: 256Mi }
            limits:   { cpu: "4", memory: 256Mi }
EOF
----

Wait a few seconds, then look at why it never becomes available:

[source,sh,role=execute]
----
oc get deploy claims-hog
oc get deploy claims-hog -o jsonpath='{range .status.conditions[?(@.type=="ReplicaFailure")]}{.reason}: {.message}{"\n"}{end}'
----

.Expected output
[source,texinfo]
----
NAME         READY   UP-TO-DATE   AVAILABLE   AGE
claims-hog   0/1     0            0           11s
FailedCreate: pods "claims-hog-79f759969d-8h666" is forbidden: exceeded quota: workshop-quota, requested: requests.cpu=4, used: requests.cpu=200m, limited: requests.cpu=3
----

The message is exact: you asked for `requests.cpu=4`, `200m` is already used, and the limit is `3` — so the pod is *forbidden*.
The Deployment exists but has a `ReplicaFailure` condition and zero pods.
Fix it by removing the greedy deployment (right-sizing its request to fit would work too):

[source,sh,role=execute]
----
oc delete deploy claims-hog
----

[NOTE]
.Work *with* the guardrails, not against them
====
The LimitRange and ResourceQuota are not there to frustrate you — they keep one team's workload from starving another's on a shared cluster.
The right instinct when you hit a quota is to *right-size your requests* (do you really need 4 cores?), not to demand the quota be raised.
Requests are a reservation; asking for more than you use wastes the whole namespace's headroom.
====

****
✔ *Verify:* the claims app shows explicit requests/limits (`oc get deploy parasol-claims -o jsonpath='{.spec.template.spec.containers[0].resources}'`), you saw the `exceeded quota: workshop-quota` message, and `claims-hog` is gone.

== 6. Promote the same image to stage and prod
⏱ *~13 min* · one image, three environments, different config

You have configured *dev* by hand, one imperative command at a time.
That does not scale to more environments.
The answer is to capture the desired configuration *as code* and promote it.
The workshop has forked a small config repository into your Git account — `{user}/claims-config` — containing a Kustomize *base* (the claims app, its config, its database) and three *overlays* that change only what differs per environment.

Clone it in the Showroom terminal:

[source,sh,role=execute,subs="attributes"]
----
cd ${HOME:-/tmp}
git clone {gitea_url}/{user}/claims-config
cd claims-config
----

Preview what the *stage* overlay will apply — note the namespace, replica count, and `APP_ENV`, all set by the overlay, and that it carries *both* deployments: the claims app (2 replicas) and its database (1):

[source,sh,role=execute]
----
oc kustomize overlays/stage | grep -E 'namespace:|replicas:|APP_ENV:|QUARKUS_LOG_LEVEL:|image:' | sort -u
----

.Expected output
[source,texinfo]
----
        image: image-registry.openshift-image-registry.svc:5000/openshift/postgresql:15-el9
        image: image-registry.openshift-image-registry.svc:5000/ogsr-parasol-images/parasol-claims:1.0
  APP_ENV: stage
  QUARKUS_LOG_LEVEL: INFO
  namespace: user1-stage
  replicas: 1
  replicas: 2
----

Now promote — apply the stage overlay, then the prod overlay.
Each stands up the claims app, its config, and its own database in the matching namespace:

[source,sh,role=execute]
----
oc apply -k overlays/stage
oc apply -k overlays/prod
----

Wait for both to roll out (the startup probe gives each a moment to boot and seed its database):

[source,sh,role=execute,subs="attributes"]
----
oc -n {user}-stage rollout status deploy/parasol-claims
oc -n {user}-prod  rollout status deploy/parasol-claims
----

Now prove the point of the whole module.
Compare all three environments — *same image digest*, different config:

[source,sh,role=execute,subs="attributes"]
----
for ns in {user}-dev {user}-stage {user}-prod; do
  printf "%-16s replicas=%s  APP_ENV=%-5s  digest=%s\n" "$ns" \
    "$(oc -n $ns get deploy parasol-claims -o jsonpath='{.status.readyReplicas}')" \
    "$(oc -n $ns get cm claims-config -o jsonpath='{.data.APP_ENV}')" \
    "$(oc -n $ns get pod -l app=parasol-claims -o jsonpath='{.items[0].status.containerStatuses[0].imageID}' | sed 's#.*@##' | cut -c1-16)"
done
----

.Expected output
[source,texinfo]
----
user1-dev        replicas=1  APP_ENV=dev    digest=sha256:0995ab38
user1-stage      replicas=2  APP_ENV=stage  digest=sha256:0995ab38
user1-prod       replicas=3  APP_ENV=prod   digest=sha256:0995ab38
----

The *identical* image digest runs in all three namespaces — the exact bytes, not just the same tag.
Only the replica count and the configuration differ.
That is promotion done right: you ship what you tested, and the environment supplies the settings.

[NOTE]
====
If one environment shows a *different* digest here, that is the mutable-tag trap in action, not a broken
promotion: the shared `parasol-claims:1.0` tag was rebuilt after some nodes had already cached the older
image, and `imagePullPolicy: IfNotPresent` let an earlier-scheduled pod keep the cached bytes. Delete the
odd pod out (`oc delete pod -l app=parasol-claims -n <that-ns>`) and the fresh pull converges the digests —
and remember the production-grade fix you will meet in the supply-chain module: promote by *digest*, not tag.
====

The log level is a real, running difference too.
`stage` runs at `INFO` and `prod` at `WARN`, so the prod pods are noticeably quieter — most of the startup banner stage logs is suppressed in prod:

[source,sh,role=execute,subs="attributes"]
----
echo "stage (INFO):"; oc -n {user}-stage logs -l app=parasol-claims --tail=100 | grep -c 'io.quarkus'
echo "prod  (WARN):"; oc -n {user}-prod  logs -l app=parasol-claims --tail=100 | grep -c 'io.quarkus'
----

Stage prints its full `INFO` startup banner — 11 matching lines in our capture — while prod at `WARN` keeps only a handful of unavoidable startup lines (5 in ours).
The `INFO` chatter is gone, not every last line; what matters is the clear stage ≫ prod drop.

== 7. [OPTIONAL] Real secret management with External Secrets
⏱ *~5 min* · read / instructor-led — no hands-on on this cluster

You saw it plainly: a Kubernetes Secret is base64, not encryption, and it lives in the cluster.
For anything beyond a workshop, the *source of truth* for a credential belongs in a dedicated secrets manager — HashiCorp Vault, or a cloud provider's secrets service — and should be *synced* into the cluster on demand, never copied by hand.

On OpenShift the tool for that is the *External Secrets Operator* (ESO).
You write a small `ExternalSecret` object that names a key in the external manager; the operator fetches it and produces an ordinary Kubernetes Secret that your app consumes *unchanged* — the app never knows the difference.
A minimal `ExternalSecret` looks like this:

[source,yaml]
----
apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: claims-creds
spec:
  refreshInterval: 1h
  secretStoreRef:              # points at a configured Vault / cloud store
    name: parasol-vault
    kind: SecretStore
  target:
    name: claims-creds         # the Secret the operator will create — same name your app uses
  data:
    - secretKey: POSTGRESQL_PASSWORD
      remoteRef:
        key: parasol/claims-db
        property: password
----

[IMPORTANT]
.This is a platform add-on, not installed here
====
The External Secrets Operator is a cluster-wide operator that a platform team installs and wires to your secrets manager.
It is *not* installed on this workshop cluster, so there are no hands-on steps — treat this as the pattern to reach for in a real environment, and see your instructor's demo if one is scheduled.
The key idea to carry forward: your app keeps consuming a normal Secret; *where that Secret comes from* becomes a platform concern, not an application one.
====
