== Exercise 1 shows *two* pods, and `oc logs --previous` errors

Symptom:: After the exercise-1 break, `oc get pods -l app=parasol-claims` shows *two* pods — one `1/1 Running` and one `0/1` whose `STATUS` flips between `Running`, `Error`, and `CrashLoopBackOff` — and a plain `oc logs -l app=parasol-claims --previous` errors with `previous terminated container "…" not found`.
Cause:: The readiness probe you add first *gates* the rolling update: the new pod cannot reach the database, never passes readiness, and is held at `0/1`, while the old, healthy pod keeps serving.
So there are two pods, and `oc logs -l …` matches *both* — but the healthy old pod has no previous (crashed) container, which is exactly what triggers the error.
The `STATUS` column oscillates because the broken container keeps booting, failing, and restarting; the honest signal is the `READY` column staying `0/1`.
Fix:: Read the *newest* (broken) pod by name instead of by label:
+
[source,sh,role=execute]
----
BAD=$(oc get pods -l app=parasol-claims --sort-by=.metadata.creationTimestamp -o jsonpath='{.items[-1:].metadata.name}')
oc logs "$BAD" --previous --tail=40 | grep -iE 'Failed to start|Caused by'
----
+
The last `Caused by:` is the real cause (here, `UnknownHostException: wrong-db-host`).
Remove the bad override to recover: `oc set env deploy/parasol-claims QUARKUS_DATASOURCE_JDBC_URL-` (then `oc rollout status deploy/parasol-claims`).

== I removed an environment variable and the app broke

Symptom:: After `oc set env deploy/parasol-claims SOMEVAR-`, the app fails to connect to the database.
Cause:: Once you wire a ConfigMap or Secret with `oc set env --from`, each key becomes an environment variable *sourced from* that object.
Removing it with `SOMEVAR-` deletes the sourced variable too — the app loses that setting and falls back to a default that does not point at your database.
Fix:: Re-inject the source; it is idempotent:
+
[source,sh,role=execute]
----
oc set env deploy/parasol-claims --from=configmap/claims-config
oc set env deploy/parasol-claims --from=secret/claims-creds
----
+
Only remove a variable with `NAME-` when you intend the app to stop having it.
To *change* a value, edit the ConfigMap/Secret instead.

== The Route returns `503` right after a config change

Symptom:: `curl` to the claims Route returns `503` for a few seconds after an `oc set env`/`oc set probe`/`oc set resources` command.
Cause:: Each of those commands triggers a rolling update.
While the new pod is booting (and, once you add a startup probe, until it passes), there may be a brief window with no Ready pod — the Route has nowhere to send traffic.
Fix:: Wait for the rollout to finish, then retry:
+
[source,sh,role=execute]
----
oc rollout status deploy/parasol-claims
curl -s -o /dev/null -w '%{http_code}\n' "http://$(oc get route parasol-claims -o jsonpath='{.spec.host}')/api/claims"
----
+
A persistent `503` (not transient) means the pod never becomes Ready — check `oc get pods` and the readiness probe path.

== I broke the readiness probe but the Route stayed `200`

Symptom:: In exercise 4 you pointed readiness at a `404` path, but the Route kept serving `200`.
Cause:: A probe change is a rolling update, and the rollout is gated by readiness: the new (broken) pod never becomes Ready, so it is held out of the Service while the *old*, healthy pod keeps serving.
This is the platform protecting users — working as intended.
Fix:: This is not a failure.
To see the drained `503` state, roll the whole deployment onto the broken probe by scaling to zero and back to one (deleting individual pods will not work — the old ReplicaSet keeps recreating a *ready* pod):
+
[source,sh,role=execute]
----
oc scale deploy/parasol-claims --replicas=0
oc scale deploy/parasol-claims --replicas=1
sleep 40
oc get endpoints parasol-claims   # ENDPOINTS: <none>
----
+
Then fix readiness: `oc set probe deploy/parasol-claims --readiness --get-url=http://:8080/q/health/ready`.

== `exceeded quota` — my deployment has zero pods

Symptom:: A deployment shows `0/1` available and a `ReplicaFailure` condition; events say `exceeded quota: workshop-quota`.
Cause:: The pod's total CPU/memory *request* would push the namespace over its `ResourceQuota` (here, `requests.cpu=3`).
The scheduler never gets the pod — admission refuses it.
Fix:: This is the deliberate lesson in exercise 5.
Read the exact message, then right-size or remove the greedy workload:
+
[source,sh,role=execute]
----
oc get deploy claims-hog -o jsonpath='{range .status.conditions[?(@.type=="ReplicaFailure")]}{.message}{"\n"}{end}'
oc delete deploy claims-hog
----
+
For a real workload, lower `requests.cpu`/`requests.memory` to what it actually uses — do not assume the quota should be raised.

== `must be less than or equal to cpu limit` when setting a big request

Symptom:: Applying a Deployment with a large `requests.cpu` and *no* limit fails with `Invalid value: "4": must be less than or equal to cpu limit of 500m`.
Cause:: The namespace LimitRange supplies a *default* limit of `500m` when you set none, and Kubernetes requires `request <= limit`.
Your request (`4`) exceeds the defaulted limit (`500m`).
Fix:: Set an explicit limit `>=` your request (as the lab's `claims-hog` does with `limits.cpu: "4"`).
Then the pod is valid per-container and is refused by the *quota* instead — which is the behavior the exercise wants to show.

== `oc create configmap ... already exists` after a reset

Symptom:: After `ws reset {module-id}`, re-running exercise 2 fails with `configmaps "claims-config" already exists`.
Cause:: The reset purge removes workloads but not the ConfigMaps/Secrets the lab created, so `claims-config`/`claims-creds` survive the reset.
Fix:: Delete them before re-creating (or edit in place):
+
[source,sh,role=execute,subs="attributes"]
----
oc delete configmap claims-config --ignore-not-found -n {user}-dev
oc delete secret claims-creds --ignore-not-found -n {user}-dev
----

== The mounted config file has no effect

Symptom:: You mounted a properties file but its settings (e.g. `access-log`) do not take effect.
Cause:: Either `QUARKUS_CONFIG_LOCATIONS` does not point at the mounted path, or an environment variable is overriding the same property (env vars outrank file config in Quarkus).
Fix:: Confirm the file is present and the env var points at it:
+
[source,sh,role=execute]
----
oc exec deploy/parasol-claims -- cat /mnt/claims-config/application.properties
oc set env deploy/parasol-claims --list | grep QUARKUS_CONFIG_LOCATIONS
----
+
The path in `QUARKUS_CONFIG_LOCATIONS` must be the exact mounted file (or its directory).
Remember that a property also set via an environment variable will win over the file.

== `git clone` of my config fork fails

Symptom:: In exercise 6, `git clone {gitea_url}/{user}/claims-config` fails, though `oc` commands work.
Cause:: The fork is a *public* repo, so clone needs no credentials — a failure is almost always network (proxy/TLS interception) rather than auth, or the fork job has not finished.
Fix:: Confirm the Git server is reachable and the fork exists:
+
[source,sh,role=execute,subs="attributes"]
----
curl -sI {gitea_url}/{user}/claims-config | head -1
----
+
A `200`/`301` means the repo is there — retry the clone.
If it is missing, the fork job has not run; ask your instructor to `ws reset {module-id} --user {user}`.
