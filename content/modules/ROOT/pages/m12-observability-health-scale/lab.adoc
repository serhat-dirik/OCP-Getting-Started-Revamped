Your claims service is already *instrumented and running under a steady trickle of traffic* — a load generator continuously lists claims, reads a claim's history, files a new claim, and looks up a claim that does not exist.
That traffic is what lights up every signal you are about to read.
You will consume the three signals, alert on the service, break it on purpose to watch the alert fire, trace a request, then make the service *autoscale* and *survive a node drain*.

[NOTE]

====
Keep two things open: a *terminal* logged in as `{user}` (the CLI tab of each step runs there), and the {ocp_console_url}[OpenShift web console^] (the Console tab).

The *Observe* section of the console (Metrics, Alerting, Targets, Traces, Dashboards) is the home base for this module.

== 1. Read the golden signals and your custom metric
⏱ *~18 min* · find request rate, errors and latency — and the business metric the team added

The claims service exposes Prometheus metrics at `/q/metrics`, and a *ServiceMonitor* (shipped with your environment) already tells user-workload monitoring to scrape them.
So the same numbers exist in two places: raw at the app, and aggregated-and-queryable in the platform's monitoring.
You will look at both — that gap *is* the ServiceMonitor's job.

Start at the source.
The raw endpoint is the ground truth every other view is built from.

[source,sh,role=execute]
----
curl -s "$CLAIMS/q/metrics" | grep -E '^claims_created_total|http_server_requests_seconds_count' | head

`http_server_requests_seconds_count` is the framework's built-in HTTP timer — one series per method + status + route, which is *traffic* and *errors* in one metric (see the steady stream of `status="404"` from the load generator looking up a claim that does not exist).
And `claims_created_total` is a *business* metric the team added by hand — the platform has no idea what a claim is, but a one-line counter in the code makes "claims filed" a first-class, alertable signal.

Now the aggregated view.

.Why PromQL rate math lives in the console, not `curl`

Running ad-hoc PromQL from the terminal needs a cluster-monitoring view role most attendees do not have, so treat the console as the PromQL surface and `curl /q/metrics` as the raw-truth surface.

== 2. Alert on what matters
⏱ *~10 min* · turn "the 5xx rate is too high" into a rule that pages you

A dashboard only helps when someone is looking at it.
An *alert* watches a metric for you and fires when it crosses a line.
You author it as a *PrometheusRule* in your own namespace — user-workload monitoring scopes it to `{user}-dev`, so it is *your* alert, invisible to other tenants.

The rule below fires when the claims API's *server-error* (5xx) rate stays above 0.1/s for two minutes.
At the healthy baseline the load generator produces only 404s (client errors), so the rule loads *armed but inactive* — exactly what you want.

[tabs]
====
Console::
+
--
. In the masthead, click the *+* (*Import YAML*) button.
// [CAPTURE-VERIFY] unified console 4.21 — the "+" / Import YAML action is in the masthead
. Paste the rule, set the namespace to `{user}-dev`, and click *Create*:
+
[source,yaml,subs="attributes"]
----
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: parasol-claims-alerts
  namespace: {user}-dev
  labels:
    app: parasol-claims
spec:
  groups:
    - name: parasol-claims.rules
      rules:
        - alert: ParasolClaimsErrorRateHigh
          expr: sum(rate(http_server_requests_seconds_count{status=~"5.."}[5m])) > 0.1
          for: 2m
          labels:
            severity: warning
          annotations:
            summary: "parasol-claims is returning server errors"
            description: "The claims API 5xx rate has exceeded 0.1/s for 2m."
----
. Open *Observe* → *Alerting* → *Alerting rules* and filter for `ParasolClaimsErrorRateHigh`.
Its state is *Inactive* (there are no 5xx yet).
// [CAPTURE-VERIFY] Observe -> Alerting -> Alerting rules lists the user-defined rule as Inactive
// media-pass: screenshot m12-observability-health-scale-05-observe-alerting-inactive.png — Observe -> Alerting -> Alerting rules, ParasolClaimsErrorRateHigh Inactive (the "before" to the 04-alert-firing shot). Alt: "Observe Alerting rules list showing the just-created ParasolClaimsErrorRateHigh rule Inactive at the healthy baseline."
--
CLI::
+

== 3. Break it, and watch the alert catch it
⏱ *~12 min* · take the database away, read all three signals, then recover

Now earn the alert.
The claims service depends on its database; take the database away and the API starts returning 500s.
You will watch the alert climb from *Inactive* through *Pending* to *Firing*, and see how each of the three signals reads during a real fault.

Break it — scale the database to zero:

[tabs]
====
Console::
+
--
Open *Workloads* → *Deployments* → `claims-db`, and use *Actions* → *Edit Pod count* (or the down-arrow on the pod ring) to set it to *0*.
// [CAPTURE-VERIFY] Deployments -> claims-db -> pod count control sets replicas to 0
--
CLI::
+
--
[source,sh,role=execute]
----
oc scale deploy/claims-db -n $NS --replicas=0
----
--
====

The database is gone — but making the alert fire is a *race against the app's own health check*.
A one-request-at-a-time loop loses that race: within about a minute the failing app trips its own `/q/health/ready` probe, Kubernetes pulls the pod out of the Service endpoints, and the metrics scrape *freezes* with only a couple of dozen 5xx recorded — a 5-minute rate of roughly `0.08/s`, just *under* your `0.1/s` line, so the alert flaps in and out of *Pending* and never fires.
Win the race with a *concurrent burst* instead — twenty workers hammering the failing API at once — so you bank hundreds of server-errors in the first half-minute, while the pod is still being scraped, and pin the 5-minute rate well above the line.
Run it from the terminal (it works whichever lane you picked) and watch the sampled status go *200* while the old connection pool is still warm, then a steady stream of *500* once it breaks:

[source,sh,role=execute]
----
# 20 workers hammer the failing API so the 5xx rate clears 0.1/s *before* the app
# trips its readiness probe (~1 min in) and drops out of the metrics scrape:
end=$(( $(date +%s) + 90 ))
for w in $(seq 1 20); do
  ( while [ "$(date +%s)" -lt "$end" ]; do curl -s -o /dev/null "$CLAIMS/api/claims"; done ) &
done
# meanwhile, sample the status once a second so you can watch 200 -> 500:
while [ "$(date +%s)" -lt "$end" ]; do
  curl -s -o /dev/null -w 'GET /api/claims -> HTTP %{http_code}\n' "$CLAIMS/api/claims"
  sleep 1
done
wait
----

.Expected output (the once-a-second sampled line; the 20 workers push silently in the background)
[%collapsible]
====
[source,texinfo]
----
GET /api/claims -> HTTP 200
GET /api/claims -> HTTP 200
GET /api/claims -> HTTP 500
GET /api/claims -> HTTP 500
GET /api/claims -> HTTP 500
GET /api/claims -> HTTP 500
GET /api/claims -> HTTP 500
----
====

Now watch the alert catch it.
The live *Inactive → Pending → Firing* progression lives in the platform's alerting engine, not on the `PrometheusRule` object — the CR carries only your rule's *spec*, and reading the ruler directly needs a cluster-monitoring role you do not have.
So read the state where the platform exposes it to you — the console:

. Open *Observe* → *Alerting* → *Alerting rules* and find `ParasolClaimsErrorRateHigh`.
. Within about a minute it moves to *Pending*, and after its 2-minute `for:` window, to *Firing*.
// [CAPTURE-VERIFY] Observe -> Alerting shows ParasolClaimsErrorRateHigh Pending -> Firing while claims-db is at 0

While it climbs, read the *three signals* on the same fault — this is the whole module in ninety seconds:

* *Metrics* say *what*: in *Observe → Metrics*, `sum(rate(http_server_requests_seconds_count{status=~"5.."}[5m]))` rises off zero and keeps climbing.
That rising line is what tripped the alert.
* *Logs* say… almost nothing useful.
Run `oc logs deploy/parasol-claims -n $NS --tail=20`.
A healthy claims request logs nothing, and even the failures are terse — which is precisely why you do not operate this service by tailing logs.
* *Traces* say *where* — you will use them in the next exercise; the failing requests are recorded there with their status.

[NOTE]
.Measured: how long until it fires
====
With the concurrent burst, the 5xx rate cleared `0.1/s` within about *40 seconds* and climbed to roughly *12/s* — so `ParasolClaimsErrorRateHigh` reached *Pending* about *1 minute* after the database went down and *Firing* about two minutes after that (its `for: 2m` window), close to *3 minutes* end to end.
It fires with a wide margin, not a squeaker.
A single serial `curl` loop, by contrast, peaks at only ~*0.08/s*: the failing pod trips its readiness probe and drops out of the scrape before enough server-errors are banked, so the rate never clears the line and the alert stays *Inactive*.
That is why the burst is concurrent — and the two-minute `for:` window is deliberate, so a brief blip never pages anyone.
====

// media-pass: screenshot m12-observability-health-scale-04-alert-firing.png — Observe -> Alerting, ParasolClaimsErrorRateHigh Firing (HIGH; the signature visual). Alt: "Observe Alerting view showing the ParasolClaimsErrorRateHigh alert Firing with warning severity while claims-db is scaled to zero."

Recover.
Your scale-down stands until you undo it — the entry environment does not auto-heal edits to its own workloads — so bring the database back yourself.
That is necessary but *not sufficient*: the claims app is still holding dead connections in its pool, so it keeps erroring until you restart it too:

[source,sh,role=execute]
----
oc scale deploy/claims-db -n $NS --replicas=1
oc rollout status deploy/claims-db -n $NS
oc rollout restart deploy/parasol-claims -n $NS
oc rollout status deploy/parasol-claims -n $NS
----

Give it a moment, then confirm the service is healthy again — *HTTP 200*.
The alert does not clear the instant you recover: the 5-minute rate has to decay back below the threshold, so `ParasolClaimsErrorRateHigh` lingers in *Firing* for several minutes after the fix before it returns to *Inactive* — the harder you pushed, the more 5xx there are to drain from the window, so it can take the better part of ten minutes.

[source,sh,role=execute]
----
curl -s -o /dev/null -w 'GET /api/claims -> HTTP %{http_code}\n' "$CLAIMS/api/claims"
oc get prometheusrule parasol-claims-alerts -n $NS
----

[.checkpoint]
****
✔ *Verify:* you saw `ParasolClaimsErrorRateHigh` go *Firing* while the database was down, watched the 5xx-rate metric climb while the logs stayed nearly silent, and recovered the service (database back *and* app restarted) so the claims list returns *HTTP 200* again.
That "restart to drop the poisoned connection pool" step is the fix worth remembering.

== 4. Follow a request with a trace
⏱ *~15 min* · see the path a request takes, and rank requests by where the time goes

Metrics told you the 5xx rate was up; they cannot tell you *which* request or *where* it spent its time.
A *trace* does.
The claims service emits an OpenTelemetry trace for every request to the shared collector, which forwards to *Tempo*; the *Cluster Observability Operator*'s distributed-tracing plugin surfaces them in the console.

This is a product-UI beat — reading a trace waterfall has no terminal equivalent — so there is one path: the console.

. In the {ocp_console_url}[OpenShift web console^], open *Observe* → *Traces*.
// [CAPTURE-VERIFY] COO distributed-tracing UIPlugin adds Observe -> Traces; confirm exact nav label + Tempo/tenant selector
. Select the `tempo-traces` instance and the `parasol-claims` service, or paste the TraceQL query directly:
+
[.copy]
`{ resource.service.name = "parasol-claims" }`
. You get a stream of recent traces.
*Sort by duration* and open the slowest one.
Each trace is one request — its HTTP method, route (`/api/claims/\{claimNumber}/history`), status, and total time.

// media-pass: screenshot/gif m12-observability-health-scale-02-observe-traces.png — Observe -> Traces, TraceQL for parasol-claims, sort by duration, open a trace to its HTTP span (HIGH; prefer a <30s gif of the search->open flow). Alt: "Observe Traces view with a TraceQL query for service.name parasol-claims, a trace list sorted by duration, and one trace expanded to its HTTP server span."

[NOTE]
.The metric that points at its own trace
====
Look again at `/q/metrics` from exercise 1: each counter line ends with an *exemplar* — `# {span_id="…",trace_id="…"}`.
That is a direct link from an aggregate metric to *one exact trace* that contributed to it.
It is how you get from "the latency graph has a spike" to "*this* request is why" in one click, and it is the modern glue between the metrics and traces signals.

[source,sh,role=execute]
----
curl -s "$CLAIMS/q/metrics" | grep -m1 'trace_id'
----
====

*About the N+1.*
The `GET /api/claims/\{claimNumber}/history` endpoint is deliberately inefficient — it loads a claim's audit events with *one database query per event* instead of one query for all of them (a classic *N+1*).
At the warm, in-cluster baseline this costs only a few milliseconds, so it hides in the averages; against a remote or loaded database those extra round-trips are exactly the tail latency a trace helps you find.
The trace tells you *which* endpoint is the outlier; the code tells you *why*.

[NOTE]
.Go deeper: seeing each database query as its own span
====
With database-span instrumentation enabled in the build, that N+1 renders as a *staircase of `SELECT` spans* under the request — the anti-pattern becomes visible, not inferred.
This module's image emits the HTTP request span today; per-query database spans are a build-time flag on the claims image tracked as an enhancement.
Until then, rank by duration and confirm the cause in the code.
====

== 5. Autoscale under load with an HPA
⏱ *~15 min* · add replicas automatically when CPU climbs, and watch it happen

The claims service runs at a fixed replica count.
A *HorizontalPodAutoscaler (HPA)* changes that: it watches CPU utilization against the container's CPU *request* and adds replicas when the app is busy, removing them when it is idle.
The entry state already set a CPU request (200m) on the app — that is the denominator the HPA needs.

Create an HPA that targets *60% CPU utilization*, floor *2* replicas, ceiling *4*:

[tabs]
====
Console::
+
--
. Open *Workloads* → *Deployments* → `parasol-claims` → *Actions* → *Add HorizontalPodAutoscaler*.
// [CAPTURE-VERIFY] Deployments -> parasol-claims -> Actions -> Add HorizontalPodAutoscaler form
. Set *Minimum pods* `2`, *Maximum pods* `4`, *CPU utilization* `60%`, and *Create*.
The app immediately scales to the floor of 2.
--
CLI::
+
--
[source,sh,role=execute]
----
oc autoscale deploy/parasol-claims --cpu-percent=60 --min=2 --max=4 -n $NS
oc get hpa parasol-claims -n $NS
----

parasol-claims   Deployment/parasol-claims   cpu: 65%/60%    2         4         4
----
=====
--
====

On this run the burst drove CPU well past the 60% target — peaking around *118%* — and once the load actually landed the HPA scaled to the *ceiling of 4* within about *10 seconds*, then settled toward the 60% target spread across four pods.
Expect the burst to take up to a minute to bite: its container image must pull and start before it can generate any load, so `REPLICAS` sits at 2 (and `TARGETS` may even dip) for a short while first — the scale-out is fast, the *warm-up* is what you wait on.
Stop the load and clean up the burst pod:

[source,sh,role=execute]
----
oc delete pod claims-burst -n $NS
----

[NOTE]
.Scale-down is deliberately slow
====
Replicas will *not* drop back to 2 the moment load stops — the HPA holds them for a *5-minute stabilization window* by default, so a brief lull does not trigger a shrink-then-grow flap.
Watching `oc get hpa` you will see `TARGETS` fall below 60% while `REPLICAS` stays at 4 for several minutes.
That is correct behavior, not a stuck autoscaler.
====

== 6. Survive a node drain with a PodDisruptionBudget
⏱ *~12 min* · guarantee a minimum of healthy pods through voluntary disruption

Autoscaling handles load.
Node *maintenance* is the other thing that moves your pods: every patch cycle, nodes are cordoned and drained.
A *PodDisruptionBudget (PDB)* is the contract that stops a drain from taking your service down — "you may evict my pods, but never leave me fewer than one healthy."

Create the budget:

[tabs]
====
Console::
+
--
Use *+* (*Import YAML*), paste the budget, set the namespace to `{user}-dev`, and *Create*:

[source,yaml,subs="attributes"]
----
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: parasol-claims
  namespace: {user}-dev
spec:
  minAvailable: 1
  selector:
    matchLabels:
      app: parasol-claims
----
--
CLI::
+
--
[source,sh,role=execute]
----
oc create pdb parasol-claims --selector=app=parasol-claims --min-available=1 -n $NS
oc get pdb parasol-claims -n $NS
----

.Expected output (ALLOWED DISRUPTIONS = current healthy − minAvailable, so it tracks the live replica count)
[%collapsible]
=====
[source,texinfo]
----
poddisruptionbudget.policy/parasol-claims created
NAME             MIN AVAILABLE   MAX UNAVAILABLE   ALLOWED DISRUPTIONS   AGE
parasol-claims   1               N/A               3                     1s
----
=====
--
====

`ALLOWED DISRUPTIONS` is the whole story, and it tracks your *current* healthy pod count.
The HorizontalPodAutoscaler from the previous exercise is likely still holding *four* replicas (the 5-minute scale-down window), so with `minAvailable: 1` the platform will let *three* be evicted at a time — never the last one.
As the HPA settles back to the floor of *two*, that number falls to *one*: the budget is a live promise, not a fixed count.

[IMPORTANT]
.The node drain is an instructor-run demonstration
====
Draining a real node moves *every* tenant's pods on it, so it is run once by the instructor against the shared cluster — do not drain a node yourself.
Watch what happens to *your* service while they do: `oc get pods -n {user}-dev -w`.
As the node drains, one `parasol-claims` pod is evicted and a replacement schedules elsewhere, but a second pod is *held* until the first replacement is healthy — the budget paces the drain so your claims API never drops below one serving pod.
The mechanism is the eviction API refusing the disruptive request: on the build run, evicting past the budget returned `Cannot evict pod as it would violate the pod's disruption budget`.
====

[.checkpoint]
****
✔ *Verify:* `oc get pdb parasol-claims -n {user}-dev` shows `MIN AVAILABLE 1` and a non-zero `ALLOWED DISRUPTIONS`.
You can state what the platform does when a drain would breach that budget — it *waits* rather than evicting the last healthy pod — and why that turns node maintenance from an outage risk into a routine operation.
****
