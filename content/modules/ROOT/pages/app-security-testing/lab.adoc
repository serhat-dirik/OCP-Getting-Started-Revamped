== 2. Gate 1 — SAST: the hard-coded credential
⏱ *~12 min* · read the SonarQube finding, externalize the secret, watch the gate go green

SAST (Static Application Security Testing) analyzes the *source you wrote*. Read *why* it failed — in SonarQube's own dashboard, then confirm from the terminal.

[tabs]

Console::
+

Open {sonarqube_url}[the SonarQube dashboard^] and select the *parasol-claims-{user}* project.
The quality gate shows *Failed*, with the failing condition *Vulnerabilities is greater than 0*.
Open *Issues* → the one *Vulnerability*: rule *S2068 — "Credentials should not be hard-coded"*, on `PartnerGateway.java`, flagging the `partnerApiPassword` literal. [CAPTURE-VERIFY: exact dashboard labels on SonarQube {sonarqube_version}.]
--
CLI::
+
--
[source,sh,role=execute]
----
oc get taskrun -n $NS -l tekton.dev/pipelineTask=sast-sonar --sort-by=.metadata.creationTimestamp \
  -o name | tail -1 | xargs -I{} oc logs -n $NS {} --all-containers | grep -iE "QUALITY GATE|S2068|dashboard"
----

.Expected output (the SonarQube gate failed on the credential)
[source,texinfo]
----
QUALITY GATE STATUS: FAILED - View details on http://sonarqube.../dashboard?id=parasol-claims-user1
   fix: resolve the new issues (e.g. remove the hardcoded credential — rule S2068), then re-run.
----
--
====

The finding is exact: a password *compiled into the source*. The fix is the one M04 teaches — *externalize it to configuration*, so the value comes from a Secret or environment variable, never a literal. Apply the fix and commit it to `seed-appsec`:

[source,sh,role=execute]
----
cd ~/parasol-claims
cat > src/main/java/com/parasol/claims/PartnerGateway.java <<'EOF'
package com.parasol.claims;

import org.eclipse.microprofile.config.inject.ConfigProperty;
import jakarta.enterprise.context.ApplicationScoped;

/** The partner API credential is read from configuration (a Secret / env var in production),
 *  never hard-coded — the config-externalization pattern from M04. */
@ApplicationScoped
public class PartnerGateway {

    @ConfigProperty(name = "partner.api.password", defaultValue = "")
    String partnerApiPassword;

    private final String partnerEndpoint = "https://partners.example.com/claims";

    public String endpoint() { return partnerEndpoint; }
    public String credential() { return partnerApiPassword; }
}
EOF
git commit -am "fix(SAST): externalize the partner API credential (@ConfigProperty)"
git push origin seed-appsec
----

Now *re-run* the pipeline (Pipelines console → the last run → *Actions* → *Rerun*, or `oc create` the PipelineRun again). SAST re-analyzes: with no literal secret, `vulnerabilities` drops to *0* and the quality gate passes — so the run advances to the *next* gate, `sca-trivy`, which now fails.

====
✔ *Verify:* The SonarQube quality gate for `parasol-claims-{user}` is now *Passed* (0 vulnerabilities). On the re-run, `sast-sonar` is *green* and the pipeline advanced — it now fails at `sca-trivy`.

== 3. Gate 2 — SCA: the vulnerable dependency
⏱ *~10 min* · read the Trivy finding, drop the dependency, re-run

SCA (Software Composition Analysis) checks the *dependencies you pulled in*. Read the Trivy finding from the TaskRun log:

[source,sh,role=execute]
----
oc get taskrun -n $NS -l tekton.dev/pipelineTask=sca-trivy --sort-by=.metadata.creationTimestamp \
  -o name | tail -1 | xargs -I{} oc logs -n $NS {} --all-containers | grep -iE "commons-collections|CVE-2015|CRITICAL|HIGH|SCA scan FAILED"
----

.Expected output (Trivy flags a critical RCE in a transitive-era library)
[source,texinfo]
----
Total: 2 (HIGH: 1, CRITICAL: 1)
│ commons-collections:commons-collections │ CVE-2015-7501 │ CRITICAL │ ... 3.2.1 → 3.2.2 │ InvokerTransformer ...
│                                         │ CVE-2015-6420 │ HIGH     │ ...               │ Insecure Deserialization ...
❌ SCA scan FAILED — a HIGH,CRITICAL dependency CVE was found.
----

`commons-collections` 3.2.1 carries *CVE-2015-7501* — the infamous *InvokerTransformer* deserialization *RCE* (Remote Code Execution) gadget. Your code never calls it; it's a landmine you inherited by declaring the dependency. The fix is to *remove it*. Trivy also wrote a *CycloneDX SBOM* (`trivy-sbom.json`) into the workspace — the full bill of materials for the build. Drop the dependency and commit:

[source,sh,role=execute]
----
cd ~/parasol-claims
# remove the seeded commons-collections block from pom.xml
awk '/M27 SCA seed/{skip=1} skip&&/<\/dependency>/{skip=0;next} skip{next} {print}' pom.xml > pom.xml.new && mv pom.xml.new pom.xml
grep -q commons-collections pom.xml && echo "STILL PRESENT" || echo "commons-collections removed"
git commit -am "fix(SCA): remove vulnerable commons-collections 3.2.1 (CVE-2015-7501)"
git push origin seed-appsec
----

*Re-run* the pipeline. SCA is now clean, the build proceeds — and fails at the *next* gate, `image-scan`.

[.checkpoint]
====
✔ *Verify:* `pom.xml` no longer declares `commons-collections`. On the re-run, `sca-trivy` is *green*, the image builds, and the pipeline now fails at `image-scan` (RHACS).
====

== 4. Gate 3 — Image scan: log4shell, and defense in depth
⏱ *~12 min* · read the RHACS finding, stop baking the flaw in, understand why a source scan missed it

This is the module's key beat. The *image scan* (RHACS, shift-right) failed on *log4shell* — but your *SCA* scan (Trivy, shift-left) was *clean*. How? Because log4shell is *not in `pom.xml`*: the seeded `Containerfile` bakes a vulnerable `log4j-core` 2.14.1 jar *directly into the image* at build time. A source scan has nothing to find; the image scan inspects the *assembled image* and catches it. That gap is exactly why you scan at *both* layers.

Read the finding — in the RHACS console, then the TaskRun log:

[tabs]
====
Console::
+
--
Open {acs_console_url}[the RHACS console^] → *Violations*. Find the `parasol-claims` image violation for policy *"Block Log4Shell (CVE-2021-44228) at build"*, severity *Critical*. Open it to see the CVE, the component (`log4j-core` 2.14.1), and the image digest. [CAPTURE-VERIFY: exact Violations column labels on RHACS {acs_version}.]
--
CLI::
+
--
[source,sh,role=execute]
----
oc get taskrun -n $NS -l tekton.dev/pipelineTask=image-scan --sort-by=.metadata.creationTimestamp \
  -o name | tail -1 | xargs -I{} oc logs -n $NS {} --all-containers | grep -iE "log4|CVE-2021-44228|Block Log4Shell|FAIL"
----

.Expected output (the image scan blocks log4shell — the SCA source scan two stages earlier was clean of it)
[source,texinfo]
----
Policy check results for image: ...parasol-claims:latest
(TOTAL: 2, LOW: 1, MEDIUM: 0, HIGH: 0, CRITICAL: 1)
| Block Log4Shell (CVE-2021-44228) at build | CRITICAL | X (BREAKS BUILD) |
|   - CVE-2021-44228 (CVSS 10) (severity Critical) found in component
|     'org.apache.logging.log4j:log4j-core' (version 2.14.1)                |
ERROR: failed policies found: 1 policies violated that are failing the check
❌ image check FAILED — an enforced build policy was violated (e.g. Log4Shell).
----
--
====

Now *stop baking it in* — remove the seeded `Containerfile` step — and commit:

[source,sh,role=execute]
----
cd ~/parasol-claims
# remove the seeded log4j-copy step from the Containerfile
awk '/M27 image-scan seed/{skip=1} skip&&/outputDirectory/{skip=0;next} skip{next} {print}' Containerfile > Containerfile.new && mv Containerfile.new Containerfile
grep -q "log4j-core:2.14.1" Containerfile && echo "STILL PRESENT" || echo "log4j-copy step removed"
git commit -am "fix(image): stop baking log4j-core 2.14.1 into the image"
git push origin seed-appsec
----

*Re-run.* The rebuilt image has no log4shell, RHACS passes, and the pipeline fails at the *next* gate, `deployment-check`.

[NOTE]
.Why this beats "just run a dependency scanner"
====
If you *only* ran SCA on the source, this vulnerable image would have shipped — the flaw was invisible there. If you *only* scanned the image, you'd miss things a source scan catches cheaply (a declared-but-transitive CVE, before you ever build). *Defense in depth* is running both, so a blind spot in one layer is covered by the next. Same idea, applied again in M08: the image is also *signed and attested*, so it must be both *scanned clean* and *provably yours*.
====

== 5. Gate 4 — Config check: the insecure Deployment
⏱ *~10 min* · read the roxctl violations, restore the hardened manifest, re-run

Config security checks the *manifest you're about to apply*, before it reaches the cluster. The seeded `deploy/parasol-claims-deployment.yaml` runs *privileged*, as *root*, adds *SYS_ADMIN* and *NET_ADMIN*, and sets *no resource limits*. RHACS `roxctl deployment check` flagged every one. Read the finding:

[source,sh,role=execute]
----
oc get taskrun -n $NS -l tekton.dev/pipelineTask=deployment-check --sort-by=.metadata.creationTimestamp \
  -o name | tail -1 | xargs -I{} oc logs -n $NS {} --all-containers | grep -iE "POLICY|Privileged|CAP_SYS_ADMIN|limit|TOTAL|deployment check FAILED"
----

.Expected output (RHACS deploy-time policies flag the insecure manifest)
[source,texinfo]
----
|         CAP_SYS_ADMIN capability added          |  MEDIUM  | ... parasol-claims ...
|   Container with privilege escalation allowed   |  MEDIUM  | ... parasol-claims ...
|    No CPU request or memory limit specified     |  MEDIUM  | ... parasol-claims ...
|              Privileged Container               |  MEDIUM  | ... parasol-claims ...
(TOTAL: 6, LOW: 1, MEDIUM: 5, HIGH: 0, CRITICAL: 0)
❌ deployment check FAILED — 5 MEDIUM+ deploy-policy violation(s).
----

"You ship your configuration, not just your code." The fix is to *restore the hardened manifest* — least-privilege securityContext (non-root, drop `ALL` capabilities, no privilege escalation, read-only root filesystem), real resource requests and limits, and no auto-mounted service-account token:

[source,sh,role=execute]
----
cd ~/parasol-claims
cat > deploy/parasol-claims-deployment.yaml <<'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: parasol-claims
  labels: {app: parasol-claims}
spec:
  replicas: 1
  selector: {matchLabels: {app: parasol-claims}}
  template:
    metadata: {labels: {app: parasol-claims}}
    spec:
      automountServiceAccountToken: false
      containers:
        - name: parasol-claims
          image: image-registry.openshift-image-registry.svc:5000/parasol/parasol-claims:1.0.0
          ports: [{containerPort: 8080}]
          resources:
            requests: {cpu: 100m, memory: 256Mi}
            limits: {cpu: "1", memory: 512Mi}
          securityContext:
            runAsNonRoot: true
            allowPrivilegeEscalation: false
            readOnlyRootFilesystem: true
            capabilities: {drop: [ALL]}
            seccompProfile: {type: RuntimeDefault}
          volumeMounts: [{name: tmp, mountPath: /tmp}]
      volumes: [{name: tmp, emptyDir: {}}]
EOF
git commit -am "fix(config): restore the hardened Deployment manifest"
git push origin seed-appsec
----

*Re-run.* The hardened manifest passes the check, the pipeline *deploys the image and creates the Route for you* (no manual `oc expose`) — and then hits the *final* gate, `dast-zap`.

== 6. Gate 5 — DAST: the missing security headers
⏱ *~12 min* · read the ZAP finding, add the headers, and watch the whole run go green

DAST (Dynamic Application Security Testing) exercises the *running* app — something you can only do *after* deploy. The `zap-baseline` stage ran a ZAP (Zed Attack Proxy) scan against the live service and found it never sets its HTTP security headers, so a browser is free to MIME-sniff it, frame it, and leak its URLs. Read the finding:

[source,sh,role=execute]
----
oc get taskrun -n $NS -l tekton.dev/pipelineTask=dast-zap --sort-by=.metadata.creationTimestamp \
  -o name | tail -1 | xargs -I{} oc logs -n $NS {} --all-containers | grep -iE "WARN-NEW|FAIL-NEW|DAST FAILED"
----

.Expected output (ZAP flags the missing headers on the RUNNING app — a check only possible post-deploy)
[source,texinfo]
----
WARN-NEW: Content Security Policy (CSP) Header Not Set [10038] x 3
WARN-NEW: Storable and Cacheable Content [10049] x 3
WARN-NEW: Permissions Policy Header Not Set [10063] x 3
FAIL-NEW: 0   WARN-NEW: 3   INFO: 0   IGNORE: 0   PASS: 64
❌ DAST FAILED — ZAP reported a gating alert on the running app.
----

The `dast-zap` stage runs with `FAIL_ON_WARN=true`, so these *warnings* fail the build. ZAP found three missing response headers: no Content-Security-Policy, no Permissions-Policy, and cacheable responses (claim data a browser could store). The fix is to *set the headers* in Quarkus configuration — one line per header, applied to every response. Append them to `application.properties` and commit:

[source,sh,role=execute]
----
cd ~/parasol-claims
cat >> src/main/resources/application.properties <<'EOF'

# HTTP security response headers (M27 DAST fix)
quarkus.http.header."X-Content-Type-Options".value=nosniff
quarkus.http.header."Content-Security-Policy".value=default-src 'self'; frame-ancestors 'none'
quarkus.http.header."X-Frame-Options".value=DENY
quarkus.http.header."Referrer-Policy".value=no-referrer
quarkus.http.header."Permissions-Policy".value=geolocation=(), camera=(), microphone=()
quarkus.http.header."Strict-Transport-Security".value=max-age=31536000; includeSubDomains
quarkus.http.header."Cache-Control".value=no-store
EOF
git commit -am "fix(DAST): add HTTP security response headers"
git push origin seed-appsec
----

*Re-run one final time.* The rebuilt app sets its headers, ZAP finds nothing to warn about, and — for the first time — *every gate is green*: SAST, SCA, unit test, build, image scan, sign, config check, deploy, DAST. The scanned, signed image is deployed with its Route.

[.checkpoint]
