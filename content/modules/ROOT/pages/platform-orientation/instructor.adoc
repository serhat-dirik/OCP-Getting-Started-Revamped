This module is the workshop's on-ramp: for many attendees it is their first hour on OpenShift.
Optimize for confidence, not coverage — the two ideas that must land are *desired state* and *cattle, not pets*.
Everything else is reinforcement.

== Timing

Total hands-on: *~60–67 min* (spec target 60–75 min).
Add ~5–10 min of in-room concept discussion up front.

.How this timing was measured

The *Measured CLI core* column is real: each `oc` step was executed as a workshop user (`user5`) on the build cluster (OCP {ocp_version}) on 2026-07-08 and timed.

The *Attendee time* column adds realistic reading, console-navigation, and checkpoint overhead on top of that core — it is what a first-time attendee takes, not just command runtime.

== Pre-flight checklist

Run these *before the session* (as an admin or as the sample user).
All should pass.

. *Entry state materializes and verifies* for a sample user:
+
[source,sh,role=execute]
----
ws start platform-orientation --user user1
ws verify platform-orientation --user user1 --entry-only
----
+
Expect `✅ all 4 checks passed` (namespace, entry marker, quota, Gitea account).

. *A user can log in and pull the image.*
Log in as a sample user and deploy once to confirm registry permissions and to warm the image on a node:
+
[source,sh,role=execute]
----
oc new-app --image=image-registry.openshift-image-registry.svc:5000/ogsr-parasol-images/parasol-web:1.0 --name=preflight -n user1-dev
oc rollout status deploy/preflight -n user1-dev
oc delete all -l app=preflight -n user1-dev
----
+
Expect `deployment "preflight" successfully rolled out`.
If it sticks in `ImagePullBackOff`, the puller RoleBinding is missing — see troubleshooting.

. *Lightspeed answers* (if the cluster has it).
Open the *Lightspeed* chat bubble at the bottom-right of the console (its drawer auto-opens on the first visit) and ask "Why is my pod restarting?"; confirm you get a bulleted answer naming `CrashLoopBackOff`/`OOMKilled`/probe failures with `oc` commands.
If Lightspeed is absent, plan to use the CLI fallback and tell attendees up front.

. *Console is the unified layout.*
Confirm the navigation is grouped under headings such as *Core platform* (with *Topology* under *Workloads*), and that the masthead *Quick create* (+) menu and the *Lightspeed* chat bubble (bottom-right) are present.
The lab teaches the *unified navigation*.
The standalone Developer perspective is off by default but is *admin-configurable* — check whether it is enabled on your cluster before the session:
+
[source,sh,role=execute]
----
oc get consoles.operator.openshift.io cluster -o jsonpath='{.spec.customization.perspectives}{"\n"}'
----
+
An empty result is the unified default (no perspective switcher).
A `dev` entry with `"state":"Enabled"` — the state on the current workshop cluster — makes a perspective switcher appear at the top of the nav; attendees still follow every step from the unified navigation.
An admin toggles it under *Administration → Cluster Settings → Configuration → Console → Customize → Perspectives*.

== Demo-mode notes

Beyond the Say/Show/Do arc in the demo flavor:

* *Two windows, side by side.*
Console (Topology) on the left, terminal on the right.
The kill-a-Pod beat lands hardest when you delete from the terminal and the Topology ring visibly loses and regains a segment — do not do it from the console, or you lose the "I broke it from outside" framing.
* *Pre-warm the image.*
Deploy and delete `parasol-web` once on the node(s) you will demo on, so the live deploy shows a Pod starting in ~4 s, not an 18 s image pull.
Dead air during a pull kills the moment.
* *Scale to 3, then kill — do not skip to kill.*
The self-heal is only impressive if the audience has seen the three-segment donut first.
Let them count to three, then take one away.
* *Lightspeed answers vary.*
The wording changes run to run (it is a live model).
Rehearse the exact question, and have the CLI fallback (`oc get events`, `oc logs`) ready in the terminal in case the panel is slow or the cluster lacks Lightspeed.
Never promise a specific sentence.
* *Keep it to the four beats.*
Tour → deploy → kill → Lightspeed.
Resist adding Routes/logs to the demo; those are the hands-on payoff.

== Top questions attendees ask

*1. "What's the difference between Kubernetes and OpenShift, really?"*::
Kubernetes is the engine: the object model (Pods/Deployments/Services) and the reconcile loop.
OpenShift is Kubernetes plus what an organization needs to use it safely on day one — the web console and catalog, Routes (built-in ingress), secure-by-default containers (non-root, random UID), integrated builds/pipelines/GitOps, and Red Hat support.
Everything they did today used a Kubernetes primitive *plus* an OpenShift convenience (`oc new-app`, `oc expose`, the Topology view).

*2. "Why does my container run as a random user ID like 1001010000?"*::
Security.
OpenShift refuses to run containers as root by default; instead each project gets a range of high UIDs and the platform assigns one.
It also drops Linux capabilities and applies the `restricted-v2` security profile.
The practical consequence for image authors — write images that tolerate an arbitrary non-root UID and group-writable paths — is covered in *Ways to Build & Deliver Apps* and *Multi-Tenancy & Workload Security*.
Today it is enough that they *notice* it and know it is deliberate, not a bug.

*3. "Route or Ingress — which should I use?"*::
A *Route* is OpenShift's mature, native way to publish a Service on the cluster domain, with TLS built in — the lab creates an *edge* Route so the app answers over HTTPS, and most OpenShift apps use Routes this way.
Kubernetes *Ingress* also works (OpenShift implements it via the same router), and the newer *Gateway API* is the strategic direction.
For a first app, Route is the right answer; the trade-offs are a full module later (*Networking for Dev & DevOps*).

*4. "What's an operator? People keep saying it."*::
An operator is just the reconcile loop applied to *operations*: a controller that watches a custom resource and drives real state to match it — the same "desired vs actual" idea they saw with Deployments, but for installing and running software (databases, the very platform services on this cluster).
They do not need to write one; they should recognize that the pattern under self-healing is the pattern under everything on OpenShift.
(Writing one is an advanced specialty in its own right — beyond what this workshop asks of them.)

*5. "Is Lightspeed sending my cluster data off somewhere?"*::
Fair question, and worth answering straight.
OpenShift Lightspeed sends the text of your question — and context you explicitly attach — to a configured large language model to get an answer; it is not silently exfiltrating your workloads.
*Where* that model runs is a cluster-admin choice: it can be a hosted model service or a model running in your own environment.
Point them to the OpenShift Lightspeed documentation for the data-handling specifics, and note that the same "who can call which model, and what leaves the cluster" question is exactly what the AI modules later in the workshop govern.

== Watchouts

* *First-login OAuth delay.*
The very first console login for a freshly-provisioned user can take 10–30 s (identity provider warm-up) and occasionally shows a transient error page — a refresh fixes it.
Warn attendees so a slow first login does not read as "it's broken."
(Troubleshooting entry included.)
* *First image pull is the slow step.*
The 410 MB image takes ~18 s to pull the first time on a node; subsequent Pods on that node start in seconds.
This is why exercise 2 can feel slower than the rest.
* *Harmless deprecation warning.*
`oc get all` and `oc status` print a deprecation warning about a legacy workload API this lab does not use (you use *Deployments*) — the exact line is in the troubleshooting guide.
Tell attendees to ignore it.
* *Multi-port Service and the Route.*
The Service has ports 8080 and 8443, so `oc create route edge parasol-web --service=parasol-web` needs `--port=8080` or it cannot choose a port.
The lab uses the explicit flag; if an attendee omits it, they will get a confusing result.
* *Per-user Route hosts.*
Route hosts are `parasol-web-<user>-dev.<domain>` — unique per project, so attendees never collide as long as everyone stays in their own `{user}-dev`.
If someone works in the wrong project, that is where collisions and "why is my app someone else's" confusion come from.
* *`ws reset` on a macOS laptop.*
If an *instructor* runs `ws reset` from a Mac, the workload purge may silently no-op (a BSD `sed` portability issue in the CLI, tracked with the platform team); it works in the Linux Showroom terminal that attendees use.
Verify with `oc get all -n <user>-dev` after a local reset.
(Troubleshooting entry included.)
* *Lightspeed nondeterminism / absence.*
Answers vary run to run, and the cluster may not have Lightspeed at all.
The lab and demo both carry a CLI fallback; use it rather than fighting the panel.
