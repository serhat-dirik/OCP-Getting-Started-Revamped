You are the platform engineer for Parasol's *claims* application.
Its services — `parasol-web` → `parasol-claims` → `parasol-fraud` (with a `claims-db`) — are already running in `{user}-mesh`, but *un-meshed*: no proxies, no mutual TLS, no traffic control.

[demo-block]
[TIME 4m]

----
+
"I sent a hundred requests from the claims pod. The mesh's own metrics show about ninety went to v1 and ten to v2 — the exact split I asked for, and I can pin *test users* to v2 with a header while everyone else stays on v1."

[demo-block]
[TIME 3m]
Say:: "Now the nightmare: the fraud service gets slow. Watch what a hanging dependency does — and what the mesh gives me to survive it."
Show:: A 5-second fault delay injected on the fraud route; the claims→fraud call now taking 5 seconds; the resilience policy (timeout, retry, circuit breaker) that's already in place.

+
[source,sh,role=execute]
----
# re-apply the fraud VirtualService with a 5s fault.delay (exercise 5), then send one call:
oc exec $CLAIMS -c parasol-claims -n $NS -- curl -s -m8 -o /dev/null -w "claims felt: %{time_total}s\n" http://parasol-fraud:8080/api/fraud/score/CLM-1001
# then re-apply the clean (fault-free) route
----
+
"Five seconds — the caller *feels* every bit of that hang. This is why the route carries a timeout, a retry, and a circuit breaker: so a slow dependency fails *fast* instead of taking the caller's threads down with it. And note the honest bit — Istio injects the fault *ahead* of the router, so it tests how the *caller* copes; the timeout fires on a genuinely slow upstream."

[demo-block]
[TIME 4m]

[source,sh,role=execute]

+
"Claims gets a `200`. The web tier — a perfectly valid pod, one namespace over from nothing — gets `403 RBAC: access denied`. Not by IP, not by namespace: by *cryptographic identity*. That's 'only claims may call fraud,' enforced."

== 1. Enroll the app into the mesh
⏱ *~12 min* · add the injection label, watch the native sidecar appear

Start by confirming the world you're in: a shared control plane, and your app running *without* proxies.

[source,sh,role=execute]
----
oc get istio default -o jsonpath='control plane: Istio {.spec.version}  status={.status.state}{"\n"}'
oc get pods -n $NS -o custom-columns='POD:.metadata.name,READY:.status.containerStatuses[*].ready,CONTAINERS:.spec.containers[*].name'
----

.Expected output (a healthy control plane; one container per pod — no sidecars yet)
[source,texinfo]
----
control plane: Istio v1.28.6  status=Healthy
POD                                    READY   CONTAINERS
claims-db-...                          true    postgresql
demo-client-...                        true    client
parasol-claims-...                     true    parasol-claims
parasol-fraud-...                      true    parasol-fraud
parasol-web-...                        true    parasol-web
partner-tcp-backend-...                true    tcp-backend
----

Every app pod has *one* container — your application, and nothing else.
Your namespace is already a *discovered* mesh tenant (the platform labelled it `istio-discovery=enabled`), so istiod is watching it — but discovery isn't injection. To get a proxy, a workload needs the *injection label* `sidecar.istio.io/inject: "true"` on its *pod template*.

[NOTE]
.Why per-workload, not "label the namespace"?
====
You may have seen mesh tutorials that enroll by labelling the *namespace* `istio-injection=enabled`. That's a *cluster-scoped* edit to the Namespace object, which a namespace tenant isn't allowed to make — try it and you'll be refused:

[source,sh,role=execute]
----
oc label namespace $NS istio-injection=enabled
----

[source,texinfo]
----
Error from server (Forbidden): namespaces "user4-mesh" is forbidden: User "user4"
cannot patch resource "namespaces" in API group "" in the namespace "user4-mesh"
----

That's expected, not a problem. As a namespace owner you enroll *per workload* — the injection label on each Deployment's pod template — which is finer-grained anyway: you choose exactly which workloads join the mesh. That's the path below.
====

Enroll the three app tiers by adding the injection label to their pod templates.
`claims-db` (a database, no mesh benefit) and `demo-client` (your external test client) stay *out*.

Now look at what changed — the app pods are `2/2`, and the extra container is the mesh proxy:

[source,sh,role=execute]
----
oc get pods -n $NS -o custom-columns='POD:.metadata.name,READY:.status.containerStatuses[*].ready,INIT:.spec.initContainers[*].name'
----

.Expected output (meshed tiers gain istio-proxy; demo-client + db stay single-container)
[source,texinfo]
----

claims-db-...                          true    <none>
demo-client-...                        true    <none>
parasol-claims-...                     true    istio-validation,istio-proxy
parasol-fraud-...                      true    istio-validation,istio-proxy
parasol-web-...                        true    istio-validation,istio-proxy
----

Notice `istio-proxy` is an *init container*, not a regular one — that's the OpenShift Service Mesh 3 *native sidecar*.
Confirm its always-on restart policy (what makes it start before your app and stop after it):

[source,sh,role=execute]
----
POD=$(oc get pod -n $NS -l app=parasol-claims -o jsonpath='{.items[0].metadata.name}')
oc get pod $POD -n $NS -o jsonpath='{range .spec.initContainers[*]}{.name}{" restartPolicy="}{.restartPolicy}{"\n"}{end}'
----

.Expected output (istio-proxy is a native sidecar — an init container that never exits)
[source,texinfo]
----
istio-validation restartPolicy=
istio-proxy restartPolicy=Always
----

The `restartPolicy: Always` on an init container is the Kubernetes native-sidecar pattern: the proxy runs for the whole life of the pod, but starts *first* so no app traffic escapes unproxied.

== 2. Prove mutual TLS with a STRICT policy
⏱ *~12 min* · encryption you can prove by watching a plaintext client get refused

Meshed workloads now talk over *mutual TLS* automatically — but the default mode is *PERMISSIVE*, which still *accepts* plaintext from outside the mesh so nothing breaks on day one.
See that first: your un-meshed `demo-client` can still reach the meshed `parasol-web`.

[source,sh,role=execute]
----
oc exec deploy/demo-client -n $NS -- curl -s -m5 -o /dev/null -w "demo-client -> web: HTTP %{http_code}\n" http://parasol-web:8080/api/claims
----

.Expected output (PERMISSIVE accepts the plaintext client)
[source,texinfo]
----
demo-client -> web: HTTP 200
----

Now require encryption. A `PeerAuthentication` in `STRICT` mode tells every proxy in the namespace to *refuse* any connection that isn't mutual TLS:

// [CAPTURE-VERIFY] Console: +Add (or Import YAML) -> paste the PeerAuthentication -> Create (OCP 4.21) -- CLI authoritative.
[tabs]
====
Console::
+
--
Open *+Add* → *Import YAML* (project `{user}-mesh`), paste the manifest from the *CLI* tab, and *Create*.
--
CLI::
+
--
[source,sh,role=execute]
----
oc apply -n $NS -f - <<'EOF'
apiVersion: security.istio.io/v1
kind: PeerAuthentication
metadata:
  name: default
spec:
  mtls:
    mode: STRICT
EOF
----
--
====

.Expected output
[source,texinfo]
----
peerauthentication.security.istio.io/default created
----

Give the proxies a few seconds to pick up the new policy, then try the un-meshed client again — it now gets *refused* at the connection level:

[source,sh,role=execute]
----
sleep 5
oc exec deploy/demo-client -n $NS -- sh -c 'curl -sS -m5 http://parasol-web:8080/api/claims; echo "curl exit=$?"'
----

.Expected output (plaintext is now rejected — the proxy resets the connection)
[source,texinfo]
----
curl: (56) Recv failure: Connection reset by peer
curl exit=56
----

That reset *is* the proof: the `parasol-web` proxy now demands mutual TLS and drops a plaintext caller.
Meanwhile a *meshed* caller — whose proxy speaks mutual TLS — sails through. Drive a call from *inside* the claims pod so it egresses through claims' real sidecar:

[source,sh,role=execute]
----
CLAIMS=$(oc get pod -n $NS -l app=parasol-claims -o jsonpath='{.items[0].metadata.name}')
oc exec $CLAIMS -c parasol-claims -n $NS -- curl -s -m5 -o /dev/null -w "claims -> fraud (meshed): HTTP %{http_code}\n" http://parasol-fraud:8080/api/fraud/score/CLM-1001
----

.Expected output (mesh-to-mesh over mutual TLS still works)
[source,texinfo]
----
claims -> fraud (meshed): HTTP 200
----

The encryption is identity-bound. Read the *workload identity* off the claims sidecar's certificate — a *SPIFFE* URI naming the namespace and ServiceAccount, the name the mesh authenticates:

[source,sh,role=execute]
----
istioctl proxy-config secret $CLAIMS.$NS -o json \
  | oc exec -i deploy/demo-client -n $NS -- sh -c 'cat > /tmp/c.json; python3 -c "import json,base64;d=json.load(open(\"/tmp/c.json\"));c=[x for x in d[\"dynamicActiveSecrets\"] if x[\"name\"]==\"default\"][0][\"secret\"][\"tlsCertificate\"][\"certificateChain\"][\"inlineBytes\"];open(\"/tmp/c.pem\",\"wb\").write(base64.b64decode(c))"; openssl x509 -in /tmp/c.pem -noout -ext subjectAltName'
----

.Expected output (the claims workload's cryptographic identity)
[source,texinfo]
----
X509v3 Subject Alternative Name: critical
    URI:spiffe://cluster.local/ns/user4-mesh/sa/parasol-claims
----

Keep that identity in mind — `sa/parasol-claims` is exactly what the AuthorizationPolicy in exercise 6 will allow.

Finally, *revert to PERMISSIVE* by removing the STRICT policy. The rest of the lab uses the un-meshed `demo-client` to drive traffic, and (like the module's end state) leaves mutual TLS at its automatic default:

[source,sh,role=execute]
----
oc delete peerauthentication default -n $NS
sleep 3
oc exec deploy/demo-client -n $NS -- curl -s -m5 -o /dev/null -w "demo-client -> web after revert: HTTP %{http_code}\n" http://parasol-web:8080/api/claims
----

.Expected output (plaintext accepted again under PERMISSIVE)
[source,texinfo]
----
demo-client -> web after revert: HTTP 200
----

== 3. See the mesh in Kiali
⏱ *~12 min* · turn live traffic into a service graph — the picture `oc get` can't give you

Kiali is the mesh's console: it reads the proxies' telemetry and draws a live *graph* of what's calling what, with health and mutual-TLS badges.
A graph needs *traffic*, so first generate some. Because the sample app doesn't yet make its own downstream calls, *you* drive the chain — `demo-client → web`, and `claims → fraud` from inside the meshed claims pod:

[source,sh,role=execute]
----
CLAIMS=$(oc get pod -n $NS -l app=parasol-claims -o jsonpath='{.items[0].metadata.name}')
# 60 requests into the front door, and 60 on the claims->fraud hop (each loop runs inside the pod):
oc exec deploy/demo-client -n $NS -- sh -c 'for i in $(seq 1 60); do curl -s -o /dev/null http://parasol-web:8080/api/claims; done'
oc exec $CLAIMS -c parasol-claims -n $NS -- sh -c 'for i in $(seq 1 60); do curl -s -o /dev/null http://parasol-fraud:8080/api/fraud/score/CLM-$i; done'
echo "traffic sent"
----

Ground that the traffic really flowed *through the mesh* before you open the console — your `istioctl` client matches the mesh, and the claims sidecar recorded its calls to fraud in `istio_requests_total`:

[source,sh,role=execute]
----
istioctl version --remote=false
oc exec $CLAIMS -c istio-proxy -n $NS -- pilot-agent request GET 'stats?filter=istio_requests_total' 2>/dev/null | grep -c parasol-fraud
----

.Expected output (the client matches the control plane; the claims proxy has a live fraud-request metric series)
[source,texinfo]
----
client version: 1.28.6
1
----

[NOTE]
.Why `--remote=false`
====
The bare `istioctl version` also reports the *control-plane* and *data-plane* builds, but reading those lists cluster-scoped webhook and `istio-system` objects your project role deliberately doesn't include — so as an attendee it stalls on a permissions error.
`--remote=false` prints the client you just installed; the per-pod `istioctl proxy-config` reads later in the lab are how you inspect what the data plane is actually running.
====

Now open Kiali and *see* it.

[NOTE]
.Kiali is a product console — open it directly (single path)
====
Kiali has no OpenShift-console equivalent, so this step is UI-only.
Open `\https://kiali-istio-system.{cluster_domain}` in a new tab and sign in with your cluster identity (`{user}`). Kiali scopes what you see to your OpenShift permissions, so you'll see *your* `{user}-mesh` namespace.

====
The graph, the rates, the health, and the mTLS badges are *real mesh telemetry*.

Each request still egresses through the caller's real sidecar, so the mesh measures and secures it exactly as it would an app-originated call — which is why the graph and the padlocks are genuine.

== 5. Add resilience: faults, timeouts, and a circuit breaker
⏱ *~12 min* · make a slow dependency fail fast instead of hanging the caller

A slow `parasol-fraud` is the cascading-failure trigger. The mesh lets you *simulate* that failure and *configure* the defenses against it. First, baseline the healthy latency:

[source,sh,role=execute]

[source,texinfo]

Now *inject a 5-second delay* on the fraud route with a `VirtualService` fault, and add the resilience policy — a request *timeout*, *retries*, and (on the `DestinationRule`) a *circuit breaker* via connection limits and outlier detection:

[NOTE]
.The honest truth about fault injection vs. the route timeout
====
You set `timeout: 3s`, yet the faulted call returned after *5* seconds, not 3. That's *correct* Istio behavior, and it's worth knowing: Istio applies *fault injection* in a filter that runs *ahead* of the router, so an injected delay (or abort) is *not* subject to that same route's `timeout` or `retries`. Fault injection is a tool to test *how your caller copes* with a slow/failing dependency — not a way to exercise the route's own timeout.

To see the *timeout* actually fire, the *upstream* has to be genuinely slow (a real slow endpoint), which the sample fraud service isn't. So in this lab the timeout/retry/circuit-breaker settings are *real and applied* (you'll confirm them below), and *observing them fire live* is the piece that waits on a slower upstream. That's the honest boundary — the policy is real; the live "watch it trip" needs app behavior the sample doesn't have yet.
====

That's a real circuit breaker: after *3* consecutive 5xx errors the proxy ejects the offending fraud pod for *30s*, and `maxConnections: 10` caps concurrent connections so a slow dependency can't exhaust the caller.
