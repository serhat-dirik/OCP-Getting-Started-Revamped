== Demo arc — The batch queue, and why AI batch is just batch (10 min)

[NOTE]

apiVersion: batch/v1

      containers:

EOF
done

oc -n $NS get workloads -o custom-columns='WORKLOAD:.metadata.name,ADMITTED:.status.conditions[?(@.type=="Admitted")].status'
----

"Two admitted, three pending. The ClusterQueue only has room for two — nobody's month-end run can eat the whole cluster."

oc -n $NS apply -f - <<EOF
apiVersion: batch/v1

oc -n $NS get workloads -o custom-columns='WORKLOAD:.metadata.name,PRIORITY:.spec.priority,ADMITTED:.status.conditions[?(@.type=="Admitted")].status,PREEMPTED:.status.conditions[?(@.type=="Preempted")].status'

"There it is — the high-priority job preempted a low one, kicked it back into the queue, and took its place. That eviction *is* admission control. No spreadsheet, no Slack argument — policy as code."

[demo-block]
[TIME 3m]
Say:: "Last thing. Everyone asks how AI workloads fit. The punchline: an inference job is *the same object*."

[source,sh,role=execute]

oc -n $NS apply -f - <<'EOF'
apiVersion: batch/v1
kind: Job
metadata: {name: fraud-inference, labels: {kueue.x-k8s.io/queue-name: user-queue, kueue.x-k8s.io/priority-class: batch-high}}
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: infer
          image: registry.access.redhat.com/ubi9/ubi:latest
          resources: {requests: {cpu: 200m, memory: 256Mi}, limits: {cpu: 200m, memory: 256Mi}}
          env:
            - {name: TOKEN, valueFrom: {secretKeyRef: {name: maas-credentials, key: GENAI_API_KEY}}}
            - {name: BASE,  valueFrom: {configMapKeyRef: {name: maas-config, key: endpoint}}}
            - {name: MODEL, valueFrom: {configMapKeyRef: {name: maas-config, key: model}}}
          command: ["/bin/sh","-c"]
          args:
            - |
              tail -n +2 /data/claims.csv | head -4 | while IFS=, read -r id c type region amount status; do

                  -d "{\"model\":\"$MODEL\",\"temperature\":0,\"max_tokens\":24,\"messages\":[{\"role\":\"user\",\"content\":\"$P\"}]}" "$BASE/chat/completions")
                r=$(echo "$b" | sed -n 's/.*"content":"\([^"]*\)".*/\1/p' | grep -oiE 'low|medium|high' | head -1)
                echo "$id -> fraud-risk: ${r:-unknown}"
              done
          volumeMounts: [{name: data, mountPath: /data}]
      volumes: [{name: data, persistentVolumeClaim: {claimName: claims-data}}]
EOF
oc -n $NS wait --for=condition=complete job/fraud-inference --timeout=120s
oc -n $NS logs -l job-name=fraud-inference --prefix=false | grep fraud-risk
----
+
"Same label, same queue, same admission and the same preemption. The machinery that keeps a claims batch fair is the machinery that will keep your teams from fighting over GPUs. *AI batch is just batch.*"

== 1. Run a monthly-statement Job
⏱ *~10 min* · a bounded batch that runs to completion, in waves

Your project has a seeded dataset — 200 claim records on a shared volume (`claims-data`), the input every batch job reads.
Confirm it is there, then run the monthly statement.

[source,sh,role=execute]
----
oc -n $NS get pvc claims-data
----

.Expected output
[source,texinfo]
----
NAME          STATUS   VOLUME       CAPACITY   ACCESS MODES   STORAGECLASS                         VOLUMEATTRIBUTESCLASS   AGE
claims-data   Bound    pvc-1db28…   1Gi        RWX            ocs-external-storagecluster-cephfs   <unset>                 30s
----

That volume is *ReadWriteMany* (RWX) on purpose — the statement Job runs *three Pods at once*, the scheduler spreads them across nodes, and they all read the same file simultaneously.
A single-writer (RWO) volume could not do that; this is the storage-module access modes made concrete.

Now the Job.
It asks for `completions: 6` (six Pods must succeed) at `parallelism: 3` (three at a time), so it runs in *two waves*.
Each Pod reads the dataset and totals the claims:

// [CAPTURE-VERIFY] masthead "+" (Quick create) > "Import YAML"; Workloads > Jobs list + Completions column (OCP 4.21) — CLI heredoc authoritative.
[tabs]
====
Console::
+
--
With your `{user}-batch` project selected, use the masthead *+* (*Quick create*) → *Import YAML*, paste the manifest from the *CLI* tab, and click *Create*.
Then watch it under *Workloads* → *Jobs* → `monthly-statement`: the *Completions* count climbs `0 of 6 → 3 of 6 → 6 of 6` across the two waves.
--
CLI::
+
--
[source,sh,role=execute]
----
oc -n $NS apply -f - <<'EOF'
apiVersion: batch/v1
kind: Job
metadata:
  name: monthly-statement
spec:
  parallelism: 3          # up to 3 Pods at once
  completions: 6          # 6 successful Pods = done (two waves of 3)
  backoffLimit: 4
  template:
    metadata: {labels: {app: monthly-statement}}
    spec:
      restartPolicy: Never
      containers:
        - name: statement
          image: registry.access.redhat.com/ubi9/ubi-minimal:latest
          resources: {requests: {cpu: 100m, memory: 128Mi}, limits: {cpu: 300m, memory: 256Mi}}
          command: ["/bin/sh","-c"]
          args:
            - |
              set -eu
              DATA=/data/claims.csv
              total=$(tail -n +2 "$DATA" | wc -l | tr -d ' ')
              sum=$(tail -n +2 "$DATA" | awk -F, '{s+=$5} END{printf "%.0f", s}')
              echo "worker ${HOSTNAME}: processed ${total} claims, total exposure USD ${sum}"
              sleep 12
              echo "segment complete"
          volumeMounts: [{name: data, mountPath: /data}]
      volumes:
        - name: data
          persistentVolumeClaim: {claimName: claims-data}
EOF
----
--
====

Watch it advance through its two waves — the `COMPLETIONS` column climbs `0/6 → 3/6 → 6/6`:

[source,sh,role=execute]
----
oc -n $NS get job monthly-statement -w
----

Press *Ctrl-C* once it shows `6/6`.
On the build cluster the whole Job took *~44&nbsp;s*: the first three Pods ran, finished at about `3/6`, then the second three ran to `6/6`.
Read one Pod's log — every worker saw the same 200 claims and the same total:

[source,sh,role=execute]
----
oc -n $NS logs -l app=monthly-statement --tail=2 | head -2
----

.Expected output
[source,texinfo]
----
worker monthly-statement-hmsd2: processed 200 claims, total exposure USD 2853700
segment complete
----

[.checkpoint]
****
✔ *Verify:* `oc -n $NS get job monthly-statement` shows `COMPLETIONS 6/6`, and a Pod log reports `200 claims, total exposure USD 2853700`.
That is batch: a bounded amount of work that ran to completion, in parallel, and then *stopped*.
****

== 2. Break it, read it, fix it
⏱ *~10 min* · a failing Job, backoff, and why Jobs are immutable

Batch fails differently from a web request — there is no user to see an error page, so the Job controller's *retry and give-up* behavior is what you rely on.
Make a Job fail on purpose.
This one points at the wrong input file:

[source,sh,role=execute]
----
oc -n $NS apply -f - <<'EOF'
apiVersion: batch/v1
kind: Job
metadata:
  name: statement-broken
spec:
  completions: 1
  backoffLimit: 2          # retry twice, then give up
  template:
    metadata: {labels: {app: statement-broken}}
    spec:
      restartPolicy: Never
      containers:
        - name: statement
          image: registry.access.redhat.com/ubi9/ubi-minimal:latest
          resources: {requests: {cpu: 100m, memory: 128Mi}, limits: {cpu: 300m, memory: 256Mi}}
          command: ["/bin/sh","-c"]
          args:
            - |
              set -eu
              DATA=/data/claims-2026.csv        # BUG: the seeded file is /data/claims.csv
              test -f "$DATA" || { echo "ERROR: input file $DATA not found"; exit 1; }
              tail -n +2 "$DATA" | wc -l
          volumeMounts: [{name: data, mountPath: /data}]
      volumes:
        - name: data
          persistentVolumeClaim: {claimName: claims-data}
EOF
----

Watch it fail.
`backoffLimit: 2` means *three* attempts total (the first, plus two retries), each retried after an *exponential* backoff (~10&nbsp;s, then ~20&nbsp;s):

[source,sh,role=execute]
----
oc -n $NS get job statement-broken -w
----

After ~60–70&nbsp;s it stops retrying and reports `1/1` never reached.
Press *Ctrl-C*, then read the events and the Pods — they tell you what happened:

[source,sh,role=execute]
----
oc -n $NS describe job statement-broken | sed -n '/Events:/,$p'
oc -n $NS get pods -l app=statement-broken
----

.Expected output
[source,texinfo]
----
Events:
  Type     Reason                Age   From            Message
  ----     ------                ----  ----            -------
  Normal   SuccessfulCreate      94s   job-controller  Created pod: statement-broken-2zffq
  Normal   SuccessfulCreate      77s   job-controller  Created pod: statement-broken-vjtqr
  Normal   SuccessfulCreate      45s   job-controller  Created pod: statement-broken-b4phg
  Warning  BackoffLimitExceeded  33s   job-controller  Job has reached the specified backoff limit

NAME                     READY   STATUS   RESTARTS   AGE
statement-broken-2zffq   0/1     Error    0          94s
statement-broken-vjtqr   0/1     Error    0          77s
statement-broken-b4phg   0/1     Error    0          45s
----

Three `Error` Pods, then `BackoffLimitExceeded`.
Confirm the actual cause from a Pod log:

[source,sh,role=execute]
----
oc -n $NS logs -l app=statement-broken --tail=1 | head -1
----

It says `ERROR: input file /data/claims-2026.csv not found` — the file name is wrong.
Now try the instinctive fix — edit the Job in place:

[source,sh,role=execute]
----
oc -n $NS patch job statement-broken --type=strategic \
  -p '{"spec":{"template":{"spec":{"containers":[{"name":"statement","command":["/bin/sh","-c","echo fixed"]}]}}}}'
----

It refuses — and OpenShift prints the *entire* Pod template it would not change, so you get a wall of JSON.
Do not be alarmed: you did not break anything else.
The signal is the very last line:

.Expected output (JSON dump elided — the last line is the point)
[source,texinfo]
----
The Job "statement-broken" is invalid: spec.template: Invalid value: {"labels":{…},"Spec":{…}}: field is immutable
----

*A Job's Pod template is immutable.*
You cannot edit a Job's command once it exists — which is the platform telling you that batch definitions belong in files and Git, not in live hand-edits.
The fix is to delete and re-create with the correct path:

[source,sh,role=execute]
----
oc -n $NS delete job statement-broken
oc -n $NS apply -f - <<'EOF'
apiVersion: batch/v1
kind: Job
metadata: {name: statement-broken}
spec:
  completions: 1
  backoffLimit: 2
  template:
    metadata: {labels: {app: statement-broken}}
    spec:
      restartPolicy: Never
      containers:
        - name: statement
          image: registry.access.redhat.com/ubi9/ubi-minimal:latest
          resources: {requests: {cpu: 100m, memory: 128Mi}, limits: {cpu: 300m, memory: 256Mi}}
          command: ["/bin/sh","-c"]
          args:
            - |
              set -eu
              DATA=/data/claims.csv             # fixed path
              tail -n +2 "$DATA" | wc -l
          volumeMounts: [{name: data, mountPath: /data}]
      volumes:
        - name: data
          persistentVolumeClaim: {claimName: claims-data}
EOF
oc -n $NS wait --for=condition=complete job/statement-broken --timeout=60s
oc -n $NS logs -l app=statement-broken --tail=1
----

It prints `200` — the claim count.
Fixed.

[.checkpoint]
****
✔ *Verify:* You saw three `Error` Pods and a `BackoffLimitExceeded` event; the in-place edit failed with `field is immutable`; and the re-created Job completed, printing `200`.
****

== 3. Shard the dataset with an Indexed Job
⏱ *~8 min* · split the work across Pods, deterministically

`parallelism` runs many Pods, but in the statement Job every Pod did the *same* work.
To split a dataset into *shards* — each Pod handling a distinct slice — use an *Indexed* Job.
Each Pod gets a fixed index in `$JOB_COMPLETION_INDEX`, and you use that index to pick your slice:

// [CAPTURE-VERIFY] "Import YAML"; Workloads > Jobs (Indexed completions, statement-sharded-0..4) (OCP 4.21) — CLI heredoc authoritative.
[tabs]
====
Console::
+
--
With your `{user}-batch` project selected, create it via the masthead *+* → *Import YAML* (paste the manifest from the *CLI* tab, then *Create*), and watch the five indexed Pods (`statement-sharded-0`…`-4`) under *Workloads* → *Jobs* → `statement-sharded`.
--
CLI::
+
--
[source,sh,role=execute]
----
oc -n $NS apply -f - <<'EOF'
apiVersion: batch/v1
kind: Job
metadata: {name: statement-sharded}
spec:
  completionMode: Indexed   # each Pod gets a fixed index 0..N-1
  completions: 5            # 5 shards
  parallelism: 5            # all at once
  backoffLimit: 2
  template:
    metadata: {labels: {app: statement-sharded}}
    spec:
      restartPolicy: Never
      containers:
        - name: shard
          image: registry.access.redhat.com/ubi9/ubi-minimal:latest
          resources: {requests: {cpu: 100m, memory: 128Mi}, limits: {cpu: 300m, memory: 256Mi}}
          command: ["/bin/sh","-c"]
          args:
            - |
              set -eu
              DATA=/data/claims.csv
              IDX="${JOB_COMPLETION_INDEX}"     # injected by Kubernetes for Indexed Jobs
              SHARDS=5
              out=$(tail -n +2 "$DATA" | awk -F, -v idx="$IDX" -v n="$SHARDS" '
                { id=substr($1,5)+0; if ((id-1)%n==idx){ c++; s+=$5 } }
                END { printf "%d %d", c+0, s+0 }')
              set -- $out
              echo "shard ${IDX}/${SHARDS} on ${HOSTNAME}: ${1} claims, subtotal USD ${2}"
          volumeMounts: [{name: data, mountPath: /data}]
      volumes:
        - name: data
          persistentVolumeClaim: {claimName: claims-data}
EOF
oc -n $NS wait --for=condition=complete job/statement-sharded --timeout=90s
----
--
====

Each shard processed a *different* fifth of the 200 claims.
Read them in order:

[source,sh,role=execute]
----
oc -n $NS logs -l app=statement-sharded --prefix=false --tail=1 | sort
----

.Expected output
[source,texinfo]
----
shard 0/5 on statement-sharded-0: 40 claims, subtotal USD 559780
shard 1/5 on statement-sharded-1: 40 claims, subtotal USD 565260
shard 2/5 on statement-sharded-2: 40 claims, subtotal USD 570740
shard 3/5 on statement-sharded-3: 40 claims, subtotal USD 576220
shard 4/5 on statement-sharded-4: 40 claims, subtotal USD 581700
----

Five shards, 40 claims each, no overlap — and the five subtotals add up to `2,853,700`, exactly the total the monthly-statement Job computed in exercise 1.
That is map/reduce's "map" step with nothing but the Job controller: notice the Pod *names* even carry their index (`statement-sharded-0` … `-4`).

[.checkpoint]
****
✔ *Verify:* All five shards completed, each reporting `40 claims`, and the subtotals sum to the exercise-1 total — proof the dataset was partitioned cleanly across Pods by `$JOB_COMPLETION_INDEX`.

== 4. Schedule it with a CronJob
⏱ *~8 min* · nightly batch, without overlap — and a suspend switch

Real "nightly batch" runs on a schedule.
A *CronJob* wraps a Job template in cron syntax.
The three fields that keep it out of trouble are `concurrencyPolicy: Forbid` (never overlap runs), `startingDeadlineSeconds` (skip a missed tick rather than pile up), and `suspend` (pause without deleting).
So you don't wait until 2&nbsp;a.m., this one runs *every minute* — the real schedule is in the comment:

// [CAPTURE-VERIFY] "Import YAML"; Workloads > CronJobs list (OCP 4.21) — CLI heredoc authoritative.
[tabs]
====
Console::
+
--
With your `{user}-batch` project selected, create it via the masthead *+* → *Import YAML* (paste the manifest from the *CLI* tab, then *Create*).
It appears under *Workloads* → *CronJobs* → `nightly-statement`, and a new `nightly-statement-…` Job shows up there at the top of each minute.
--
CLI::
+
--
[source,sh,role=execute]
----
oc -n $NS apply -f - <<'EOF'
apiVersion: batch/v1
kind: CronJob
metadata: {name: nightly-statement}
spec:
  schedule: "*/1 * * * *"        # every minute FOR THE LAB (real nightly would be "0 2 * * *")
  concurrencyPolicy: Forbid      # never let a slow run overlap the next tick
  startingDeadlineSeconds: 30    # if the controller misses the tick by >30s, skip this run
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 1
  jobTemplate:
    metadata: {labels: {app: nightly-statement}}   # label the Jobs this CronJob creates so the watch below can select them
    spec:
      backoffLimit: 2
      template:
        metadata: {labels: {app: nightly-statement}}
        spec:
          restartPolicy: Never
          containers:
            - name: statement
              image: registry.access.redhat.com/ubi9/ubi-minimal:latest
              resources: {requests: {cpu: 100m, memory: 128Mi}, limits: {cpu: 300m, memory: 256Mi}}
              command: ["/bin/sh","-c"]
              args:
                - |
                  set -eu
                  DATA=/data/claims.csv
                  total=$(tail -n +2 "$DATA" | wc -l | tr -d ' ')
                  sum=$(tail -n +2 "$DATA" | awk -F, '{s+=$5} END{printf "%.0f", s}')
                  echo "nightly run: processed ${total} claims, total exposure USD ${sum}"
              volumeMounts: [{name: data, mountPath: /data}]
          volumes:
            - name: data
              persistentVolumeClaim: {claimName: claims-data}
EOF
----
--
====

The CronJob does nothing until the next tick.
Watch for it to fire — the list starts empty, then at the top of the next minute a `nightly-statement-<number>` Job appears and runs to completion:

[source,sh,role=execute]
----
oc -n $NS get jobs -l app=nightly-statement -w
----

Once a `nightly-statement-<number>` Job appears and completes (press *Ctrl-C*), read its output — it is your statement run, on a schedule:

[source,sh,role=execute]
----
oc -n $NS logs -l app=nightly-statement --tail=1 | head -1
----

.Expected output
[source,texinfo]
----
nightly run: processed 200 claims, total exposure USD 2853700
----

Now the *suspend* switch — how you pause a schedule during a freeze or an incident without deleting it:

[source,sh,role=execute]
----
oc -n $NS patch cronjob nightly-statement --type=merge -p '{"spec":{"suspend":true}}'
oc -n $NS get cronjob nightly-statement
----

.Expected output
[source,texinfo]
----
NAME                SCHEDULE      TIMEZONE   SUSPEND   ACTIVE   LAST SCHEDULE   AGE
nightly-statement   */1 * * * *   <none>     True      0        60s             90s
----

With `SUSPEND: True`, the schedule is paused — no new Jobs are created on the next tick, and nothing is lost.
Flip it back with `suspend:false` when you are ready.

[.checkpoint]

== 5. Put batch under admission control with Kueue
⏱ *~15 min* · watch a queue admit, hold, and preempt — the heart of the module

So far every Job the scheduler could place, it ran.
At organization scale that is exactly the problem: the biggest batch wins and everyone else waits.
*Red Hat build of Kueue* fixes that with admission control.
Your project is already wired for it — a `LocalQueue` named `user-queue` points at a `ClusterQueue` (`cq-{user}`) whose quota is deliberately tiny: about *two* of our sample Pods.
Confirm your queue is live:

[source,sh,role=execute]
----
oc -n $NS get localqueue user-queue
----

.Expected output (the `CLUSTERQUEUE` column shows *your* queue — `cq-user8` here was captured as user8)
[source,texinfo]
----
NAME         CLUSTERQUEUE   PENDING WORKLOADS   ADMITTED WORKLOADS
user-queue   cq-user8       0                   0
----

A Job joins the queue with one label, `kueue.x-k8s.io/queue-name: user-queue`.
Submit *five* low-priority jobs at once, each requesting `200m` CPU and `256Mi` — two of which fill the quota:

[source,sh,role=execute]
----
for i in 1 2 3 4 5; do
oc -n $NS apply -f - <<EOF
apiVersion: batch/v1
kind: Job
metadata:
  name: batch-low-$i
  labels:
    kueue.x-k8s.io/queue-name: user-queue
    kueue.x-k8s.io/priority-class: batch-low
spec:
  parallelism: 1
  completions: 1
  backoffLimit: 0
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: work
          image: registry.access.redhat.com/ubi9/ubi-minimal:latest
          resources: {requests: {cpu: 200m, memory: 256Mi}, limits: {cpu: 200m, memory: 256Mi}}
          command: ["/bin/sh","-c","echo batch-low-$i running; sleep 300"]
EOF
done
----

Kueue intercepts each labeled Job, *suspends* it, and creates a *Workload* object to track it.
Give it a few seconds, then look at the Workloads — the real view of the queue:

[source,sh,role=execute]
----
sleep 8
oc -n $NS get workloads -o custom-columns='WORKLOAD:.metadata.name,PRIORITY:.spec.priority,ADMITTED:.status.conditions[?(@.type=="Admitted")].status'
----

.Expected output
[source,texinfo]
----
WORKLOAD                PRIORITY   ADMITTED
job-batch-low-1-52c22   100        True
job-batch-low-2-5e4d0   100        True
job-batch-low-3-717ff   100        <none>
job-batch-low-4-f12d5   100        <none>
job-batch-low-5-ace1f   100        <none>
----

*Two admitted, three pending.*
The two admitted jobs' Pods are running; the other three are held suspended — their quota isn't available.
Confirm the Jobs themselves: the admitted two have `suspend: false`, the pending three `suspend: true`:

[source,sh,role=execute]
----
oc -n $NS get jobs -o custom-columns='JOB:.metadata.name,SUSPEND:.spec.suspend'
----

Now the moment the whole module is built around.
The fraud team has an *urgent* scoring job — higher priority.
Submit *one* `batch-high` job into the same full queue:

[source,sh,role=execute]
----
oc -n $NS apply -f - <<EOF
apiVersion: batch/v1
kind: Job
metadata:
  name: batch-high-1
  labels:
    kueue.x-k8s.io/queue-name: user-queue
    kueue.x-k8s.io/priority-class: batch-high
spec:
  parallelism: 1
  completions: 1
  backoffLimit: 0
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: work
          image: registry.access.redhat.com/ubi9/ubi-minimal:latest
          resources: {requests: {cpu: 200m, memory: 256Mi}, limits: {cpu: 200m, memory: 256Mi}}
          command: ["/bin/sh","-c","echo batch-high-1 running; sleep 300"]
EOF
sleep 12
oc -n $NS get workloads -o custom-columns='WORKLOAD:.metadata.name,PRIORITY:.spec.priority,ADMITTED:.status.conditions[?(@.type=="Admitted")].status,PREEMPTED:.status.conditions[?(@.type=="Preempted")].status'
----

.Expected output
[source,texinfo]
----
WORKLOAD                 PRIORITY   ADMITTED   PREEMPTED
job-batch-high-1-73396   1000       True       <none>
job-batch-low-1-d881e    100        True       <none>
job-batch-low-2-0a64a    100        False      True
job-batch-low-3-a9bbc    100        <none>     <none>
job-batch-low-4-3cd1e    100        <none>     <none>
job-batch-low-5-70dc1    100        <none>     <none>
----

The high-priority Workload is `Admitted`, and one previously-running low-priority Workload (`batch-low-2` here) is now `PREEMPTED` — evicted from its slot and sent back into the queue to make room.
See exactly why in its conditions:

[source,sh,role=execute]
----
VICTIM=$(oc -n $NS get workloads -o jsonpath='{range .items[*]}{.metadata.name}{" "}{.status.conditions[?(@.type=="Preempted")].status}{"\n"}{end}' | awk '$2=="True"{print $1}' | head -1)
oc -n $NS get workload "$VICTIM" -o jsonpath='{range .status.conditions[*]}{.type}={.status} ({.reason}){"\n"}{end}'
----

.Expected output
[source,texinfo]
----
QuotaReserved=False (Pending)
Evicted=True (Preempted)
Admitted=False (NoReservation)
Preempted=True (InClusterQueue)
Requeued=True (Preempted)
----

`Evicted (Preempted)` then `Requeued` — the low-priority job wasn't lost, it was *put back in line*.
When the high-priority job finishes and frees quota, this one gets admitted again.
That is the entire promise of admission control: urgent work gets in, nothing is dropped, and no single team can monopolize the cluster.

Clean up the queue before the next exercise:

[source,sh,role=execute]
----
oc -n $NS delete jobs -l kueue.x-k8s.io/queue-name=user-queue
----

****

****

== 6. Run AI batch inference — through the same queue
⏱ *~10 min* · an inference Job is admitted, queued, and preempted like any other

Here is the payoff.
Score every claim for fraud risk by calling a model — a *batch-inference* Job.
It is an ordinary Job: it reads the dataset, calls a *Models-as-a-Service* endpoint, writes one-line verdicts, and stops.
Its endpoint and model come from a ConfigMap, and its bearer token from a Secret, both already in your project:

[source,sh,role=execute]
----
oc -n $NS get configmap maas-config -o jsonpath='endpoint={.data.endpoint}{"\n"}model={.data.model}{"\n"}'
oc -n $NS get secret maas-credentials -o jsonpath='token bytes: {.data.GENAI_API_KEY}' | wc -c
----

Now the inference Job.
Note the `kueue.x-k8s.io/queue-name` label — *it goes through the same LocalQueue as everything else*.
It is Indexed with two workers, each scoring a few claims at `temperature 0` (so the answers are reproducible):

[source,sh,role=execute]
----
oc -n $NS apply -f - <<'EOF'
apiVersion: batch/v1
kind: Job
metadata:
  name: fraud-inference
  labels:
    kueue.x-k8s.io/queue-name: user-queue        # admission-controlled, same as any batch
    kueue.x-k8s.io/priority-class: batch-high
spec:
  completionMode: Indexed
  completions: 2
  parallelism: 2
  backoffLimit: 2
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: infer
          image: registry.access.redhat.com/ubi9/ubi:latest
          resources: {requests: {cpu: 200m, memory: 256Mi}, limits: {cpu: 200m, memory: 256Mi}}
          env:
            - {name: TOKEN, valueFrom: {secretKeyRef: {name: maas-credentials, key: GENAI_API_KEY}}}
            - {name: BASE,  valueFrom: {configMapKeyRef: {name: maas-config, key: endpoint}}}
            - {name: MODEL, valueFrom: {configMapKeyRef: {name: maas-config, key: model}}}
          command: ["/bin/sh","-c"]
          args:
            - |
              set -eu
              DATA=/data/claims.csv
              IDX="${JOB_COMPLETION_INDEX}"
              OUT="/data/fraud-risk-${IDX}.txt"; : > "$OUT"
              tail -n +2 "$DATA" | awk -v idx="$IDX" 'NR%2==idx' | head -3 | while IFS=, read -r id claimant type region amount status; do
                P="/no_think You are a fraud-triage assistant. Reply with exactly one word (low, medium, or high) for the fraud risk of this claim: type=${type} region=${region} amount=${amount} status=${status}."
                body=$(curl -s --max-time 40 -H "Authorization: Bearer $TOKEN" -H 'Content-Type: application/json' \
                  -d "{\"model\":\"$MODEL\",\"temperature\":0,\"max_tokens\":24,\"messages\":[{\"role\":\"user\",\"content\":\"$P\"}]}" \
                  "$BASE/chat/completions") || body=""
                risk=$(echo "$body" | sed -n 's/.*"content":"\([^"]*\)".*/\1/p' | grep -oiE 'low|medium|high' | head -1)
                echo "${id} (${type}/${amount}) -> fraud-risk: ${risk:-unknown}" | tee -a "$OUT"
              done
              echo "worker ${IDX} done"
          volumeMounts: [{name: data, mountPath: /data}]
      volumes:
        - name: data
          persistentVolumeClaim: {claimName: claims-data}
EOF
oc -n $NS wait --for=condition=complete job/fraud-inference --timeout=120s
----

It finishes in well under a minute.
Read the risk verdicts the model returned:

[source,sh,role=execute]
----
oc -n $NS logs -l job-name=fraud-inference --prefix=false | grep fraud-risk | sort
----

.Expected output
[source,texinfo]
----
CLM-0001 (home/637) -> fraud-risk: medium
CLM-0002 (life/774) -> fraud-risk: medium
CLM-0003 (travel/911) -> fraud-risk: medium
CLM-0004 (commercial/1048) -> fraud-risk: medium
CLM-0005 (auto/1185) -> fraud-risk: medium
CLM-0006 (home/1322) -> fraud-risk: medium
----

Now confirm the crucial point — this AI job went through Kueue exactly like the statement run:

[source,sh,role=execute]
----
oc -n $NS get workloads -o custom-columns='WORKLOAD:.metadata.name,ADMITTED:.status.conditions[?(@.type=="Admitted")].status,FINISHED:.status.conditions[?(@.type=="Finished")].status' | grep infer
----

A Workload, `Admitted`, then `Finished` — the same object, the same queue, the same admission control as every other Job in this lab.
That is the whole thesis: *AI batch is just batch*.
The machinery that keeps a claims statement run fair is the machinery that will govern your GPU fleet.

[.checkpoint]
****
✔ *Verify:* The `fraud-inference` Job completed, its Pods wrote `fraud-risk: <low|medium|high>` verdicts for the claims, and its Workload shows `Admitted` — proof the inference ran through the LocalQueue like any batch.
****
