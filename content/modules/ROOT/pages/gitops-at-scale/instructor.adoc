The module's signature beat is exercise 4: a bad release that the canary's analysis *aborts* on its own, the stable version never leaving production (route `200` throughout).
Both canaries are fast (~2 min each, pod-ratio pause + a job-provider analysis), so the whole ship-canary-rollback arc lands *live* with no waiting to narrate around.
Two seams need your attention up front: attendees *create the ApplicationSet in the Argo CD UI* (no CLI path from the terminal), and the ApplicationSet *outlives a namespace reset* (delete it before `ws prep`).

== Timing

Total hands-on: *~70 min* (spec target 90 min; add ~12–15 min of concept up front).

[cols="1,4,1,2",options="header"]
|===
| # | Exercise group | Attendee time | Measured core (as user7, {ocp_version})

| 1 | One ApplicationSet generates the fleet | ~15 min | CLI download *~290 MB / ~30–60s* (once) + token + `appset create`; generation *~8s*; prod converges (db + migration + Rollout Healthy) in *~50s*
| 2 | Sync waves and hooks | ~10 min | read-only (`oc get`/`oc logs`) — seconds
| 3 | Canary with automated analysis | ~20 min | one Gitea commit + Refresh; canary *25%→pause 20s→50%→analysis ~45s→100%* ≈ *2 min*
| 4 | Break and fix (auto-rollback) | ~20 min | knob + Gitea commit; canary aborts at analysis in *~90s*; git-revert recovery *~30s*
| 5 | Blue-green (concept + challenge) | ~5 min | reading; optional hands-on blue-green in the Challenge

[NOTE]
.How this timing was measured
====
Every exercise was performed as `user7` on the build cluster (OCP {ocp_version}, OpenShift GitOps {gitops_version}, Argo CD {argocd_version}, Argo Rollouts GA via a cluster-scoped `RolloutManager`) on 2026-07-11, creating the ApplicationSet with the served `argocd` CLI + a per-user token (the way the attendee does — the UI has no appset screen) and shipping new versions by committing to the `{user}/claims-config` fork, then timing each operation.

* *ApplicationSet (exercise 1)* — the served CLI is *~290 MB* (a one-time ~30–60&nbsp;s fetch); `~/argocd appset create` then generated the three Applications in *~8 seconds* (dev + stage *adopted* by name, prod *created*); production converged in *~50 seconds* (claims-db, the wave-1 migration Job, then the Rollout to `4/4` Healthy) on a warm image cache.
On a *cold* node add the image pull for the first user of the day.
* *Canary (exercise 3)* — from the Gitea commit to `100%` Healthy was *~2 minutes*: `setWeight 25` (1 of 4 pods), a *20 s* pause, `setWeight 50` (2 of 4), the job-provider analysis (*~45 s*: two 200-probes at a 15 s interval plus Job startup), then `setWeight 100`.
* *Break/abort (exercise 4)* — with `gitops-at-scale-canary-control` = `fail`, the canary reached the analysis step and *aborted* about *90 seconds* after the commit; the route answered `200` throughout and the stable ReplicaSet never scaled down.
* *Fix (exercise 4)* — clearing the knob and reverting the image in Git returned the Rollout to Healthy in *~30 seconds* (reverting to the version that is already stable clears the abort without a fresh canary).
====

== Pre-flight checklist

Run these *before the session* (as cluster admin unless noted).

. *A sample user's entry state materializes and verifies.*
+
[source,sh,role=execute]
----
ws start gitops-at-scale --user user1
ws verify gitops-at-scale --user user1 --entry-only
----
+
Expect `✅ all 14 checks passed`: the `user1-gitops` namespace + marker, the student Argo CD instance reachable, `proj-user1`, the `user1/claims-config` fork carrying the `rollouts/` overlay *and* `applicationset.yaml`, the analysis prereqs in `user1-prod` (the `claims-analysis` ServiceAccount + `gitops-at-scale-canary-control` knob), claims GitOps-managed in `user1-dev`/`user1-stage`, and *no Rollout yet in `user1-prod`* (standing prod up is the lab).

. *The cluster Argo Rollouts controller is serving.*
+
[source,sh,role=execute]
----
oc get rolloutmanager -n openshift-gitops
oc get pods -n openshift-gitops | grep rollouts
----
+
Expect a `RolloutManager` (cluster-scoped) and a running `argo-rollouts` controller pod.
A `Rollout` only ever goes Healthy if this controller processes it — if it is missing, the `progressive-delivery` platform stack was not applied.

. *Attendees can create ApplicationSets — controller, RBAC, and the served CLI.*
+
[source,sh,role=execute]
----
oc get pods -n ogsr-student-gitops | grep applicationset
argocd admin settings rbac can user1 create applicationsets 'proj-user1/x' --namespace ogsr-student-gitops
DOMAIN=$(oc get ingresses.config.openshift.io cluster -o jsonpath='{.spec.domain}')
curl -ks -o /dev/null -w 'CLI served: HTTP %{http_code}\n' -r 0-1 \
  "https://student-gitops-server-student-gitops.${DOMAIN}/download/argocd-linux-amd64"
----
+
Expect the `student-gitops-applicationset-controller` pod `Running`, the RBAC check `Yes`, and `CLI served: HTTP 206` (the server ships the `argocd` binary the attendee downloads in exercise 1).
Attendees create the ApplicationSet from the *terminal with the CLI* (the UI has no appset screen — see Watchouts); all three must be in place, and `ws verify gitops-at-scale --entry-only` gates the CLI-served one.

. *The analysis knob is present and defaults to pass.*
+
[source,sh,role=execute]
----
oc get configmap gitops-at-scale-canary-control -n user1-prod -o jsonpath='verdict={.data.verdict}{"\n"}'
----
+
Expect `verdict=pass`.
Exercise 4 flips it to `fail` and back; it is entry-owned and defaults to `pass` on every `ws prep`.

. *(For a demo) pre-solve a sample user — it blocks ~4–6 min by design.*
+
[source,sh,role=execute]
----
ws solve gitops-at-scale --user user1     # stands prod up as a canary Rollout; BLOCKS until Synced/Healthy
----
+
[IMPORTANT]
.`ws solve gitops-at-scale` blocks for ~4–6 minutes — that is correct, not a hang
=====
`ws solve gitops-at-scale` materializes the end state — production running claims as a canary *Rollout* (with the wave-ordered migration) on top of the always-present dev/stage — and *waits* for it to be Synced/Healthy before returning.
It does not print "demo-ready" until the prod Rollout is actually `4/4` Healthy (image pull + claims-db + the migration Job + the Rollout, all in prod).
Do not Ctrl-C it thinking it stalled.
=====

== Demo-mode notes

Beyond the Say/Show/Do arc in the demo flavor:

* *Pre-solve and pre-sign-in before the room.*
Run `ws solve gitops-at-scale --user {user}` early (blocks ~4–6 min) so a finished, Healthy prod Rollout is your opening visual, and sign in to the student Argo CD UI as `{user}` ahead of time — open `claims-prod-{user}` and switch to its *Rollout* view (the canary dashboard) so beats 2–3 are one click away.
* *Both canaries are fast — the whole arc lands live.*
The green canary (beat 2) and the aborting canary (beat 3) each take ~2 minutes; there is nothing to wait out.
Let the Rollout view animate the 25% → 50% → analysis → 100% on screen, then the abort snapping back to stable.
* *Drive by Git, narrate by terminal.*
Ship each version with a one-line Gitea edit (the image tag) so the audience sees "GitOps"; keep the terminal poll running so the step/phase is legible while the UI animates.
For the abort beat, `oc patch … verdict=fail` *before* the commit — the solve prod app is `selfHeal: false`, so the knob holds while you drive.
* *Have a fallback.*
If the shared student controller is sluggish and a sync lags, narrate from the Rollout view's history (the previous green canary's revisions) rather than waiting on a fresh one.

== Top questions attendees ask

*1. "Is the canary really sending 25% of requests to the new version?"*::
Roughly, yes — by *pod ratio*, not by counting requests.
At `setWeight: 25` with 4 replicas the controller runs *1 canary pod and 3 stable pods*, and the single Service load-balances across all four, so about a quarter of traffic hits the new version.
*Exact* request percentages, header-based routing, or mirroring need a *traffic router* — an OpenShift Route plugin or a service mesh (the Networking and Service Mesh modules) — which this platform deliberately does not require, so the lab installs cleanly with no external plugin.

*2. "The canary aborted — why didn't production go down?"*::
Because a Rollout keeps the *stable* ReplicaSet running the whole time and only scales the *canary* up to the current weight.
When analysis fails, it *aborts*: the canary pods are scaled back to zero and the stable pods — which never stopped serving — carry all traffic again.
The route answers `200` throughout (the lab proves it).
"Roll back" here means "the new version never got promoted," not "tear down and redeploy."

*3. "How do I create an ApplicationSet — there's no button in the UI?"*::
You create it from the *terminal with the Argo CD CLI*, not the UI — this Argo CD version (3.4) has no create-ApplicationSet screen (nor a read view for them).
Exercise 1 downloads the CLI the student server ships (`/download/argocd-linux-amd64`), gets a token from `/api/v1/session` scoped to `proj-{user}` by the attendee's per-user *local Argo account*, then runs `~/argocd appset create applicationset.yaml`.
The token flow (not `argocd login`) is deliberate: interactive login negotiates gRPC over the edge route and does not complete reliably; `ARGOCD_OPTS='--grpc-web --insecure'` plus a REST token does.

*4. "Canary or blue-green — which should we actually use?"*::
Canary when you want to *limit blast radius* and can *measure* health during the rollout (a bad build fails analysis at 50%, not 100%); its cost is a small extra slice.
Blue-green when you want a *clean instant cutover* after a full pre-flight on a preview URL, and can afford *double capacity* for the switch window.
Argo Rollouts serves both; the wrap-up has the decision guide, and the Challenge lets them stand up a blue-green Rollout.

*5. "The failing analysis is a knob — isn't that faked?"*::
The knob makes the failure *deterministic* for a timed lab, but the analysis is real: the same Job also *probes the canary's live readiness endpoint*, so a genuinely broken canary (one that never becomes ready, or whose database wiring is wrong) fails the analysis on its own merits too.
The knob simply lets you reproduce, on demand, the *signal* a real regression would produce — without shipping a genuinely broken image and hoping it fails the same way every run.

== Watchouts

* *The ApplicationSet is created from the CLI, not the UI (Argo {argocd_version} has no appset screen).*
The Showroom terminal has `oc`, `git`, and `helm` but no `argocd`/Rollouts plugin, and attendees cannot `oc apply` into `ogsr-student-gitops` — so exercise 1 fetches the CLI the student Argo server *serves* (`/download/argocd-linux-amd64`, ~290&nbsp;MB) and authenticates with a per-user *local Argo account* token via `/api/v1/session` (`--grpc-web --insecure`).
Pre-flight check 3 confirms the server serves the CLI and the account + RBAC exist; the verify's `argocd CLI served` check gates it too.
The ~290&nbsp;MB download is a one-time ~30–60&nbsp;s wait the first time an attendee runs exercise 1 — set the expectation.
Do *not* steer attendees to `argocd login` (interactive gRPC over the route is unreliable — the lab uses a REST token) or to `oc apply` into `ogsr-student-gitops`.
* *Reset handles the ApplicationSet — `ws prep` is enough.*
`ws reset`/`ws prep` purge the attendee's *ApplicationSet first* (in the shared control plane), then its Applications, then the namespaces — so a surviving ApplicationSet cannot re-generate what was purged, and a full reset leaves nothing to resurrect (verified).
Attendees do not need a manual step; if one prefers, `~/argocd appset delete claims-{user}` removes it and prunes its apps in one shot.
* *Watch the canary with `oc get rollout`, not a plugin.*
There is no Argo Rollouts CLI plugin (`oc argo rollouts`) in the attendee terminal, so the lab observes the canary with plain `oc get rollout … -o jsonpath` (step/phase) plus the *Argo CD Rollout view* for the visual.
If you demo from your laptop and have the plugin installed, `oc argo rollouts get rollout parasol-claims -n {user}-prod --watch` is a nicer live view — but do not put it in attendee steps; it will not be found.
* *The analysis knob is entry-owned; a mid-lab `ws start`/`ws prep` resets it to `pass`.*
`gitops-at-scale-canary-control` is materialized by the entry app (with `selfHeal: false`), so a manual `oc patch … verdict=fail` *holds* through the lab.
But re-running `ws start`/`ws prep`/a full entry re-sync while a room is mid-exercise-4 will reset it to `pass` and a canary that should abort will instead promote.
Set the knob and ship within the same exercise; if a canary unexpectedly promotes, re-check `oc get cm gitops-at-scale-canary-control -n {user}-prod -o jsonpath='{.data.verdict}'` before assuming a bug.
* *`ws prep` reports the GitOps Fundamentals end-state workloads as "N workloads but no gitops-at-scale marker" — cosmetic.*
Because GitOps at Scale's entry state deploys claims into `{user}-dev`/`{user}-stage` (workloads) while the marker lives in `{user}-gitops`, `ws prep gitops-at-scale` lists those namespaces as "has N workload(s) but no gitops-at-scale marker."
It is accurate and harmless — the marker is simply in a different namespace — and `ws prep --yes` proceeds correctly.
Do not "fix" it by chasing a missing marker in dev/stage.
* *Per-user prod quota during a canary.*
A canary transiently runs stable + canary pods (up to ~5 app pods: 4 stable + 1 canary at 25%, or 4 + 2 at 50%) plus a short analysis Job pod, on top of claims-db.
This fits the `{user}-prod` quota at the shipped replica count (4); if you raise `spec.replicas`, re-check the quota headroom for a full room running canaries at once.

== Provenance

GitOps at Scale is built fresh on OpenShift GitOps {gitops_version} (Argo CD {argocd_version}) with *Argo Rollouts* (GA, cluster-scoped `RolloutManager`), against the Parasol `claims-config` repo (Kustomize base + dev/stage/prod overlays + a `rollouts/` progressive-delivery overlay + `applicationset.yaml`), on the shared *student* Argo CD instance with per-user AppProjects, `applicationsets` RBAC, and per-user *local Argo accounts* + a server-served CLI (Argo {argocd_version} has no ApplicationSet UI).
Every command, output, and timing — the CLI download (~290 MB), the token-scoped `~/argocd appset create` and the `PermissionDenied` a neighbour's project returns, the ApplicationSet adopting dev/stage and generating prod (~8s), the wave-ordered convergence (db → migration Job → Rollout, ~50s), the `1.0→1.1` canary (25/50/100 pod-ratio with a green job-provider `AnalysisRun`, ~2 min), the forced abort (`RolloutAborted … Metric "canary-health" assessed Failed`, stable preserved, route `200` throughout, ~90s), the git-revert recovery (~24s), and a scratch blue-green preview/promote — was performed and measured on OpenShift {ocp_version} as `user7` (G3 fix round; earlier beats as `user4`) on 2026-07-11.
The ApplicationSet uses a *list* generator (prod → `rollouts/`) so the fleet is generated from one object and production runs progressive delivery; structure and narrative adapt the OpenShiftDemos advanced-gitops and argo-rollouts workshops (credit per the reuse policy).
