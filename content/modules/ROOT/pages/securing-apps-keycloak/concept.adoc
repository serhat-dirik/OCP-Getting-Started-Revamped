[.lead]
Parasol's claims API answers to anyone who can reach it on the network.
Type the URL, get every claimant's name, amount, and adjuster â€” no login, no token, no questions asked.
That was fine when "on the network" meant a locked server room; it is a data breach when the network is a shared cluster, a partner integration, and a mobile app.

This module puts a *front door* on the Parasol apps â€” a login for the web frontend, bearer tokens for the API, a machine identity for the batch job â€” using *Red Hat build of Keycloak*, the identity and single sign-on (SSO) product that ships with your OpenShift subscription.
You will not write a password check, a session store, or a login page.
You will point your apps at Keycloak, describe *who may do what*, and let the platform enforce it.

== Authentication is not authorization

Two questions hide inside every "is this allowed?" and confusing them is the most common security bug there is.

* *Authentication (authN)* â€” *who are you?* Proving identity.
A login, a token, a certificate.
* *Authorization (authZ)* â€” *what may you do?* Checking permission.
A role, a group, an ownership rule.

A valid token answers only the first question.
The Parasol `viewer` user can *log in* perfectly â€” correct password, valid token â€” and still must not be allowed to list claims, because listing claims needs the `claims-adjuster` role they do not have.
You will see this split as two different HTTP status codes in the lab: *401 Unauthorized* means "I don't know who you are" (authN failed), and *403 Forbidden* means "I know exactly who you are, and the answer is no" (authZ failed).
Keep them straight and half of application security falls into place.

== The four flows, and when each applies

OAuth 2.0 and OpenID Connect (OIDC) define a handful of *flows* â€” scripted conversations for getting a token.
It is tempting to learn all of them; resist.
Four flows cover almost everything an enterprise app needs, and this module teaches exactly those four.

* *Authorization code + PKCE* protects the *web frontend*.
The browser is redirected to Keycloak, the human logs in *there*, and the app receives a short-lived *authorization code* it swaps for tokens.
The app never handles the password.
*PKCE* (Proof Key for Code Exchange, "pixie") adds a one-time secret per login so a stolen code is useless â€” mandatory for browser and mobile clients that cannot keep a secret.
* *Bearer tokens* protect the *API*.
The caller attaches a token (`Authorization: Bearer â€¦`); the API validates its signature, issuer, expiry, and audience, then reads its roles to decide.
No redirect, no session â€” every request stands alone.
* *Client credentials* is for *machines*.
The Jobs, Batch & Queued Workloads batch job has no human and no browser; it authenticates as *itself* with a client id and secret and gets a token that represents the *service*, not a person.
* *Token exchange (RFC 8693)* is for *one service calling another on a user's behalf*.
The claims service takes the user's token and asks Keycloak to *re-audience* it for the fraud service â€” same user, narrower scope.
The crucial property, which you will prove in the lab: exchange can *downscope* but never *escalate*.

[NOTE]
.A JWT is a signed, readable claim â€” not an opaque secret
====
The tokens in this module are *JSON Web Tokens* (JWT, "jot"): three base64url segments â€” header, payload, signature â€” joined by dots.
The payload is *not encrypted*; anyone can decode it and read the claims (`iss`, `aud`, `exp`, `preferred_username`, `realm_access.roles`).
What they *cannot* do is forge one, because the signature is Keycloak's and the API checks it against Keycloak's public keys.
"Debug tokens like an adult" in the lab is exactly this: decode the payload, read the claims, and stop guessing.
====

* A *realm* is a self-contained world of identity: its own users, roles, clients, keys, and login page.
This workshop gives *every attendee their own realm* (`realm-<your-user>`) on one shared Keycloak, so your changes never touch anyone else's â€” the same isolation pattern a real platform team uses to separate business units or environments.
* A *client* is an application that talks to the realm.
Clients come in two flavors that matter: *public* clients (the browser app â€” cannot keep a secret, so it uses PKCE) and *confidential* clients (a backend â€” holds a client secret).
* A *role* is a named permission.
`claims-adjuster` is a *realm role*, and it rides inside the token under `realm_access.roles` â€” where the app reads it to decide.
* A *user* is a person who can log in.
Your realm is seeded with two: `adjuster`, who holds `claims-adjuster`, and `viewer`, who holds nothing â€” the two sides of every authorization check.

[NOTE]
.Why realm roles need one line of mapping
====
Keycloak puts *realm* roles under the `realm_access.roles` claim â€” not the top-level `roles` claim some identity providers use.
So a framework's role check has to be told where to look: in Quarkus that is `quarkus.oidc.roles.role-claim-path=realm_access/roles`.
The Parasol images already ship this setting, which is why `claims-adjuster` resolves straight from the token in the lab.
It is called out here because forgetting it is the classic trap: a user who genuinely *has* the role still gets *403*, and you burn an afternoon before checking where the role actually lives in the token.
====

Authentication is a *solved problem*, and getting it slightly wrong is catastrophic â€” a weak reset flow or a missing token check is the whole breach.
A dedicated identity provider gives you password policies, multi-factor authentication, account lockout, social and enterprise federation, token issuance, and an audit trail â€” built, tested, and patched by people who do only this.
Your job shrinks to *describing* who may do what and pointing your apps at it.

And here is the punchline the room's leadership should hear: *Red Hat build of Keycloak is included with your OpenShift subscription.*
The same product backs OpenShift's own console login.
SSO for your applications is a *platform capability you already own*, not a line item to purchase â€” and a team treating "add login" as a procurement project is usually re-buying what the subscription already includes.

== How this shows up in real orgs

* *"Trusted network" stops being your security model.*
The open claims API is the shape of most internal breaches: something got onto the network and everything trusted it.
A bearer-protected API trusts a *token it can validate*, not a network location â€” the foundation of zero-trust, and the difference between one compromised pod and a full data exfiltration.
* *One login, every app â€” and one place to cut it off.*
When identity lives in Keycloak, a new hire gets access to every Parasol app at once, and a departing employee loses it at once.
The alternative â€” a separate user table per app â€” is how orphaned accounts and "we forgot that system existed" breaches happen.
* *Machines are identities too.*
The batch job having its *own* credential, distinct from any human, means you can see what it did, rotate its secret, and revoke it without paging a person â€” and no engineer is tempted to run it under their own login.
* *Delegation without impersonation.*
Token exchange lets the claims service call the fraud service *as the user* while provably *narrowing* what that call can do.
That is how you build a chain of services that each see only what they need â€” and can *prove* an escalation is impossible, not just promise it.
