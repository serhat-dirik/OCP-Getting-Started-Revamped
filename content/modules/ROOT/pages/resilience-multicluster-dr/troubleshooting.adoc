Every entry below was either hit while building the module on the cluster or is a known failure mode to watch for.
Set the shortcuts first: `CLIENT={user}-client`, `SITEA={user}-site-a`, `SITEB={user}-site-b`.

== The client log shows `HTTP 404 served-by-site=none`

Symptom:: `oc logs deploy/claims-client -n $CLIENT` prints `HTTP 404 served-by-site=none` once a second.
Cause:: This is the *entry state*, not a bug. The ingress gateway (`claims-stable`) is up, but you haven't wired the failover routing yet — there's no `VirtualService` telling the gateway where to send traffic, so it returns `404` (no route).
Fix:: Apply the three failover resources from exercise 2 (the `ServiceEntry`, `DestinationRule`, and `VirtualService`). Within a few seconds the log flips to `HTTP 200 served-by-site=A`:
+
[source,sh,role=execute]
----
oc get serviceentry,destinationrule,virtualservice -n $CLIENT
----
+
If those are absent, you're still at entry — build them (exercise 2). If they exist but you still get `404`, check the `VirtualService` `gateways:` list names `claims-gateway` and its `hosts:` is `["*"]`.

== Traffic splits between A and B instead of preferring the primary

Symptom:: After wiring the routing, the client log alternates `served-by-site=A` and `served-by-site=B` roughly 50/50, instead of steadily serving the primary A.
Cause:: The ingress gateway lost its *locality pin*. The gateway pod carries `istio-locality: east.site-a`, and the `DestinationRule` `localityLbSetting.enabled: true` uses it to make site A priority 0 (primary) and site B priority 1 (failover). Without the locality, the two sites are *equal* and the mesh round-robins them (active-active).
Fix:: Confirm the gateway pod still has the locality label; if it was edited away, re-materialize the entry state:
+
[source,sh,role=execute]
----
oc get pod -n $CLIENT -l istio=claims-gateway -o jsonpath='{.items[0].metadata.labels.istio-locality}{"\n"}'
# expect: east.site-a ; if empty, re-materialize:  ws reset resilience-multicluster-dr --user {user}
----
+
The label must be present *at pod creation* (the injector reads it then); editing a running pod's label won't take — restart the gateway (or `ws reset`).

== The failover happens, but there's a brief `503` during a long outage

Symptom:: After scaling site A to 0, the client flips to B and serves `200`s, but *once* — around 15–20 s into the outage — a single `HTTP 503 served-by-site=none` appears, then it's back to B.
Cause:: `outlierDetection.baseEjectionTime` expired and the mesh *re-probed* the dead primary (site A is priority 0, so Envoy periodically re-checks it). That one probe hit the still-down site and returned `503` before being re-ejected.
Fix:: Lengthen `baseEjectionTime` so the failed primary stays ejected through a typical outage (the entry chart uses `30s`, which covers a ~25 s failover drill with no re-probe). It's a trade-off — a longer ejection keeps the service available during the outage but slower to fail *back* after recovery:
+
[source,sh,role=execute]
----
oc get destinationrule claims-global -n $CLIENT -o jsonpath='baseEjectionTime={.spec.trafficPolicy.outlierDetection.baseEjectionTime}{"\n"}'
----
+
A single transient `503` on a re-probe is cosmetic (the very next request is served by B); tune it to your outage profile.

== Site A scaled to 0 but the client keeps showing `A` for ~30 s

Symptom:: You scaled site A to 0, but the client log stays on `served-by-site=A` for around thirty seconds before flipping to B.
Cause:: The claims pods are *ignoring `SIGTERM`* and lingering through their termination grace period (Kubernetes default is 30 s), so they keep serving until they're force-killed — the site isn't actually "gone" yet.
Fix:: The workshop responder handles `SIGTERM` and uses a short `terminationGracePeriodSeconds` so `--replicas=0` removes it promptly. If you swapped the responder image, ensure it exits on `SIGTERM` (and set a short grace period):
+
[source,sh,role=execute]
----
oc get deploy parasol-claims -n $SITEA -o jsonpath='grace={.spec.template.spec.terminationGracePeriodSeconds}{"\n"}'
----
+
A quick sanity check: `oc get pods -n $SITEA` right after scaling to 0 should show the pods `Terminating` and *gone within a few seconds*, not lingering `Running`.

== The failover to site B never happens (client keeps erroring)

Symptom:: You scaled site A to 0, but the client log shows errors (`503`/`no healthy upstream`) and never flips to `served-by-site=B`.
Cause:: The mesh can't reach site B. Usually one of: (1) `{user}-site-b` isn't a *mesh tenant* (`istio-discovery=enabled` missing), so istiod doesn't build config for it; (2) the `ServiceEntry` endpoint for site B has the wrong FQDN; or (3) site B's `parasol-claims` isn't running.
Fix:: Check all three:
+
[source,sh,role=execute]
----
oc get ns $SITEB -o jsonpath='site-b discovery={.metadata.labels.istio-discovery}{"\n"}'   # want: enabled
oc get serviceentry claims-global -n $CLIENT -o jsonpath='{range .spec.endpoints[*]}{.address}{" "}{.locality}{"\n"}{end}'
oc get deploy parasol-claims -n $SITEB
----
+
Site B's namespace must be `istio-discovery=enabled` (the workshop layer sets it), the `ServiceEntry` endpoint address must be `claims.{user}-site-b.svc.cluster.local`, and site B's Deployment must have ready replicas.

== `[ADD-ON]` The Skupper `Site` router won't start — `exceeded quota`

Symptom:: You created a Skupper `Site` in a site namespace, but the `Site` stays `Pending`/`Not Running`, and the `skupper-router` Deployment shows `ReplicaFailure: FailedCreate ... pods "skupper-router-..." is forbidden: exceeded quota: workshop-quota`.
Cause:: A Skupper `Site` starts a router pod (~1 CPU) *on top of* the meshed claims pods — and each meshed pod's istio-proxy sidecar itself reserves CPU/memory. If the site namespace's `ResourceQuota` is sized only for the claims tier, there's no room for the router.
Fix:: The workshop layer sizes the site namespaces with headroom for both. If your namespace quota was reverted to a smaller value (or you tightened it), widen it, or scale the claims tier down briefly to free room:
+
[source,sh,role=execute]
----
oc get resourcequota workshop-quota -n $SITEB -o jsonpath='used-limits.cpu={.status.used.limits\.cpu} hard={.status.hard.limits\.cpu}{"\n"}'
oc describe rs -n $SITEB -l app.kubernetes.io/part-of=skupper | sed -n '/Events/,$p' | head
----
+
If `used` is at (or above) `hard`, the quota is the blocker — widen `limits.cpu`/`limits.memory` on `workshop-quota`.

== `[ADD-ON]` The RHSI `AccessToken` won't form a `Link`

Symptom:: You created the `AccessToken` in DC-B, but `oc get link -n $SITEB` never reaches `Ready` (stays blank or `False`).
Cause:: The token redemption is incomplete or wrong. The `AccessToken` needs the grant's *url*, *code*, and *ca* copied *exactly* — and the `ca` is a *multi-line* PEM certificate that's easily truncated or mis-indented when copied by hand.
Fix:: Confirm the grant is `Ready` and re-run the *scripted* copy (it re-indents the CA correctly), then wait on the link:
+
[source,sh,role=execute]
----
oc get accessgrant dc-a-grant -n $SITEA -o jsonpath='grant Ready={.status.conditions[?(@.type=="Ready")].status}{"\n"}'
URL=$(oc get accessgrant dc-a-grant -n $SITEA -o jsonpath='{.status.url}')
CODE=$(oc get accessgrant dc-a-grant -n $SITEA -o jsonpath='{.status.code}')
CA=$(oc get accessgrant dc-a-grant -n $SITEA -o jsonpath='{.status.ca}')
{ echo "apiVersion: skupper.io/v2alpha1"; echo "kind: AccessToken"; echo "metadata: {name: dc-a-token, namespace: $SITEB}";
  echo "spec:"; echo "  url: $URL"; echo "  code: $CODE"; echo "  ca: |"; echo "$CA" | sed 's/^/    /'; } | oc apply -f -
oc get link dc-a-token -n $SITEB
----
+
Never hand-type the CA. If the grant has *expired* (past its `expirationWindow`) or hit `redemptionsAllowed`, recreate the `AccessGrant` first.

== `[ADD-ON]` `curl` to `claims-remote` fails even though the sites are linked

Symptom:: The `Link` is `Ready`, but `curl http://claims-remote:8080/` from DC-A fails — either `could not resolve host` or `Connection refused`.
Cause:: Two possibilities. (1) The `Listener` isn't created yet, so its local `claims-remote` Service doesn't exist (the name doesn't resolve). (2) The `Connector` and `Listener` don't share the *same* routing key, so they never matched — the endpoint exists but routes nowhere.
Fix:: Confirm the Listener's Service exists and that Connector/Listener matched on the *same* routing key:
+
[source,sh,role=execute]
----
oc get svc claims-remote -n $SITEA                                                        # the Listener's local endpoint must exist
oc get listener claims-remote -n $SITEA -o jsonpath='matched={.status.hasMatchingConnector} key={.spec.routingKey}{"\n"}'
oc get connector claims -n $SITEB -o jsonpath='matched={.status.hasMatchingListener} key={.spec.routingKey}{"\n"}'
----
+
Both `matched` must be `true` and both `key` must be identical (`claims-cross`). A routing-key typo is the usual cause — the objects are `Ready` individually but never bind.

== `ws verify` fails on `istio-discovery=enabled` for a namespace

Symptom:: `ws verify resilience-multicluster-dr` reports `❌ {user}-site-b is istio-discovery=enabled (mesh tenant)` even though the entry state materialized.
Cause:: One of the three namespaces lost its `istio-discovery=enabled` label. That label is *workshop-layer-owned* (`per-user-resilience.yaml`) and is what makes the namespace a tenant of the shared istiod. If the workshop layer is stale (doesn't yet carry all three M21 namespaces), the label can be missing.
Fix:: Ensure the workshop layer labels *all three* namespaces, then re-sync it (instructor/admin):
+
[source,sh,role=execute]
----
for ns in $CLIENT $SITEA $SITEB; do disc=$(oc get ns $ns -o jsonpath='{.metadata.labels.istio-discovery}'); echo "$ns istio-discovery=$disc"; done
----
+
All three must read `enabled`. If one is missing, the `gitops/workshop-config` layer (per-user-resilience) needs to be current and synced — this is a platform-layer fix, not a per-user one.
