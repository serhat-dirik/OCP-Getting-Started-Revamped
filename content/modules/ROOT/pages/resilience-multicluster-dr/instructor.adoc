This module has one spine and it climbs: *prove in-site resiliency (kill a pod, the PDB holds) → wire cross-site failover on the mesh → fail an entire site and watch the client flip A→B with no dropped requests → recover → (add-on) link two sites over an RHSI VAN*.
The beat that lands hardest is the *failover* (exercise 2): the attendee scales the whole primary site to *zero* and the client's live log *keeps printing `200`* while the site marker flips to `B`. It makes "cross-site failover" concrete instead of a slide — hold on it, and keep the client log tailing on screen the whole time.
The reframe to state plainly: this module's *graded work is application-level failover* — designing a service to survive a whole-site loss. Platform backup/restore (OADP) and GitOps re-materialization are the *DR safety net* underneath, covered as a *concept* (an instructor mention is fine; there is no hands-on backup lab).

The single most useful thing to internalize before delivering: *the failover is driven by three mesh resources plus the gateway's locality.* The `ServiceEntry` declares both sites, the `DestinationRule` prefers the local (primary) site and ejects a failing one, the `VirtualService` retries so failover drops no requests — and the *ingress gateway is pinned to site A's locality*, which is what makes A the primary and B the failover. If traffic ever splits A/B instead of preferring A, the gateway lost its `istio-locality` label.

== Pre-flight checklist

Run these *before the session* (as cluster-admin, `KUBECONFIG` on the workshop cluster). The sample user below is `user1` (use any real attendee user).

. *A sample user's entry state materializes and verifies.*
+
[source,sh,role=execute]
----
ws start resilience-multicluster-dr --user user1
ws verify resilience-multicluster-dr --user user1 --entry-only
----
+
Expect all entry checks green: the three namespaces present *and* `istio-discovery=enabled`, the `ws-entry-resilience-multicluster-dr` marker, `parasol-claims` resilient in *both* sites (≥2 replicas + PDB + HPA) and ready, the `claims-client` and `claims-gateway` up, and the clean-slate assertion (no failover routing, no `ws-solve-resilience-multicluster-dr` marker yet).

. *The shared Service Mesh is healthy and all three namespaces are mesh tenants.*
+
[source,sh,role=execute]
----
oc get istio default -o jsonpath='control plane: Istio {.spec.version}  state={.status.state}{"\n"}'
for ns in client site-a site-b; do
  disc=$(oc get ns user1-$ns -o jsonpath='{.metadata.labels.istio-discovery}')
  echo "user1-$ns istio-discovery=$disc"
done
----
+
Expect `state=Healthy` and `istio-discovery=enabled` on *all three*. *This is the check that matters most* — if any of the three namespaces is not a mesh tenant, the failover misbehaves (the mesh can't route to a site it doesn't discover). All three carry the label from the workshop layer (`per-user-resilience.yaml`).

. *A full failover round-trips (not just "the operators are installed").*
+
[source,sh,role=execute]
----
ws solve resilience-multicluster-dr --user user1
ws verify resilience-multicluster-dr --user user1
----
+
`ws solve` renders the failover routing; `ws verify` (full mode) asserts it *and actively proves the failover* — it briefly scales site A to zero, confirms the client is served by site B, and restores site A. Expect all checks green, ending with `FAILOVER proven: site-a down -> the client is served by site-b`. *This is the check that matters* — a "Ready" mesh with a mis-wired `DestinationRule` passes every object check but fails the actual failover. When done, `ws reset resilience-multicluster-dr --user user1` returns the user to the entry state for the attendee to build the routing themselves.

. *Skupper (RHSI) is healthy — only if you're running the `[ADD-ON]`.*
+
[source,sh,role=execute]
----
oc get pods -n skupper -l app.kubernetes.io/name=skupper-controller
oc get crd sites.skupper.io -o jsonpath='RHSI CRDs present: {.spec.versions[0].name}{"\n"}'
----
+
Expect the `skupper-controller` *Running* and `v2alpha1`. If absent, the `service-interconnect` stack isn't synced — the RHSI section can't run (skip it; it's an `[ADD-ON]`). Also confirm the *site namespaces have quota headroom for the Skupper router* (~1 CPU on top of the meshed claims pods) — see Watchouts.

== Instructor demo — the 11-minute arc (demo flavor)

The demo flavor renders a three-beat Say/Show/Do arc: *show the resilient primary + the steady client log* (3 min), *fail the whole primary site — the client flips A→B with no drops* (5 min), *recover — traffic returns to A* (3 min).

* *Run it from `ws solve resilience-multicluster-dr --user user1`* so the sites, client, gateway, *and failover routing* are all up before you start.
* *Keep the client log tailing on screen* — `oc logs -f deploy/claims-client -n user1-client` — large font. That live `SITE=A`→`SITE=B` flip *is* the demo; don't cover it with other windows.
* *The money moment is beat 2:* `oc scale ... --replicas=0` on site A — narrate "the entire primary site is *gone*" — and let the audience watch the log keep printing `200` while the marker flips to `B`. Let the "no dropped requests" land.
* *Beat 3 (recovery) is quick but worth it* — failover *and* failback, both automatic, is the complete story.
* *Keep the RHSI `[ADD-ON]` for the workshop*, not the 11-minute demo — mention it as "how you'd connect two real data centers," and point to exercise 3.

== Demo-mode notes

* *Don't cover the client log.* The entire payoff is visual and lives in that one tailing terminal. Arrange windows so it's always visible.
* *If traffic splits A/B instead of preferring A,* the gateway lost its `istio-locality` label — re-materialize (`ws reset` then `ws solve`). Locality is what makes A the primary; without it the mesh round-robins both sites (active-active), and the "flip" isn't clean.
* *The brief gap at failover is normal, an error line is not.* Between the last `A` and the first `B` the once-a-second loop may skip a second (the failover request took a retry hop) — but you should see *no* `404`/`503`. If you see errors, the `VirtualService` retries or the `DestinationRule` outlier detection aren't right.
* *Failback takes ~30 s of `baseEjectionTime`, not instantly.* After `--replicas=3`, give it about half a minute; the ejection has to expire before the mesh trusts site A again. Narrate the wait rather than clicking around.

== Top questions attendees ask

*1. "The whole primary site went to zero — how did the client not drop a single request?"*::
Two mesh features cooperate. *Outlier detection* on the `DestinationRule` notices site A's requests failing and *ejects* it from the load-balancing pool, so new requests go to site B. And *retries* on the `VirtualService` catch the handful of in-flight requests that raced the ejection — a request that lands on the just-failed site is retried and lands on site B, returning `200`. The client sees a clean flip from A to B, not an outage. Without the retries you'd see a brief burst of errors at the moment of failure; with them, the failover drops no requests.

*2. "Why does traffic prefer site A instead of splitting 50/50 across A and B?"*::
Because the ingress gateway is *pinned to site A's locality* (an `istio-locality` label on the gateway pod), and the `DestinationRule` turns on `localityLbSetting`. Istio's locality-aware load balancing sends traffic to the *nearest* locality first — site A is priority 0, site B is the failover at priority 1 — so you get *primary/backup*, not active-active. Remove the locality pin and the same resources would round-robin both sites. This is the one deliberate "the gateway lives at the primary site" choice.

*3. "In the RHSI section, the site-b Skupper router won't schedule — why?"*::
Almost always *quota*. A Skupper `Site` starts a router pod (~1 CPU) in each site namespace, *on top of* the meshed claims pods (each of which carries an istio-proxy sidecar that itself reserves CPU/memory). If the site namespace's `ResourceQuota` is sized only for the claims tier, the router is refused with `exceeded quota`. The workshop layer sizes the site namespaces with headroom for both — if you see `FailedCreate ... exceeded quota` on `skupper-router`, the namespace quota is too tight (or was reverted to an older, smaller value). Widen `limits.cpu`/`limits.memory` on the site's `workshop-quota`.

*4. "Is this a real multi-cluster failover, or simulated?"*::
It's a *single-cluster simulation*: three namespaces stand in for the client and two sites, all in one mesh. What's *real* is the mechanism — the `ServiceEntry`/`DestinationRule`/`VirtualService` failover routing, the locality-aware load balancing, the outlier detection and retries, and (in the add-on) the Skupper VAN. The *identical* resources work across two real clusters in two regions: you'd point the ServiceEntry endpoints at the other cluster's services (reached over the RHSI VAN) and pin the gateway to the local region. The lab teaches the multi-site pattern; the topology is simulated, the plumbing is real.

*5. "Where's the OADP backup/restore that this module used to have?"*::
Reframed. The *graded lab* is now *application-level failover* — the part that's genuinely the app team's job. Platform *disaster recovery* — OADP backup/restore of data plus GitOps re-materialization of the platform — is covered as the *DR safety net concept* in the concept page (an instructor mention or a quick `Backup`/`Restore` show-and-tell is fine). The distinction to land: the platform *has your back* for DR; *your* job is to design the service to be resilient and to fail over — which is what the lab builds.

== Watchouts

* *[CRITICAL] All three namespaces must be mesh tenants (`istio-discovery=enabled`).*
The workshop layer (`per-user-resilience.yaml`) labels `{user}-client`, `{user}-site-a`, and `{user}-site-b` with `istio-discovery=enabled` so the *one shared istiod* builds config for all three. If a namespace loses that label (a stale workshop layer, a manual edit), the mesh can't route to that site and the failover misbehaves — the failover *to* site B needs site B discovered. Pre-flight check 2 asserts all three; fix the workshop layer if any is missing.
* *[BUILD SURPRISE] The gateway's `istio-locality` label is what makes site A the primary.*
The ingress gateway pod carries `istio-locality: east.site-a`. That is the *only* thing that makes the mesh prefer site A (priority 0) and treat site B as the failover. If it's absent, traffic round-robins A/B (active-active) and the "flip" isn't a clean primary→backup story. It's set by the entry chart; if you edit the gateway, preserve the label.
* *[BUILD SURPRISE] The site quota must leave room for the Skupper router (RHSI `[ADD-ON]`).*
Each Skupper `Site` adds a router pod (~1 CPU) to a site namespace that already runs the *meshed* claims tier (claims pods + istio-proxy sidecars are CPU-heavy). Size the site `workshop-quota` with headroom, or the router is refused with `exceeded quota` and the Link never forms. This is only a concern for the add-on.
* *The responder dies fast on "scale to 0" by design.*
The claims responder handles `SIGTERM` and its pod has a short `terminationGracePeriodSeconds`, so `--replicas=0` removes the site *promptly* and the failover is crisp. A workload that ignores `SIGTERM` would linger through its grace period and the failover would look delayed — worth knowing if you swap the responder image.
* *`baseEjectionTime` tuning is a trade-off.*
The `DestinationRule` uses `baseEjectionTime: 30s` — a middle ground. Shorter (e.g. `5s`) fails *back* to the primary faster but re-probes the dead site during a longer outage (a brief blip); longer (e.g. `120s`) keeps the service available during the outage but slow to fail back. The challenge section has attendees feel both.
* *`ws reset resilience-multicluster-dr` is slow — run it before the session.*
Reset re-materializes all three namespaces via Argo; the CLI waits for the sync. It always completes, but don't run it live mid-session — prep worlds ahead of time.
