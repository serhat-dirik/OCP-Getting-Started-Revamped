== What you built

You took a fluent-but-blind model and turned it into an agent Parasol's staff can *trust*: one that reads *live* claims, *grounds* its answers in the policy corpus with citations, admits when it doesn't know — and ships on OpenShift exactly like every other service.

* You *met* a running agent (`parasol-agent`) whose readiness probe pings both MCP tool servers, so a *Ready* agent had already proven its wiring — no token spent.
* You saw *grounding is engineered*: the same claim answered *ungrounded* (the model printing the tool call as text) or *grounded* (the tool actually running) depending only on *how you asked*.
* You watched *MCP tools are your APIs*: the agent called `get_claim`, `list_claims_by_status`, and `get_claim_history` on your claims service, and `search_policies` on your policy corpus — reporting exactly which tools it used.
* You *grounded with RAG and citations* (`POL-AUTO-01`, `POL-CLAIM-01`), *combined* a claim fact with a policy rule in one answer, and got an honest _"not found"_ for a claim that doesn't exist.
* You *observed* the agent — per-answer tokens and tool calls, Prometheus GenAI metrics — and confirmed it is a plain Deployment with a Route and probes that *self-heals* its tool wiring on restart.

== Map it to your org

Discuss with the person next to you:

. *Pick one system you own that a support person asks about all day.* What would its `get_claim` be — the one tool that returns a grounded fact? Wrapping it as an MCP tool is a smaller job than most teams fear; what is stopping you?
. *Where would an ungrounded answer hurt you most?* You saw the same question grounded or not depending on phrasing. In your world, what is the *review gate* — do you require a *citation* on every AI answer, and would you ship an agent that couldn't produce one?
. *Who owns the guardrail?* If an agent in your org touched a Social Security number or account data, is the protection a *platform layer* everyone inherits, or code each team re-writes (and sometimes forgets)? What would it take to make it the former?

== When NOT to use this

Honesty keeps this credible — an agent is not always the answer:

* *Don't use an agent when a plain API call will do.* If you *know* the user wants claim `CLM-1004`'s status, call `get_claim` directly — you don't need a model to decide that. Agents earn their cost when the *question is open-ended* and the right tool isn't known in advance. A model in the loop is latency, tokens, and nondeterminism you should add *deliberately*.
* *Don't RAG what fits in the prompt.* Retrieval is machinery. If the whole policy is three sentences, put it in the system prompt and skip the vector database. RAG is for corpora too big to prompt, not for looking clever.
* *Don't ship an ungrounded agent for anything that matters.* If it can't cite a source or call a tool, it is *guessing*. For a claim, a policy, a payout, or anything touching money or regulation, an ungrounded answer is a liability, not a feature.
* *Don't put an agent on regulated data without guardrails.* PII and safety filtering is not a "later" for an agent that reads real customer records. If the `[ADD-ON]` guardrail layer isn't there yet, keep the agent off regulated data until it is.
* *Don't demo at temperature 0.7 over live data.* Reproducibility is a choice: *temperature 0*, *seeded data*, *deterministic tools*. A "creative" agent over a production database is how you get surprised in front of a customer.

== The next platform layer `[ADD-ON]`

You built the *graded* spine — a Quarkus LangChain4j agent, MCP tools, deterministic RAG — deliberately the simplest thing that teaches the ideas. Three `[ADD-ON]` layers (a separate subscription, and in places a GPU) turn it into production, each slotting in *behind an interface the agent already uses*:

* *Guardrails* — the *TrustyAI FMS Guardrails Orchestrator* (generally available) runs a *regex-detector sidecar* that blocks or masks PII (Social Security numbers, credit cards, emails) and unsafe content as a *platform policy*, so no agent re-implements it. Adopt it the moment an agent touches regulated data.
* *A Llama Stack runtime* — Red Hat OpenShift AI ships *Llama Stack* (Technology Preview) as a shared agent runtime with the Responses API, MCP registries, and RAG providers. Adopt it when you want a *product surface* for agents rather than hand-wiring each service.
* *A vector database* — Milvus or PostgreSQL/pgvector holding document *embeddings* for semantic search at scale, behind the same `search_policies` contract. Adopt it when your corpus outgrows a handful of documents.

The engineering is knowing these exist and reaching for them *when the need is real* — not building all of it on day one.
