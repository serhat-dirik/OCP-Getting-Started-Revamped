In this lab you take the developer inner loop into the cluster.
You open Parasol's claims service in a browser IDE that starts in under two minutes, run it in Quarkus dev mode against the *real* in-cluster database, change a live endpoint and watch it hot-reload, extend the workspace definition with a Redis cache, push your change to Git, and step through a running request in a debugger.
Everything happens in your own projects — `{user}-devspaces` for the workspace, `{user}-dev` for the app and database — so nothing you do can affect another attendee.

[IMPORTANT]
.Two terminals — know which one you are in
====
This lab uses two different terminals:

* The *workspace terminal* — inside the browser IDE (Red Hat OpenShift Dev Spaces, "Dev Spaces" from here on).
*Almost every command runs here.*
These blocks are copy-enabled; paste them into the IDE terminal.
* The *Showroom terminal* — the terminal tab beside these instructions, logged in as `{user}`.
Only steps explicitly labelled *Showroom terminal* run here (they use `oc` to look at the cluster from the outside).
Those are click-to-run.
====

== 1. Open your workspace from your fork
⏱ *~12 min* · one click to a correct, running environment

The workshop setup has already forked the claims service into your own Git account, at `{user}/parasol-claims`, and its repository root contains a `devfile.yaml` — the workspace contract.
Opening a *factory URL* of the shape `{devspaces_url}/#<git-repo-url>` tells Dev Spaces to clone that repo and build the workspace its devfile describes.

Open your workspace:

. Open the {devspaces_url}[Dev Spaces dashboard^] and sign in as `{user}` (same credentials as the console).
The first sign-in provisions your workspace project, `{user}-devspaces`, automatically.
. Open your workspace with this factory URL (it clones *your* fork):
+
[.copy]
`{devspaces_url}/#{gitea_url}/{user}/parasol-claims`
+
. If Dev Spaces asks *"Do you trust the authors of this repository?"*, click *Continue* — it is your own fork.
. Wait for the workspace to start.
The *first* start pulls the universal developer image and can take around a minute and a half; later starts take about twenty seconds.
When it finishes, the che-code editor (VS Code in your browser) opens with the `parasol-claims` project loaded.

[NOTE]
.Two harmless prompts on the way in
====
If you have opened this workspace before, Dev Spaces does not overwrite it — it creates a *new* workspace with a random suffix (for example `parasol-claims-titq`).
That is fine; you are working in the newest one.
And the first time you open a terminal, che-code may ask you to *Trust Workspace & Continue* (or show a *Restricted Mode* banner) — trust it, so the terminal and language tools work.
====

.The che-code editor after the factory URL, with the parasol-claims project loaded
image::m03-devspaces-inner-loop/m03-devspaces-inner-loop-01-workspace-loaded.png[che-code editor (VS Code in the browser): the parasol-claims project in the Explorer with devfile.yaml, and an integrated terminal showing java -version reporting openjdk 21]

Take two minutes to get oriented, then confirm your toolchain.
Open a terminal in the IDE (menu *Terminal → New Terminal*) and check the Java version:

[source,sh]
----
java -version
----

[source,texinfo]
----
openjdk version "21.0.11" 2026-04-21 LTS
OpenJDK Runtime Environment (Red_Hat-21.0.11.0.10-1) (build 21.0.11+10-LTS)
----

The universal developer image ships several JDKs and *defaults to 17*, but this project targets *Java 21*.
The devfile pins JDK 21 (via `JAVA_HOME` and `PATH`) so your build and your editor both use it — a one-line change in Git that everyone on the team inherits, instead of a per-laptop chase.

== 2. Wire dev mode to the in-cluster database
⏱ *~15 min* · the deliberate break-and-fix — a dev instance needs a real database

Quarkus *dev mode* (`./mvnw quarkus:dev`) runs the app with live reload.
Start it the naive way first, in the *workspace terminal*, and watch it fail on purpose:

[source,sh]
----
cd /projects/parasol-claims
./mvnw quarkus:dev
----

After the dependencies download, it stops with an error:

.Expected output (the deliberate failure)
[%collapsible]
=====
[source,texinfo]
----
ERROR [...DockerClientProviderStrategy] Could not find a valid Docker environment. ...
WARN  [...DevServicesDatasourceProcessor] Please configure the datasource URL for
      default datasource or ensure the Docker daemon is up and running.
ERROR [io.quarkus.runtime.Application] Failed to start application:
  Datasource '<default>' was deactivated automatically because its URL is not set.
  To activate the datasource, set configuration property 'quarkus.datasource.jdbc.url'.
----
=====

Read what happened, because it is the whole lesson.
The claims app needs a database.
With no database configured, Quarkus tried to *start one for you* (a throwaway container, called a Dev Service) — but a workspace has no container runtime inside it, so that failed, and the app refused to start without a datasource URL.

You do not want a throwaway database anyway.
You want the *real* `claims-db` that is already running in your `{user}-dev` project — the same one the deployed claims app uses.
Your workspace can reach it by ordinary in-cluster DNS, across namespaces: `claims-db.{user}-dev.svc`.

Press *Ctrl+C* to stop the failed attempt, then wire the datasource with environment variables and start dev mode again:

[source,sh,subs="attributes"]
----
export QUARKUS_DATASOURCE_JDBC_URL=jdbc:postgresql://claims-db.{user}-dev.svc:5432/parasol
export QUARKUS_DATASOURCE_USERNAME=parasol
export QUARKUS_DATASOURCE_PASSWORD=parasol
export QUARKUS_HIBERNATE_ORM_SCHEMA_MANAGEMENT_STRATEGY=update
./mvnw quarkus:dev
----

This time it starts.
Watch for these lines:

.Expected output
[source,texinfo,subs="attributes"]
----
INFO  [io.quarkus] parasol-claims 1.0.0 on JVM (powered by Quarkus {quarkus_version}) started in 6.6s. Listening on: http://0.0.0.0:8080
INFO  [io.quarkus] Profile dev activated. Live Coding activated.
INFO  [io.quarkus] Installed features: [..., hibernate-orm, jdbc-postgresql, ...]
----

[NOTE]
.Why environment variables, not an edit to the code
====
The claims service reads its database URL, username, and password from environment variables, so the *same image* runs in dev, stage, and prod with only configuration changing.
You just used that: you pointed a dev instance at the cluster database without touching a line of code.
The schema strategy `update` tells your dev instance to *use* the existing tables, not drop and reseed them — so you never clobber the shared database.
Externalizing configuration like this is the subject of a later module (Config, Secrets & Multi-Environment — a separate module).
====

With dev mode running, confirm your workspace instance is talking to the in-cluster database.
In the workspace terminal, in a *second* IDE terminal tab (leave dev mode running in the first):

[source,sh]
----
curl -s http://localhost:8080/api/claims | head -c 200
----

You should see real claims — `CLM-1001`, `CLM-1002`, … — the 30 seeded rows served from `claims-db`.

[.checkpoint]
****
✔ *Verify:* Dev mode logs `Profile dev activated. Live Coding activated`, and `curl http://localhost:8080/api/claims` returns seeded claims.
If you get `Connection refused`, dev mode has not finished starting — wait for the `Listening on` line.
If the app fails with a datasource error, re-check the `export` lines (host `claims-db.{user}-dev.svc`).
****

== 3. Change a live endpoint and watch it hot-reload
⏱ *~12 min* · the inner loop at full speed

Now the payoff: change the running service without rebuilding or restarting it.
Leave dev mode running.

In the IDE Explorer, open `src/main/java/com/parasol/claims/ClaimResource.java`.
Add a small new endpoint just before the final closing brace of the class:

[source,java]
----
    @GET
    @Path("/ping")
    @Produces(MediaType.TEXT_PLAIN)
    public String ping() {
        return "parasol-claims dev mode: hot reload works";
    }
----

Save the file (*Ctrl+S*).
You did *not* restart anything.
Now hit the new endpoint from your second terminal tab:

[source,sh]
----
curl http://localhost:8080/api/claims/ping
----

The first request triggers Quarkus to recompile just what changed and reload it in place — you will see it in the dev-mode terminal:

.Expected output (in the dev-mode terminal)
[source,texinfo]
----
INFO  [io.quarkus.deployment.dev.RuntimeUpdatesProcessor] Restarting quarkus due to changes in ClaimResource$NewClaim.class, ClaimResource.class, ClaimResource$StatusUpdate.class.
INFO  [io.quarkus.deployment.dev.RuntimeUpdatesProcessor] Live reload total time: 2.2s
----

— and `curl` prints `parasol-claims dev mode: hot reload works`.
That endpoint did not exist a moment ago.
No image was built, no Pod was replaced; the loop was *seconds*.

Prove it is the *same shared database* as the deployed app.
Create a claim through your dev instance, then read it back through the *deployed* app's public Route — from the *Showroom terminal* this time:

[source,sh]
----
# workspace terminal: create a claim via your dev instance
curl -s -X POST http://localhost:8080/api/claims \
  -H 'Content-Type: application/json' \
  -d '{"claimant":"Dev Mode Demo","type":"auto","amount":500.00}'
----

[source,sh,role=execute,subs="attributes"]
----
# Showroom terminal: read it back through the DEPLOYED app's Route
CLAIMS_HOST=$(oc -n {user}-dev get route parasol-claims -o jsonpath='{.spec.host}')
curl -s "http://$CLAIMS_HOST/api/claims" | tr ',' '\n' | grep -A1 "Dev Mode Demo"
----

Your new claim, written by the workspace dev instance, is visible through the separately-deployed app — because both point at the one `claims-db`.

[.checkpoint]
****
✔ *Verify:* `curl .../api/claims/ping` returns your string, the dev-mode terminal shows a `Live reload` line, and the claim you POSTed appears through the deployed Route.
****

== 4. Extend the workspace: add a Redis cache to the devfile
⏱ *~15 min* · the devfile is yours to change — restart the workspace from it

A workspace is not fixed — it is whatever its devfile says.
Add a second container (a Redis cache the app could use) and a command to talk to it, then restart the workspace *from the devfile* so the new container appears.

Stop dev mode (*Ctrl+C*).
In the IDE Explorer, open `devfile.yaml` at the repository root.
Under `components:`, add a `cache` component, and under `commands:`, add a `cache-ping` command:

[source,yaml]
----
  # add under components:
  - name: cache
    container:
      # pinned by digest: tags move, digests don't — a devfile exists to be reproducible
      image: registry.redhat.io/rhel9/redis-7@sha256:4e9f428f53b7ee73796aa398f3c9fb340a43453685b2042baba6f9b782fc1467
      memoryLimit: 256Mi
      endpoints:
        - name: redis
          targetPort: 6379
          exposure: internal
----

[source,yaml]
----
  # add under commands:
  - id: cache-ping
    exec:
      label: Ping the Redis cache
      component: cache
      commandLine: redis-cli ping
----

Save the file.
Now restart the workspace so Dev Spaces rebuilds it from the updated devfile:

. Open the command palette (*F1* or *Ctrl+Shift+P*).
. Run *Dev Spaces: Restart Workspace from Local Devfile*.

When the workspace comes back, the `cache` container is part of it.
Confirm it by running the devfile command you just declared: open the command palette (*F1*) → *Tasks: Run Task* → *devfile: Ping the Redis cache*.
It executes `redis-cli ping` *inside the `cache` container* (the `redis-cli` binary lives there, not in your `tools` terminal).

.Expected output
[source,texinfo]
----
PONG
----

The two containers share the workspace Pod's network, so your app — and your terminal — can reach Redis at `localhost:6379`.
Prove it from the *workspace terminal* without any Redis tooling, using bash's built-in TCP support:

[source,sh]
----
exec 3<>/dev/tcp/localhost/6379 && printf 'PING\r\n' >&3 && head -c 7 <&3
----

.Expected output
[source,texinfo]
----
+PONG
----

No service, no route — just a sidecar you declared in Git.

[.checkpoint]
****
✔ *Verify:* After the restart, the *devfile: Ping the Redis cache* task returns `PONG`, and the workspace now runs a `cache` container (the terminal TCP probe answers `+PONG` too).
If the restart fails, your devfile has a YAML error — check the indentation of the two blocks you added (see xref:m03-devspaces-inner-loop/troubleshooting.adoc[troubleshooting]).
****

== 5. Commit and push your change
⏱ *~10 min* · the hand-off from inner loop to outer loop

Your `/ping` change lives only in the workspace.
Commit it and push it to your fork — this is the moment the inner loop hands off to the outer loop that the rest of the workshop builds (pipelines, GitOps).

In the workspace terminal, first tell git who you are — a fresh workspace container has no git identity yet (set it once per workspace; the values only label your commits):

[source,sh,subs="attributes"]
----
git config --global user.email "{user}@parasol.example.com"
git config --global user.name "{user}"
----

Then commit and push:

[source,sh]
----
cd /projects/parasol-claims
git add -A
git commit -m "Add /api/claims/ping endpoint"
git push origin main
----

The *first* push will likely fail asking for credentials:

.Expected output (the auth wall)
[source,texinfo]
----
fatal: could not read Username for 'https://...': No such device or address
----

That is expected: your workspace can *read* the public repo but has no credentials to *write* to it yet.
Give it your workshop credentials once, then push again:

[source,sh,subs="attributes"]
----
git config --global credential.helper store
printf 'https://{user}:YOUR_PASSWORD@%s\n' "$(echo {gitea_url} | sed 's#https://##')" > ~/.git-credentials
chmod 600 ~/.git-credentials
git push origin main
----

Replace `YOUR_PASSWORD` with your workshop password.
The push succeeds:

.Expected output
[source,texinfo,subs="attributes"]
----
To https://.../{user}/parasol-claims.git
   3bf5a39..c270dc8  main -> main
----

[TIP]
.The IDE has a Git UI too
====
You do not have to use the terminal.
The che-code *Source Control* panel (the branch icon in the left rail) stages, commits, and pushes with buttons, and it prompts for your username and password in a dialog the first time you push — no credential file needed.
Use whichever you prefer; the result is the same commit on your fork.
====

[.checkpoint]
****
✔ *Verify:* `git push` reports your commit was written to `main`.
Open your fork in the {gitea_url}[Git server^] (browse to `{user}/parasol-claims`) and confirm the latest commit is your `/ping` change.
****

== 6. Debug a live request
⏱ *~12 min* · attach a debugger to the running service

Quarkus dev mode opens a Java debug port (5005) automatically, so you can attach a debugger to the *running* dev instance and step through a real request — the JVM is genuinely listening for one.

[IMPORTANT]
.The graphical debugger needs the Java extension; a terminal fallback always works
====
The breakpoint-and-step (GUI) half of this exercise depends on che-code having the *Language Support for Java* + *Debugger for Java* extensions.
Your fork ships a `.vscode/extensions.json` that recommends them, so che-code prompts *"Do you want to install the recommended extensions?"* the first time you open the workspace — click *Install*.
If this cluster's extension registry is offline and that install is unavailable, the GUI flow is instructor-demo / optional — use the guaranteed `jdb` terminal path below instead; it needs no extension and proves the same thing.
====

Start dev mode again (wired to the database, as in exercise 2 — the environment variables are gone after a restart):

[source,sh,subs="attributes"]
----
cd /projects/parasol-claims
export QUARKUS_DATASOURCE_JDBC_URL=jdbc:postgresql://claims-db.{user}-dev.svc:5432/parasol
export QUARKUS_DATASOURCE_USERNAME=parasol
export QUARKUS_DATASOURCE_PASSWORD=parasol
export QUARKUS_HIBERNATE_ORM_SCHEMA_MANAGEMENT_STRATEGY=update
./mvnw quarkus:dev
----

Confirm the debug port is live — you will see this line in the dev-mode output:

[source,texinfo]
----
Listening for transport dt_socket at address: 5005
----

=== Attach the IDE debugger (if the Java extension is installed)

. In the IDE Explorer, open `ClaimResource.java` and click in the left margin next to the first line of the `list(...)` method (`PanacheQuery<Claim> query = Claim.findAll(...)`) to set a *breakpoint* (a red dot appears).
. Open the *Run and Debug* view (the play-with-bug icon in the left rail).
. Choose the pre-supplied *Attach to Quarkus (5005)* configuration — it ships in your fork's `.vscode/launch.json`, so you do not have to build one — and start it.
. From your second terminal tab, hit the endpoint to trigger the breakpoint:
+
[source,sh]
----
curl http://localhost:8080/api/claims
----
. Execution pauses on your breakpoint.
Inspect variables, step over a few lines, then resume.

// (screenshot uncaptured/blocked: the GUI Java debugger is not installable out of the box on this cluster — see media-manifest.md)

=== Guaranteed fallback: attach `jdb` from the terminal

This path needs no IDE extension — it uses the JDK's own command-line debugger (`jdb`, shipped with the JDK) to prove 5005 is a real, attachable debug port.
In a *second* workspace terminal tab (leave dev mode running in the first):

[source,sh]
----
jdb -attach localhost:5005
----

It connects to the running JVM.
At the `>` prompt, list the live threads, set a breakpoint on the list method, and continue:

[source,text]
----
> threads
> stop in com.parasol.claims.ClaimResource.list
> cont
----

The thread list shows the Quarkus main thread and the vert.x event-loop threads — you are attached to the real, running service, not a copy.
Now trigger the endpoint from a *third* terminal tab:

[source,sh]
----
curl http://localhost:8080/api/claims
----

`jdb` halts at the method.
`locals` prints the method's variables and `where` the call stack; `cont` resumes and `exit` detaches.
Same idea as the IDE debugger — a debugger attached to a live in-cluster request — with nothing to install.

[.checkpoint]
****
✔ *Verify:* Either a `curl` to `/api/claims` pauses on your breakpoint in the IDE with the method's variables visible, *or* `jdb -attach localhost:5005` connects and `threads` lists the running Quarkus and vert.x threads (and stops at `ClaimResource.list` on the next request once you set the breakpoint).
If neither attaches, confirm dev mode is running and printed the `Listening for transport dt_socket at address: 5005` line.
****

What to notice, and why it matters for how you think about the platform:

* *One devfile, more than one target.*
The showcase workspace uses a devfile with multiple build targets (a `devfile.yaml` and a registry-specific variant) — the same contract idea you used, scaled to a bigger toolchain.
* *Declarative tasks.*
Build, run, and test are devfile *commands* with task groups — there is no magic "Run" button, just named tasks anyone can discover, exactly like your `cache-ping`.
* *The device is a governed workload.*
The Android device runs as a virtual machine on the cluster; the workspace's service account has *no* rights to create VMs, so the device lifecycle is a governed, separate concern — a preview of how the platform keeps blast radius small (covered in the Virtualization and Multi-Tenancy modules — separate modules).

[NOTE]
.Why this is a recorded demo, not hands-on here
====
The Android device streams from a virtual machine, which needs OpenShift Virtualization (bare-metal nodes).
This workshop's cluster does not run it, so the showcase is a video by default; where a `virt`-capable cluster exists, it can be run live.
The take-home below gives you a purely local way to explore the same inner-loop idea.

== 8. Take-home: the same loop, locally
⏱ *~3 min* · read

In-cluster workspaces are one end of a spectrum.
When you want the inner loop entirely on your own machine — on a plane, air-gapped, or just offline — two Red Hat tools give you the same shape:

* *Podman Desktop* — build and run containers (and small Kubernetes/Pod setups) locally; the natural home for the container half of the inner loop without a cluster.
* *Red Hat OpenShift Local* — a single-node OpenShift on your laptop, for when you need the *platform* (Routes, operators, the console) and not just containers.

Both run the same images and, in OpenShift Local's case, the same objects you have been using all along — the skills transfer directly.
When you are done for the day, your Dev Spaces workspace idles automatically after 30 minutes of inactivity to free cluster resources, and resumes where you left off.
