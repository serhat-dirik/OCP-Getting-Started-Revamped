The one idea under all of it: the inner loop moved into the cluster without getting slower.

== Map to your org

Take these back to how your teams work today:

. *How long until a new hire ships their first change?*
You went from a link to a running, editable, database-connected service in a couple of minutes.
For a real team of yours: measure the actual elapsed time from "laptop handed over" to "first pull request merged."
How much of that is environment setup — and what would it be worth to make it a link?
. *What does your laptop fleet cost — in money and in drift?*
Every developer laptop is a licensed, secured, backed-up, drifting machine.
Count them; then ask how many of those toolchains genuinely need to be local versus could be a workspace the platform team maintains once, in Git.
. *Where does "works on my machine" still cost you?*
Think of the last bug that turned out to be a version or config difference between a developer's machine and production.
If the toolchain were a reviewed devfile that the pipeline used too, would that class of bug still exist?

== When *not* to do it this way

In-cluster workspaces are a strong default for server-side development, but they are not universal.
Be honest about the edges:

* *Air-gapped or offline development.*
If developers routinely work without cluster connectivity — on aircraft, in secure facilities, at the edge — a browser IDE that lives in the cluster is exactly the wrong tool.
The take-home local options (Podman Desktop, OpenShift Local) fit those cases.
* *Plugins and tooling that need local hardware.*
A workflow that depends on a local GPU, a USB device, a hardware security key reader, or a desktop-only IDE plugin will not fit a browser workspace.
Mobile/desktop app development that needs a local simulator is the classic example — the Android showcase runs the device *in the cluster* precisely because it cannot be local.
* *Very heavy native toolchains.*
Some builds (large C++/Rust monorepos, giant asset pipelines) want more local disk and CPU than a shared per-user workspace should hold.
Size the workspace honestly, or keep those builds on dedicated machines.
* *A workspace is not a server.*
Dev mode is for iterating, not for hosting.
Your workspace idles after 30 minutes and is capped at two running at once per user — deliberately, because it is a *development* environment, not a place to run production.

.Clean up
====
Your workspace idles automatically after 30 minutes of inactivity and resumes where you left off — you do not need to stop it.
To return your world to a fresh claims app + database (and remove your workspace), ask your instructor for a reset (`ws reset {module-id} --user {user}`).
Your Gitea fork keeps the commits you pushed — that is your work, not the module's; delete the fork in Gitea first if you want to redo the lab's code changes from scratch.
Leave everything running if you are continuing to another module — each module materializes its own world.
