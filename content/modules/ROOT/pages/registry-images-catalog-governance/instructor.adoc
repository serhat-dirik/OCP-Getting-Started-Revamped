This module has two beats that land, and they bracket the hands-on.
The *promotion by digest* is exercise 2 — `prod` and `1.0` resolving to the *identical* `sha256` — which makes "an ImageStream pins a tag to immutable content" concrete and provable, not abstract.
The *governance read* is exercise 6 — the whole cluster-wide surface (where images may come from, what's pruned, what the catalog offers) read live through `platform-observer`, then demonstrated by you.
Everything between is the image lifecycle — scheduled imports, pull secrets, a namespaced Template — that makes those two beats legible.

The single most common confusion is *"my scheduled-import stream isn't in my project."*
It's the exercise-3 gotcha: `oc import-image {user}-dev/ext-ubi` doesn't parse the `namespace/` prefix (unlike `oc tag`) — it creates the stream in the user's *current* project. The lab flags this in an IMPORTANT block and uses `-n $NS`; pre-empt it only if someone strays from the written command.

== Instructor demo — govern the cluster-wide supply (exercise 6)

All four operations are *cluster singletons* — they affect *every* tenant on a shared cluster.
*Only mutate a disposable or clearly-safe cluster*; on a shared workshop cluster, announce a tight window and revert immediately, or use a pre-recorded capture.

=== Block a registry for import (instant, live-safe with care)

`allowedRegistriesForImport` is admission-level — it limits which registries a *normal user* may import from, with *no* node reboot. It's the one cluster-wide control safe to toggle live in a tight window (it blocks imports for *all* concurrent users, so announce it).

[source,sh,role=execute]
----
oc patch image.config.openshift.io/cluster --type=merge \
  -p '{"spec":{"allowedRegistriesForImport":[{"domainName":"registry.redhat.io"},{"domainName":"registry.access.redhat.com"}]}}'
# now attempt an unapproved import as a normal-user action:
oc import-image demo-blocked -n user3-dev --from=docker.io/library/nginx:latest --confirm
----

// [CAPTURE-VERIFY] exact apiserver denial on OCP 4.21 — a Forbidden admission error stating the docker.io registry is not permitted by the import policy. Capture the precise string on your demo cluster; it was NOT applied on the shared build cluster (singleton).
Talk track: the import is *refused* by admission with a `Forbidden` error naming `docker.io` as a registry the import policy doesn't allow — *instantly*, with no node change. Cluster-admins are exempt (the policy limits normal users), so demonstrate the denial from a user context or note the exemption. Then revert:

[source,sh,role=execute]
----
oc patch image.config.openshift.io/cluster --type=json -p '[{"op":"remove","path":"/spec/allowedRegistriesForImport"}]'
oc delete is demo-blocked -n user3-dev 2>/dev/null
----

=== Mirror for disconnected — talk-through / pre-recorded

`registrySources.allowedRegistries` (pull/push enforcement on every node) and an *ImageDigestMirrorSet (IDMS)* / *ImageTagMirrorSet (ITMS)* are how an air-gapped cluster runs — every image reference is mirror-served.

[source,sh,role=execute]
----
# DO NOT run live on a shared cluster — this re-renders /etc/containers/registries.conf on EVERY
# node via the Machine Config Operator and ROLLS THE POOL (drain/reboot). Show the manifest, explain it.
cat <<'EOF'
apiVersion: config.openshift.io/v1
kind: ImageDigestMirrorSet
metadata:
  name: parasol-mirror
spec:
  imageDigestMirrors:
    - source: registry.redhat.io/ubi9
      mirrors:
        - mirror.internal.parasol.example.com/ubi9
EOF
----

Talk track: applying `registrySources` or an IDMS/ITMS triggers the Machine Config Operator to re-render every node and roll the pool — minutes of disruption that reschedules every tenant's pods, so it's *never* run mid-event. IDMS mirrors *by digest*, ITMS *by tag*. The older `ImageContentSourcePolicy` did this before OpenShift 4.13 and is *deprecated* — convert it with `oc adm migrate icsp`. Frame this as the disconnected-install mechanism; pre-record it or show it on a scratch cluster.

  abstract: The blessed starting point for a Parasol claims service.

Talk track: `skippedImagestreams`/`skippedTemplates` remove *one* sample from the catalog; `managementState: Removed` would delete *all* of them and break every module that uses the catalog (never do it on a shared cluster). == Demo-mode notes

Beyond the Say/Show/Do arc in the demo flavor:

* *Run the demo from the entry state, not `ws solve`.*
The story is *building* the governed supply chain from the seeded stream. `ws solve registry-images-catalog-governance` renders the finished end state (promoted tag, scheduled import, Template, pull-secret Deployment) — use it to *show the destination* or pre-stage, never as the demo's start.
* *Land the digest-equality line (beat 1).*
"Same `sha256` on both — production runs the image we tested, and I can prove it." Say it while the two identical digest lines are on screen. That equality *is* the point of an ImageStream promotion.
* *Announce and revert the allowed-registries beat (beat 2).*
It blocks imports for *every* concurrent user for as long as it's set. Announce a tight window, apply, show the denial, revert *immediately*, and confirm `allowedRegistriesForImport` is empty again. On any shared cluster, prefer a pre-recorded capture.
* *Keep the Template beat namespaced.*
"One Template, one namespace" is the contrast with the cluster-wide samples operator. If asked "could I do this for everyone?" — yes, that's a `ConsoleSample` (cluster-scoped), which is the instructor segment, not a namespaced Template.

== Top questions attendees ask

*1. "Why use an ImageStream at all — why not just reference the image directly?"*::
Three reasons you can point at from the lab: it *pins a moving tag to an immutable digest* (so `prod` is provably the bits you tested — you saw `prod` and `1.0` share a `sha256`), it can *re-import on a schedule* (the `ext-ubi` stream stays current on its own), and `referencePolicy: Local` makes it a *pull-through cache* (workloads pull from the internal registry, not the source). A bare image reference gives you none of those — the tag can silently move, nothing re-imports, and every pod pulls from the source.

*2. "My scheduled-import stream isn't in my project — where did it go?"*::
`oc import-image` does *not* read a `namespace/name` positional prefix the way `oc tag` does. `oc import-image {user}-dev/ext-ubi …` creates the stream in your *current* project (whatever `oc project` shows), not `{user}-dev`. Always pass `-n $NS` with a *bare* stream name. On the build cluster, running it from the `default` project created `ext-ubi` in `default`; with `-n user3-dev` it landed correctly. This is exercise 3's IMPORTANT block.

*3. "Can I push to the internal registry from my laptop with `podman push`?"*::
Not on this cluster — the registry has *no external route* (`defaultRoute` is off), so it's reachable only in-cluster. Everything runs through the API: `oc tag`, `oc import-image`, in-cluster builds. Enabling `defaultRoute` is possible but adds an internet-facing surface and TLS/pull-secret handling — a deliberate platform decision, not a default. The route-free posture is the common one for shared and locked-down clusters.

*4. "The pull-secret step didn't actually pull from the private registry — is it broken?"*::
No — the sample host (`registry.parasol.example.com`) is an *inert placeholder* that doesn't resolve, so a real external pull can't happen. The lab teaches the *mechanics* — linking a secret to a ServiceAccount `--for=pull` and naming it in a pod's `imagePullSecrets` — which are identical for a *real* private registry (or Quay). The demo Deployment runs the *internal* claims image so the pod is genuinely `Ready`; the pull-secret reference is the artifact being wired, not a live external pull. That honesty is deliberate — the transferable skill is the wiring.

*5. "The allowed-registries policy is cluster-wide — how do I demo it without breaking everyone?"*::
`allowedRegistriesForImport` is admission-level and *instant* (no node reboot), but it *does* apply to every concurrent user's imports while it's set. Announce a tight window, apply, show one denied `oc import-image`, and revert immediately — or use a pre-recorded capture. *Never* demo `registrySources` or IDMS/ITMS live: those re-render every node and roll the pool, disrupting every tenant. Those are talk-through / pre-recorded only.

== Watchouts

* *`oc import-image` uses the current project, not a `namespace/` prefix.*
The single most common stumble (question 2). The lab uses `-n $NS`; if a user's stream vanishes, check `oc get is ext-ubi -n default` (or wherever their `oc project` points). Harmless, but confusing — pre-empt it if you see it.
* *Never mutate a cluster singleton on a shared cluster without a plan.*
`allowedRegistriesForImport`, `registrySources`, IDMS/ITMS, the samples `Config`, `ConsoleSample` — all singletons. The build harness blocks these writes for good reason. On a shared workshop cluster, keep exercise 6 *read-only* for attendees and revert any instructor change immediately.
* *`managementState: Removed` on the samples operator is a foot-gun.*
It deletes *all* 60 image streams and 51 templates — the catalog the Build & Deliver and other modules depend on. The disable-a-sample demo MUST use `skippedImagestreams`/`skippedTemplates`. Never `Removed` on a shared cluster.
* *The nightly pruner can delete unreferenced images mid-event.*
`ImagePruner/cluster` runs daily at midnight, keeping 3 tag revisions and sparing images younger than an hour. On a multi-day event, entry-state images that aren't referenced by a running workload can vanish overnight. Mitigation is an ops decision: keep them referenced, raise `keepTagRevisions`, or suspend the pruner for the event.
* *Registry storage is one RWO PVC.*
Thirty users' multi-day image churn lands on a single volume with a `Recreate` (one-pod) registry. A full registry blocks *every* push and build cluster-wide. Monitor fill on long events; size the PVC accordingly at bootstrap.
