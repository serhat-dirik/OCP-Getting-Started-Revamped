== 1. Tour the internal registry and your ImageStream
⏱ *~10 min* · read the registry topology, the seeded stream, digests, and pull-through

Start by seeing *where* your images live.
The cluster's internal registry is an in-cluster Service — read its address and note it has *no external route*:

[source,sh,role=execute]
----
oc get image.config.openshift.io/cluster -o jsonpath='internal registry: {.status.internalRegistryHostname}{"\n"}'
oc get route -n openshift-image-registry 2>&1 | tail -1
----

.Expected output (an in-cluster Service on :5000; no route means route-free)
[source,texinfo]
----
internal registry: image-registry.openshift-image-registry.svc:5000
No resources found in openshift-image-registry namespace.
----

`image-registry.openshift-image-registry.svc:5000` is reachable only from *inside* the cluster — there's no route, so everything you do with images runs through the API (`oc tag`, `oc import-image`), never a `docker push` from your laptop.

Now look at *your* image — the `parasol-claims` ImageStream, seeded with one tag:

// [CAPTURE-VERIFY] Builds → ImageStreams → parasol-claims (project {user}-dev) shows the tag + the sha256 (OCP 4.21) — CLI authoritative.
[tabs]
====
Console::
+
--
In the {ocp_console_url}[web console^] (project `{user}-dev`), open *Builds* → *ImageStreams* → `parasol-claims`.
The detail page lists the *tag* (`1.0`), the *image repository* inside the cluster, and the resolved *`sha256`* digest.
--
CLI::
+
--
[source,sh,role=execute]
----
oc get is parasol-claims -n $NS
oc describe is parasol-claims -n $NS | head -20
----
--
====

.Expected output (one tag, resolved to an immutable digest)
[source,texinfo]
----
NAME             IMAGE REPOSITORY                                                            TAGS   UPDATED
parasol-claims   image-registry.openshift-image-registry.svc:5000/user3-dev/parasol-claims   1.0    6 minutes ago
...
1.0
  tagged from image-registry.openshift-image-registry.svc:5000/parasol-images/parasol-claims:1.0
    prefer registry pullthrough when referencing this tag

  * image-registry.openshift-image-registry.svc:5000/parasol-images/parasol-claims@sha256:0995ab385748d9fdcc999e394e6cc072b76cbef1136fd705864741983f7614f7

Two lines matter.
*"tagged from …parasol-claims:1.0"* is the *source* — the shared build this stream tracks.
*"prefer registry pullthrough when referencing this tag"* is `referencePolicy: Local` at work: workloads pull the bits *from the internal registry*, which caches them, instead of reaching the source every time.
And the `* …@sha256:…` line is the *immutable digest* the tag currently resolves to.

Confirm the two properties directly — the source reference and the pull-through policy:

[source,sh,role=execute]
----
oc get is parasol-claims -n $NS -o jsonpath='{range .spec.tags[*]}{.name}{" -> "}{.from.name}{"  referencePolicy="}{.referencePolicy.type}{"\n"}{end}'
----

.Expected output
[source,texinfo]
----
1.0 -> image-registry.openshift-image-registry.svc:5000/parasol-images/parasol-claims:1.0  referencePolicy=Local
----

Finally, peek at a *curated* stream the platform team shipped for everyone — the shared `java-21` builder image in the `openshift` namespace.
It's the same mechanics you just read, applied as *governance* (you'll return to this in exercise 6):

[source,sh,role=execute]
----
oc get is java-21 -n openshift -o jsonpath='{range .spec.tags[*]}{.name}{" scheduled="}{.importPolicy.scheduled}{" referencePolicy="}{.referencePolicy.type}{"\n"}{end}'
----

.Expected output (a scheduled, locally-cached builder — the platform curating your catalog)
[source,texinfo]
----
latest scheduled=true referencePolicy=Local
----

== 2. Tag and promote an image
⏱ *~10 min* · a promotion is a pointer to immutable content, not a rebuild

Parasol tested `1.0` and now wants it in production.
You *won't* rebuild and you *won't* re-push a tag — you'll *promote* by adding a `prod` tag that points at the *exact digest* `1.0` resolves to.
That's the whole value of an ImageStream: `prod` becomes a provable pointer to the image you tested.

// [CAPTURE-VERIFY] Builds → ImageStreams → parasol-claims → Actions → Edit ImageStream (YAML) can add a spec.tags entry (OCP 4.21) — CLI authoritative and far simpler here.
[tabs]
====
Console::
+
--
Open *Builds* → *ImageStreams* → `parasol-claims` → *Actions* → *Edit ImageStream* (YAML).
Add a second entry under `spec.tags` named `prod` whose `from.kind` is `ImageStreamImage` and `from.name` is `parasol-claims@<the sha256 you saw in exercise 1>`, then *Save*.
The CLI tab does exactly this in one command — reach for it here.
--
CLI::
+
--
[source,sh,role=execute]
----
oc tag $NS/parasol-claims:1.0 $NS/parasol-claims:prod
----
--
====

.Expected output (the promote resolves to a digest, not a tag)
[source,texinfo]
----
Tag user3-dev/parasol-claims:prod set to user3-dev/parasol-claims@sha256:0995ab385748d9fdcc999e394e6cc072b76cbef1136fd705864741983f7614f7.
----

Now *prove* the promotion is honest — `prod` and `1.0` must resolve to the *same* digest:

[source,sh,role=execute]
----
oc get is parasol-claims -n $NS -o jsonpath='{range .status.tags[*]}{.tag}{" -> "}{.items[0].dockerImageReference}{"\n"}{end}'
----

.Expected output (identical `sha256` on both — same bits, two names)
[source,texinfo]
----
1.0: image-registry.openshift-image-registry.svc:5000/parasol-images/parasol-claims@sha256:0995ab385748d9fdcc999e394e6cc072b76cbef1136fd705864741983f7614f7
prod: image-registry.openshift-image-registry.svc:5000/parasol-images/parasol-claims@sha256:0995ab385748d9fdcc999e394e6cc072b76cbef1136fd705864741983f7614f7
----

Same `sha256` on both lines.
`prod` is now a name for the *exact image* you tested as `1.0` — if someone re-pushes `1.0` upstream later, your `prod` still points at the digest you promoted, because a digest is content-addressed and can never move.
That is a promotion you can put in an audit.

[NOTE]
.Why not just re-tag at the registry?
====
Re-pushing a `prod` tag at a plain registry means "whatever `prod` points to *now*" — it can silently change.
Promoting through an ImageStream records the *digest*, so `prod` is pinned to immutable content.
The tag is a convenient *name*; the digest is the *identity*.
====

[.checkpoint]
****
✔ *Verify:* `oc tag` added a `prod` tag to `parasol-claims`, and `prod` resolves to the *same `sha256`* as `1.0` — a promotion by immutable digest, not a rebuild or a mutable re-push.
****

== 3. Re-import an external base image on a schedule

Create one that tracks Red Hat's UBI base image and re-checks it periodically:

[tabs]
====
Console::
+
--
Open *Builds* → *ImageStreams* → *Create ImageStream* → *YAML view*, and create a stream named `ext-ubi` with a tag whose `from` is `registry.access.redhat.com/ubi9/ubi:latest` and whose `importPolicy.scheduled` is `true`, then *Create*.

CLI::
+

[source,sh,role=execute]
----

----
--
====

.Expected output (the stream is created and the base image imported)
[source,texinfo]
----
imagestream.image.openshift.io/ext-ubi imported

Name:			ext-ubi
Namespace:		user3-dev
----

[IMPORTANT]
.`oc import-image` uses your *current project*, not a `namespace/` prefix
====
Unlike `oc tag`, `oc import-image` does *not* read a `namespace/name` positional prefix — `oc import-image {user}-dev/ext-ubi` would create the stream in whatever project you're *currently* in, not `{user}-dev`.
Always pass `-n $NS` with a *bare* stream name (as above) so it lands where you mean.
====

Confirm the schedule is set — this is what makes the re-import automatic:

[source,sh,role=execute]
----
oc get is ext-ubi -n $NS -o jsonpath='{range .spec.tags[*]}{.name}{" scheduled="}{.importPolicy.scheduled}{"\n"}{end}'
oc get is ext-ubi -n $NS -o jsonpath='resolved: {.status.tags[0].items[0].dockerImageReference}{"\n"}'
----

.Expected output
[source,texinfo]
----
latest scheduled=true
resolved: registry.access.redhat.com/ubi9/ubi@sha256:bcfca170da4fe08c0b70aa76ca4ee63f0e724db1574712cbc6c6a77fea6b21dc
----

`scheduled=true` means OpenShift periodically re-checks `ubi9/ubi:latest` and pulls the new digest when it moves.

*Now the break-and-fix.* Imports fail for real reasons — a typo in the registry path, a tag that doesn't exist — and the *import status* tells you exactly why.
Try an import with a deliberately wrong repository name and *read the error*:

[source,sh,role=execute]
----
oc import-image ext-broken -n $NS --from=registry.access.redhat.com/ubi9/does-not-exist:latest --confirm
----

.Expected output (the import is created but its status carries the failure)
[source,texinfo]
----
  ! error: Import failed (InternalError): Internal error occurred: registry.access.redhat.com/ubi9/does-not-exist:latest: name unknown: Repo not found
      Less than a second ago

error: imported completed with errors
----

*"name unknown: Repo not found"* — the registry has no such repository.
That one line is the answer to almost every failed import: read the import status, and it names the cause (bad repo, missing tag, or — as you'll see in exercise 6 — a registry the *policy* doesn't allow).
Fix it by importing the *correct* reference, and clean up the broken stream:

[source,sh,role=execute]
----

oc import-image ext-ubi -n $NS --from=registry.access.redhat.com/ubi9/ubi:latest --confirm | grep -i 'imported'
----

.Expected output
[source,texinfo]
----
imagestream.image.openshift.io "ext-broken" deleted
imagestream.image.openshift.io/ext-ubi imported
----

== 4. Pull from a private registry with a pull secret
⏱ *~12 min* · the mechanics of authenticating a workload to a registry it doesn't own

Public images (like the UBI base above) need no credentials.
A *private* registry does — and the credential is a *pull secret* the workload presents when it pulls.
Your namespace already has a sample one; inspect it (it's safe to decode — it's a non-secret teaching sample):

[source,sh,role=execute]
----
oc get secret parasol-registry-creds -n $NS -o jsonpath='type={.type}{"\n"}'
oc get secret parasol-registry-creds -n $NS -o jsonpath='{.data.\.dockerconfigjson}' | base64 -d; echo
----

.Expected output (a dockerconfigjson: a host, a username, an auth token)
[source,texinfo]
----
type=kubernetes.io/dockerconfigjson
{"auths":{"registry.parasol.example.com":{"username":"parasol-puller","password":"parasol","auth":"cGFyYXNvbC1wdWxsZXI6cGFyYXNvbA==","email":"puller@parasol.example.com"}}}
----

[NOTE]
.This is an *inert* sample — the host doesn't resolve
====
`registry.parasol.example.com` is a placeholder that doesn't exist, and the credential is a teaching sample, not a real secret.
So you can't perform a real *pull* from it — but you *can* learn the two mechanics that wire a pull secret to a workload, which is the transferable skill.
On a real private registry (or Red Hat Quay) you'd create this secret with `oc create secret docker-registry` using genuine credentials, and the *same* two mechanics apply unchanged.
====

There are *two* ways to attach a pull secret; you'll use both.

*Mechanic A — link it to a ServiceAccount.*
Every pod runs as a ServiceAccount; linking the secret to a SA `--for=pull` means *every* pod that SA runs can authenticate to that registry, no per-pod config:

[source,sh,role=execute]
----
oc secrets link default parasol-registry-creds --for=pull -n $NS
oc get sa default -n $NS -o jsonpath='default SA pull secrets: {.imagePullSecrets[*].name}{"\n"}'
----

.Expected output (the secret is now on the ServiceAccount)
[source,texinfo]
----
default SA pull secrets: default-dockercfg-... parasol-registry-creds
----

*Mechanic B — name it on the pod.*
The GitOps-clean form is to name the pull secret directly in a workload's `spec.template.spec.imagePullSecrets`.
Deploy a small workload that references the secret this way.
(It runs the *internal* claims image so the pod is genuinely `Ready` — the pull-secret reference is the artifact you're wiring, not a real external pull):

// [CAPTURE-VERIFY] Workloads → Deployments → Create → the Deployment YAML accepts spec.template.spec.imagePullSecrets (OCP 4.21) — CLI authoritative.
[tabs]
====
Console::
+
--
Open *Workloads* → *Deployments* → *Create Deployment* → *YAML view*, paste the manifest from the *CLI* tab, and *Create*.
Note the `imagePullSecrets` entry naming `parasol-registry-creds` in the pod template.
--
CLI::
+
--
[source,sh,role=execute]
----
oc apply -n $NS -f - <<'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: parasol-registry-demo
  labels:
    app: parasol-registry-demo
spec:
  replicas: 1
  selector:
    matchLabels:
      app: parasol-registry-demo
  template:
    metadata:
      labels:
        app: parasol-registry-demo
    spec:
      imagePullSecrets:
        - name: parasol-registry-creds
      containers:
        - name: claims
          image: image-registry.openshift-image-registry.svc:5000/parasol-images/parasol-claims:1.0
          ports:
            - containerPort: 8080
          resources:
            requests: {cpu: 50m, memory: 128Mi}
            limits: {cpu: 200m, memory: 256Mi}
          securityContext:
            runAsNonRoot: true
            allowPrivilegeEscalation: false
            capabilities: {drop: [ALL]}
            seccompProfile: {type: RuntimeDefault}
EOF
oc rollout status deploy/parasol-registry-demo -n $NS --timeout=90s
----
--
====

.Expected output (the workload is Ready, carrying the pull-secret reference)
[source,texinfo]
----
deployment.apps/parasol-registry-demo created
deployment "parasol-registry-demo" successfully rolled out
----

.Expected output

NAME                                     READY   STATUS    RESTARTS   AGE
parasol-registry-demo-...                1/1     Running   0          5s

✔ *Verify:* You decoded the sample dockerconfigjson pull secret, *linked* it to the `default` ServiceAccount for pull (mechanic A), and deployed a workload that *names* it in `imagePullSecrets` (mechanic B) — the pod is `Running` with the pull-secret reference wired. == 5. Publish your own Developer Catalog entry
⏱ *~12 min* · a namespaced Template appears in *your* catalog and nobody else's

The Developer Catalog isn't fixed — you can *add* to it.
A `Template` you create in your *own* project becomes a one-click catalog entry in *your* Developer Catalog only (contrast the cluster-wide samples operator, which you'll read in exercise 6).

== 6. Read the governance surface (instructor demo)
⏱ *~10 min* · watch — where images may come from, and what the catalog offers

Everything so far was *yours* to change, in your own namespace.
The controls that govern *the whole cluster* — which registries images may come from, what the catalog offers — are cluster singletons.
You *read* them here (the read-only `platform-observer` role grants you cluster-wide *read*); *setting* them is the instructor's segment, because one change affects every tenant.

First, the *trust boundary* — where images may come from.
Read the cluster `Image` config and the mirroring objects:

[source,sh,role=execute]
----
oc get image.config.openshift.io/cluster -o jsonpath='allowedRegistriesForImport={.spec.allowedRegistriesForImport}{"\n"}registrySources={.spec.registrySources}{"\n"}'
oc get imagedigestmirrorset,imagetagmirrorset -A
----

.Expected output (open by default here — no restriction, no mirrors)
[source,texinfo]
----
allowedRegistriesForImport=
registrySources=
No resources found
----

Both are *empty* on this cluster — images may come from anywhere, and nothing is mirrored.
Those are the fields the instructor will demonstrate: `allowedRegistriesForImport` (instant, admission-level) and `registrySources` / *ImageDigestMirrorSet (IDMS)* / *ImageTagMirrorSet (ITMS)* (heavy — they re-render every node).

Next, *retention* — the nightly pruner that reclaims registry storage:

[source,sh,role=execute]
----
oc get imagepruner/cluster -o jsonpath='keepTagRevisions={.spec.keepTagRevisions} suspend={.spec.suspend}{"\n"}'
oc get cronjob image-pruner -n openshift-image-registry
----

.Expected output (a daily pruner keeping the last 3 revisions)
[source,texinfo]
----
keepTagRevisions=3 suspend=false
NAME           SCHEDULE    SUSPEND   ACTIVE   LAST SCHEDULE   AGE
image-pruner   0 0 * * *   False     0        101m            4d
----

`0 0 * * *` is midnight daily; it keeps the last 3 tag revisions and spares anything younger than an hour.
On a multi-day event, an *unreferenced* image can be pruned overnight — which is why the images you use stay *referenced* by a running workload.

Finally, the *catalog supply* — what developers can pick from.
Read the samples operator, the operator catalog sources, and the cluster samples:

[source,sh,role=execute]
----

echo "sample imagestreams: $(oc get is -n openshift --no-headers | wc -l | tr -d ' ')  sample templates: $(oc get templates -n openshift --no-headers | wc -l | tr -d ' ')"
oc get catalogsources -A
----

.Expected output (a managed samples operator; the default catalogs plus a curated internal one)
[source,texinfo]
----
samples: Managed
sample imagestreams: 60  sample templates: 51
NAMESPACE               NAME                  DISPLAY                         TYPE   PUBLISHER
gitea-operator          redhat-rhpds-gitea    Red Hat Demo Platform (Gitea)   grpc   Red Hat Demo Platform
