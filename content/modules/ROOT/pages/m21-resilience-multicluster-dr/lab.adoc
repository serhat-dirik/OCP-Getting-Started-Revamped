== 1. Prove in-site resiliency — survive a pod loss
⏱ *~12 min* · meet the resilient claims service and watch it absorb a pod kill

Before a service can fail *over* to another site, it has to be resilient *where it lives*. Meet the primary site's claims service — the M12/M16 primitives, made concrete.

[tabs]
====
Console::
+
--
In the {ocp_console_url}[OpenShift web console^] (project `{user}-site-a`), open *Workloads → Deployments*. You'll see `parasol-claims` at *3 of 3* pods. Open it and note its *Pods* land on *different nodes*. Under *Workloads*, the `parasol-claims` *HorizontalPodAutoscaler* and *PodDisruptionBudget* show the resilience policy.
The *CLI* tab reads the same facts — use it to capture the exact fields.
--
CLI::
+
--
[source,sh,role=execute]
----
oc get deploy,pdb,hpa -n $SITEA
----
--
====

.Expected output (a 3-replica claims service, a PDB keeping 2 available, an HPA holding the floor at 3)
[source,texinfo]
----
NAME                             READY   UP-TO-DATE   AVAILABLE   AGE
deployment.apps/parasol-claims   3/3     3            3           11m

NAME                                        MIN AVAILABLE   MAX UNAVAILABLE   ALLOWED DISRUPTIONS   AGE
poddisruptionbudget.policy/parasol-claims   2               N/A               1                     11m

NAME                                                 REFERENCE                   TARGETS       MINPODS   MAXPODS   REPLICAS
horizontalpodautoscaler.autoscaling/parasol-claims   Deployment/parasol-claims   cpu: 4%/80%   3         6         3
----

Read that as a resilience posture: *three* replicas so losing one is a non-event; a `PodDisruptionBudget` that keeps *two* available during any voluntary disruption (`ALLOWED DISRUPTIONS: 1` — a node drain evicts *at most one* at a time); and a `HorizontalPodAutoscaler` holding the floor at three, bursting to six. Confirm the replicas are *spread across nodes*, so one node loss can't take them all:

[source,sh,role=execute]
----
oc get pods -n $SITEA -l app=parasol-claims -o custom-columns='POD:.metadata.name,STATUS:.status.phase,NODE:.spec.nodeName'
----

.Expected output (three replicas on three distinct nodes — topology spread + anti-affinity at work)
[source,texinfo]
----
POD                               STATUS    NODE
parasol-claims-65d8cc85fd-4wwmk   Running   control-plane-cluster-example-1
parasol-claims-65d8cc85fd-v2wth   Running   control-plane-cluster-example-2
parasol-claims-65d8cc85fd-z5r22   Running   control-plane-cluster-example-3
----

Now *break it on purpose*. Kill one replica and watch the service never dip — the Deployment replaces the pod, and the PDB guarantees a quorum stays up throughout:

[source,sh,role=execute]
----
VICTIM=$(oc get pod -n $SITEA -l app=parasol-claims -o jsonpath='{.items[0].metadata.name}')
oc delete pod $VICTIM -n $SITEA
oc get pods -n $SITEA -l app=parasol-claims
----

.Expected output (one pod deleted; the other two keep running and a replacement is already up)
[source,texinfo]
----
pod "parasol-claims-65d8cc85fd-4wwmk" deleted
NAME                              READY   STATUS    RESTARTS   AGE
parasol-claims-65d8cc85fd-2t2hf   1/1     Running   0          6s
parasol-claims-65d8cc85fd-v2wth   1/1     Running   0          2m
parasol-claims-65d8cc85fd-z5r22   1/1     Running   0          2m
----

Two replicas never stopped serving, and a fresh pod (`2t2hf`, six seconds old) already replaced the one you killed. *That* is what replicas + a PDB + spread buy you: a pod or node loss is a non-event. But it is precisely what they *cannot* do — survive the loss of the *whole site* — that the next exercise is about.

[NOTE,role=instructor-demo]
.`[INSTRUCTOR-DEMO]` — drain a node, watch the PDB pace it
====
A great five-minute live beat: the instructor *cordons and drains a node* the claims service runs on (`oc adm drain <node> --ignore-daemonsets --delete-emptydir-data`). The PodDisruptionBudget *paces* the eviction — one pod at a time — so a quorum stays up throughout, and attendees watch `oc get pods -n {user}-site-a -w` absorb it with no outage. The lesson to name: this is *in-site* resilience; the next exercise is the failure it can't survive.
====

[.checkpoint]
****
✔ *Verify:* You found the resilient primary claims service — `parasol-claims` at *3/3* with a `PodDisruptionBudget` (min available 2), a `HorizontalPodAutoscaler` (floor 3, ceiling 6), and replicas *spread across three nodes* — then *killed a pod* and watched the service stay up (two replicas never dipped, a replacement appeared in seconds). In-site resiliency, proven.
****

== 2. Cross-site failover with Service Mesh — fail an entire site
⏱ *~25 min* · wire a stable endpoint that fails over from the primary site to the secondary automatically

This is the module's centerpiece. You'll put a *stable endpoint* in front of both sites, teach the mesh to prefer the primary and *fail over to the secondary automatically*, then *fail the whole primary site* and watch the client keep working.

First, meet the client and prove the problem. The client calls a *stable endpoint* (`claims-stable`, the mesh ingress gateway) once a second and logs which site answered. Tail its log:

[source,sh,role=execute]
----
oc logs deploy/claims-client -n $CLIENT --tail=5
----

.Expected output (the stable endpoint exists, but nothing routes through it yet — `404`)
[source,texinfo]
----
10:22:14  HTTP 404  served-by-site=none
10:22:15  HTTP 404  served-by-site=none
10:22:16  HTTP 404  served-by-site=none
----

The gateway is up, but it has *no route* — it doesn't yet know about your two sites. That's what you wire now, with three mesh resources in the client namespace:

* a *ServiceEntry* that declares one logical claims host whose two endpoints are site A and site B, each tagged with its *locality*;
* a *DestinationRule* that turns on *locality-aware load balancing* (prefer the primary) and *outlier detection* (eject a failing site);
* a *VirtualService* on the gateway that routes the stable endpoint to that host, with *retries* so failover drops no requests.

// [CAPTURE-VERIFY] Console: +Add → Import YAML (project {user}-client) → paste all three resources → Create (OCP 4.21). CLI authoritative.
[tabs]
====
Console::
+
--
Open *+Add* → *Import YAML* (project `{user}-client`), paste *all three* resources from the *CLI* tab (separated by `---`), and *Create*.
--
CLI::
+
--
[source,sh,role=execute]
----
oc apply -f - <<EOF
apiVersion: networking.istio.io/v1
kind: ServiceEntry
metadata: {name: claims-global, namespace: $CLIENT}
spec:
  hosts: [claims.parasol.internal]
  location: MESH_INTERNAL
  ports: [{number: 8080, name: http, protocol: HTTP}]
  resolution: DNS
  endpoints:
    - {address: claims.$SITEA.svc.cluster.local, locality: east/site-a, ports: {http: 8080}}
    - {address: claims.$SITEB.svc.cluster.local, locality: east/site-b, ports: {http: 8080}}
---
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata: {name: claims-global, namespace: $CLIENT}
spec:
  host: claims.parasol.internal
  trafficPolicy:
    loadBalancer: {localityLbSetting: {enabled: true}}
    outlierDetection:
      consecutive5xxErrors: 1
      consecutiveGatewayErrors: 1
      interval: 5s
      baseEjectionTime: 30s
      maxEjectionPercent: 100
---
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata: {name: claims-edge, namespace: $CLIENT}
spec:
  hosts: ["*"]
  gateways: [claims-gateway]
  http:
    - route: [{destination: {host: claims.parasol.internal, port: {number: 8080}}}]
      retries: {attempts: 3, perTryTimeout: 2s, retryOn: "connect-failure,refused-stream,reset,gateway-error,5xx"}
EOF
----
--
====

.Expected output
[source,texinfo]
----
serviceentry.networking.istio.io/claims-global created
destinationrule.networking.istio.io/claims-global created
virtualservice.networking.istio.io/claims-edge created
----

Give the mesh a few seconds to program the gateway, then look at the client log again — it now serves *site A*, the primary:

[source,sh,role=execute]
----
sleep 8
oc logs deploy/claims-client -n $CLIENT --tail=5
----

.Expected output (the stable endpoint now routes — steadily to the primary site A)
[source,texinfo]
----
10:24:03  HTTP 200  served-by-site=A
10:24:04  HTTP 200  served-by-site=A
10:24:05  HTTP 200  served-by-site=A
----

Why *A* and not a mix of A and B? Because the ingress gateway is *co-located with site A's locality*, and `localityLbSetting` makes the mesh *prefer the local site* — site A is priority 0, site B is the failover. Confirm the gateway's Envoy really has both sites as endpoints, and that the outlier-detection failover policy is programmed:

[source,sh,role=execute]
----
GW=$(oc get pod -n $CLIENT -l istio=claims-gateway -o jsonpath='{.items[0].metadata.name}')
istioctl proxy-config endpoints $GW.$CLIENT --cluster "outbound|8080||claims.parasol.internal"
----

.Expected output (both sites are endpoints of the one logical claims host, both healthy)
[source,texinfo]
----
ENDPOINT                STATUS      OUTLIER CHECK     CLUSTER
172.231.18.106:8080     HEALTHY     OK                outbound|8080||claims.parasol.internal
172.231.92.201:8080     HEALTHY     OK                outbound|8080||claims.parasol.internal
----

Now *the failover*. In a *second terminal*, start tailing the client log live so you can watch it in real time:

[source,sh,role=execute]
----
oc logs -f deploy/claims-client -n $CLIENT --tail=3
----

Then, back in your *first* terminal, *fail the entire primary site* — scale site A to zero. There is no site A anymore:

[source,sh,role=execute]
----
oc scale deploy/parasol-claims -n $SITEA --replicas=0
----

Watch the tailing log. Within a few seconds the site marker *flips from A to B* — and *every line is still `HTTP 200`*:

.Expected output (the live client log across the failover — the whole primary site is gone, not one request dropped)
[source,texinfo]
----
10:27:12  HTTP 200  served-by-site=A
10:27:13  HTTP 200  served-by-site=A     <- site A scaled to 0 here
10:27:16  HTTP 200  served-by-site=B     <- failover: the client flips to site B
10:27:17  HTTP 200  served-by-site=B
10:27:18  HTTP 200  served-by-site=B
----

That is *automatic cross-site failover*. The client never changed its address; the mesh detected site A's requests failing, *ejected* it (outlier detection), and *retried* the in-flight requests onto site B — so the client saw a clean flip from A to B, not an outage. The whole primary site vanished and *no request was dropped*.

Now *recover*. Bring site A back, and once the mesh trusts it again (about thirty seconds — the outlier-detection ejection expiring), traffic *returns to the primary*:

[source,sh,role=execute]
----
oc scale deploy/parasol-claims -n $SITEA --replicas=3
oc rollout status deploy/parasol-claims -n $SITEA --timeout=90s
----

Back in the tailing log, watch the marker flip *back to A*:

.Expected output (failback — traffic returns to the recovered primary, still all 200)
[source,texinfo]
----
10:27:45  HTTP 200  served-by-site=B
10:27:47  HTTP 200  served-by-site=A     <- site A recovered; traffic returns to the primary
10:27:48  HTTP 200  served-by-site=A
----

Stop the tailing log (`Ctrl-C`). You just failed and recovered a whole site with the client none the wiser.

[NOTE]
.Why the brief pause, and why no `404`/`503`
====
Between the last `A` and the first `B` you may see a *one- or two-second gap* in the log (the failover request took a retry hop, so the once-a-second loop skipped a beat) — but *no error line*. That's the retries doing their job: the request that raced site A's ejection was retried onto site B and returned `200`. Tune `baseEjectionTime` up and the failed primary stays ejected longer (fewer re-probes mid-outage); tune it down and traffic returns to the primary faster after recovery. Thirty seconds is a good middle.
====

[.checkpoint]
****
✔ *Verify:* You wired cross-site failover on the mesh — a *ServiceEntry* declaring both sites with localities, a *DestinationRule* (locality-aware load balancing + outlier detection), and a *VirtualService* with retries on the ingress gateway — saw the client steadily served by the *primary* site A, then *scaled the entire site A to zero* and watched the client *fail over to site B with every request still `200`*, and finally *recovered* site A and watched traffic *return* to it. Automatic failover and failback, zero client change.
****

== 3. `[ADD-ON]` Connect two sites over a VAN with Red Hat Service Interconnect
⏱ *~20 min* · link two data centers into one Virtual Application Network and read a remote service locally

[NOTE]
.This whole exercise is a flagged, skippable `[ADD-ON]`
====
Red Hat Service Interconnect is a *separate subscription*. Nothing earlier in this module depends on it. Skip it freely if RHSI isn't part of your engagement. It's included, fully hands-on, because *connecting* separate sites is what makes cross-site failover possible in the first place when the sites are real, distant clusters.
====

The failover you just built assumes the two sites can *reach* each other. On one cluster that's free. Now treat `{user}-site-a` and `{user}-site-b` as *two separate data centers* — DC-A and DC-B — with *no shared network*, and link them with a *Virtual Application Network* (VAN): a mutual-TLS Layer-7 overlay that connects *just the services that need it*, over a single outbound connection, with no VPN.

Each site becomes a *Site*. Make DC-A *linkable* (`linkAccess: default`) so DC-B can connect to it:

[tabs]
====
Console::
+
--
Open *+Add* → *Import YAML*, paste *both* `Site` resources from the *CLI* tab, and *Create* (once in project `{user}-site-a`, once in `{user}-site-b`).
--
CLI::
+
--
[source,sh,role=execute]
----
oc apply -f - <<EOF
apiVersion: skupper.io/v2alpha1
kind: Site
metadata: {name: dc-a, namespace: $SITEA}
spec: {linkAccess: default}
---
apiVersion: skupper.io/v2alpha1
kind: Site
metadata: {name: dc-b, namespace: $SITEB}
spec: {}
EOF
oc wait --for=condition=Ready site/dc-a -n $SITEA --timeout=180s
oc wait --for=condition=Ready site/dc-b -n $SITEB --timeout=180s
----
--
====

.Expected output (both sites Ready; a Skupper router pod starts in each)
[source,texinfo]
----
site.skupper.io/dc-a condition met
site.skupper.io/dc-b condition met
----

Now *expose* DC-B's claims service across the VAN with a *Connector* (it binds the local `parasol-claims` pods to a *routing key*), and *consume* it in DC-A with a *Listener* (a local endpoint `claims-remote:8080` bound to the *same* routing key):

// [CAPTURE-VERIFY] Console: Networking → Service Interconnect (the RHSI console plugin) shows the sites/connectors/listeners; or Import YAML. CLI authoritative — the console view is for visualizing.
[tabs]
====
Console::
+
--
Open *+Add* → *Import YAML* and create the `Connector` (in `{user}-site-b`) and the `Listener` (in `{user}-site-a`) from the *CLI* tab. The {ocp_console_url}[web console^] *Networking → Service Interconnect* view visualizes them once linked.
--
CLI::
+
--
[source,sh,role=execute]
----
oc apply -f - <<EOF
apiVersion: skupper.io/v2alpha1
kind: Connector
metadata: {name: claims, namespace: $SITEB}
spec: {routingKey: claims-cross, selector: app=parasol-claims, port: 8080}
---
apiVersion: skupper.io/v2alpha1
kind: Listener
metadata: {name: claims-remote, namespace: $SITEA}
spec: {routingKey: claims-cross, host: claims-remote, port: 8080}
EOF
----
--
====

Now *link* the two sites. This is the step the field warns about — a *token exchange*: the linkable site (DC-A) issues an *AccessGrant* (a short-lived, redeemable token), and DC-B *redeems* it with an *AccessToken* to form the `Link`. Issue the grant:

[source,sh,role=execute]
----
oc apply -f - <<EOF
apiVersion: skupper.io/v2alpha1
kind: AccessGrant
metadata: {name: dc-a-grant, namespace: $SITEA}
spec: {redemptionsAllowed: 5, expirationWindow: 1h}
EOF
oc wait --for=condition=Ready accessgrant/dc-a-grant -n $SITEA --timeout=60s
----

The grant publishes its *url*, *code*, and *ca* into its `status`. *Script the redemption* — copy those three fields into an `AccessToken` in DC-B (doing this by hand is the classic RHSI footgun; the multi-line `ca` in particular):

[source,sh,role=execute]
----
URL=$(oc get accessgrant dc-a-grant -n $SITEA -o jsonpath='{.status.url}')
CODE=$(oc get accessgrant dc-a-grant -n $SITEA -o jsonpath='{.status.code}')
CA=$(oc get accessgrant dc-a-grant -n $SITEA -o jsonpath='{.status.ca}')
{
  echo "apiVersion: skupper.io/v2alpha1"
  echo "kind: AccessToken"
  echo "metadata: {name: dc-a-token, namespace: $SITEB}"
  echo "spec:"
  echo "  url: $URL"
  echo "  code: $CODE"
  echo "  ca: |"
  echo "$CA" | sed 's/^/    /'
} | oc apply -f -
oc wait --for=condition=Ready link/dc-a-token -n $SITEB --timeout=120s
----

.Expected output (the token is redeemed and the Link forms — the two DCs are now one VAN)
[source,texinfo]
----
accesstoken.skupper.io/dc-a-token created
link.skupper.io/dc-a-token condition met
----

Confirm the VAN is wired end-to-end — the Connector and Listener found each other across the link, and both DCs are in one network:

[source,sh,role=execute]
----
oc get listener claims-remote -n $SITEA -o jsonpath='listener matched={.status.hasMatchingConnector} key={.spec.routingKey}{"\n"}'
oc get connector claims -n $SITEB -o jsonpath='connector matched={.status.hasMatchingListener} key={.spec.routingKey}{"\n"}'
oc get site dc-a -n $SITEA -o jsonpath='sites in network={.status.sitesInNetwork}{"\n"}'
----

.Expected output (Listener and Connector matched across the VAN; 2 sites in the network)
[source,texinfo]
----
listener matched=true key=claims-cross
connector matched=true key=claims-cross
sites in network=2
----

Now the payoff — read DC-B's claims service from *inside DC-A*, through the *local* `claims-remote` address the VAN provides, with the app unchanged:

[source,sh,role=execute]
----

REMOTE=$(oc exec $PODA -c claims -n $SITEA -- curl -s -m5 http://claims-remote:8080/)
LOCAL=$(oc exec $PODA -c claims -n $SITEA -- curl -s -m5 http://claims:8080/)
echo "remote over the VAN (claims-remote): $REMOTE"
echo "local in DC-A (claims):              $LOCAL"
----

`SITE=B` through `claims-remote` — DC-A read DC-B's claims service over a mutual-TLS Layer-7 link, through a *local* address, with *no VPN and no shared network*.
