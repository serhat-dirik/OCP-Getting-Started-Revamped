âŹ± *~10 min* Â· the heart of the module â€” desired state defends itself

CLI::
+

.Expected output
[source,texinfo]

parasol-web-6475f4bbb5-4qspj   1/1     Running   0          21s   worker-...-1
parasol-web-6475f4bbb5-chnfm   1/1     Running   0          21s   worker-...-2

----

Notice the three Pods landed on *three different worker nodes* â€” the scheduler spread them for you, so losing one node does not take out all your replicas.

=== Break it: kill a Pod

This is the deliberate break-and-fix of the module.
You are going to delete a running Pod â€” the exact event that used to mean a 2 a.m. page â€” and watch the platform absorb it.

This is the classic *watch-in-one, act-in-the-other* beat, so use *both* cockpit terminals.

First, switch to *Terminal 2* (the second terminal tab) and start watching the Pods â€” copy the command below, paste it into Terminal 2, and leave it running:

[source,sh]
----
oc get pods -w
----

Then come back to *Terminal 1* and delete one Pod (click *Run*):

[source,sh,role=execute]
----
oc delete pod "$(oc get pods -l deployment=parasol-web -o jsonpath='{.items[0].metadata.name}')"
----

Watch *Terminal 2*: the deleted Pod goes `Terminating`, and *before it is even gone* a new Pod appears in `ContainerCreating` and then `Running`.
When you have seen it, press *Ctrl+C* in Terminal 2 to stop the watch.
Your desired state is three; actual briefly dropped to two; the ReplicaSet reconciled it back to three.

Confirm with the events:

[source,sh,role=execute]
----
oc get events --sort-by=.lastTimestamp | grep -E "Killing|SuccessfulCreate" | tail -4
----

.Expected output
[source,texinfo]
----
Normal   Killing            pod/parasol-web-6475f4bbb5-4qspj    Stopping container parasol-web
Normal   SuccessfulCreate   replicaset/parasol-web-6475f4bbb5   Created pod: parasol-web-6475f4bbb5-t7l6q
----

There it is in the platform's own words: `SuccessfulCreate` from the *ReplicaSet*.
No human created that Pod.

=== Break it harder (optional): scale to zero

Want to feel desired state from the other side?
Set it to zero, watch every Pod disappear, then bring it back:

[source,sh,role=execute]
----
oc scale deployment/parasol-web --replicas=0 && oc get pods
----

[source,sh,role=execute]
----
oc scale deployment/parasol-web --replicas=3 && oc rollout status deployment/parasol-web
----

The platform did exactly what you declared each time â€” down to nothing, back to three.
That is the point: the number you set is the number the platform defends.

[.checkpoint]
****
âś” *Verify:* After deleting a Pod, `oc get pods` still shows *3* `Running` Pods (with one much younger than the others), and `oc get events` contains a `SuccessfulCreate` from the ReplicaSet.

== 4. Publish the app with a Route
âŹ± *~7 min* Â· make the claims portal reachable from a browser

Your Service load-balances the three Pods, but only *inside* the cluster.
A *Route* publishes it at a real URL on the cluster domain.

[tabs]
====
Console::
+
--
. In the navigation, open *Networking* â†’ *Routes* and click *Create Route*.
. Name the Route `parasol-web`, select the `parasol-web` *Service*, and set *Target port* to *8080*.
. Check *Secure Route*, set *TLS termination* to *Edge* and *Insecure traffic* to *Allow*, then click *Create*.
. Go back to *Topology* (under *Workloads*), click the `parasol-web` node, and click the *Open URL* arrow that now appears on the top-right of the node.
+
NOTE: In this console version you create a Route from *Networking â†’ Routes* â€” there is no *Actions â†’ Create Route* item on the Topology node, and a workload with no Route shows "No Routes found for this resource." in its side-panel *Resources* tab.
--
CLI::
+
--
The Service exposes two ports (8080 for HTTP, 8443 for HTTPS). Publish it over TLS with an *edge* Route: the router terminates HTTPS with the cluster's certificate and forwards to the app's HTTP port 8080.

[source,sh,role=execute]
----
oc create route edge parasol-web --service=parasol-web --port=8080 --insecure-policy=Allow
----

[source,sh,role=execute]
----
oc get route parasol-web
----

.Expected output
[source,texinfo,subs="attributes"]
----
NAME          HOST/PORT                                 PATH   SERVICES      PORT   TERMINATION   WILDCARD
parasol-web   parasol-web-{user}-dev.{cluster_domain}          parasol-web   8080   edge/Allow    None
----

[NOTE]
.Why an edge Route (and `--port=8080`)
=====
Browsers reach anything on `*.apps` over HTTPS, so a plain HTTP Route lands on the router's "Application is not available" page.
An *edge* Route terminates TLS at the router with the cluster's wildcard certificate, so `https://â€¦` just works; `--insecure-policy=Allow` also answers plain `http://` (instead of redirecting), which keeps the `curl` below simple.
Because the Service has more than one port, `--port=8080` picks the HTTP port the router forwards to.
The Route's host is built from the Route name and your project â€” `parasol-web-{user}-dev.{cluster_domain}` â€” so it is unique to you; no two attendees collide.
=====
--
====

Open your app at https://parasol-web-{user}-dev.{cluster_domain}/[the claims portal^].
You should see the *Parasol Insurance â€” Claims Portal* landing page.
The seeded claims data is served at `/api/claims`:

[source,sh,role=execute,subs="attributes"]
----
curl -s http://parasol-web-{user}-dev.{cluster_domain}/api/claims | head -c 300
----

.Expected output
[source,texinfo]
----
[{"id":"CLM-1001","policyholder":"Alice Nguyen","type":"Auto","status":"Under Review","amount":4200.0,...}]
----

[.checkpoint]
****
âś” *Verify:* The Route host `parasol-web-{user}-dev.{cluster_domain}` opens the Parasol claims portal in your browser (HTTP 200), and `curl .../api/claims` returns a JSON array of claims.

== 7. Ask the cluster: OpenShift Lightspeed
âŹ± *~8 min* Â· get an answer without leaving the console

*OpenShift Lightspeed* is the assistant built into the OpenShift web console â€” it answers OpenShift and Kubernetes questions in plain language and points you at the right `oc` commands and docs.
(It is distinct from *Developer Lightspeed for Red Hat Developer Hub* and *Developer Lightspeed for the Migration Toolkit for Applications*, which you meet in later modules.)

. Open the *Lightspeed* chat bubble at the bottom-right of the console (its drawer may already be open â€” it opens automatically on your first visit).
. Ask a real operational question:
+
[.copy]
`Why is my pod restarting?`
. Read the answer.
It should list the usual causes â€” `CrashLoopBackOff`, `OOMKilled`, a failing liveness probe, `ImagePullBackOff`, node pressure â€” and, importantly, hand you the exact commands to investigate (`oc get events`, `oc logs`, `oc describe pod`) with links to OpenShift documentation.
. Ask a follow-up about *your* app, for example:
+
[.copy]
`How do I see the logs for a deployment called parasol-web?`

The point is not that the AI is magical â€” it is that the fastest path from "something's wrong" to "here's the command to run" is now one question, in the same window where you fix things.
Later modules point this same idea at Parasol's *own* applications.

[NOTE]

If Lightspeed answers with a permissions error, your user has not been granted query access â€” ask your instructor (the workshop setup grants it to all attendees automatically).
