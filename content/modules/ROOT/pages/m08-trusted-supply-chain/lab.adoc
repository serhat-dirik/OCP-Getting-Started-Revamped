== 1. Run the supply-chain pipeline — and watch the gate catch a CVE
⏱ *~20 min* · start the run early, then read how it is wired while it builds

Your project has a Pipeline the platform team seeded — `parasol-claims-supply-chain`.
It is the build-and-deploy shape from the previous module with *three trust controls* added.
Read it before you run it:

[source,sh,role=execute]
----

----

.Expected output (abridged)
[%collapsible]

[source,texinfo]
----

----
====

Four Tasks in order: `fetch-source` → `sbom-report` → `build-image` → `acs-scan`.
Notice the default `git-revision` is *`seed-vulnerable`* — a branch of your fork the entry state seeded with a known-bad dependency, so the gate has something real to catch.
Now look at *where the Tasks come from*:

[source,sh,role=execute]
----
oc get pipeline parasol-claims-supply-chain -n $NS \
  -o jsonpath='{range .spec.tasks[*]}{.name}{" -> "}{.taskRef.resolver}{" "}{.taskRef.params[1].value}{"/"}{.taskRef.params[2].value}{"\n"}{end}'
----

.Expected output
[source,texinfo]
----

sbom-report ->  /
build-image -> cluster buildah/openshift-pipelines
acs-scan -> cluster acs-image-check/parasol-tasks
----

Two Tasks come from the shipped catalog (`git-clone`, `buildah`), one is the curated *`acs-image-check`* from Parasol's `parasol-tasks` library (the scan gate), and `sbom-report` is defined inline in the Pipeline.
The scan gate is the curated Task — the platform team standardized _one_ way to fail a build on a vulnerability.

Now *run* it.
A PipelineRun attaches a fresh workspace and raises memory for the two Maven/build-heavy steps above the namespace default:

[source,sh,role=execute]
----
oc create -n $NS -f - <<'EOF'
apiVersion: tekton.dev/v1
kind: PipelineRun
metadata:
  generateName: supply-chain-run-
spec:
  pipelineRef:
    name: parasol-claims-supply-chain
  taskRunTemplate:
    serviceAccountName: pipeline      # operator-provided SA: can build + push images here
  workspaces:
    - name: shared-workspace
      volumeClaimTemplate:            # one PVC per run, auto-pruned with the run
        spec:
          accessModes: [ReadWriteOnce]
          resources:
            requests:
              storage: 1Gi
  taskRunSpecs:                       # more memory than the 1Gi namespace default for the builds
    - pipelineTaskName: sbom-report
      computeResources:
        requests: {memory: 1Gi}
        limits: {memory: 1536Mi}
    - pipelineTaskName: build-image
      computeResources:
        requests: {memory: 1536Mi}
        limits: {memory: 2Gi}
EOF
----

Follow it live.
The build downloads Maven dependencies, so expect *roughly 6–12 minutes* (longer the first time on a node):

[source,sh,role=execute]
----
tkn pipelinerun logs --last -f -n $NS
----

[TIP]
.Use the wait — read exercise 2's SBOM section now
====
This is a real build.
Rather than watch dependencies download, *leave this streaming* (or press *Ctrl-C* — the run keeps going) and read ahead.
Come back when the logs reach the `acs-scan` step — that is the beat you want to watch.
You can re-attach any time with `tkn pipelinerun logs --last -f -n $NS`.
====

The build *succeeds* — and then the `acs-scan` step turns the run *red*.

.The scan step's output (abridged — the real RHACS table has six columns)

----

[image-check] Policy check results for image: .../parasol-claims:latest
[image-check] (TOTAL: 2, LOW: 1, MEDIUM: 0, HIGH: 0, CRITICAL: 1)
[image-check]
[image-check] | POLICY                                    | SEVERITY | BREAKS BUILD | VIOLATION
[image-check] | Block Log4Shell (CVE-2021-44228) at build | CRITICAL |      X       | CVE-2021-44228 (CVSS 10) (severity Critical) found in
[image-check] |                                           |          |              | component 'org.apache.logging.log4j:log4j-core' (version 2.14.1)

Read it the way a security-minded engineer would.

== 2. Find what's inside: the software bill of materials
⏱ *~15 min* · generate the CycloneDX SBOM and locate the exact vulnerable component

The scan gate told you _something_ is wrong.
The *SBOM* — the software bill of materials — tells you _exactly what_, and it is the artifact you will search the day the next Log4Shell lands.
Your pipeline already generated one (the `sbom-report` step); now generate the same bill of materials yourself and inspect it.

Clone your fork's `seed-vulnerable` branch and produce a *CycloneDX* SBOM.
This is the identical plugin invocation the pipeline's `sbom-report` step uses, so it works whether or not the project's `pom.xml` declares the plugin:

[source,sh,role=execute]
----
# your fork's in-cluster URL, derived from $NS — no typing your username
git clone -b seed-vulnerable http://gitea.gitea.svc:3000/${NS%-cicd}/parasol-claims.git ~/parasol-claims
cd ~/parasol-claims
mvn -B -ntp org.cyclonedx:cyclonedx-maven-plugin:2.9.2:makeBom
----

.Expected output (abridged)
[source,texinfo]
----
[INFO] --- cyclonedx:2.9.2:makeBom (default-cli) @ parasol-claims ---
[INFO] CycloneDX: Resolving Dependencies
[INFO] CycloneDX: Creating BOM version 1.6 with 225 component(s)
[INFO] CycloneDX: Writing and validating BOM (JSON): .../target/bom.json
[INFO] BUILD SUCCESS
----

That wrote `target/bom.json` — a *CycloneDX* SBOM listing all *225* components in the build.
CycloneDX is an open SBOM standard; each component carries a *purl* (package URL) that names it exactly.
Now search it for the vulnerable dependency:

[source,sh,role=execute]
----
jq -r '.components[] | select(.name=="log4j-core") | "\(.group):\(.name)@\(.version)  \(.purl)"' target/bom.json
----

.Expected output
[source,texinfo]
----
org.apache.logging.log4j:log4j-core@2.14.1  pkg:maven/org.apache.logging.log4j/log4j-core@2.14.1?type=jar
----

There is the offender, named precisely by its purl: `log4j-core` version `2.14.1`.
That is the whole value proposition of an SBOM — the question "is the Log4Shell library in this image?" is a one-line query against a file, not a manual audit of the source.

[NOTE]
.The pipeline generated the same SBOM — you can read its finding too
====
The pipeline's `sbom-report` step ran this exact `makeBom` and then scanned the result.
Read what it reported for your last failed run:

[source,sh,role=execute]
----
SBOM_TR=$(oc get taskrun -n $NS -l tekton.dev/pipelineTask=sbom-report \
  --sort-by=.metadata.creationTimestamp -o jsonpath='{.items[-1:].metadata.name}')
tkn taskrun logs "$SBOM_TR" -n $NS | grep -iE 'Creating BOM|log4j-core'
----

.Expected output
[source,texinfo]
----
[cyclonedx] [INFO] CycloneDX: Creating BOM version 1.6 with 225 component(s)
[cyclonedx] ⚠️  the SBOM contains log4j-core@2.14.1 — CVE-2021-44228 (Log4Shell). Fix: drop this dependency.
----

The pipeline generates the bill of materials on *every* build; you just reproduced it by hand to see what it contains.
====

== 3. Fix the vulnerability and watch the gate go green
⏱ *~20 min* · remove the dependency, re-run, and watch the scan pass — and the image get signed

You know the exact problem: a `log4j-core` 2.14.1 dependency that should not be there.
The fix is to *remove it* — and then prove the fix by re-running the pipeline and watching the gate pass.
You make the change in *your own fork*, on the `seed-vulnerable` branch the pipeline builds.

In your *Gitea tab* ({gitea_url}/{user}/parasol-claims[{user}/parasol-claims^]):

. Switch to the *`seed-vulnerable`* branch (the branch selector, top-left of the file list).
. Open `pom.xml` and click the *pencil* (Edit) icon.
. Find and *delete* this block (near the end of the `<dependencies>` section):
+
[source,xml]
----
    <!-- M08 seeded CVE (do not fix on this branch): log4j-core 2.14.1 = CVE-2021-44228 Log4Shell. -->
    <dependency>
      <groupId>org.apache.logging.log4j</groupId>
      <artifactId>log4j-core</artifactId>
      <version>2.14.1</version>
    </dependency>
----
. Scroll down and *Commit Changes* directly to the `seed-vulnerable` branch (commit message e.g. "fix: remove vulnerable log4j-core dependency").

// media-pass: screenshot m08-trusted-supply-chain-02-gitea-pom-edit.png — the Gitea pom.xml editor on
// the seed-vulnerable branch with the log4j-core <dependency> block selected for deletion. See media-manifest.md.

[NOTE]
.Why remove it, rather than upgrade it
====
Upgrading `log4j-core` to a patched version would also clear the CVE.
But `parasol-claims` does not actually _use_ `log4j-core` — it was seeded purely to trip the gate — so the honest fix is to *remove the dependency you do not need*.
Fewer dependencies is fewer supply-chain risk: the smallest attack surface is the code you did not ship.
====

Now re-run the pipeline against your fork — the default `git-revision` is already `seed-vulnerable`, so this rebuilds the branch you just fixed:

          resources: {requests: {storage: 1Gi}}
  taskRunSpecs:

[TIP]
.Optional hardening: bump the base image
====
The seeded branch also pins an *older* UBI OpenJDK tag (`openjdk-21:1.18`), which carries base-operating-system CVEs that a newer tag has patched.
Bumping it (in the `Containerfile`, `FROM ...openjdk-21:1.23`) is good hygiene — *complementary hardening* — but it is *not* what turned this gate green.
The gate is scoped to the "Supply Chain Security" policy category, which keys on the Log4Shell dependency, not on drifting base-OS CVEs.
Try it if you like: bump the tag, re-run, and confirm the gate still passes (it does either way) — proof that the dependency, not the base image, was the gate.

== 4. Signing and provenance: prove who built the image
⏱ *~15 min* · Tekton Chains signed your build automatically — see the signature and its SLSA provenance

The scan proves the image is _clean_.
Signing proves it is _yours_ — that the image you are about to run is byte-for-byte the one your pipeline produced, built by a builder you trust.
*Tekton Chains* — a component of OpenShift Pipelines — watches every build TaskRun and *signs the image it produced*, with no per-build wiring.
Confirm it signed your builds:

[source,sh,role=execute]
----
oc get taskrun -n $NS -l tekton.dev/pipelineTask=build-image \
  -o jsonpath='{range .items[*]}{.metadata.name}{"  signed="}{.metadata.annotations.chains\.tekton\.dev/signed}{"\n"}{end}'
----

.Expected output
[source,texinfo]
----
supply-chain-run-xxxxx-build-image     signed=true
supply-chain-fixed-xxxxx-build-image   signed=true
----

Every build is `signed=true` — even the vulnerable one.
That is correct: signing answers "did _we_ build this," which is a _different_ question from "is it clean" (the scan) and "what is inside" (the SBOM).
You want all three; here they all happened automatically.

Chains stores the signature and the provenance *in the registry, beside the image* — as extra tags derived from the image digest:

[source,sh,role=execute]
----
oc get imagestream parasol-claims -n $NS \
  -o jsonpath='{range .status.tags[*]}{.tag}{"\n"}{end}'
----

.Expected output
[source,texinfo]
----
latest
sha256-<your-image-digest>.att
sha256-<your-image-digest>.sig
----

The `.sig` tag is the *signature*; the `.att` tag is the *attestation* — a signed *SLSA provenance* statement describing how the image was built.
They travel with the image: copy the image to another registry and the signature comes too.

[NOTE,role=instructor-demo]
.🧑‍🏫 Instructor demo — verify the signature and read the provenance with cosign
====
`cosign` is the tool that _checks_ what Chains _produced_.
It is not installed in the attendee terminal, so the instructor runs it (see the instructor guide for the one-line verification pod).
Against the in-namespace public key (`chains-cosign-pub`, copied in by the entry state), verification *passes*:

.cosign verify + verify-attestation output
[%collapsible]
=====
[source,texinfo]
----
$ cosign verify --key cosign.pub <image>@sha256:<digest>
Verification for ...parasol-claims@sha256:<digest> --
The following checks were performed on each of these signatures:
  - The cosign claims were validated
  - The signatures were verified against the specified public key

$ cosign verify-attestation --key cosign.pub --type slsaprovenance <image>@sha256:<digest>
  ... verified against the specified public key
  predicateType: https://slsa.dev/provenance/v0.2
  buildType:     tekton.dev/v1/TaskRun
  builder.id:    https://tekton.dev/chains/v2
----
=====

The attestation is real *SLSA provenance*: it names the builder (`tekton.dev/chains/v2`), the build type (a Tekton TaskRun), the source, and the start/finish timestamps — a machine-checkable answer to "prove this image came from your pipeline."
An unsigned image, or one signed with a different key, fails this check with a non-zero exit code — which is exactly what the admission control in exercise 5 relies on.
====

[NOTE]
.Keyless signing — the enterprise path
====
This lab signs with a *cosign key pair* (the private key lives in the cluster; the public key verifies).
The enterprise alternative is *keyless* signing: *Red Hat Trusted Artifact Signer* (RHTAS {rhtas_version}, built on the sigstore project — Fulcio + Rekor) issues a *short-lived certificate* tied to the build's identity and records the signature in a public *transparency log*, so there is no long-lived key to steal or rotate.
RHTAS is installed on this cluster and the instructor can show the keyless flow; the key-based path you used here is the deterministic one for a lab.
====

== 5. Only signed images run: admission control
⏱ *~10 min* · the cluster refuses to pull an unsigned image into your namespace

Signing is worthless if nobody checks the signature.
The final control is *admission*: OpenShift's native `ImagePolicy` (a `config.openshift.io/v1` resource, shipped with OpenShift {ocp_image_policy_version}) tells the node to *verify a signature before it will pull an image* into a namespace.
An image that never went through your signing pipeline is *refused at pull time*.

Because an `ImagePolicy` changes cluster-node behavior (the node's container runtime re-reads its signature-verification config), it is applied by the instructor cluster-wide in this workshop — but the effect lands in *your* namespace, and you verify it.

[NOTE,role=instructor-demo]
.🧑‍🏫 Instructor demo — the cluster blocks an unsigned image
====
The instructor applies an `ImagePolicy` scoped to your `parasol-claims` image, pinned to the same cosign public key Chains signs with:

[source,yaml]
----
apiVersion: config.openshift.io/v1
kind: ImagePolicy
metadata:
  name: only-signed-claims
  namespace: <you>-cicd
spec:
  scopes:
    - image-registry.openshift-image-registry.svc:5000/<you>-cicd/parasol-claims
  policy:
    rootOfTrust:
      policyType: PublicKey              # <1>
      publicKey:
        keyData: <base64 cosign.pub>     # <2>
    signedIdentity:
      matchPolicy: MatchRepoDigestOrExact
----
<1> Key-based verification (the lab's path); the keyless alternative is `policyType: FulcioCAWithRekor`.
<2> The base64-encoded cosign public key — the same `chains-cosign-pub` your namespace already holds.

With the policy in place:

* Deploying your *signed* `parasol-claims` image *works* — the node verifies the signature against the key and pulls it.
* Deploying an *unsigned* image (say a stock `docker.io` image) into the same namespace is *refused* — the pod stays `ImagePullBackOff`, and its events read `SignatureValidationFailed: ... A signature was required, but no signature exists`.

That is the whole point of admission: it does not matter whether anyone _remembered_ to run the scan gate — the cluster will not pull an image it cannot verify.
====

Confirm the pieces that make this possible are already in *your* namespace — the public key the policy trusts, and a signed image for it to admit:

[source,sh,role=execute]
----
echo "trust key present:  $(oc get configmap chains-cosign-pub -n $NS -o name)"
echo "signed image tags:  $(oc get imagestream parasol-claims -n $NS -o jsonpath='{range .status.tags[*]}{.tag}{" "}{end}')"
----

.Expected output
[source,texinfo]
----
trust key present:  configmap/chains-cosign-pub
signed image tags:  latest sha256-....att sha256-....sig
----

The key that admission verifies against, and the signature it verifies — both are here.
`ImagePolicy` is the switch that turns "we sign our images" into "the cluster will not run one we did not sign."
