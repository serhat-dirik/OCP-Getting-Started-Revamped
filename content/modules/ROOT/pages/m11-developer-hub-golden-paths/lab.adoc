=== Break and fix: run the template again with the same name
The golden path publishes to a *real* Git server, and a Git server will not hold two repositories with the same name.
Prove it: go back to *Create ‚Üí New Parasol microservice*, fill in the *same* name `parasol-policy-{user}` and the *same* org `{user}-svcs`, and *Create*.

This time the task *fails at the Publish step*.
Open the failed step's log and read it:

[source,texinfo]
----
Publish to Gitea
  Error: Unable to create repository, 409 Conflict,
  {"message":"The repository with the same name already exists."}
----

// media-pass: screenshot m11-developer-hub-golden-paths-06-scaffold-409.png ‚Äî the scaffolder task with
// the Publish step failed (red) and the 409 "repository with the same name already exists" error in the
// step log. Embed here ‚Äî the deliberate break. See media-manifest.md.

That is the golden path being honest: it *fetched* the skeleton fine, but *publishing* into a repository that already exists is refused ‚Äî no silent overwrite of your existing service.
The *fix* is what you would do in real life: give the new service a *unique* name (`parasol-quote-{user}`, `parasol-billing-{user}`, ‚Ä¶), or delete the old repo first if you truly meant to replace it.
(The workshop clears your `{user}-svcs` slate on every `ws prep m11` ‚Äî emptying the org and deregistering its catalog entry ‚Äî so a fresh run always starts empty.)

*The image is trusted and non-root.*

The `USER root` belongs to the *build* stage ‚Äî it compiles the app, then is discarded; only the final `USER 185` runtime stage is shipped and run, so the image on the cluster runs non-root.

== 4. Deploy your scaffolded service into {user}-dev
‚è± *~10 min* ¬∑ collect the build and run the new service

The build you started in exercise 3 has been running on the cluster while you toured the repo.
Collect it ‚Äî poll until it reports `Complete` (the first build on a fresh node pulls the UBI9 Java 21 builder image and runs a full Maven build, so on the build cluster this took about *8‚Äì9 minutes* end to end):

[source,sh,role=execute]
----
for i in $(seq 1 90); do
  B=$(oc get build -n $DEV -l buildconfig=$SVC --sort-by=.metadata.creationTimestamp -o jsonpath='{.items[-1:].metadata.name}' 2>/dev/null)
  P=$(oc get build "$B" -n $DEV -o jsonpath='{.status.phase}' 2>/dev/null)
  echo "  $(date +%T)  ${B:-build}: ${P:-Pending}"
  case "$P" in
    Complete) echo "  ‚úî image built and pushed to the internal registry"; break;;
    Failed|Error|Cancelled) echo "  ‚úñ build $P ‚Äî see: oc logs build/$B -n $DEV"; break;;
  esac
  sleep 12
done
----

  ‚úî image built and pushed to the internal registry

With the image in your namespace's registry, deploy and expose it.
`oc new-app` creates a `Deployment` + `Service` from the image tag; `oc create route edge` publishes it over HTTPS as a `Route`:

[source,sh,role=execute]
----
oc new-app ${SVC}:latest -n $DEV
oc create route edge ${SVC} --service=${SVC} -n $DEV --port=8080 --insecure-policy=Allow
oc rollout status deploy/${SVC} -n $DEV --timeout=120s
----

====

Now prove the scaffolded service is live ‚Äî and that the paved road's observability is real, not just wired.
Call the starter endpoint, the readiness probe, and the metrics endpoint:

[source,sh,role=execute]
----
ROUTE=$(oc get route $SVC -n $DEV -o jsonpath='{.spec.host}')
curl -s "http://$ROUTE/api/info" | jq -Sc .
curl -s -o /dev/null -w "readiness /q/health/ready : %{http_code}\n" "http://$ROUTE/q/health/ready"
curl -s -o /dev/null -w "metrics   /q/metrics      : %{http_code}\n" "http://$ROUTE/q/metrics"
----

.Expected output
[source,texinfo]
----
{"service":"parasol-policy-user1","status":"ready"}
readiness /q/health/ready : 200
metrics   /q/metrics      : 200
----

[NOTE]
.A `503` for a few seconds right after creating the route is normal
====
Immediately after you create the Route, it needs a moment to program and the pod a moment to finish starting.
If `/api/info` returns `503` on the first try, wait a few seconds and retry ‚Äî it becomes `200` once the pod is ready.
====

Stop and notice what you did *not* do.
You never wrote a Dockerfile, never chose a base image, never added a health endpoint or a metrics dependency, never set up tracing.
You filled in a form, and minutes later a trusted-image, health-checked, metrics-emitting Parasol service is running in your namespace ‚Äî the golden path carried every one of those decisions for you.

== Challenge (optional)
‚è± *no hand-holding*

You scaffolded, inspected, and ran a golden-path service.
Try these on your own:

* *Wire the platform probe to the app's health endpoint.*
`oc new-app` did not add a Kubernetes readiness probe ‚Äî but the app already serves one.
Connect them: `oc set probe deploy/$SVC --readiness --get-url=http://:8080/q/health/ready --initial-delay-seconds=10 -n $DEV`, then confirm the pod is gated on real readiness (`oc get pod -n $DEV -l deployment=$SVC`).
Then *break it* ‚Äî point the probe at `/q/health/WRONG`, watch the pod go `NotReady`, read the event, and fix it.
* *Make your first change on the paved road.*
Edit `InfoResource.java` to add a field to the JSON (say `"team": "claims"`), commit, and rebuild with another `oc start-build $SVC --from-dir=. -n $DEV` ‚Äî the image redeploys and `/api/info` shows your change.
* *Open the service in Dev Spaces.*
The scaffold shipped a `devfile.yaml`. From your Dev Spaces module knowledge, open the repo in a Java 21 workspace and run `./mvnw quarkus:dev` (a separate module covers Dev Spaces in depth).
* *Scaffold a second service under a new name.*
Run the template again with a *unique* name and watch it register beside the first ‚Äî reason about what the platform team would change in *one* place to give every future service a new standard (hint: the template's skeleton).

When you are done, run `ws verify {module-id}` to confirm your scaffold outcome, or `ws prep {module-id} --yes` to empty your `{user}-svcs` org, deregister its catalog entry, and purge `{user}-dev` for a clean start.

[NOTE]
.Clean up when you are done
====
`ws prep {module-id} --yes` returns everything to a clean entry state: it empties your `{user}-svcs` scaffold org and deregisters its catalog entry (so the template can publish and register fresh), and purges `{user}-dev` (removing the service you built and ran).
Your seeded Parasol forks in your personal Git namespace are never touched.
