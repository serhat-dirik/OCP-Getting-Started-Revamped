== 1. Import the claims service from Git (Source-to-Image)
⏱ *~15 min* · the paved road: source in, running image out

You have no container image for `parasol-claims` — only source.
Source-to-Image (S2I) takes your Git repository plus a language *builder image* and produces a runnable image for you, no Dockerfile required.
The workshop provides a *Java 21* builder (`java-21`) in the shared catalog; because the claims service is a Java 21 application, that builder is the right match.

[tabs]
====
Console::
+
--
. Go to *+Add* → *Import from Git*.
. In *Git Repo URL*, paste your fork:
+
[.copy%interactive]
`{gitea_url}/{user}/parasol-claims`
. The console inspects the repo and selects a *Builder Image*.
Confirm it is *Java 21* — if a different Java builder is auto-selected, open the *Builder Image* dropdown and choose *Java 21* explicitly.
. Leave the *Name* as `parasol-claims`, *Target port* `8080`, and *Resource type* *Deployment*.
. Click *Create*.
You land in *Topology*; the `parasol-claims` node shows a build starting.

Grounded from CLI: `oc new-app java-21~<git>` selects openshift/java-21:latest (description "Build and run Java 21 applications ... on UBI 9"); BuildConfig strategy=Source; build pushes parasol-claims:latest.

[IMPORTANT]
.Your app builds, then can't fully start — that is expected
====
Once the build finishes, the Deployment runs your image — and the Pod goes into `CrashLoopBackOff`.
That is *not* a build failure; the build succeeded.
The claims service needs a *PostgreSQL database* to start (it creates its schema and loads seed data at boot), and there is no database yet.
Confirm what it is complaining about:

[source,sh,role=execute]
----
oc logs deployment/parasol-claims --tail=15 | grep -iE 'datasource|connection|refused' | head -3
----

Leave it crash-looping — you will give it a database in exercise 4 and wire it up in exercise 5.
This is a real dependency, and fixing it is the payoff at the end of the lab.
====

== 2. The same app, the Dockerfile way — and why you'd choose it
⏱ *~10 min* · full control, at the cost of owning a Containerfile

S2I built the image *for* you from a builder image you did not write.
Sometimes you need control S2I does not give — a specific base image, a system package, an exact build.
Your `parasol-claims` repo also ships a `Containerfile` (the vendor-neutral name for a Dockerfile), so you can build the *same source* with the *Docker* strategy and compare.

[NOTE]
.`Containerfile`, not `Dockerfile`
====
The Docker build strategy looks for a file named `Dockerfile` by default.
Parasol's repo uses the vendor-neutral name `Containerfile`, so you must point the build at it explicitly.
This is a real, common gotcha — `oc new-app --strategy=docker` will refuse the repo ("No Dockerfile was found") until you tell it the path.
====

[tabs]
====
Console::
+
--
. Go to *+Add* → *Import from Git* again.
. Paste the same URL: `{gitea_url}/{user}/parasol-claims`.
. Under *Import Strategy*, choose *Dockerfile*, and set the *Dockerfile path* to `Containerfile`.
. Set the *Name* to `claims-docker` (so it does not collide with your S2I app), and click *Create*.

// [CAPTURE-VERIFY] Confirm on 4.21: that Import-from-Git offers a "Dockerfile" import strategy with an editable Dockerfile-path field, and that setting it to "Containerfile" is accepted. Grounded from CLI: a Docker-strategy BuildConfig with dockerStrategy.dockerfilePath=Containerfile builds the repo's multi-stage Containerfile to completion (~95s) and pushes claims-docker:latest.
--
CLI::
+
--
`oc new-app --strategy=docker` inspects the repo first and refuses it, because the file is named `Containerfile`:

[source,sh,role=execute,subs="attributes"]
----
oc new-app --strategy=docker {gitea_url}/{user}/parasol-claims.git --name=claims-docker
----

.Expected output
[source,texinfo]
----
error: No Dockerfile was found in the repository and the requested build strategy is 'docker'
----

Create the build the explicit way — a BuildConfig that names the `Containerfile`.
This applies an ImageStream to hold the result and a Docker-strategy BuildConfig whose
`dockerfilePath` points at `Containerfile`:

[source,sh,role=execute,subs="attributes"]
----
oc apply -f - <<'EOF'
apiVersion: image.openshift.io/v1
kind: ImageStream
metadata:
  name: claims-docker
---
apiVersion: build.openshift.io/v1
kind: BuildConfig
metadata:
  name: claims-docker
spec:
  source:
    type: Git
    git:
      uri: "{gitea_url}/{user}/parasol-claims.git"
  strategy:
    type: Docker
    dockerStrategy:
      dockerfilePath: Containerfile
  output:
    to:
      kind: ImageStreamTag
      name: claims-docker:latest
EOF
----

Start the build and watch the multi-stage `Containerfile` run:

[source,sh,role=execute]
----
oc start-build claims-docker && oc logs -f bc/claims-docker
----
--
====

Now compare the two BuildConfigs side by side:

[source,sh,role=execute]
----
oc get bc parasol-claims claims-docker -o custom-columns=NAME:.metadata.name,STRATEGY:.spec.strategy.type
----

.Expected output
[source,texinfo]
----
NAME             STRATEGY
parasol-claims   Source
claims-docker    Docker
----

Same source, two strategies.
The *Source* (S2I) build used the platform's Java 21 builder — you conform to its runtime.
The *Docker* build ran the repo's `Containerfile`, which pins its *own* base image (`ubi9/openjdk-21`) — you control the runtime, and you own that file forever.

[TIP]
.Which did you just prove you'd rather maintain?
====
Both produced a working image.
S2I needed *nothing* from you but the source; the Docker path needed a hand-written, hand-patched `Containerfile`.
For most services, that is the whole argument for S2I — and the wrap-up returns to when the Docker path genuinely earns its keep.
====

You do not need `claims-docker` past this comparison — clean it up so it does not clutter your project:

[source,sh,role=execute]
----
oc delete bc,is claims-docker
----

[.checkpoint]
****
✔ *Verify:* `oc get bc` showed `parasol-claims` as `Source` and `claims-docker` as `Docker` before you deleted the latter.
You understand the trade-off: S2I = the platform owns the build; Dockerfile = you own the base image and the file.
****

== 3. Polyglot: import the notifications service (Node.js)
⏱ *~10 min* · the same on-ramp, a different language

S2I is not just for Java.
The `parasol-notifications` service ships the *same* small API in two languages — Node.js and Python — in `node/` and `python/` subdirectories of one repo.
You import from a subdirectory with a *context directory*.
Node.js is the paved-road path here (it has zero dependencies, so nothing to download); the Python path is noted at the end if you want to try it.

[tabs]
====
Console::
+
--
. *+Add* → *Import from Git*, URL `{gitea_url}/{user}/parasol-notifications`.
. Expand *Show advanced Git options* and set *Context dir* to `/node`.
. Confirm the builder is *Node.js*, *Name* `parasol-notifications`, *Target port* `8080`.
. Click *Create*.

// [CAPTURE-VERIFY] Confirm on 4.21: the "Context dir" field under advanced Git options; that setting it to /node selects the Node.js builder and builds the node subdir. Grounded from CLI: `oc new-app nodejs:22-ubi9~<git> --context-dir=node` builds and pushes in ~45s (zero-dependency stdlib app, no npm registry needed).
--
CLI::
+

[source,sh,role=execute,subs="attributes"]

oc new-app nodejs:22-ubi9~{gitea_url}/{user}/parasol-notifications.git \

Watch it build — this one is quick, because the Node app uses only the standard library (no `npm install`):

`oc new-app python:3.12-ubi8~{gitea_url}/{user}/parasol-notifications.git --context-dir=python --name=notifications-py`.

== 5. Wire the claims service to its database
⏱ *~10 min* · the break-and-fix payoff — and a preview of config

The database is running; the claims service is still crash-looping because it does not yet know where the database is.
The claims image reads its datasource from environment variables (`POSTGRESQL_HOST`, `POSTGRESQL_DATABASE`, `POSTGRESQL_USER`, `POSTGRESQL_PASSWORD`) — and the `claims-db` Secret holds exactly those keys.
So wiring them is one command: pull every key from the Secret into the Deployment as environment.

[tabs]
====
Console::
+
--
. In *Topology*, click the `parasol-claims` node → *Actions* → *Edit Deployment* (or open the *Environment* tab on the Deployment).
. Add *All values from a ConfigMap or Secret* → select the `claims-db` Secret.
. Save.
The Deployment rolls out a new Pod with the database environment; it starts, connects, and goes Ready.

// [CAPTURE-VERIFY] Confirm on 4.21: the Deployment "Environment" tab "Add from ConfigMap/Secret" control and that selecting the claims-db secret injects its four keys. Grounded from CLI: `oc set env deploy/parasol-claims --from=secret/claims-db` injects POSTGRESQL_HOST/DATABASE/USER/PASSWORD; claims then reaches Ready and /q/health/ready returns 200.
--
CLI::
+
--
[source,sh,role=execute]
----
oc set env deployment/parasol-claims --from=secret/claims-db
----

.Expected output
[source,texinfo]
----
deployment.apps/parasol-claims updated
----

Watch the crash loop turn into a healthy Pod:

[source,sh,role=execute]
----
oc rollout status deployment/parasol-claims
----
--
====

Publish the claims API and prove it works end to end:

[source,sh,role=execute]
----
oc expose service/parasol-claims
----

[source,sh,role=execute,subs="attributes"]
----
curl -s http://parasol-claims-{user}-dev.{cluster_domain}/api/claims | head -c 200
----

.Expected output
[source,texinfo]
----
[{"claimNumber":"CLM-1001","claimant":"Alice Nguyen","type":"auto","status":"UnderReview","amount":4200.00,...}]
----

There it is: a service that arrived as *source code* is now a running, database-backed API you built yourself — and the crash loop is gone because you gave it what it declared it needed.

[NOTE]
.You just previewed two later modules
====
Injecting configuration from a Secret instead of baking it into the image is *config externalization* — the whole subject of the next module (*Config, Secrets & Multi-Environment* — a separate module).
And this database is *ephemeral*: restart its Pod and the data is gone.
Making it survive is the *Storage & Stateful Apps* module (a separate module).
You are standing exactly where those modules begin.
====
