== 1. Import the claims service from Git (Source-to-Image)
⏱ *~15 min* · the paved road: source in, running image out

You have no container image for `parasol-claims` — only source.
Source-to-Image (S2I) takes your Git repository plus a language *builder image* and produces a runnable image for you, no Dockerfile required.
The workshop provides a *Java 21* builder (`java-21`) in the shared catalog; because the claims service is a Java 21 application, that builder is the right match.

[tabs]
====
Console::
+
--
. Open the masthead *Quick create* (+) menu and choose *Import from Git*.
. In *Git Repo URL*, paste your fork:
+
[.copy%interactive]
`{gitea_url}/{user}/parasol-claims`
. The console tries to inspect the repo and reports that it *cannot reach it* — a yellow warning reading "the repository is unreachable" and "Unable to detect the import strategy".
That is expected here (see the note below); you choose the builder yourself.
. Under *Import Strategy*, select *Builder Image*, then pick the *Java 21* tile (leave the version dropdown at *latest*).
. Type the *Name* `parasol-claims`, and leave *Target port* `8080` and *Resource type* *Deployment*.
. Click *Create*.
You land in *Topology*; the `parasol-claims` node shows a build starting.

[NOTE]
.Why the console can't detect the builder here
=====
The console runs its repository inspection from the console backend, which on this workshop cluster cannot reach the in-cluster Gitea host — so it shows "repository is unreachable" / "Unable to detect the import strategy" even though the repo is public and the build clones it fine.
Because the console cannot introspect the source, *you* tell it which builder to use: *Import Strategy → Builder Image → Java 21*.
That is exactly what the `java-21~` prefix expresses in the CLI's `oc new-app java-21~<repo>` on the *CLI* tab — you are naming the builder by hand either way.
=====

[TIP]
.Why the first build is slow — and how real orgs speed it up
====
Almost all of that wait is *Maven downloading the dependency tree from the public internet* (`repo1.maven.org`), not compiling — a cold build with an empty cache fetches hundreds of jars.
Two techniques an organization uses to make this fast, neither of which changes your code:

* *An internal artifact repository (Maven mirror).*
A platform team runs a repository manager — Sonatype Nexus or JFrog Artifactory — inside the network and points builds at it with a `settings.xml` `<mirror>` (baked into the builder image or mounted from a ConfigMap).
Dependencies then come from the LAN cache at wire speed, and the build no longer depends on the public internet being up.
* *Incremental builds.*
Source-to-Image can *reuse* the previous build's artifacts (the `.m2` repository) instead of downloading them again — set `spec.strategy.sourceStrategy.incremental: true` on the BuildConfig, and the second build onward skips most of the download.

Slow builds read as "the platform is inefficient," but the real cause is dependency egress — and both fixes are platform-team decisions, not app changes.
====

Now look at *how* the platform recorded the build — a first-class object, not a shell script:

[source,sh,role=execute]
----
oc get buildconfig parasol-claims -o jsonpath='{.spec.strategy.type}{"\n"}'
----

[source,sh,role=execute]
----
oc get istag parasol-claims:latest -o jsonpath='pushed image: {.image.dockerImageReference}{"\n"}'
----

The BuildConfig's strategy is `Source` — that is S2I.
The ImageStream now has a `parasol-claims:latest` tag pointing at the image you just built, living in the cluster's internal registry.
The `pushed image:` line prints a *digest* (`…/parasol-claims@sha256:…`), not the `:latest` tag — that digest is the immutable content address S2I pushed, and it is exactly what `:latest` resolves to right now. Referencing an image by digest is what guarantees every replica runs the identical bytes you just built.

[IMPORTANT]
.Your app builds, then can't fully start — that is expected
====
Once the build finishes, the Deployment runs your image — and the Pod goes into `CrashLoopBackOff`.
That is *not* a build failure; the build succeeded.
The claims service needs a *PostgreSQL database* to start (it creates its schema and loads seed data at boot), and there is no database yet.
Confirm what it is complaining about:

[source,sh,role=execute]
----
oc logs deployment/parasol-claims --tail=40 | grep -iE 'UnknownHost|PSQLException|connection attempt|datasource' | head -3
----

Leave it crash-looping — you will give it a database in exercise 4 and wire it up in exercise 5.
This is a real dependency, and fixing it is the payoff at the end of the lab.
====

****
✔ *Verify:* `oc get buildconfig parasol-claims` exists with strategy `Source`, the build reached `Complete` (`oc get builds`), and `oc get istag parasol-claims:latest` shows a pushed image.
The `parasol-claims` Pod is *not* Ready yet (no database) — that is expected.
If the build fails to clone, see xref:m02-build-deliver/troubleshooting.adoc[the troubleshooting page].

== 2. The same app, the Dockerfile way — and why you'd choose it
⏱ *~10 min* · full control, at the cost of owning the build recipe

S2I built the image *for* you from a builder image you did not write.
Sometimes you need control S2I does not give — a specific base image, a system package, an exact build.
Your `parasol-claims` repo ships a `Dockerfile` — a multi-stage build recipe (a fat build stage, a lean runtime stage) — so you can build the *same source* with the *Docker* strategy and compare the two on-ramps.

[NOTE]
.`Dockerfile` or `Containerfile`?
====
The Docker build strategy looks for a file named `Dockerfile` by default, so the command below just works.
`Containerfile` is Red Hat's vendor-neutral name for the very same file, and this repo ships both (identical) so either tool finds a recipe.
If a repo has *only* a `Containerfile`, the Docker strategy will not auto-detect it — you point the build at it explicitly with a BuildConfig `dockerfilePath: Containerfile` (or the console's *Dockerfile path* field).
The xref:m02-build-deliver/troubleshooting.adoc[troubleshooting page] shows that fallback.
====

[tabs]
====
Console::
+
--
. Open the masthead *Quick create* (+) menu → *Import from Git* again.
. Paste the same URL: `{gitea_url}/{user}/parasol-claims`.
The same *unreachable-repository* warning appears — ignore it and choose the strategy yourself, exactly as in exercise 1.
. Under *Import Strategy*, choose *Dockerfile*.
The *Dockerfile path* field defaults to `Dockerfile` — the exact file in the repo, so leave it.
. Set the *Name* to `claims-docker` (so it does not collide with your S2I app), and click *Create*.
--
CLI::
+
--
Because the repo has a `Dockerfile`, `--strategy=docker` finds it and builds — nothing else to specify:

[source,sh,role=execute,subs="attributes"]
----
oc new-app --strategy=docker {gitea_url}/{user}/parasol-claims.git --name=claims-docker
----

Like the S2I import, this creates four objects — an ImageStream, a *Docker*-strategy BuildConfig, a Deployment, and a Service — and schedules the build.
Follow the multi-stage build (it compiles in the JDK 21 build stage, then copies the artifact into the lean runtime stage):

[source,sh,role=execute]
----
oc logs -f bc/claims-docker
----
--
====

Like your S2I app in exercise 1, the `claims-docker` Deployment will crash-loop until it has a database — expected, and it does not affect the build you came here to compare.

Now compare the two BuildConfigs side by side:

[source,sh,role=execute]
----
oc get bc parasol-claims claims-docker -o custom-columns=NAME:.metadata.name,STRATEGY:.spec.strategy.type
----

.Expected output
[source,texinfo]
----
NAME             STRATEGY
parasol-claims   Source
claims-docker    Docker
----

Same source, two strategies.
The *Source* (S2I) build used the platform's Java 21 builder — you conform to its runtime.
The *Docker* build ran the repo's own multi-stage recipe, which pins its *own* base images (`ubi9/openjdk-21` to build, `ubi9/openjdk-21-runtime` to run) — you control the runtime, and you own that file forever.

[TIP]
.Which did you just prove you'd rather maintain?
====
Both produced a working image.
S2I needed *nothing* from you but the source; the Docker path needed a hand-written, hand-patched build recipe.
For most services, that is the whole argument for S2I — and the wrap-up returns to when the Docker path genuinely earns its keep.
====

You do not need `claims-docker` past this comparison — clean it up so it does not clutter your project:

[source,sh,role=execute]
----
oc delete all -l app=claims-docker
----

[.checkpoint]
****
✔ *Verify:* `oc get bc` showed `parasol-claims` as `Source` and `claims-docker` as `Docker` before you deleted the latter.
You understand the trade-off: S2I = the platform owns the build; Dockerfile = you own the base image and the recipe.
****

== 5. Wire the claims service to its database
⏱ *~10 min* · the break-and-fix payoff — and a preview of config

The database is running; the claims service is still crash-looping because it does not yet know where the database is.
The claims image reads its datasource from environment variables (`POSTGRESQL_HOST`, `POSTGRESQL_DATABASE`, `POSTGRESQL_USER`, `POSTGRESQL_PASSWORD`) — and the `claims-db` Secret holds exactly those keys.
So wiring them is one command: pull every key from the Secret into the Deployment as environment.

[tabs]
====
Console::
+
--
. In *Topology* (under *Workloads*), click the `parasol-claims` node → *Actions* → *Edit Deployment* (or open the Deployment's *Environment* tab).
. In the *All values from existing ConfigMaps or Secrets (envFrom)* section, add the `claims-db` Secret — this pulls *every* key from it, the same as the CLI's `--from=secret/claims-db`.
. Save.
The Deployment rolls out a new Pod with the database environment; it starts, connects, and goes Ready.
--
CLI::
+
--
[source,sh,role=execute]
----
oc set env deployment/parasol-claims --from=secret/claims-db
----

.Expected output
[source,texinfo]
----
deployment.apps/parasol-claims updated
----

Watch the crash loop turn into a healthy Pod:

[source,sh,role=execute]
----

----
--
====

The rollout replaces the crash-looping Pod with one that carries the database environment.

There it is: a service that arrived as *source code* is now a running, database-backed API you built yourself — and the crash loop is gone because you gave it what it declared it needed.

Now open *Topology* and make it match the picture in your head.
By default it looks scattered: `claims-db` sits inside an application group named *parasol* (its catalog template stamped the `app.kubernetes.io/part-of=parasol` label), while `parasol-claims` and `parasol-notifications` — which `oc new-app` labelled only with their own names — float as separate, ungrouped nodes, and nothing draws the line from the claims service to the database it depends on.
Two labels fix that, and they are worth knowing because this is exactly how the console decides what to group and what to connect:

[source,sh,role=execute]
----
# group all three under one "parasol" application, and draw claims -> claims-db
oc label deploy/parasol-claims deploy/parasol-notifications svc/parasol-claims svc/parasol-notifications \
  app.kubernetes.io/part-of=parasol --overwrite
oc annotate deploy/parasol-claims app.openshift.io/connects-to='["claims-db"]' --overwrite
----

Refresh *Topology*: one *parasol* application now holds all three workloads, with an arrow from `parasol-claims` to `claims-db`.
The `part-of` label groups; the `connects-to` annotation connects — your topology finally matches your mental model.
