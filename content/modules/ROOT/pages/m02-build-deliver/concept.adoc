[.lead]
The claims service arrives as *source code*, not a finished image — and OpenShift gives you four different on-ramps to get it running.

Parasol's development team just handed you the `parasol-claims` service: a Quarkus REST API in a Git repository, no container image anywhere.
Your job is to get it building and running on the platform — and to understand *why* OpenShift offers several ways to do that, so you pick the right on-ramp for each app instead of forcing everything down one path.

== What you'll learn

* The *build spectrum* — deploy-from-image, import-from-Git (Source-to-Image), Dockerfile/Containerfile, and pipelines — and when each one is the right choice.
* Where your base images come from: *Universal Base Image (UBI)*, Red Hat *trusted content*, and why provenance matters before you ship to production.
* Which runtimes your OpenShift subscription *already includes*, so you build on a supported paved road instead of assembling one yourself.
* Why OpenShift runs your container as a *random non-root user*, and what that means for how an image is built.
* How the *developer catalog* is the platform's front door — templates, samples, and operator-backed services in one place.

== Four on-ramps: the build spectrum

You have source code.
The platform's job is to turn that into a running, reachable application.
There is no single "right" way to do that, because apps arrive in different shapes: sometimes you already have an image, sometimes you have source and want the platform to figure out the build, sometimes you need precise control over the container.
OpenShift gives you four on-ramps, from least to most control:

Deploy from image:: You *already have* a built container image (in a registry).
The platform just runs it — no build at all.
This is what you did for the claims *portal* in the first module.
Fastest path; you own the image and its supply chain elsewhere.
Import from Git (Source-to-Image, S2I):: You have *source code* and want the platform to build the image *for you*.
S2I takes your Git repo plus a language *builder image* (for example the workshop's Java 21 builder) and produces a runnable image — no Dockerfile required.
This is the paved road for "I just want my app built and running."
Dockerfile / Containerfile:: You have source *and* a `Dockerfile` (or the vendor-neutral `Containerfile`) describing exactly how to build the image.
The platform runs that build.
You get full control of the base image, build steps, and layering — at the cost of owning that file.
Pipelines (build as a workflow):: When a build is really a *workflow* — test, scan, build, sign, deploy — you graduate from a single BuildConfig to a pipeline.
That is a later module (*Pipelines & CI/CD* — a separate module); this module is the single-build foundation it stands on.

The honest summary, which the wrap-up returns to: *S2I is the fastest way to go from source to a running app, and the right default for most Java, Node.js, and Python services.*
Reach for a Dockerfile when you need something S2I cannot give you — a specific base image, a system package, an unusual build step.
Reach for a pipeline when the build is more than a build.

== Where your image comes from: provenance and trusted content

Every container starts from a *base image*.
Where that base image comes from is not a detail — it is the foundation of everything you ship, and the first question a security review will ask.

Red Hat's answer is the *Universal Base Image (UBI)*: freely redistributable, production-grade base images (full, minimal, micro, and language-specific flavors) that you can build on and run anywhere, with Red Hat's maintenance behind them.
The language builder images this module uses — the Java 21 builder, and the Node.js and Python builders — are UBI-based.
You can see the two Red Hat registries behave differently, which tells you something real about provenance:

* `registry.access.redhat.com` serves UBI *anonymously* — a pod on this cluster pulls a UBI image with no credentials (HTTP 200).
* `registry.redhat.io` serves the fuller catalog but is *authenticated* — the same pull without credentials is refused (HTTP 401); the cluster's global pull secret is what lets it through.

[NOTE]
.Why this matters later
====
"Which base images are we allowed to use, and can we prove where they came from?" is the whole subject of software supply chain security — signing images, generating a Bill of Materials, and scanning for vulnerabilities.
That is a dedicated later module (*Software Supply Chain Security* — a separate module).
Here, the takeaway is smaller and load-bearing: *start from trusted content*, and know the difference between an image you can prove the origin of and one you pulled off the internet.
====

The payoff of staying on Red Hat's runtimes is that your OpenShift subscription *already includes* the paved road: Quarkus (the Java framework `parasol-claims` is built with, currently {quarkus_version}), JBoss Web Server (Tomcat) for traditional Java web apps, and the UBI language images for Java, Node.js, and Python.
You are not assembling a runtime from scratch and maintaining it — you are building on one that is supported and patched for you.

[TIP]
.The workshop ships its own Java builder
====
This cluster carries a workshop-provided *Java 21* builder image in the shared catalog, on top of the ones OpenShift samples install by default.
That is a small taste of a bigger idea: platform teams *curate* which builder images and base images their developers use, so everyone starts from the same blessed, current runtime.
Who owns that list — and how they keep it current — is the subject of the *Platform-as-a-Product / golden paths* module (a separate module).
====

== Containers run as a random non-root user

In the first module you saw your Pod run as a high, random user ID like `1000920000` instead of root.
That is not cosmetic, and it shapes how you *build* images.

OpenShift's default security profile refuses to run a container as root.
Instead it assigns your namespace a range of high UIDs and runs your container as an arbitrary one from that range — a *different* UID than the image's author expected.
An image that only works as root, or as one specific hardcoded user, will fail here.

The fix is a property of the image, and `parasol-claims` already has it.
Its `Containerfile` ends with a *numeric* user and *group-writable* paths:

[source,dockerfile]
----
COPY --from=build --chown=185:0 /build/target/quarkus-app/ /deployments/
EXPOSE 8080
USER 185
----

`USER 185` is numeric (OpenShift can verify it is non-root without resolving a name), and everything the app needs is owned by *group 0* — so whatever random UID OpenShift injects, that UID is a member of group 0 and can still read and run the app.
Write images this way and they run unchanged as any UID.
The full security story — Security Context Constraints, the `restricted-v2` profile, and multi-tenancy — is a dedicated later module (*Multi-Tenancy & Workload Security* — a separate module).

== The developer catalog is the platform's front door

Everything above — builder images, templates, sample apps, and ready-to-run services backed by operators — is surfaced in one place: the *developer catalog*, reached from *+Add* in the console.
It is where a developer goes to answer "what can I deploy here, and how?" without filing a ticket.

In the lab you use the catalog twice: once implicitly, when import-from-Git picks the Java 21 builder, and once explicitly, when you deploy a *PostgreSQL* database for the claims service from a Parasol catalog *template*.
A template is a parameterized recipe — fill in a few values, get a set of objects — and the platform team decides which templates appear.

[NOTE]
.The catalog is curated, and that is the point
====
A stock cluster's catalog also lists things you should *not* reach for — end-of-life database versions, deprecated middleware, sample apps that ship insecure defaults.
Part of running a platform is *curating* the catalog so developers fall into the pit of success: the tiles they see are the ones they should use.
The database template you deploy in the lab exists precisely because the stock PostgreSQL templates ship an outdated engine on a deprecated workload type — so Parasol ships its own.
Catalog governance is a later module (*Platform-as-a-Product* — a separate module); notice here that *someone chose* what is in front of you.
====

== How this shows up in real orgs

* **"Who owns our base images?"**
On OpenShift the answer can be "the platform team, via the catalog" instead of "every team, copy-pasting `FROM` lines from the internet."
The build spectrum lets most teams stay on the paved road (S2I + curated builders) and only the teams that genuinely need a custom `Containerfile` maintain one.
* **The Dockerfile is a liability you take on deliberately.**
Every `Containerfile` your organization owns is a base image to patch, a build to maintain, and a security surface to review.
S2I exists so you write *zero* of them for the common case.
Count how many Dockerfiles your teams maintain today, and ask how many actually need to.
* **Provenance becomes a requirement, not a preference.**
The first time an auditor or a CVE asks "where did this image come from?", "we pulled it off Docker Hub" is not an answer.
Starting from UBI and trusted content is how teams answer that question before it is asked.
* **Non-root is not optional.**
Images that assume root run fine on a laptop and fail on a hardened platform.
The teams that adopt OpenShift smoothly are the ones whose images already tolerate an arbitrary UID — a small discipline that pays off across every environment.
