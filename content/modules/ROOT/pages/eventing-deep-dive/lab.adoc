== Demo arc — Filtered routing, a woken consumer, and a dead-letter catch (12 min)

[NOTE]
.Before you present
====
Run `ws solve eventing-deep-dive --user {user}` a few minutes ahead so the *end state* is materialized: the Broker, the seeded `PingSource`, the `claims-consumer` and `claims-dlq` services, and the filtered Triggers (`claims-fraud-review` on `type`, `claims-audit` on `claimpriority`) are all present.

Warm both services once (a `curl` to each URL) so the demo isn't waiting on a cold start mid-sentence; the *first* broker POST still shows the ~14 s wake if you let a service settle back to zero, which is a great beat if you want it.

[demo-block]
[TIME 4m]
Say:: "Parasol's claim events land in a *broker* — a hub. Nobody calls a consumer directly. Instead, *triggers* subscribe with a *filter*: this one wants `claim.submitted`, that one wants high-priority claims. Watch me drop one claim event into the broker and see it *routed*, and see the consumer *wake from zero* to display the full CloudEvents envelope."
Show:: The `claims-consumer` event-display page; a `curl` POST to the broker returning `202`; the consumer page showing a new `☁️ cloudevents.Event` with `type`, `source`, and the claim `Data`.
Do:: Send a claim event through the broker and show it displayed:
+
[source,sh,role=execute]
----
NS=user7-dev
BROKER=$(oc get broker default -n $NS -o jsonpath='{.status.address.url}')
oc exec deploy/demo-client -n $NS -- sh -c 'curl -s -k -m 30 -o /dev/null -w "claim -> broker: HTTP %{http_code} in %{time_total}s\n" \
  -X POST -H "Ce-Id: claim-9001" -H "Ce-Specversion: 1.0" -H "Ce-Type: com.parasol.claim.submitted" \
  -H "Ce-Source: /demo/claims" -H "Content-Type: application/json" \
  -d "{\"claimId\":\"CLM-9001\",\"amount\":7300,\"fraudScore\":0.91}" "'$BROKER'"'
----
+
"Two-oh-two — accepted. The trigger routed it, and the consumer woke from zero to *display* it — envelope and all. The producer never knew a consumer existed."

[demo-block]
[TIME 4m]
Say:: "Filters decide *who cares*. I'll send two events: a `claim.submitted` and an unrelated `claim.rejected`. The fraud-review trigger filters on `type=claim.submitted` — so the rejected one is simply *dropped* by that trigger. Routing lives in the trigger, not in anyone's code."
Show:: `kn trigger list` with the filters; two POSTs; the consumer displaying the `submitted` event but *not* the filtered-out one on the fraud path.
Do:: Show attribute filtering — matching delivered, non-matching dropped:
+
[source,sh,role=execute]
----
kn trigger list -n $NS
oc exec deploy/demo-client -n $NS -- sh -c 'curl -s -k -m 20 -o /dev/null -w "rejected (filtered out): HTTP %{http_code} in %{time_total}s\n" \
  -X POST -H "Ce-Id: rej-1" -H "Ce-Specversion: 1.0" -H "Ce-Type: com.parasol.claim.rejected" \
  -H "Ce-Source: /demo/claims" -H "Content-Type: application/json" -d "{\"claimId\":\"CLM-REJ\"}" "'$BROKER'"'
----
+
"The rejected event got a `202` too — the broker accepted it — but the fraud-review trigger's filter *dropped* it in a tenth of a second: no delivery, no woken consumer. Change routing by editing a trigger, never by redeploying an app."

[demo-block]
[TIME 4m]
Say:: "Now the hard part: what happens when a consumer *fails*? I'll point the fraud-review trigger at an endpoint the consumer *rejects*, so every delivery 404s. Watch Knative *retry with backoff*, then *dead-letter* the exhausted event to a separate sink — and watch me *open the dead event* and read exactly why it died."
Show:: A re-applied trigger pointing at `/process`; a POST that takes ~8 s (the retries); the `claims-dlq` page showing the dead event with `knativeerrorcode: 404` and `knativeerrordest`.
Do:: Break the consumer, watch the retries and the dead-letter, then read the dead event:
+
[source,sh,role=execute]
----
# repoint fraud-review's delivery at a path the consumer 404s, keeping retries + the dead-letter sink:
oc apply -n $NS -f - <<EOF
apiVersion: eventing.knative.dev/v1
kind: Trigger
metadata:
  name: claims-fraud-review
spec:
  broker: default
  filter: { attributes: { type: com.parasol.claim.submitted } }
  subscriber: { uri: http://claims-consumer.$NS.svc.cluster.local/process }
  delivery:
    retry: 3
    backoffPolicy: exponential
    backoffDelay: PT1S
    deadLetterSink: { ref: { apiVersion: serving.knative.dev/v1, kind: Service, name: claims-dlq } }
EOF
oc exec deploy/demo-client -n $NS -- sh -c 'curl -s -k -m 90 -o /dev/null -w "fails -> retries -> DLQ: HTTP %{http_code} in %{time_total}s\n" \
  -X POST -H "Ce-Id: dead-99" -H "Ce-Specversion: 1.0" -H "Ce-Type: com.parasol.claim.submitted" \
  -H "Ce-Source: /demo/claims" -H "Content-Type: application/json" -d "{\"claimId\":\"CLM-DEAD-99\"}" "'$BROKER'"'
oc logs deploy/claims-dlq-v1-deployment -c showcase -n $NS --tail=20 | grep -iE 'id:|knativeerror'
----
+
"Eight seconds — that's *one try plus three retries* with exponential backoff. The event failed every time, so instead of *dropping* it, Knative dead-lettered it to `claims-dlq` — stamped `knativeerrorcode: 404` and the destination that kept failing. No event silently lost, and a diagnosable record of why."

[NOTE]

Restore the consumer with one re-apply (subscriber back to `claims-consumer`), and the fraud path is healthy again — the dead-letter sink is *insurance*, not a normal path.
`ws prep eventing-deep-dive --yes` returns the world to the entry state (baseline Trigger only, no filtered/DLQ triggers).
If the room is engaged, `oc get pingsource claim-ticker -n $NS -o yaml` shows the *seeded source* firing every 5 minutes — a good segue to "the real move is to make the *app* emit these events, and repoint the triggers at them."
====

== 2. The consumer is a real CloudEvents processor
⏱ *~10 min* · send it an event directly, and watch it display the envelope

Before routing anything, meet the consumer. Unlike the Serverless eventing *taste* — where the app had no handler and every delivery returned `404` — `claims-consumer` is a *real* CloudEvents consumer (the OpenShift Serverless `showcase` / *event-display*): it returns `HTTP 200` on a plain `GET`, `204` on a CloudEvents `POST`, and *displays* every event it receives.

First, a plain `GET` — this *wakes it from zero*, so the first request pays a cold start (the Serverless lesson, still true). Drive traffic from the in-cluster `demo-client`:

[source,sh,role=execute]
----
CONSUMER=$(oc get ksvc claims-consumer -n $NS -o jsonpath='{.status.url}')
oc exec deploy/demo-client -n $NS -- curl -s -k -o /dev/null -w "GET cold: HTTP %{http_code} in %{time_total}s\n" "$CONSUMER"
oc exec deploy/demo-client -n $NS -- curl -s -k -o /dev/null -w "GET warm: HTTP %{http_code} in %{time_total}s\n" "$CONSUMER"
----

.Expected output (a real HTTP 200 — a cold start on the first request, then warm)
[source,texinfo]
----
GET cold: HTTP 200 in 14.280834s
GET warm: HTTP 200 in 0.039679s
----

That `200` is the first honest difference from Serverless: this consumer *serves*. Now send it a *CloudEvent* directly — a binary-mode `POST` with the `Ce-*` headers that carry the envelope — and watch the response code:

[source,sh,role=execute]
----
oc exec deploy/demo-client -n $NS -- sh -c 'curl -s -k -o /dev/null -w "POST CloudEvent: HTTP %{http_code} in %{time_total}s\n" \
  -X POST \
  -H "Ce-Id: direct-1" -H "Ce-Specversion: 1.0" \
  -H "Ce-Type: com.parasol.claim.submitted" -H "Ce-Source: /demo/direct" \
  -H "Content-Type: application/json" \
  -d "{\"claimId\":\"CLM-9001\",\"amount\":7300,\"fraudScore\":0.91}" "'"$CONSUMER"'"'
----

.Expected output (204 No Content — the CloudEvent was accepted and processed)
[source,texinfo]
----
POST CloudEvent: HTTP 204 in 0.406753s
----

`204` means "processed, nothing to return" — the consumer *received and handled* the event. Now *see* what it did with it. The consumer *displays* every event; read its log (its *event-display* page shows the same thing in the browser):

[source,sh,role=execute]

[source,texinfo]

Validation: valid

  datacontenttype: application/json

There is the *CloudEvents envelope*: the *context attributes* (`type`, `source`, `id`, `specversion`) that make the event routable, and your `Data` payload. `Validation: valid` means it is a well-formed CloudEvent. == 3. Route an event through the Broker — and wake a consumer from zero
⏱ *~12 min* · POST to the broker, watch it route to the consumer, which scales up on the event

In exercise 2 you called the consumer *directly*. That's not eventing — that's a direct call. Now do it *properly*: POST the event to the *Broker*, and let the `claims-events` *Trigger* route it to the consumer. The producer (you) touches only the broker; the trigger owns delivery.

Let the consumer settle back to *zero* first, so you can watch it *wake on the event*:

[source,sh,role=execute]
----
# wait for scale-to-zero (about a minute of no traffic), then confirm no consumer pod:
sleep 90
oc get pods -n $NS -l app=claims-consumer --no-headers | grep -c . || echo 0
----

.Expected output (the consumer is holding zero pods)
[source,texinfo]
----
0
----

Now POST a CloudEvent to the *broker ingress* — not the consumer. Because the in-memory broker delivers *synchronously*, the `202` comes back only after the trigger delivers the event — and delivery has to *wake the consumer from zero*, so this first POST takes about *fourteen seconds* (the Serverless cold start, now on the delivery path). Give `curl` room with `-m 30`:

[source,sh,role=execute]
----
oc exec deploy/demo-client -n $NS -- sh -c 'curl -s -k -m 30 -o /dev/null -w "claim -> broker (cold): HTTP %{http_code} in %{time_total}s\n" \
  -X POST -H "Ce-Id: broker-1" -H "Ce-Specversion: 1.0" \
  -H "Ce-Type: com.parasol.claim.submitted" -H "Ce-Source: /demo/broker" \
  -H "Content-Type: application/json" \
  -d "{\"claimId\":\"CLM-7777\",\"amount\":5100}" "'"$BROKER"'"'
# the consumer woke to receive it:
oc get pods -n $NS -l app=claims-consumer
----

.Expected output (202 Accepted — after the consumer cold-started to receive the delivery)
[source,texinfo]
----
claim -> broker (cold): HTTP 202 in 13.814156s
NAME                                             READY   STATUS    RESTARTS   AGE
claims-consumer-v1-deployment-...                2/2     Running   0          14s
----

The consumer *woke from zero on an event* — no client was waiting on it directly; the trigger's delivery scaled it up. Send a *second* event now that the consumer is warm, and watch the difference:

[source,sh,role=execute]
----
oc exec deploy/demo-client -n $NS -- sh -c 'curl -s -k -m 30 -o /dev/null -w "claim -> broker (warm): HTTP %{http_code} in %{time_total}s\n" \
  -X POST -H "Ce-Id: broker-2" -H "Ce-Specversion: 1.0" \
  -H "Ce-Type: com.parasol.claim.submitted" -H "Ce-Source: /demo/broker" \
  -H "Content-Type: application/json" \
  -d "{\"claimId\":\"CLM-7778\",\"amount\":9200}" "'"$BROKER"'"'
oc logs deploy/claims-consumer-v1-deployment -c showcase -n $NS --tail=40 | grep -E 'id: broker-'
----

.Expected output (warm delivery is sub-second; both broker-routed events reached the consumer)
[source,texinfo]
----
claim -> broker (warm): HTTP 202 in 0.639711s
  id: broker-1
  id: broker-2
----

[NOTE]
.Why the first broker POST is slow — the in-memory broker is *synchronous*
====
The default `MTChannelBasedBroker` is backed by an *in-memory channel*, and it delivers *synchronously*: your POST's `202` comes back only after the trigger *completes* delivery. When the consumer is scaled to zero, that means waiting for its cold start (~14 s here) — the same JVM cold start from Serverless, now paid on the delivery path. Warm, it's sub-second.
This synchronous, in-memory behavior is exactly why the *Kafka* broker exists (durable, buffered, asynchronous) for events you can't afford to lose — the broker-backing decision you'll meet in exercise 6. One practical consequence: never give the producer a `curl` timeout *shorter* than a possible cold start, or you'll cancel a delivery that was about to succeed.
====

== 4. Route by attribute — trigger filters
⏱ *~15 min* · add filtered triggers, and watch matching events delivered while non-matching are dropped

The baseline `claims-events` trigger is a *catch-all* — no filter, so it delivers *every* broker event to the consumer. Real routing is *selective*: fraud-review wants `claim.submitted` events; an audit path wants only *high-priority* claims. That selection is an *attribute filter* on a trigger — and each trigger filters *independently*.

Add a *fraud-review* trigger that filters on the CloudEvent `type`, and an *audit* trigger that filters on the `claimpriority` *extension attribute* the seeded source stamps:

[tabs]
====
Console::
+
--
In *Serverless → Eventing*, create two *Triggers* on the `default` broker, each with a *subscriber* of `claims-consumer`:

* `claims-fraud-review` — filter *attribute* `type` = `com.parasol.claim.submitted`
* `claims-audit` — filter *attribute* `claimpriority` = `high`

Or use *+Add → Import YAML* with the manifests from the *CLI* tab.
--
CLI::
+
--
[source,sh,role=execute]
----
# fraud-review: route claim-submitted events (filter on the CloudEvent type)
kn trigger create claims-fraud-review -n $NS --broker default \
  --filter type=com.parasol.claim.submitted --sink ksvc:claims-consumer
# audit: route only high-priority claims (filter on a business extension attribute)
kn trigger create claims-audit -n $NS --broker default \
  --filter claimpriority=high --sink ksvc:claims-consumer
----
--
====

.Expected output (two filtered triggers created, alongside the baseline catch-all)
[source,texinfo]
----
Trigger 'claims-fraud-review' successfully created in namespace 'user7-dev'.
Trigger 'claims-audit' successfully created in namespace 'user7-dev'.
----

Confirm all three triggers and their filters. The consumer displays an event *once per trigger that matches it*, so you can *count* the matches to see filtering at work:

[source,sh,role=execute]
----
oc get trigger -n $NS -o custom-columns='NAME:.metadata.name,FILTER:.spec.filter.attributes,SINK:.spec.subscriber.ref.name'
----

.Expected output (a catch-all plus two attribute-filtered triggers, all to claims-consumer)
[source,texinfo]
----
NAME                  FILTER                                  SINK
claims-audit          map[claimpriority:high]                 claims-consumer
claims-events         <none>                                  claims-consumer
claims-fraud-review   map[type:com.parasol.claim.submitted]   claims-consumer
----

Now prove the filters. Send *three* events and count how many triggers each one matches (the consumer displays it once per match):

* a plain `claim.submitted` — matches the catch-all *and* fraud-review → *2*
* a `claim.rejected` — matches *only* the catch-all; both filters drop it → *1*
* a high-priority `claim.submitted` — matches all three → *3*

[source,sh,role=execute]
----
for kind in "cnt-submitted:com.parasol.claim.submitted:" \
            "cnt-rejected:com.parasol.claim.rejected:" \
            "cnt-highval:com.parasol.claim.submitted:high"; do
  id=${kind%%:*}; rest=${kind#*:}; typ=${rest%%:*}; prio=${rest##*:}
  ph=""; [ -n "$prio" ] && ph="-H ce-claimpriority:$prio"
  oc exec deploy/demo-client -n $NS -- sh -c "curl -s -k -m 30 -o /dev/null -w '$id: HTTP %{http_code}\n' \
    -X POST -H 'Ce-Id: $id' -H 'Ce-Specversion: 1.0' -H 'Ce-Type: $typ' $ph \
    -H 'Ce-Source: /demo' -H 'Content-Type: application/json' -d '{\"claimId\":\"$id\"}' '$BROKER'"
done
sleep 5
for id in cnt-submitted cnt-rejected cnt-highval; do
  echo "$id displayed: $(oc logs deploy/claims-consumer-v1-deployment -c showcase -n $NS --tail=200 | grep -c "id: $id")x"
done
----

.Expected output (the match count = how many triggers' filters the event satisfied)
[source,texinfo]
----
cnt-submitted: HTTP 202
cnt-rejected: HTTP 202
cnt-highval: HTTP 202
cnt-submitted displayed: 2x
cnt-rejected displayed: 1x
cnt-highval displayed: 3x

== 5. Survive a failing consumer — retries and a dead-letter sink
⏱ *~15 min* · add delivery guarantees, break the consumer, and catch the dead events

Routing works — but what happens when a consumer *fails*? By default, a failed delivery is *retried a few times and then dropped*. For events you can't afford to lose, a trigger's `delivery` spec adds *retries with backoff* and a *dead-letter sink* (DLQ): the events that still fail land in a separate sink, *stamped with why*.

Give the `claims-fraud-review` trigger a delivery spec — retry three times with exponential backoff, and dead-letter to `claims-dlq`. Delivery guarantees aren't a `kn trigger` flag, so apply the trigger as YAML:

[tabs]
====
Console::
+
--
In *Serverless → Eventing → Triggers*, edit `claims-fraud-review` (or *+Add → Import YAML*) and set its `spec.delivery`: `retry: 3`, `backoffPolicy: exponential`, `backoffDelay: PT1S`, and a `deadLetterSink` referencing the `claims-dlq` service — the manifest is in the *CLI* tab.
--
CLI::
+
--
[source,sh,role=execute]
----
oc apply -n $NS -f - <<EOF
apiVersion: eventing.knative.dev/v1
kind: Trigger
metadata:
  name: claims-fraud-review
spec:
  broker: default
  filter:
    attributes:
      type: com.parasol.claim.submitted
  subscriber:
    uri: http://claims-consumer.$NS.svc.cluster.local     # the working consumer
  delivery:
    retry: 3                       # try 3 more times before giving up
    backoffPolicy: exponential     # 1s, 2s, 4s between tries
    backoffDelay: PT1S
    deadLetterSink:                # where exhausted events go instead of being dropped
      ref:
        apiVersion: serving.knative.dev/v1
        kind: Service
        name: claims-dlq
EOF
----
The `$NS` shell variable you set at the start expands to your `{user}-dev`, so the subscriber URL is correct for your namespace.

[NOTE]
.The dead-letter sink is insurance, not a normal path
====
A healthy trigger *never* uses its dead-letter sink — events flow straight to the subscriber. The DLQ only catches events that *exhaust their retries*, which means a dead-letter sink filling up is a *signal*: a consumer is failing. In production you'd *alert* on dead-letter arrivals and *drain* the sink (re-process the events once the consumer is healthy). The retry count and backoff are tunable per trigger; three retries with exponential backoff is a sane default, not a law.
====

Now the decisions, which you take back to your own systems:

*Broker vs Channel.* You used a *Broker + Triggers* — the right tool for *distributing events to many consumers with filtering*. Knative's other primitive, a *Channel + Subscriptions*, is a *linear pipe* with no per-subscriber filter — reach for it for an *ordered processing pipeline*, not fan-out. A broker is built *on* a channel, so this is a choice of the abstraction you program against.

*In-memory vs Kafka `[ADD-ON]`.* Your broker is *in-memory*: free with Serverless, synchronous, and *not durable* (events are lost if the broker restarts) — perfect for dev and best-effort internal events. For events you *cannot lose*, the *Kafka-backed broker* (from *Streams for Apache Kafka* {streams_kafka_version}, a *separate subscription*) adds durability, replay, and ordering.

[cols="1,2,2",options="header"]
|===
| | *In-memory broker* (this lab) | *Kafka broker* `[ADD-ON]`
| Durability | None — lost on restart | Persistent — survives restarts
| Replay | No | Yes
| Reach for it when | Dev, demos, best-effort events | Events you can't lose; replay/audit; high throughput
|===

[NOTE]
.`[ADD-ON]` — the Kafka path (only if you own a Streams subscription)
====
Swapping to a Kafka broker is a broker-class change, not an app change: create a `KnativeKafka` custom resource (enabling the Kafka broker), then create a broker with the Kafka broker class instead of the in-memory default. Your *triggers, filters, and consumers are unchanged* — that's the point of the broker abstraction. The durable, replayable Kafka path adds a topic per broker, a `Topic` custom resource for direct produce/consume, and lag-based autoscaling (KEDA) for consumers. It requires the Streams subscription, so it's entirely optional and never on the default path.
====

*Choreography vs Orchestration — workflows with SonataFlow `[ADD-ON]`.* Everything you built is *choreography*: each consumer reacts to events on its own, with no conductor. That's beautifully decoupled and *hard to reason about* once a business process has branches, retries across steps, and timeouts. *Orchestration* adds a conductor: *OpenShift Serverless Logic* {serverless_logic_version} (a *SonataFlow* runtime on Knative) runs a declarative *workflow* — a Parasol claims-approval flow would auto-approve below a threshold, route the rest to review, and take a *timeout* branch if review is slow — with a queryable instance per claim.

[NOTE]
.Why SonataFlow is a concept here, not a graded step
====
OpenShift Serverless Logic is *generally available* (since OpenShift Serverless 1.33) with a console, a `kn-workflow` CLI, and a VS Code extension. It's presented here as a *concept and `[ADD-ON]`* for one honest reason: a per-user SonataFlow platform (the workflow runtime *plus its own state database*) is significant footprint for every seat, and the graded core of this module is the *eventing* substrate. The decision is what matters and travels with you: *choreography* (triggers) for decoupled fan-out; *orchestration* (SonataFlow) when a *process* needs a conductor that can branch, call services, and time out. The wrap-up's decision guide places it against Pipelines (Pipelines Fundamentals) and Jobs (Jobs, Batch & Queued Workloads).
====
