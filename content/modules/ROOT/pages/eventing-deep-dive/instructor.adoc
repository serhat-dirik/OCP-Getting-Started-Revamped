This module has one spine and it builds: *inspect a working event path → meet a real CloudEvents consumer → route your own event through the broker (waking a consumer from zero) → filter by attribute → survive a failing consumer with retries + a dead-letter sink → decide the backings*.
The beat that lands hardest is the *dead-letter* (exercise 5): a *deliberately broken* consumer whose events *retry for ~8 seconds and then dead-letter*, and you *open the dead event* and read `knativeerrorcode: 404`. It makes "delivery guarantees" concrete instead of a slide.
The beat to frame most carefully is the *seeded source* (exercise 1): the event *source* is a `PingSource` ticker, not a real app emit, because `parasol-claims` doesn't emit CloudEvents yet. Everything downstream — broker, triggers, filtering, the consumer that *displays* events, retries, DLQ — is *real*. Do not let the seeded source read as "fake"; it's the Serverless-`404` honesty, one layer up.

The single most useful thing to internalize before delivering: *the in-memory broker delivers synchronously*. A POST's `202` returns only after delivery completes, so the *first* POST to a cold consumer takes ~14 s (the consumer waking) and a client with a *short* timeout will *cancel a delivery that was about to succeed*. Budget the timeouts, and don't let anyone "fix" the 14 s.

== Instructor demo — the 12-minute arc (demo flavor)

The demo flavor renders a three-beat Say/Show/Do arc: *route a claim through the broker + wake the consumer* (4 min), *attribute filtering — matching delivered, non-matching dropped* (4 min), *break the consumer → retries → dead-letter, and read the dead event* (4 min).

* *Run it from `ws solve eventing-deep-dive --user user4`.* Unlike Serverless, the demo starts from the *end* state — the filtered Triggers are already present, so you spend the time *showing behavior*, not building wiring. `ws solve` materializes `claims-fraud-review` (filter `type` + DLQ), `claims-audit` (filter `claimpriority`), and the baseline.
* *Open two browser tabs: the `claims-consumer` and `claims-dlq` event-display pages.* The *displayed CloudEvent envelope* is the whole visual — a terminal `202` is not the story; the consumer *showing the event* is.
* *Warm both services before you start* (a `curl` to each URL), so the demo isn't waiting on a cold start mid-sentence. If you *want* the ~14 s wake as a beat, let the consumer settle to zero first and narrate it ("watch it wake from zero on the event").
* *Land the dead-letter (beat 3).* Re-apply the trigger with the `/process` path, send a claim, let the ~8 s of retries run, then *open the `claims-dlq` page and read `knativeerrorcode: 404`*. That's the closer: "the event wasn't lost — here's why it died."

== Demo-mode notes

* *Don't cancel the failing POST.* Beat 3's POST takes ~8 s (retries) because the in-memory broker is *synchronous*. Use `-m 90` and let it finish — a short timeout *cancels the delivery* and the event never reaches the dead-letter sink (the "context canceled" trap).
* *The seeded source is honest — say so.* If asked "is this a real claim submit?", answer plainly: the *source* is a seeded `PingSource` because the app doesn't emit yet; the broker, triggers, filtering, processing, and dead-lettering are *all real*, and pointing this at real claim events is one app change away.
* *Restore between runs with one re-apply.* Re-applying the trigger with the subscriber back at `claims-consumer` heals the fraud path; `ws prep eventing-deep-dive --yes` returns to the entry state (baseline Trigger only).

== Top questions attendees ask

*1. "My first POST to the broker took ~14 seconds. Is the broker slow?"*::
No — that's the *consumer cold start*, on the delivery path. The default in-memory broker delivers *synchronously*: the `202` returns only after the trigger *completes* delivery, and if the consumer is scaled to zero it must cold-start first (~14 s for this JVM consumer, the Serverless cold start again). Warm, the same POST is sub-second. It's not the broker; it's the consumer waking. (A durable *Kafka* broker decouples the producer from this by buffering — exercise 6.)

*2. "The `claim.rejected` event returned `202` but never reached the consumer on the fraud path. Did it fail?"*::
No — `202` means the *broker accepted* it. Whether a *trigger* delivers it depends on that trigger's *filter*. `claims-fraud-review` filters `type=com.parasol.claim.submitted`, so a `rejected` event *doesn't match* and is *dropped* by that trigger — correctly. It still reaches the consumer via the *catch-all* `claims-events` (no filter), which is why the match-count is 1, not 0. A non-matching event is not an error; it's "not this trigger's concern."

*3. "Why author the retry/dead-letter Trigger as YAML instead of `kn`?"*::
Because `kn trigger create` supports `--filter` and `--sink` but *not* the `delivery` spec (retries, backoff, dead-letter sink). That's a `kn` limitation, not a lab quirk — the delivery guarantees live in `spec.delivery`, which you set via YAML (`oc apply`) or the console's Import YAML / Trigger editor. The filtering Triggers in exercise 4 *do* use `kn`; only the delivery one needs YAML.

*4. "The dead event has `knativeerrorcode: 404` — where did that come from?"*::
Knative *stamps* dead-lettered events with extension attributes recording the failure: `knativeerrorcode` (the last HTTP status the subscriber returned — `404`, because we pointed delivery at a path the consumer doesn't serve), `knativeerrordest` (the destination that kept failing), and `knativeerrordata` (any error body). That's the whole value of a dead-letter sink over dropping: the dead event is a *diagnosable record*. In this lab the `404` is deliberate (a broken path simulating a failing consumer); in production it might be a `503` from an overwhelmed service.

*5. "Is this real eventing, or a toy? The source is a ping."*::
The *source* is a seeded `PingSource` ticker on purpose — `parasol-claims` doesn't emit CloudEvents yet (net-new app work). *Everything else is real*: a real in-memory Broker, real attribute-filtered Triggers, a real CloudEvents consumer that validates and displays the envelope, real retries, and a real dead-letter sink with real error attributes. Attendees also send their *own* CloudEvents by hand, exactly as a producer would. Making the *app* emit `com.parasol.claim.submitted` and repointing the triggers at it is the one remaining step — and nothing else changes. It's the Serverless-`404` honesty: the plumbing and processing are real; the app-side emit is next.

== Watchouts

* *[FRAGILITY — flag to platform] Stale channel subscriber poisons all delivery.*
Hit at build time: after a prior `ws solve` then reset cycle on the sample user, the Broker's in-memory channel (`default-kne-trigger`) *retained a subscriber for a deleted Trigger* (`claims-audit`), and the `imc-dispatcher` returned `500` to the broker ingress for *every* event — so all producer POSTs *hung to timeout* and the scheduled `PingSource` deliveries all failed. The Trigger was gone (`oc get trigger` didn't list it) but `oc get imc default-kne-trigger -o jsonpath='{.spec.subscribers[*].name}'` still showed it. *Fix:* `ws reset eventing-deep-dive --user <user>` deletes and re-materializes the Broker, rebuilding a clean channel. *Pre-flight #3 (a real end-to-end event) catches this* — a "Ready" Broker with a poisoned channel passes the object checks but fails to deliver. Flagged to the platform team as a reset/entry-state hygiene gap (the channel's subscriber list can outlive the Trigger that owned it).
* *In-memory broker is synchronous — set the cold-start expectation up front.*
The first broker POST to a cold consumer takes ~14 s. Attendees who don't expect it read it as a hang. Say up front: "the in-memory broker delivers synchronously, so the first POST waits for the consumer to wake — that's the delivery-path cold start, and it's honest."
* *A short client timeout cancels the delivery.*
Because delivery is synchronous, a `curl -m` *shorter* than the delivery time (cold start, or the ~8 s retry sequence) *cancels the request*, and the broker logs "context canceled" — the event may *not* dead-letter. The lab uses `-m 30` for the cold path and `-m 90` for the retry/DLQ POST for exactly this reason. If an attendee's DLQ beat "doesn't produce a dead event," check their timeout first.
* *`404` is retriable here — the dead-letter is genuine.*
The break points delivery at a `404` path, and this Knative version *retries* a `404` (one try + three retries, ~8 s of backoff) before dead-lettering — so the beat shows real retries, not an instant drop. If a future version treats `404` as non-retriable, switch the broken subscriber to a `503`-returning or unreachable target to keep the retry beat.
* *Per-user eventing footprint.*
The seeded `PingSource` fires *every 5 min* per user and each broker POST can wake a scale-to-zero consumer. Light per user, but it multiplies across a cohort; the in-memory broker's control plane is shared. Watch a busy cluster during the filtering and DLQ beats when many attendees burst at once.
* *SonataFlow and Kafka are `[ADD-ON]`, not graded.*
Exercise 6 is *reading + one observation*. Don't let a keen attendee try to stand up a SonataFlow platform or a Kafka broker mid-session — both are separate footprints (SonataFlow needs its own state DB per user; Kafka needs a Streams subscription). The *decision* is the deliverable, not the deployment.
