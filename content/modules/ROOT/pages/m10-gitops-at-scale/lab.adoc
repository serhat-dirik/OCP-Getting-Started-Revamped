== 1. From many Applications to one ApplicationSet
⏱ *~15 min* · generate a fleet of Applications from one object

Your claims app is GitOps-managed in `dev` and `stage` today — as *two hand-made Applications*.
Confirm the starting point: dev and stage run the app, and *production is empty* (standing production up is part of this module).

[source,sh,role=execute]

== 2. Sync waves and hooks: ordered convergence
⏱ *~10 min* · watch production come up in the right order — database, then migration, then app

Production did not just "appear" — it converged in a deliberate *order*.
The `rollouts` folder your prod Application points at contains a *database migration* that must run *after* the database is accepting connections and *before* the app starts.
Argo CD enforces that with *sync waves* and a *resource hook*.
Look at what ran, in your own prod namespace.

First, the migration Job — a `Sync` *hook* that Argo ran as part of the sync:

[source,sh,role=execute]
----
oc get job claims-db-migrate -n $PROD
oc logs job/claims-db-migrate -n $PROD | tail -8
----

.Expected output
[source,texinfo]
----
NAME                STATUS     COMPLETIONS   DURATION   AGE
claims-db-migrate   Complete   1/1           12s        3m
▶ waiting for claims-db:5432 to accept connections…
claims-db:5432 - accepting connections
▶ applying idempotent schema migration (schema_version marker)…
CREATE TABLE
INSERT 0 1
✅ migration applied — schema_version:
 version |          applied_at
---------+-------------------------------
       1 | 2026-07-11 02:37:24.182936+00
(1 row)
----

The Job *waited* for the database, applied an idempotent schema change, and exited.
It is a *Job*, not a Deployment, because it is an *action that runs once* — not desired state that should keep existing.
Now see the annotations that *ordered* it — the database is the default wave `0`, the migration is a `Sync` hook at wave `1`, and the app Rollout is wave `2`:

[source,sh,role=execute]
----
oc get deploy claims-db -n $PROD \
  -o jsonpath='claims-db          wave: {.metadata.annotations.argocd\.argoproj\.io/sync-wave}{" (unset = default 0)"}{"\n"}'
oc get job claims-db-migrate -n $PROD \
  -o jsonpath='claims-db-migrate  wave: {.metadata.annotations.argocd\.argoproj\.io/sync-wave}  hook: {.metadata.annotations.argocd\.argoproj\.io/hook}{"\n"}'
oc get rollout parasol-claims -n $PROD \
  -o jsonpath='parasol-claims     wave: {.metadata.annotations.argocd\.argoproj\.io/sync-wave}{"\n"}'
----

.Expected output
[source,texinfo]
----
claims-db          wave:  (unset = default 0)
claims-db-migrate  wave: 1  hook: Sync
parasol-claims     wave: 2
----

That is the whole ordering, encoded in the repo next to the app: *wave 0* (database) → *wave 1* (migration hook) → *wave 2* (application).
Argo applied each wave only after the previous one was healthy, so the app was never created against a database whose schema was not ready.
It happens the same way on every fresh sync — and because the migration is idempotent, safely on every *re-*sync too.
Confirm production is actually serving, database-backed:

[source,sh,role=execute]
----
oc get rollout parasol-claims -n $PROD
ROUTE=$(oc get route parasol-claims -n $PROD -o jsonpath='{.spec.host}')
curl -s -o /dev/null -w "prod health: %{http_code}\n" "http://$ROUTE/q/health/ready"
curl -s "http://$ROUTE/api/claims" | head -c 90; echo
----

.Expected output
[source,texinfo]
----
NAME             DESIRED   CURRENT   UP-TO-DATE   AVAILABLE   AGE
parasol-claims   4         4         4            4           4m
prod health: 200
[{"claimNumber":"CLM-1001","claimant":"Alice Nguyen","type":"auto","status":"UnderReview",...
----

[.checkpoint]
****
✔ *Verify:* the `claims-db-migrate` Job is `Complete` and its log shows it waited for the database then applied the schema; the wave annotations read *db (0) → migrate (1, Sync hook) → app (2)*; and prod runs `parasol-claims` as a Rollout `4/4` with the route answering `200`.

== 3. A canary rollout with automated analysis
⏱ *~20 min* · ship a new version to production as a canary that has to prove itself

Production runs claims `1.0` today.
You are going to ship `1.1` — but *not* to all four pods at once.
Because prod is a *Rollout*, a new image is delivered as a *canary*: a small slice first, a pause, a wider slice, an *automated analysis* check, and only then 100%.
You trigger it the GitOps way — by editing Git.

Open your fork in the *Gitea* tab at {gitea_url}/{user}/claims-config[{user}/claims-config^]:

. Navigate to `rollouts/claims-rollout.yaml` and click the *pencil* (Edit) icon.
. Find the `image:` line ending in `parasol-claims:1.0` and change the tag to `parasol-claims:1.1`.
. *Commit Changes* directly to `main` (message e.g. "prod: ship claims 1.1").

// media-pass: screenshot m10-gitops-at-scale-03-gitea-image-bump.png — the Gitea editor on
// rollouts/claims-rollout.yaml with the image tag changed 1.0 -> 1.1, Commit Changes panel visible.
// See media-manifest.md.

Back in the *Argo CD* tab, open `claims-prod-{user}` and press *Refresh* so Argo re-reads `main` and syncs the new image into the Rollout.
The Rollout controller sees a new version and starts the canary.
Switch to the app's *Rollout* view to watch it visually, and poll it from the terminal at the same time:

== 4. Break and fix: the analysis that saves production
⏱ *~20 min* · ship a bad release on purpose, and watch the canary roll itself back

A canary is only worth the toil if it *stops* a bad release.
So ship one.
You cannot reliably break a real image inside a timed lab, so production carries a deterministic *analysis control knob* — a ConfigMap in your prod namespace whose value the analysis Job honors.
Set it to `fail`, and the next canary's analysis reports a failure *exactly as if the new build had spiked errors under real traffic*.

Set the knob (this is in *your* namespace, so plain `oc` works):

[source,sh,role=execute]
----
oc patch configmap m10-canary-control -n $PROD --type merge -p '{"data":{"verdict":"fail"}}'
oc get configmap m10-canary-control -n $PROD -o jsonpath='analysis verdict is now: {.data.verdict}{"\n"}'
----

.Expected output
[source,texinfo]
----
configmap/m10-canary-control patched
analysis verdict is now: fail
----

Now ship the next build.
In the *Gitea* tab, edit `rollouts/claims-rollout.yaml` again and change the image tag from `parasol-claims:1.1` back to `parasol-claims:1.0` — a stand-in for "the next release candidate" — and *Commit Changes* to `main` (message e.g. "prod: ship next build").
Back in *Argo CD*, *Refresh* `claims-prod-{user}` and open the *Rollout* view.
The canary starts again — but this time watch it *abort*:

Here is the point of the whole module.
The canary *aborted* and *rolled back to the stable version on its own* — no human noticed, decided, and acted under pressure.
And production *never went down*: the stable pods kept serving every request through the entire failed rollout.
Prove it — the route answered `200` the whole time, and prod is still on the *good* version, not the aborted one:

[source,sh,role=execute]
----
ROUTE=$(oc get route parasol-claims -n $PROD -o jsonpath='{.spec.host}')
curl -s -o /dev/null -w "prod health through the abort: %{http_code}\n" "http://$ROUTE/q/health/ready"
oc get rollout parasol-claims -n $PROD \
  -o jsonpath='stable ReplicaSet still serving: {.status.stableRS}  (the canary was scaled away){"\n"}'
----

.Expected output
[source,texinfo]
----
prod health through the abort: 200
stable ReplicaSet still serving: 5485b86bcd  (the canary was scaled away)
----

Now *fix it*, the GitOps way.
A real fix is two things: clear the fault, and *revert the bad change in Git*.
Clear the knob (the regression is "resolved"), then in the *Gitea* tab change the image tag back to `parasol-claims:1.1` and *Commit Changes* to `main` (message e.g. "Revert bad release; restore 1.1").
Because you are reverting to the version that is *already* the stable one, the Rollout clears the abort and returns to `Healthy` — no new canary needed, because production is already running that version.

[source,sh,role=execute]
----
oc patch configmap m10-canary-control -n $PROD --type merge -p '{"data":{"verdict":"pass"}}'
# after committing the image revert to 1.1 in Gitea and pressing Refresh in Argo CD:
for i in $(seq 1 12); do
  P=$(oc get rollout parasol-claims -n $PROD -o jsonpath='{.status.phase}')
  echo "  $(date +%T)  rollout: $P"
  [ "$P" = "Healthy" ] && { echo "  ✔ recovered — production Healthy on the good version"; break; }
  sleep 7
done
----

.Expected output
[source,texinfo]
----
configmap/m10-canary-control patched
  06:44:15  rollout: Degraded
  06:44:38  rollout: Healthy
  ✔ recovered — production Healthy on the good version
----

Production was never at risk.

== 5. Blue-green: the other strategy
⏱ *~5 min* · know the second shape of a safe rollout, and when to pick it

Canary is one progressive-delivery strategy; *blue-green* is the other, and the same Argo Rollouts controller serves it.
Where a canary shifts traffic *gradually* (25% → 50% → 100%) and gates on analysis mid-rollout, blue-green runs *two full versions side by side* — the live one (*blue*, behind an *active* service) and the new one (*green*, behind a *preview* service) — and cuts over *all at once* when you *promote*, with a *manual gate* in between.

[cols="1,2,2",options="header"]
|===
| | Canary (what you did) | Blue-green

| Traffic shift | Gradual, analysis-gated | Instant cutover on promotion
| Extra capacity | A small canary slice | A full second copy
| Reach for it when | You want to limit blast radius and can measure health mid-rollout | You want a clean instant switch after a full pre-flight on a preview URL
|===

You do not have to choose blindly: the decision guide is in the wrap-up, and the Challenge below lets you stand up a blue-green Rollout yourself if you want to see the preview-then-promote cut live.

[.checkpoint]
****
✔ *Verify (understanding):* you can state, in one sentence each, when you would reach for a *canary* versus a *blue-green* rollout — and why request-level traffic percentages here would need a *traffic router* (Route plugin or mesh) rather than the pod-ratio splitting the canary used.
****
