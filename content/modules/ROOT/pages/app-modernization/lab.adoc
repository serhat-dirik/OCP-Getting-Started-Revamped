You are a Parasol platform engineer handed the *original claims service* — `parasol-legacy-claims`, a Java 8 Spring-on-Tomcat *WAR* (Web Application aRchive) — with a mandate to get it onto OpenShift.
Your world in `{user}-modernize` is prepared, a fork of the legacy app is seeded in your Git account, and the shared *Migration Toolkit for Applications* (MTA) 8 Hub is running.
In this lab you *analyze* the app to measure what blocks containerization, *triage* the report, *fix* an issue with *Developer Lightspeed for MTA* assisting in the IDE (or by hand where that entitlement is absent), *re-analyze* to watch the effort score fall, and *deploy* the modernized service to OpenShift.

[TIP]
.Where the paths are single vs dual — and why
====
The *MTA web console* (adding the app, running analysis, reading the report) and the *Dev Spaces* IDE fix are *product user interfaces* — you drive them in the browser, so those steps are single-path with screenshots. Where a step names a specific button or field, a *grounded command-line alternative* is given beside it so you are never stuck if a label has shifted between product versions.
The *deploy* step at the end is genuine dual-path: a `Console::` tab (the OpenShift web console) and a `CLI::` tab (`oc`) — pick either.
The MTA console URL for this workshop is {mta_url}[the MTA console^] — it opens directly (this build has no sign-in).

== 1. Meet the migration target
⏱ *~6 min* · read the legacy code and find the anti-patterns MTA will flag

Before you analyze the app, look at what makes it *legacy*. Clone your fork into the terminal and read the three files that carry the worst offenders.

[source,sh,role=execute]
----
git clone $REPO /tmp/legacy 2>/dev/null || (cd /tmp/legacy && git pull)
sed -n '6,11p' /tmp/legacy/src/main/resources/persistence.properties
grep -n "AUDIT_LOG_PATH\|System.out" /tmp/legacy/src/main/java/com/parasol/claims/legacy/config/ApplicationConfiguration.java
grep -n "java.version\|ojdbc\|tomcat-servlet" /tmp/legacy/pom.xml
----

.Expected output (abridged — one representative line per file; config baked into the build, the exact things a container can't tolerate)
[source,texinfo]
----
jdbc.driverClassName=oracle.jdbc.driver.OracleDriver
jdbc.url=jdbc:oracle:thin:@10.14.221.9:1521/CLAIMSDB
jdbc.user=claims
jdbc.password=claims
hibernate.hbm2ddl.auto=create-drop
hibernate.dialect=org.hibernate.dialect.OracleDialect
19:    private static final String AUDIT_LOG_PATH = "/opt/parasol/logs/claims-audit.log";
...
19:    <java.version>1.8</java.version>
----

You are looking at the whole modernization story: a *hardcoded database IP* and *plaintext password*, a *hardcoded filesystem path* the container can't write, `create-drop` that would *wipe the schema on every start*, a *proprietary Oracle* driver, and a pin to *Java 8*. MTA is about to find every one of these — and *score* them.

[.checkpoint]
====
✔ *Verify:* You cloned your `parasol-legacy-claims` fork and saw the compiled-in datasource IP/credentials, the hardcoded `/opt/parasol/logs/...` audit path, and the `java.version` 1.8 pin — the anti-patterns the analysis will quantify next.
====

== 2. Add the app to MTA and run the analysis
⏱ *~12 min* · point MTA at your fork and analyze it against the containerization targets

Open {mta_url}[the MTA console^] — it opens directly (no sign-in on this build). This is a *product web console* — the steps below are the console click-path; a command-line alternative follows.

. *Create the application.* Go to *Migration → Application inventory*, choose *Create new application*, name it `parasol-legacy-claims`, and under *Source code* set the *Repository URL* to your fork (`{gitea_url}/{user}/parasol-legacy-claims`) and *Branch* `main`. Save.
// [CAPTURE-VERIFY] exact field labels on the MTA {mta_version} "Create application" drawer.
. *Run the analysis.* Select the application, click *Analyze*. Choose *Source code* mode, then on *Set targets* select the five tiles *Containerization*, *Linux*, *OpenJDK* (pick sub-version *OpenJDK 21* — the default 11 changes one issue and the total-effort number), *Jakarta EE 9*, and *JBoss Web Server 6* (together these resolve to the rule labels `cloud-readiness`, `linux`, `openjdk`, `jakarta-ee`, `jws`). Advance through the wizard and *Run*.
. *Wait for it to finish.* The analysis runs as a task in the Hub; even a small source app like this takes *a few minutes* (measured ~*4–5 minutes* end to end — pod scheduling + the analyzer image pull + the rule run). The *Status* becomes *Completed*.

[NOTE]
.The same analysis from the command line (`mta-cli`)
====
MTA also ships `mta-cli` (the upstream *kantra*), which runs the identical rule engine locally or in Dev Spaces — useful in a pipeline:

[source,sh]
----
# mta-cli analyze \
#   --input ./parasol-legacy-claims \
#   --target cloud-readiness --target openjdk --target jakarta-ee \
#   --output ./mta-report --source-only
----

MTA 8's *containerless* Java mode (default when a JDK and Maven are present) needs no analyzer image pull. The console and `mta-cli` produce the same *issues* and *effort*; this lab uses the console because the report UI is where triage happens.
====

====
✔ *Verify:* Your `parasol-legacy-claims` application shows *Completed* in the MTA console with a *report available*. Open the report — you should see a non-trivial *effort* total and a set of *mandatory* issues (details in the next step).

== 3. Read the report like a consultant
⏱ *~10 min* · separate the mandatory blockers from the noise, and group by fix

A raw report is a wall of findings. Your job is to turn it into a *short work list*. Open the application's report and its *Issues* tab, sorted by *Category* then *Effort*.

Here is what `parasol-legacy-claims` actually reports (performed live against MTA {mta_version}):

[cols="2,1,1,3",options="header"]
|===
| Issue | Category | Effort | Fix (and where)

| Version of Spring not compatible with Jakarta EE 9+ | mandatory | 3 | Upgrade Spring / move to an embedded runtime (`pom.xml`)
| Hardcoded IP address | mandatory | 1 | Externalize to an env var (`persistence.properties`)
| File system - Java IO | mandatory | 1 | Log to stdout, not a host path (`ApplicationConfiguration.java`)
| The `javax` package replaced by `jakarta` | mandatory | 1 | Namespace migration (`Claim.java`, initializer)
| `javax.activation` replaced by `jakarta.activation` | mandatory | 1 | Dependency swap (`pom.xml`)
| Tomcat artifact not compatible with JBoss Web Server | mandatory | 1 | Drop the provided servlet API (`pom.xml`)
| `java.io` constructor defaults to UTF-8 (OpenJDK 21) | potential | 1 | Confirm charset assumptions
|===

The report totals *24 story points* — the per-rule efforts above, multiplied by how many places each fires. It also lists *15 technology tags* (Spring MVC, Spring DI, Spring Web, Spring Data JPA, Java Servlet): MTA doesn't just flag problems, it *inventories* what the app uses, which is gold for planning.

The consultant's read:

* *Start with mandatory.* Seven of these are the whole job; the technology tags are context, not work.
* *Group by fix, not by finding.* Three of the mandatory issues (`javax` package, `javax.activation`, the Spring/Jakarta incompatibility) are *one migration* — the `javax`→`jakarta` move. The hardcoded IP and the filesystem write are the *externalize-config* theme. Grouped, "seven issues" becomes "three pieces of work."
* *Effort is relative.* The Spring/Jakarta issue (effort 3) is the heaviest; the config externalizations (effort 1) are the cheap, high-value wins you do first.

[.checkpoint]
====
✔ *Verify:* You can name the *mandatory* blockers, point to the *highest-effort* one (Spring/Jakarta EE 9, effort 3), and group the seven issues into ~three pieces of work — the config externalization, the `javax`→`jakarta` migration, and the runtime move.
====

== 4. Fix an issue with Developer Lightspeed for MTA `[ADS]`
⏱ *~12 min* · let the AI propose the config-externalization fix — then review it critically

You'll fix the *Hardcoded IP address* issue — the cheap, high-value win. *Red Hat Developer Lightspeed for MTA* — an AI feature of the *MTA extension for VS Code / Dev Spaces* — proposes a concrete diff for a selected issue.

[IMPORTANT]
.This section is `[ADS]` — and degrades gracefully
====
Developer Lightspeed for MTA is part of the *Red Hat Advanced Developer Suite* `[ADS]`. If your cluster has it, follow the *AI path*. If it doesn't, follow the *Manual path* — you fix the *same* issue by hand and the rest of the lab is identical. Check which applies: if the command below prints a model name, the AI path is wired for you.

[source,sh,role=execute]
----
oc get configmap maas-config -n $NS -o jsonpath='{.data.model}{"\n"}' 2>/dev/null \
  && oc get secret maas-credentials -n $NS >/dev/null 2>&1 \
  && echo "AI path: Developer Lightspeed is wired." \
  || echo "Manual path: fix by hand (graceful degradation)."
----
====

*AI path (Dev Spaces + MTA extension).* Open your fork in {devspaces_url}[Dev Spaces^]. The workspace preloads the *MTA extension* and the model wiring (the `{maas_model}` model on the workshop Models-as-a-Service endpoint, mounted as the `GENAI_API_KEY`/`GENAI_MODEL`/`GENAI_ENDPOINT` environment). In the MTA extension view, open the *Hardcoded IP address* issue and choose the Developer Lightspeed *fix* action.
// [CAPTURE-VERIFY] the exact MTA-extension command name / lightbulb action in Dev Spaces {devspaces_version}.

The model returns a diff. For this issue, the real proposal (captured live against `{maas_model}`) replaces the compiled-in URL in `PersistenceConfig.java` with a runtime lookup:

[source,java]
----
// before:  dataSource.setUrl(config.getProperty("jdbc.url"));   // resolves the hardcoded jdbc:oracle:thin:@10.14.221.9:...
// after (Developer Lightspeed proposal):
dataSource.setUrl(System.getenv("CLAIMS_DB_URL"));
----

*Read the diff before accepting it.* It is correct here — it moves the value out of the build and into the environment, exactly the containerization fix — but you only *know* that because you read it. Accept it, then provide the value the way a container should, as an environment variable, not a literal.

[NOTE]
.Prove the AI wiring yourself (the grounded API path)
====
The IDE action above is a product UI, so here is the *same call* the extension makes, which you can run from the terminal to prove the key + model + endpoint work end-to-end:

[source,sh,role=execute]
----
curl -sS -k "$(oc get cm maas-config -n $NS -o jsonpath='{.data.endpoint}')/chat/completions" \
  -H "Authorization: Bearer $(oc get secret maas-credentials -n $NS -o jsonpath='{.data.GENAI_API_KEY}' | base64 -d)" \
  -H 'Content-Type: application/json' \
  -d '{"model":"'"$(oc get cm maas-config -n $NS -o jsonpath='{.data.model}')"'","temperature":0,"max_tokens":120,
       "messages":[{"role":"user","content":"Rewrite this to read the JDBC URL from env var CLAIMS_DB_URL, return only the Java line: dataSource.setUrl(\"jdbc:oracle:thin:@10.14.221.9:1521/CLAIMSDB\");"}]}' \
  | python3 -c 'import sys,json
b=sys.stdin.read()
try: print(json.loads(b)["choices"][0]["message"]["content"])
except Exception: print("MaaS did not return a completion (a 401 — your key is not scoped to the model in maas-config — is the usual cause). Raw response:\n"+b[:400])'
----

Live, this returns `HTTP 200` and the `System.getenv("CLAIMS_DB_URL")` rewrite above. It never prints your key. This is *exactly* the wiring the module fixes: the MaaS key is scoped per cluster and the model (`{maas_model}` here) is sourced on-cluster, so the same content targets a cluster whose key is scoped to a different model without a code change.
====

*Manual path (graceful degradation).* No `[ADS]` entitlement? Make the *same* change by hand in `PersistenceConfig.java` — replace the compiled-in `getProperty("jdbc.url")` with `System.getenv("CLAIMS_DB_URL")`. The point of the lesson — *externalize the config* — is identical; only the typing assistant differs.

====
✔ *Verify:* The datasource URL is no longer a literal — it reads from the `CLAIMS_DB_URL` environment variable. If you used the AI path, you *read the proposed diff* before accepting it and can say *why* it is correct. (The API block above returns `HTTP 200` from the `{maas_model}` endpoint, proving the wiring.)

== 6. Containerize and deploy the modernized service
⏱ *~12 min* · ship the modernized claims service to OpenShift — and fix a deliberate break

The modernized service is a container image (the Quarkus rebuild of the claims service). Deploy it to `{user}-modernize`. This step is *dual-path* — use the *Console* or the *CLI*.

You'll deploy it *with a deliberately wrong health-probe path first*, watch OpenShift catch it, then fix it — because a modernized app that lies about its health is worse than an honest legacy one.

[tabs]
====
Console::
+
--
In the {ocp_console_url}[OpenShift web console^] (project `{user}-modernize`), choose *+Add → Container images*.
Set the image to the modernized claims image, name it `parasol-claims-modernized`, and under *Health checks* add a *Readiness probe* — but point it at the *wrong* path `/health` on port `8080` (the real Quarkus health endpoint is `/q/health/ready`). Deploy.
// [CAPTURE-VERIFY] the +Add → Container images → Health checks pane labels on OCP {ocp_version}.
Watch *Topology*: the pod never turns ready — the readiness probe 404s on `/health`.
--
CLI::
+
--
[source,sh,role=execute]
----
# deploy with a WRONG readiness path on purpose:
oc apply -n $NS -f - <<'YAML'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: parasol-claims-modernized
  labels: {app: parasol-claims-modernized}
spec:
  replicas: 1
  selector: {matchLabels: {app: parasol-claims-modernized}}
  template:
    metadata: {labels: {app: parasol-claims-modernized}}
    spec:
      containers:
        - name: parasol-claims
          image: image-registry.openshift-image-registry.svc:5000/ogsr-parasol-images/parasol-claims:1.1
          # Run the modernized app DB-inactive: this exercise deploys the *service*; a real migration would
          # wire its datasource to a database next. Without these it CrashLoops on a missing parasol-db.
          env:
            - {name: QUARKUS_DATASOURCE_ACTIVE, value: "false"}
            - {name: QUARKUS_HIBERNATE_ORM_ACTIVE, value: "false"}
          ports: [{name: http, containerPort: 8080}]
          readinessProbe:                 # WRONG path on purpose — /health 404s
            httpGet: {path: /health, port: 8080}
            initialDelaySeconds: 10
YAML
oc rollout status deploy/parasol-claims-modernized -n $NS --timeout=60s || echo "not ready — the probe is failing (expected)"
oc get pods -n $NS -l app=parasol-claims-modernized
----
--
====

The pod runs but never becomes *Ready* — the readiness probe hits `/health`, which returns `404`. Now *fix the break*: point the probe at the real Quarkus readiness endpoint, `/q/health/ready`.

[tabs]
====
Console::
+
--
In *Topology*, open the `parasol-claims-modernized` deployment, edit the *Readiness probe*, change the path from `/health` to `/q/health/ready`, and save. The pod turns *Ready*.
// [CAPTURE-VERIFY] the edit-probe control on this OCP build.
--
CLI::
+
--
[source,sh,role=execute]
----
oc patch deploy/parasol-claims-modernized -n $NS --type json \
  -p '[{"op":"replace","path":"/spec/template/spec/containers/0/readinessProbe/httpGet/path","value":"/q/health/ready"}]'
oc rollout status deploy/parasol-claims-modernized -n $NS --timeout=120s
oc expose deploy/parasol-claims-modernized -n $NS --port 8080 2>/dev/null || true
----
--
====

[NOTE]
.This ties into the golden path you already know
====
In a real pipeline you would build this image with *Source-to-Image* (S2I) and ship it through *Tekton* and *GitOps* — the delivery machinery from Ways to Build & Deliver Apps and Pipelines Fundamentals (separate modules). Modernization ends where the *golden path* begins: a modernized app is just an app, and it rides the same rails as every other.
====
