[NOTE]
.The per-cluster model is sourced on-cluster — a fixed defect
====
The M22 entry state *sources the model and endpoint from the platform Lightspeed secret* (`openshift-lightspeed/credentials` keys `model`/`endpoint`) at sync time, falling back to the chart default when those keys are absent. This is why a cluster whose MaaS key is scoped to a *different* model still gets the *right* model in `maas-config` and in the workspace's `GENAI_MODEL` — without a code change. If `maas-config` shows the wrong model, the platform Lightspeed secret is missing its `model` key; fix the secret, then `ws reset m22 --user <u>`.
====

*5. The attendee-facing repo URL clones over HTTPS.* MTA analyzes the HTTPS Gitea *route* (recorded in the entry marker), which clones cleanly with the Hub's secure default — no `git.insecure` change needed. If you instead drive the Hub API directly against the *internal* `http` Gitea service address, you will hit `http URL used with git.insecure.enabled = FALSE` — use the HTTPS route.

== Demo-mode notes (beyond Say/Show/Do)

* *Pre-run one analysis the day before.* A live analysis is ~90 s — fine in the lab, awkward in a demo. Have the `parasol-legacy-claims` report already on screen; run a *fresh* one only if you want to show the progress-to-complete transition.
* *Warm the Dev Spaces workspace.* First workspace start pulls the universal developer image and can take 2–3 minutes. Start it before the demo so the MTA extension view is already open.
* *Have the modern app's report open in a second tab* — the *24 → 1* contrast is the strongest single visual in the module.
* *Rehearse the AI beat once.* The model is deterministic at temperature 0, but rehearse so you can narrate the diff confidently and land the "read it before you accept it" point.

== Top 5 questions attendees ask

. *"Is this the AI that auto-migrates my app?"* +
No. Developer Lightspeed for MTA proposes a *per-issue diff* that you review and accept. It compresses the typing, not the judgement. The Hub also has a separate agentic component (the Kai solution server) that is *not* configured in this workshop — the AI you use here is the *VS Code / Dev Spaces extension* feature.

. *"Why not just wrap the WAR in a container and ship it?"* +
Because the config is compiled in (a hardcoded DB IP and password), the app writes to a host filesystem path a container can't provide, and it expects a servlet container that isn't there. MTA finds each of these *before* they become a 2 a.m. incident. Lift-and-shift (rehost) buys time but pays no modernization debt down.

. *"What do the effort story points actually mean?"* +
A *relative* estimate of the work to fix an issue, summed across the app. Use them to *rank* apps and *track a burn-down* (the number drops as you fix), not to promise hours to management. `parasol-legacy-claims` scores 24; the modern Quarkus equivalent scores 1.

. *"Do all thirty attendees hammer the same Hub?"* +
Yes — the MTA Hub is *shared*, with a per-user Application in the portfolio. Each analysis runs as its own task pod; a small source app is ~90 s, so staggered starts are fine. There is no per-user MTA instance (that would be heavy); portfolio isolation is by Application, not by RBAC.

. *"What if my cluster doesn't have the Advanced Developer Suite?"* +
The AI section is `[ADS]` and *degrades gracefully*: you make the same config-externalization fix by hand, and the analyze → fix → re-analyze loop is identical. The `[OCP]` MTA core (assess / analyze / replatform) is fully functional without `[ADS]`.

== Watchouts

* *LLM nondeterminism.* The demo issue (hardcoded IP → `System.getenv`) has a stable suggestion at temperature 0, but a different issue may yield a different phrasing. Stick to the rehearsed issue for the demo; in the lab, the *review-the-diff* discipline covers any variation.
* *Dev Spaces first-start latency.* The universal developer image pull dominates the AI section's clock. Pre-warm.
* *Shared-Hub portfolio hygiene.* Attendees creating an Application named `parasol-legacy-claims` all land in the same portfolio. That's fine (each is a distinct entry), but if you re-run a cohort, delete stale Applications in the console — portfolio state lives in the Hub DB, not in `{user}-modernize`, so `ws reset` does *not* remove it.
* *The Kai solution server shows "not ready."* On a stock install the Hub's `KaiSolutionServerReady` condition is `False` (no Kai secret). That is expected and *out of scope* — do not send attendees into the Hub's not-ready AI feature; the AI beat is the *extension* in Dev Spaces.
* *Analysis returns zero issues?* The target *labels* matter. In the console the friendly targets (*Containerization*, *OpenShift*, *OpenJDK*) map to rule labels `cloud-readiness`/`linux`/`openjdk`/`jakarta-ee`/`jws`; if a run reports nothing, the target selection didn't resolve to any ruleset — re-select the targets.
